Lesson 1 · ~45 min

The HTML Boilerplate

Every HTML file starts with the same skeleton. Understanding each line — not just copying it — is the foundation everything else rests on.

The boilerplate lines are not decoration — each one fixes a specific real-world problem (encoding bugs, mobile viewport scaling, legacy browser quirks). Leaving any out causes hard-to-debug issues that only appear in certain browsers or devices.

⚓ Core Insight

The HTML boilerplate is a set of instructions to the browser about how to parse and render your document — before any content is read.

<!DOCTYPE html>
Not an HTML element — it's a declaration that tells the browser to parse the document using the modern HTML5 standard. Without it, browsers fall into "quirks mode" and render the page using legacy rules from the 1990s.
<html lang="en">
The root element that wraps the entire document. The lang attribute is critical: screen readers use it to choose the correct pronunciation engine, and search engines use it for language targeting.
<head>
Contains metadata about the document — information for the browser and search engines, not visible content for the user. Everything in <head> is invisible on the rendered page.
<meta charset="UTF-8">
Sets the character encoding to UTF-8, which supports every language script on Earth. Without this, accented characters, emoji, and non-Latin scripts display as garbled symbols (mojibake).
<meta name="viewport" ...>
Tells mobile browsers not to zoom out and shrink the page. Without width=device-width, initial-scale=1, your page will look tiny on phones and CSS media queries won't work correctly.
<title>
Sets the text shown in the browser tab, in bookmarks, and in search engine results. It's the most important SEO element on the page after the content itself.
✅ Scenario

You're creating the first page of a new website. Write the complete, valid HTML5 boilerplate — nothing more, nothing less.

HTML
<!-- 1. DOCTYPE: enable HTML5 standards mode --> <!DOCTYPE html> <!-- 2. Root element: set the document language --> <html lang="en"> <head> <!-- 3. Character encoding: must be within first 1024 bytes --> <meta charset="UTF-8"> <!-- 4. Viewport: prevent mobile browsers from shrinking the page --> <meta name="viewport" content="width=device-width, initial-scale=1.0"> <!-- 5. Title: shown in the browser tab and search results --> <title>My First Page</title> </head> <body> <!-- All visible content goes here --> <p>Hello, world!</p> </body> </html>

Lesson 2 · ~45 min

Nesting, Parent–Child Relationships & Comments

HTML elements nest inside each other to create a tree. Understanding parent–child relationships and correct nesting is what separates valid markup from broken layouts.

Incorrect nesting is one of the most common causes of layout bugs. Browsers attempt to auto-correct invalid HTML — but the correction they apply is often not what you intended, producing mysterious visual glitches that are very hard to trace.

⚓ Core Insight

Every element you open must be closed before its parent closes. Tags that overlap (e.g. <b><i></b></i>) are invalid — the inner element must close first.

Parent & Child Elements
An element that contains another element is its parent; the contained element is the child. Elements at the same level inside the same parent are siblings. CSS selectors and JS traversal rely heavily on these relationships.
Block vs. Inline Elements
Block elements (e.g. <div>, <p>, <h1>) start on a new line and take up the full available width. Inline elements (e.g. <span>, <a>, <strong>) flow within text and only take as much width as their content.
HTML Comments
Comments are not displayed by the browser: <!-- This is a comment -->. Use them to document your markup, temporarily disable sections, or leave notes for collaborators. Comments are visible in the page source — never put passwords or secrets in them.
⚠️ Common Pitfall

Block elements (<p>, <div>) cannot be placed inside inline elements (<span>, <a>). The browser will silently restructure your tree in unexpected ways if you do.

✅ Scenario

Build a simple article page structure — a heading, two paragraphs, and one word in the second paragraph made bold. Show both correct and incorrect nesting.

HTML — Correct
<article> <h1>My Article Title</h1> <p>This is the first paragraph of the article.</p> <p> This is the second paragraph. This word is <strong>important</strong>. <!-- <strong> opens and closes INSIDE the <p> --> </p> </article>
HTML — Invalid Nesting ✗
<!-- Wrong: <strong> opened inside <p> but closed outside --> <p>This word is <strong>important</p></strong> <!-- ✗ invalid -->

Mental model
Think of DOCTYPE as a "mode switch" that tells the browser which rulebook to use.
The Answer
<!DOCTYPE html> is a declaration (not an element) that instructs the browser to render the document in standards mode — using current HTML5 rules.

Without it, browsers enter quirks mode — a backwards-compatibility mode that mimics the inconsistent rendering behaviour of 1990s browsers. This causes box model differences, font sizing bugs, and other layout discrepancies that are very hard to diagnose.
DOCTYPE enables standards mode — consistent, modern rendering.
Without it: quirks mode — buggy, inconsistent legacy rendering.
It must be the very first line of the file — nothing before it, not even whitespace.
Mental model
Head = backstage instructions for the browser. Body = what the audience (user) sees on stage.
The Answer
<head> contains metadata — information about the document that isn't rendered directly: charset, viewport, title, stylesheet links, script tags, SEO meta tags.

<body> contains all visible content: text, images, links, forms — everything the user sees and interacts with.

Both are required direct children of <html>, and <head> must come before <body>.
<head> = metadata (invisible to user, critical for browser).
<body> = all visible content.
Order matters: <head> before <body>.

Lektion 1 · ~45 Min

Das HTML-Boilerplate

Jede HTML-Datei beginnt mit demselben Grundgerüst. Jede Zeile zu verstehen — nicht nur zu kopieren — ist das Fundament, auf dem alles andere aufbaut.

Die Boilerplate-Zeilen sind kein Dekor — jede löst ein bestimmtes reales Problem (Zeichenkodierungsfehler, Viewport-Skalierung auf Mobilgeräten, Veraltete-Browser-Quirks). Fehlt eine davon, entstehen schwer zu debuggende Probleme, die nur in bestimmten Browsern oder Geräten auftreten.

⚓ Kernerkenntnis

Das HTML-Boilerplate ist eine Reihe von Anweisungen an den Browser, wie er dein Dokument parsen und rendern soll — bevor ein einziges Inhaltselement gelesen wird.

<!DOCTYPE html>
Kein HTML-Element — eine Deklaration, die dem Browser mitteilt, das Dokument im modernen HTML5-Standard zu parsen. Ohne sie fällt der Browser in den „Quirks-Modus" und rendert die Seite mit Legacy-Regeln aus den 1990ern.
<html lang="de">
Das Wurzelelement, das das gesamte Dokument umschließt. Das lang-Attribut ist entscheidend: Screenreader verwenden es für die korrekte Aussprache-Engine, Suchmaschinen für das Sprach-Targeting.
<meta charset="UTF-8">
Setzt die Zeichenkodierung auf UTF-8, die jedes Sprachskript der Welt unterstützt. Ohne diese Angabe werden Umlaute, Emoji und nicht-lateinische Zeichen als unleserliche Symbole (Mojibake) angezeigt.
<meta name="viewport" ...>
Verhindert, dass mobile Browser die Seite verkleinern und herauszoomen. Ohne width=device-width, initial-scale=1 sieht deine Seite auf Smartphones winzig aus und CSS-Media-Queries funktionieren nicht korrekt.
✅ Szenario

Du erstellst die erste Seite einer neuen deutschsprachigen Website. Schreibe das vollständige, gültige HTML5-Boilerplate.

HTML
<!DOCTYPE html> <html lang="de"> <head> <meta charset="UTF-8"> <meta name="viewport" content="width=device-width, initial-scale=1.0"> <title>Meine erste Seite</title> </head> <body> <p>Hallo, Welt!</p> </body> </html>
