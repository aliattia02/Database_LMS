Lesson 1 · ~30 min

Form Structure — The Container & Labels

A form is a data pipeline from the user's browser to a server. Understanding the <form> element's attributes, and the critical relationship between labels and inputs, is the foundation of every form you'll ever build.

⚓ Core Insight

Forms have two jobs: collecting data from users, and submitting it to a server. The action tells the browser where to send the data. The method tells it how. Missing labels make forms unusable for keyboard-only and screen reader users.

<form> — Container for all form controls
Wraps all inputs, labels, and buttons. When submitted, collects all fields with a name attribute and sends them as key-value pairs.
action="/submit" — Where to send data
The URL the form data is sent to on submission. If omitted, the form submits to the current URL. In modern SPAs, this is often handled by JavaScript instead.
method="GET" vs method="POST"
GET — appends data to the URL as query parameters (?q=search+term). Bookmarkable, cached. Use for search, filters. POST — sends data in the request body, invisible in the URL. Use for login, registration, anything with sensitive data or side effects.
<fieldset> and <legend>
Group related inputs inside a <fieldset> with a <legend> describing the group. Essential for radio groups and checkboxes. Screen readers announce the legend before each control in the group.
<label for="id"> — Associates label with input
The for attribute matches the id on the input. This creates an association: clicking the label focuses the input. Screen readers read the label when the input receives focus. Every visible input needs a label.
id and name — Two different purposes
id is for the browser (links label to input, used by CSS/JS). name is for the server (it becomes the key in the submitted data). Both are needed. They can share the same value but serve different systems.
⚠️ Common Mistake

Using placeholder text as a substitute for a label. Once the user starts typing, the placeholder disappears. Screen readers don't reliably announce placeholders. Always use a real <label>, even if it's visually hidden for design reasons.

Input Types & Other Form Controls

HTML provides over 20 input types. Choosing the right type triggers the correct mobile keyboard, enables browser validation, and gives users a better experience — all for free.

TypeWhen to useBrowser behaviour
textSingle-line free textPlain text keyboard
emailEmail addressesValidates format, shows @ keyboard on mobile
passwordPasswordsMasks characters, triggers password manager
numberNumeric valuesSpinner controls, numeric keyboard on mobile
telPhone numbersShows phone keyboard on mobile, no validation
urlWeb addressesValidates URL format, shows URL keyboard
checkboxTrue/false, multi-selectToggle on/off independently
radioPick one from a setOnly one selected per name group
dateDate selectionShows native date picker
fileFile uploadOpens file browser
hiddenData to submit silentlyNot visible, not editable by user
rangeSlider within a rangeShows slider control
colorColor pickerShows native color picker
<textarea> — Multi-line text
Use for messages, comments, and any long-form text. rows and cols set the visible size (CSS is preferred for sizing). Cannot use value attribute — default text goes between the tags instead.
<select> + <option> — Dropdown
Presents a list of options in a dropdown menu. The value attribute on each <option> is what gets submitted; the text content is what the user sees. Use <optgroup label="..."> to group related options.
<button type="submit"> — Submit the form
Preferred over <input type="submit"> because it can contain HTML (icons, formatted text). Any button inside a form defaults to type="submit" — always specify the type explicitly to avoid accidental submissions.
<button type="button"> — No form action
Does nothing by default — for JavaScript interactions that should not submit the form. Use this for modal toggles, counter buttons, or any interactive UI element inside a form.
<button type="reset"> — Clear the form
Resets all fields to their initial values. Rarely used in modern UX — users find accidental resets frustrating. Only include if resetting is a genuine workflow need.
required — Field must have a value
Prevents form submission if the field is empty. The browser shows a native error message. Boolean attribute — just the word required, no value needed.
minlength / maxlength
Character count limits for text inputs. maxlength prevents typing beyond the limit. minlength blocks submission if too short. Show remaining characters with JavaScript for better UX.
min / max / step
Numeric and date range constraints. min="0" prevents negative values. max="100" sets the upper bound. step="0.01" allows two decimal places (useful for prices).
pattern — Regex validation
Validates input against a regular expression. Example: pattern="[A-Z]{2}[0-9]{4}" for a product code. Always pair with a descriptive title attribute explaining the expected format — this appears in the browser's error message.
disabled and readonly
disabled — non-interactive, not submitted with the form, visually greyed out. readonly — user cannot edit, but the value is submitted. Use disabled for options that don't apply; use readonly for values shown for reference.
ℹ️ Important

Browser validation is a convenience, not a security measure. Always validate on the server. A user can disable browser validation with novalidate on the form or by using browser dev tools.

✅ Pattern — Fieldset + legend for radio/checkbox groups

Radio buttons that belong to the same group share the same name attribute. The browser enforces single-selection within the group. Always wrap in a <fieldset> with a <legend>.

Practice Questions
Key questions for this topic — click to expand
Mental model
Imagine using a form while blindfolded, relying only on a text-to-speech reader. Without labels, you'd hear "text box, text box, text box" — no idea what to type in each. Labels are the names on the boxes.
The Answer
A <label> associated with an input (via for/id) does three things:

1. Screen readers announce the label text when the input receives focus — without it, users hear only the input type ("text field").
2. Click target — clicking the label focuses or toggles the input, making small targets (radio buttons, checkboxes) far easier to use, especially on mobile.
3. Context for everyone — placeholder text disappears on input; labels remain always visible.
Labels are the primary accessibility requirement for forms.
The for attribute must match the input's id exactly.
Wrapping the input inside the <label> tag is a valid alternative to for/id.
Mental model
When the form is submitted, all inputs are packed into an envelope and sent to the server. The name is the label on the outside of each item in that envelope — without it, the item is simply not included.
The Answer
The name attribute becomes the key in the submitted key-value pair. If name="email", the server receives email=user@example.com.

Any input without a name attribute is silently excluded from form submission — no error, no warning. This is a very common bug: a developer forgets name, wonders why the server receives no data.

The id attribute is separate — it links the label and is used by CSS/JS, but has no role in data submission.
name = key sent to the server. Without it, the field is ignored on submit.
id = browser-side identifier for CSS, JS, and label association.
They can share the same value (e.g. id="email" name="email") but serve different systems.
Mental model
GET writes the data on a postcard — everyone who handles it can see it, and you can reference it by address. POST puts the data in a sealed envelope — only the recipient opens it.
The Answer
GET — data is appended to the URL as query parameters: /search?q=html+forms&lang=en. The request has no body. The URL can be bookmarked, shared, and cached. Use for search, filter, and navigation — anything idempotent where you want the URL to represent the state.

POST — data is sent in the HTTP request body, not visible in the URL. Not cached. Not bookmarkable in that state. Use for login, registration, payment, and any action with side effects (creating or modifying data on the server).
GET: data in URL — bookmarkable, cached. For search and filters.
POST: data in body — not visible in URL. For login, registration, payments.
Never send passwords or sensitive data via GET — they appear in server logs and browser history.

Lektion 1 · ~30 Min.

Formularstruktur — Container & Labels

Ein Formular ist eine Datenpipeline vom Browser des Nutzers zu einem Server. Das <form>-Element, die kritische Beziehung zwischen Labels und Inputs sowie die Unterschiede zwischen GET und POST — das sind die Grundlagen jedes Formulars.

⚓ Kernaussage

Formulare haben zwei Aufgaben: Daten vom Nutzer sammeln und an einen Server übermitteln. action gibt an, wohin die Daten gesendet werden. method gibt an, wie. Fehlende Labels machen Formulare für Tastatur- und Screenreader-Nutzer unbrauchbar.

<form> — Container für alle Formularfelder
Umschließt alle Inputs, Labels und Buttons. Bei der Übermittlung werden alle Felder mit name-Attribut gesammelt und als Schlüssel-Wert-Paare gesendet.
action="/senden" — Wohin die Daten gehen
Die URL, an die die Formulardaten bei Übermittlung gesendet werden. Wird action weggelassen, wird an die aktuelle URL übermittelt.
method="GET" vs method="POST"
GET — hängt Daten als Query-Parameter an die URL an. Verlinkbar, gecacht. Für Suche und Filter verwenden. POST — sendet Daten im Request-Body, in der URL unsichtbar. Für Login, Registrierung und alles mit vertraulichen Daten verwenden.
<fieldset> und <legend>
Verwandte Inputs in einem <fieldset> mit beschreibender <legend> gruppieren. Unverzichtbar für Radio-Gruppen und Checkboxen. Screenreader lesen die Legend vor jedem Steuerelement der Gruppe vor.
<label for="id"> — Verknüpft Label mit Input
Das for-Attribut muss mit der id des Inputs übereinstimmen. Dadurch wird eine Verknüpfung hergestellt: Klick auf das Label fokussiert den Input. Screenreader lesen das Label, wenn der Input fokussiert wird.
id und name — Zwei verschiedene Zwecke
id ist für den Browser (verknüpft Label und Input, für CSS/JS). name ist für den Server (wird zum Schlüssel in den übermittelten Daten). Beide werden benötigt und können denselben Wert haben.
⚠️ Häufiger Fehler

placeholder-Text als Label-Ersatz verwenden. Sobald der Nutzer zu tippen beginnt, verschwindet der Placeholder. Screenreader lesen Placeholders nicht zuverlässig vor. Immer ein echtes <label> verwenden.

Input-Typen & weitere Formularelemente

HTML bietet über 20 Input-Typen. Den richtigen Typ zu wählen, aktiviert die passende mobile Tastatur, ermöglicht Browser-Validierung und verbessert die Nutzererfahrung — kostenlos.

TypVerwendungBrowser-Verhalten
textEinzeiliger FreitextStandard-Texttastatur
emailE-Mail-AdressenValidiert Format, zeigt @-Tastatur auf Mobilgeräten
passwordPasswörterVerbirgt Zeichen, aktiviert Passwort-Manager
numberNumerische WerteSpinner-Steuerelemente, Zahlentastatur auf Mobil
telTelefonnummernTelefonkästatur auf Mobil, keine Formatvalidierung
checkboxJa/Nein, MehrfachauswahlUnabhängig ein- und ausschaltbar
radioEine Option aus einer GruppeNur eine Auswahl pro name-Gruppe
dateDatumsauswahlZeigt nativen Datumsauswähler
fileDatei-UploadÖffnet Datei-Browser
hiddenDaten still übermittelnNicht sichtbar, nicht bearbeitbar
<textarea> — Mehrzeiliger Text
Für Nachrichten, Kommentare und längere Texteingaben. rows und cols setzen die sichtbare Größe (CSS ist für Styling bevorzugt). Standardtext kommt zwischen die Tags, nicht als value-Attribut.
<select> + <option> — Dropdown
Zeigt eine Liste von Optionen als Dropdown-Menü. Das value-Attribut bei <option> wird übermittelt; der Text zwischen den Tags ist das, was der Nutzer sieht. <optgroup> zum Gruppieren verwenden.
<button type="submit"> — Formular absenden
Bevorzugt gegenüber <input type="submit">, weil er HTML-Inhalt (Icons, formatierten Text) enthalten kann. Jeder Button in einem Formular hat standardmäßig type="submit" — immer explizit angeben, um versehentliches Absenden zu vermeiden.
<button type="button"> — Keine Formularaktion
Tut standardmäßig nichts — für JavaScript-Interaktionen, die das Formular nicht absenden sollen. Für Modal-Toggle-Buttons, Zähler oder andere UI-Elemente innerhalb eines Formulars verwenden.
required — Pflichtfeld
Verhindert das Absenden, wenn das Feld leer ist. Der Browser zeigt eine native Fehlermeldung. Boolesches Attribut — nur das Wort required, kein Wert nötig.
minlength / maxlength
Zeichenbegrenzungen für Texteingaben. maxlength verhindert weitere Eingabe. minlength blockiert das Absenden bei zu kurzem Inhalt.
min / max / step
Numerische und Datumsbeschränkungen. min="0" verhindert negative Werte. step="0.01" erlaubt zwei Dezimalstellen (nützlich für Preise).
disabled und readonly
disabled — nicht interaktiv, wird nicht mit dem Formular übermittelt, visuell ausgegraut. readonly — nicht bearbeitbar, wird aber übermittelt. disabled für nicht anwendbare Optionen; readonly für Referenzwerte verwenden.
ℹ️ Wichtig

Browser-Validierung ist eine Bequemlichkeit, kein Sicherheitsmechanismus. Immer serverseitig validieren. Ein Nutzer kann die Browser-Validierung mit novalidate am Formular oder über DevTools deaktivieren.

Mentales Modell
Stell dir vor, du füllst ein Formular mit verbundenen Augen aus und verlässt dich nur auf eine Sprachausgabe. Ohne Labels würdest du hören: „Textfeld, Textfeld, Textfeld" — ohne Ahnung, was in welches Feld gehört. Labels sind die Beschriftungen auf den Kästchen.
Die Antwort
Ein mit einem Input verknüpftes <label> (über for/id) erfüllt drei Aufgaben:

1. Screenreader lesen den Label-Text vor, wenn der Input fokussiert wird — ohne ihn hören Nutzer nur den Input-Typ („Textfeld").
2. Klickziel — Klick auf das Label fokussiert oder schaltet den Input um, was kleine Ziele (Radio-Buttons, Checkboxen) besser erreichbar macht.
3. Kontext für alle — Placeholder-Text verschwindet beim Tippen; Labels bleiben immer sichtbar.
Labels sind die wichtigste Barrierefreiheitsanforderung für Formulare.
Das for-Attribut muss exakt mit der id des Inputs übereinstimmen.
Den Input innerhalb des <label>-Tags einzuwickeln ist eine gültige Alternative zu for/id.
Mentales Modell
Beim Absenden des Formulars werden alle Inputs in einen Umschlag gepackt und an den Server geschickt. Der name ist das Etikett auf jedem Gegenstand in diesem Umschlag — ohne ihn wird der Gegenstand einfach nicht eingelegt.
Die Antwort
Das name-Attribut wird zum Schlüssel im übermittelten Schlüssel-Wert-Paar. Bei name="email" empfängt der Server email=nutzer@beispiel.de.

Jeder Input ohne name-Attribut wird bei der Formularübermittlung stillschweigend ausgelassen — kein Fehler, keine Warnung. Das ist ein sehr häufiger Fehler.

Das id-Attribut hat eine andere Rolle: es verknüpft Label und Input und wird von CSS/JS verwendet — es hat jedoch keinen Einfluss auf die Datenübermittlung.
name = Schlüssel, der an den Server gesendet wird. Ohne ihn wird das Feld beim Absenden ignoriert.
id = Browser-seitiger Bezeichner für CSS, JS und Label-Verknüpfung.
Beide können denselben Wert haben (z.B. id="email" name="email"), dienen aber verschiedenen Zwecken.
Mentales Modell
GET schreibt die Daten auf eine Postkarte — jeder, der sie bearbeitet, kann sie lesen, und du kannst sie per Adresse referenzieren. POST legt die Daten in einen versiegelten Umschlag — nur der Empfänger öffnet ihn.
Die Antwort
GET — Daten werden als Query-Parameter an die URL angehängt: /suche?q=html+formulare&sprache=de. Kein Request-Body. Die URL kann als Lesezeichen gespeichert, geteilt und gecacht werden. Für Suche, Filter und Navigation verwenden — alles, bei dem die URL den Zustand repräsentieren soll.

POST — Daten werden im HTTP-Request-Body gesendet, in der URL nicht sichtbar. Nicht gecacht. Für Login, Registrierung, Bezahlung und alle Aktionen mit Nebenwirkungen (Daten auf dem Server erstellen oder ändern) verwenden.
GET: Daten in URL — als Lesezeichen speicherbar, gecacht. Für Suche und Filter.
POST: Daten im Body — in URL nicht sichtbar. Für Login, Registrierung, Bezahlung.
Passwörter und vertrauliche Daten niemals per GET senden — sie erscheinen in Server-Logs und im Browser-Verlauf.
