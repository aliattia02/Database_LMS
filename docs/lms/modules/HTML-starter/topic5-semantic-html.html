Lesson 1 · ~30 min

Semantic Elements — Giving Meaning to Structure

Semantic HTML means choosing elements that describe their content role, not just their visual appearance. A <div> is a generic container with no meaning; a <nav> says "this is navigation." That distinction matters to search engines, screen readers, and every developer who reads your code later.

⚓ Core Insight

A page full of <div> elements is meaningless to a machine. Search engines use HTML structure to understand page hierarchy and topic relevance. Screen readers use landmark elements to let users jump directly to the main content, skip navigation, or list every heading on the page. Choosing the right element is free accessibility and SEO.

<header> — Page or section header
Usually contains a logo, site title, or top navigation. A page can have multiple <header> elements — one for the page and one per <article>. Not the same as <head>, which is invisible metadata.
<nav> — Navigation block
Wraps a group of navigation links — primary menus, breadcrumbs, pagination. Don't wrap every list of links in <nav> — reserve it for major site-navigation groups. Screen readers can list all <nav> landmarks so users can jump directly to them.
<main> — Primary page content
The dominant content unique to this page — not repeated elements like headers or footers. There must be only one <main> per page. Screen readers provide a keyboard shortcut to jump directly to <main>, bypassing repetitive headers.
<footer> — Page or section footer
Typically contains copyright, secondary navigation, contact links, or author information. Like <header>, a page can have multiple footers (e.g. one per article). Screen readers expose these as "contentinfo" landmarks.
<aside> — Tangentially related content
Content that is related to the surrounding page but not essential to understanding it: sidebars, pull quotes, related article lists, or advertisements. Becomes a "complementary" landmark — screen reader users can skip it or navigate to it directly.
⚠️ Common Mistake

Using <header> and <footer> just to style the top and bottom of a design. The browser cares about the element's semantic role, not its visual position. Only use these elements when the content they contain is genuinely a header or footer in the document sense.

<article> — Self-contained, reusable content
Independently distributable content — a blog post, a news story, a forum post, a product card, a comment. Ask yourself: "if I copied this to another page, would it still make sense on its own?" If yes, it's an <article>.
<section> — Thematic grouping with a heading
A thematically-related block of content. Should almost always have a heading (<h2> – <h6>) as its first child. If there's no natural heading for the group, a <div> is probably more appropriate. Think of <section> as a chapter in a document.
<figure> + <figcaption> — Media with caption
Wraps self-contained media (images, diagrams, charts, code samples) with an optional caption. The <figcaption> is announced by screen readers and associated with the figure. It provides context even when the image cannot be seen or loaded.
<time datetime=""> — Machine-readable date or time
Human-readable text goes between the tags; the datetime attribute gives the ISO 8601 machine-readable format. Example: <time datetime="2024-06-15">15 June 2024</time>. Allows browsers, search engines, and assistive tech to interpret dates accurately regardless of the display format.

Lesson 2 · ~30 min

Accessibility Basics — HTML for Everyone

About 15% of the world's population has some form of disability. Web accessibility ensures your pages work for keyboard-only users, screen reader users, and people with visual, motor, or cognitive differences. Most accessibility wins in HTML are free — they come from choosing the right element, writing good text, and keeping structure logical.

⚓ Core Insight

Accessibility isn't a special feature you add at the end — it's a quality indicator for your HTML. Code that is accessible is also more maintainable, SEO-friendly, and robust. In many countries (EU, USA, UK), accessible web content is a legal requirement for public-facing websites. Start with good HTML and most of the work is already done.

Informational images — describe the information, not the pixels
Alt text should convey the same information the image conveys. For a chart: "Bar chart showing monthly revenue grew from €40k in January to €65k in June." Not "a chart." Not "image." Not "chart.png."
Decorative images — use an empty alt=""
Purely decorative images (dividers, background textures, stock photos that add no information) should use alt="". Screen readers then skip them silently. Never omit alt entirely — without it, screen readers announce the filename ("product-hero-2024-v3-final.jpg"), which is useless and disruptive.
Functional images (icon buttons) — describe the action
When an image serves as a button or link, alt text should describe what clicking it does, not what the image looks like. <img alt="Search"> on a magnifying glass icon, not alt="magnifying glass". <img alt="Close"> on an × icon, not alt="x".
⚠️ Common Mistake

Writing redundant alt text like "Photo of a dog" or "Image showing a bar chart." Screen readers already announce the element type. Write just "A golden retriever sitting in autumn leaves" or "Bar chart showing Q4 revenue by region."

One <h1> per page — the document title
The <h1> should be the single most important heading on the page, usually the page or article title. Screen readers announce it when a page loads. Having multiple <h1> elements confuses both assistive tech and search engines about what the page is actually about.
Never skip heading levels
Going from <h1> to <h3> without an <h2> in between is a structural error. Screen reader users navigate pages by tabbing through headings — a skipped level breaks their mental model of the page structure. Think of headings as an outline: H1 → H2 → H3, not H1 → H3.
Pick headings for structure, not for size
The most common mistake: choosing a heading level because of how big it looks by default. Use CSS to change visual size. Choose heading level only based on the document hierarchy. <h2> is "a section under the page title," regardless of how you want it to look.
✅ The First Rule of ARIA

The first rule of ARIA is: don't use ARIA — use native HTML elements instead. An <input type="checkbox"> always beats <div role="checkbox">. Native elements have built-in keyboard behaviour, focus management, and accessibility announcements for free. ARIA is a repair kit for situations where native HTML can't express the right semantics.

aria-label — Adds an invisible text label
Use when there is no visible text label to associate with a control. Common case: an icon-only button. <button aria-label="Close modal">✕</button> — the screen reader announces "Close modal button" even though the visible content is just "✕".
aria-describedby — Points to a description element
Associates an element with a longer description elsewhere in the page. <input aria-describedby="pw-hint"> with <span id="pw-hint">Must be at least 8 characters</span>. Screen readers read both the label and the description when the input receives focus.
role — Overrides an element's semantic role
Tells assistive technology what kind of widget an element is. role="alert" makes a div announce dynamically inserted content immediately. role="status" makes it announce politely. Only reach for role when no semantic HTML element fits and you've verified the keyboard and focus behaviour is correct.

Mental model
Imagine giving someone a document with all the formatting stripped out — no bold, no headings, just text in a single block. You can still read it, but finding sections is hard and the structure is invisible. Semantic HTML puts the structure back in, in a form that machines can read.
The Answer
Semantic HTML is the practice of using HTML elements according to their intended meaning, not their default visual appearance.

A <div> has no inherent meaning — it's a generic block. A <nav> tells every browser, screen reader, and search engine crawler: "this is a navigation region." A <time datetime="2024-06-15"> says: "this text represents a date, here is the machine-readable version."

It matters for three reasons: accessibility (screen readers build navigation menus from landmarks and headings), SEO (search engines weight content inside meaningful elements differently), and maintainability (code intent is self-documenting).

Mental model
ARIA is a set of HTML attributes that add semantic information that native HTML alone can't express. Think of it as subtitles for content that's already there — it doesn't change what's on screen, it adds a parallel description for assistive technology.
The Answer
ARIA (Accessible Rich Internet Applications) is a specification of attributes that add semantic roles, states, and properties to HTML elements. It bridges the gap between HTML and the information assistive technologies need.

When you need it: when building custom interactive widgets that have no native HTML equivalent — a custom dropdown, a tab panel, an accordion, a live region that announces dynamic content. role="tabpanel", aria-expanded="false", aria-label="Close".

When you don't: whenever a native HTML element already exists. <button> instead of <div role="button">. <input type="checkbox"> instead of <div role="checkbox">. Native elements are always better — they come with built-in keyboard handling, focus management, and correct announcements for free.

Mental model
Headings are to a web page what a table of contents is to a book. If a book's table of contents jumped from Chapter 1 to Chapter 3 with no Chapter 2, you'd know something was wrong. A skipped heading level sends the same signal — something is broken in the structure.
The Answer
Screen reader users navigate pages by headings using keyboard shortcuts — they hear a list of headings and jump to the section they want, exactly like scanning a table of contents. A broken hierarchy (h1 → h3, skipping h2) creates confusing gaps in that mental model.

Semantische Elemente — Inhalten Bedeutung geben

Semantisches HTML bedeutet, Elemente zu wählen, die ihre inhaltliche Rolle beschreiben — nicht nur ihr visuelles Erscheinungsbild. Ein <div> ist ein generischer Container ohne Bedeutung; ein <nav> sagt: „Das ist Navigation." Diese Unterscheidung ist für Suchmaschinen, Screenreader und jeden Entwickler, der deinen Code später liest, entscheidend.

⚓ Kernaussage

Eine Seite voller <div>-Elemente ist für eine Maschine bedeutungslos. Suchmaschinen nutzen die HTML-Struktur, um die Hierarchie und thematische Relevanz einer Seite zu verstehen. Screenreader nutzen Landmarken-Elemente, damit Nutzer direkt zum Hauptinhalt springen, die Navigation überspringen oder alle Überschriften der Seite auflisten können. Das richtige Element zu wählen ist kostenlose Barrierefreiheit und SEO.

<header> — Seiten- oder Abschnitts-Kopfbereich
Enthält üblicherweise Logo, Seitentitel oder Hauptnavigation. Eine Seite kann mehrere <header>-Elemente haben — eines für die Seite und eines je <article>. Nicht identisch mit <head>, das unsichtbare Metadaten enthält.
<nav> — Navigationsbereich
Umschließt eine Gruppe von Navigationslinks — Hauptmenüs, Breadcrumbs, Seitennavigation. Nicht jede Link-Liste in <nav> einwickeln — nur für wichtige Seitennavigationsgruppen verwenden. Screenreader können alle <nav>-Landmarken auflisten, damit Nutzer direkt dorthin springen können.
<main> — Primärer Seiteninhalt
Der dominante, seitenspezifische Inhalt — keine wiederholten Elemente wie Header oder Footer. Es darf nur ein <main> pro Seite geben. Screenreader bieten eine Tastenkombination, um direkt zum <main> zu springen und sich wiederholende Kopfbereiche zu überspringen.
<footer> — Seiten- oder Abschnitts-Fußbereich
Enthält typischerweise Copyright-Angaben, sekundäre Navigation, Kontaktlinks oder Autoreninfos. Wie <header> kann eine Seite mehrere Footers haben (z.B. einen je Artikel). Screenreader behandeln diese als „contentinfo"-Landmarken.
<aside> — Ergänzende Inhalte
Inhalte, die mit dem umgebenden Inhalt verwandt, aber nicht entscheidend für das Verständnis sind: Seitenleisten, Zitatboxen, verwandte Artikellisten oder Werbeanzeigen. Wird als „complementary"-Landmarke behandelt — Screenreader-Nutzer können ihn überspringen oder direkt ansteuern.
⚠️ Häufiger Fehler

<header> und <footer> nur zum Stylen des oberen und unteren Bereichs verwenden. Den Browser interessiert die semantische Rolle des Elements, nicht seine visuelle Position. Diese Elemente nur verwenden, wenn der Inhalt tatsächlich ein Header oder Footer im dokumentarischen Sinne ist.

<article> — Eigenständiger, wiederverwendbarer Inhalt
Unabhängig verteilbarer Inhalt — ein Blogbeitrag, ein Nachrichtenartikel, ein Produktkarte, ein Kommentar. Der Schlüsseltest: Würde dieser Inhalt auf einer anderen Seite noch Sinn ergeben? Wenn ja, ist es ein <article>.
<section> — Thematische Gruppe mit Überschrift
Ein thematisch zusammengehöriger Inhaltsblock mit einer Überschrift. Sollte fast immer eine Überschrift (<h2>–<h6>) als erstes Kindelement haben. Ohne natürliche Überschrift für die Gruppe ist ein <div> wahrscheinlich besser geeignet. Denke an <section> als Kapitel in einem Dokument.
<figure> + <figcaption> — Medien mit Beschriftung
Umschließt eigenständige Medien (Bilder, Diagramme, Code-Beispiele) mit optionaler Bildunterschrift. Die <figcaption> wird von Screenreadern vorgelesen und dem Figure-Element zugeordnet. Sie bietet Kontext auch dann, wenn das Bild nicht sichtbar oder geladen ist.
<time datetime=""> — Maschinenlesbares Datum oder Uhrzeit
Menschenlesbarer Text kommt zwischen die Tags; das datetime-Attribut gibt das ISO 8601-Format für Maschinen an. Beispiel: <time datetime="2024-06-15">15. Juni 2024</time>. Ermöglicht Browser, Suchmaschinen und Hilfstechnologien, Datumsangaben unabhängig vom Anzeigeformat korrekt zu interpretieren.

Lektion 2 · ~30 Min.

Barrierefreiheitsgrundlagen — HTML für alle

Etwa 15 % der Weltbevölkerung hat eine Form von Behinderung. Barrierefreiheit im Web stellt sicher, dass deine Seiten für Nur-Tastatur-Nutzer, Screenreader-Nutzer und Menschen mit visuellen, motorischen oder kognitiven Unterschieden funktionieren. Die meisten Barrierefreiheits-Gewinne in HTML sind kostenlos — sie kommen aus der Wahl des richtigen Elements, gutem Alternativtext und logischer Struktur.

⚓ Kernaussage

Barrierefreiheit ist kein Feature, das man am Ende hinzufügt — sie ist ein Qualitätsindikator für dein HTML. Code, der barrierefrei ist, ist auch wartbarer, SEO-freundlicher und robuster. In vielen Ländern (EU, USA, UK) ist barrierefreier Webinhalt für öffentlich zugängliche Websites gesetzlich vorgeschrieben. Beginne mit gutem HTML und die meiste Arbeit ist bereits erledigt.

Informationsbilder — die Information beschreiben, nicht die Pixel
Alt-Text soll dieselbe Information vermitteln, die das Bild vermittelt. Für ein Diagramm: alt="Balkendiagramm: Monatsumsatz stieg von 40 T€ im Januar auf 65 T€ im Juni." — nicht „Diagramm", nicht „Bild", nicht den Dateinamen.
Dekorative Bilder — leeres alt="" verwenden
Rein dekorative Bilder (Trennlinien, Hintergrundtexturen, Stockfotos ohne Informationswert) erhalten alt="". Screenreader überspringen sie dann lautlos. Niemals das alt-Attribut ganz weglassen — ohne es liest der Screenreader den Dateinamen vor, was nutzlos und störend ist.
Funktionale Bilder (Icon-Buttons) — die Aktion beschreiben
Wenn ein Bild als Button oder Link dient, soll der Alt-Text beschreiben, was der Klick bewirkt — nicht wie das Bild aussieht. alt="Suche" für ein Lupensymbol, nicht alt="Lupe". alt="Schließen" für ein ×-Symbol, nicht alt="x".
⚠️ Häufiger Fehler

Redundanten Alt-Text wie „Foto eines Hundes" oder „Bild zeigt ein Balkendiagramm" schreiben. Screenreader kündigen den Elementtyp bereits an. Schreibe einfach: „Goldener Retriever sitzt in Herbstlaub" oder „Balkendiagramm: Q4-Umsatz nach Region".

Nur eine <h1> pro Seite — der Dokumententitel
Die <h1> sollte die einzelne wichtigste Überschrift der Seite sein — üblicherweise der Seiten- oder Artikeltitel. Mehrere <h1>-Elemente verwirren sowohl Hilfstechnologien als auch Suchmaschinen darüber, worum es auf der Seite eigentlich geht.
Überschriftebenen niemals überspringen
Von <h1> zu <h3> ohne <h2> dazwischen ist ein struktureller Fehler. Screenreader-Nutzer navigieren Seiten per Tastenkürzel durch Überschriften — eine übersprungene Ebene bricht ihr mentales Modell der Seitenstruktur. Denke an Überschriften wie eine Gliederung: H1 → H2 → H3, nicht H1 → H3.
Überschriften nach Struktur wählen, nicht nach Größe
Der häufigste Fehler: eine Überschriftenebene wählen, weil sie standardmäßig groß aussieht. Verwende CSS, um die visuelle Größe zu ändern. Wähle die Überschriftenebene ausschließlich nach der Dokumenthierarchie. <h2> bedeutet „ein Abschnitt unter dem Seitentitel" — unabhängig vom gewünschten Aussehen.
✅ Die erste ARIA-Regel

Die erste Regel von ARIA lautet: kein ARIA verwenden — stattdessen native HTML-Elemente benutzen. Ein <input type="checkbox"> ist immer besser als <div role="checkbox">. Native Elemente haben eingebautes Tastaturverhalten, Fokusmanagement und Barrierefreiheits-Ankündigungen — kostenlos. ARIA ist ein Reparaturset für Situationen, in denen natives HTML die richtige Semantik nicht ausdrücken kann.

aria-label — Fügt ein unsichtbares Textlabel hinzu
Verwenden, wenn kein sichtbarer Textlabel für ein Steuerelement vorhanden ist. Häufiger Fall: Icon-only-Button. <button aria-label="Modal schließen">✕</button> — der Screenreader kündigt „Modal schließen Schaltfläche" an, obwohl der sichtbare Inhalt nur „✕" ist.
aria-describedby — Verweist auf ein Beschreibungselement
Verknüpft ein Element mit einer längeren Beschreibung an anderer Stelle auf der Seite. <input aria-describedby="pw-hinweis"> mit <span id="pw-hinweis">Mindestens 8 Zeichen</span>. Screenreader lesen Label und Beschreibung vor, wenn das Eingabefeld fokussiert wird.
role — Überschreibt die semantische Rolle eines Elements
Teilt der Hilfstechnologie mit, was für ein Widget-Typ ein Element ist. role="alert" lässt einen Div dynamisch eingefügten Inhalt sofort ankündigen. role="status" kündigt ihn höflich an. Nur einsetzen, wenn kein semantisches HTML-Element passt und das Tastatur- und Fokusverhalten korrekt implementiert ist.

Mentales Modell
Stell dir vor, du gibst jemandem ein Dokument, aus dem alle Formatierungen entfernt wurden — kein Fett, keine Überschriften, nur Text in einem einzigen Block. Du kannst es noch lesen, aber Abschnitte zu finden ist schwierig und die Struktur ist unsichtbar. Semantisches HTML stellt diese Struktur in einer Form wieder her, die Maschinen lesen können.
Die Antwort
Semantisches HTML ist die Praxis, HTML-Elemente entsprechend ihrer beabsichtigten Bedeutung zu verwenden — nicht wegen ihres standardmäßigen visuellen Erscheinungsbilds.

Ein <div> hat keine inhärente Bedeutung. Ein <nav> sagt jedem Browser, Screenreader und Suchmaschinen-Crawler: „Das ist ein Navigationsbereich." Ein <time datetime="2024-06-15"> sagt: „Dieser Text repräsentiert ein Datum, hier ist die maschinenlesbare Version."

Es ist aus drei Gründen wichtig: Barrierefreiheit (Screenreader erstellen Navigationsmenüs aus Landmarken und Überschriften), SEO (Suchmaschinen gewichten Inhalte in bedeutungsvollen Elementen unterschiedlich) und Wartbarkeit (Code-Absicht ist selbstdokumentierend).
Semantisch = Elemente für ihre Bedeutung verwenden, nicht für ihre visuelle Ausgabe.
Vorteile: Barrierefreiheit, SEO und selbstdokumentierender Code.
Ein <div> ist die richtige Wahl nur, wenn kein semantisches Element passt.
Entscheidungsbaum
„Würde dieser Inhalt auf einer ganz anderen Seite noch Sinn ergeben?" Ja → <article>. „Hat diese Inhaltsgruppe ein einziges Thema und eine Überschrift?" Ja → <section>. „Ist das nur ein Stil- oder Layout-Wrapper ohne bedeutungsvolle Gruppierung?" Ja → <div>.
Die Antwort
<article> — eigenständiger, unabhängig verteilbarer Inhalt. Beispiele: Blogbeitrag, Nachrichtenartikel, Produktkarte, Nutzerkommentar. Der Schlüsseltest: Macht er ohne Kontext Sinn?

<section> — eine thematische Gruppe zusammengehöriger Inhalte mit einer Überschrift. Beispiele: „Über uns"-Bereich einer Homepage, „Kommentare"-Abschnitt eines Artikels, Kapitel in einem langen Dokument. Abschnitte stehen nicht so eigenständig wie Artikel.

<div> — ein generischer Container ohne semantische Bedeutung. Die richtige Wahl, wenn du einen Wrapper rein für CSS-Layout oder JavaScript-Targeting brauchst und keines der semantischen Elemente passt.
<article>: eigenständig, könnte auf einer anderen Seite stehen.
<section>: thematische Gruppe mit Überschrift — Teil eines größeren Ganzen.
<div>: reiner Layout-Wrapper — keine inhärente Bedeutung.
Mentales Modell
Stell dir vor, du beschreibst die Seite jemandem am Telefon. Du würdest nicht sagen: „Hier ist ein Bild." Du würdest sagen, was das Bild zeigt und warum es wichtig ist. Das ist Alt-Text — ein gesprochenes Äquivalent des visuellen Inhalts.
Die Antwort
Guter Alt-Text beschreibt die Information, die das Bild vermittelt: alt="Balkendiagramm: Q3-Umsatz 120 T€, +22 % gegenüber Q2". Er ist prägnant, spezifisch und kontextuell.

Schlechter Alt-Text: alt="Bild", alt="Foto", alt="Diagramm" oder ein Dateiname wie alt="umsatz-q3-2024-final.png". Diese vermitteln nichts.

Dekorative Bilder ohne Informationswert erhalten alt="" — ein leerer String weist Screenreader an, sie zu überspringen. Das alt-Attribut niemals ganz weglassen — der Browser gibt dann den Dateinamen aus.

Funktionale Bilder (Icon-Buttons, verlinkte Bilder): Alt-Text = die Aktion, nicht das Icon. alt="Suche", nicht alt="Lupe".
Information beschreiben, nicht Pixel: „Was kommuniziert dieses Bild?" — nicht „Wie sieht es aus?"
Dekorativ: alt="" — das Attribut ganz weglassen ist etwas anderes (und schlechter).
Icon-Buttons: die Aktion beschreiben (alt="Schließen"), nicht das Icon (alt="×").

ARIA ist wie Untertitel für Inhalte, die bereits vorhanden sind — es ändert nichts am Bildschirm, aber fügt eine parallele Beschreibung für Hilfstechnologien hinzu.
Die Antwort
ARIA (Accessible Rich Internet Applications) ist eine Spezifikation von Attributen, die semantische Rollen, Zustände und Eigenschaften zu HTML-Elementen hinzufügen. Es überbrückt die Lücke zwischen HTML und den Informationen, die Hilfstechnologien benötigen.

Wann du es brauchst: beim Erstellen benutzerdefinierter interaktiver Widgets ohne native HTML-Entsprechung — ein eigenes Dropdown, ein Tab-Panel, ein Akkordeon, eine Live-Region für dynamische Inhalte.

Wann du es nicht brauchst: wenn ein natives HTML-Element bereits existiert. <button> statt <div role="button">. <input type="checkbox"> statt <div role="checkbox">. Native Elemente sind immer besser — sie bringen eingebautes Tastaturverhalten, Fokusmanagement und korrekte Ankündigungen kostenlos mit.
ARIA fügt semantische Informationen hinzu, die natives HTML allein nicht ausdrücken kann.
Erste ARIA-Regel: es nicht verwenden — stattdessen natives HTML benutzen, wenn möglich.
Einsetzen bei eigenen Widgets (Tabs, Akkordeons, Live-Regionen) ohne native Entsprechung.
Mentales Modell
Überschriften sind für eine Webseite das, was ein Inhaltsverzeichnis für ein Buch ist. Wenn das Inhaltsverzeichnis von Kapitel 1 zu Kapitel 3 springen würde ohne Kapitel 2, wüsstest du, dass etwas nicht stimmt. Eine übersprungene Überschriftenebene sendet dasselbe Signal.
Die Antwort
Screenreader-Nutzer navigieren Seiten per Tastenkürzel durch Überschriften — sie hören eine Liste von Überschriften und springen zum gewünschten Abschnitt, genau wie beim Überfliegen eines Inhaltsverzeichnisses. Eine fehlerhafte Hierarchie (h1 → h3, h2 übersprungen) erzeugt verwirrende Lücken in diesem mentalen Modell.

Auch Suchmaschinen nutzen die Überschriftenhierarchie, um Seitenstruktur und thematische Relevanz zu verstehen. Eine einzelne <h1> pro Seite signalisiert das Hauptthema; <h2> signalisiert wichtige Abschnitte; <h3> Unterabschnitte davon.

Die Überschriftenebene soll die Dokumentgliederung widerspiegeln — nicht die gewünschte Schriftgröße. Wenn eine Überschrift größer oder kleiner aussehen soll, ändere das mit CSS. Wähle niemals eine Überschriftenebene wegen ihres standardmäßigen visuellen Gewichts.
Überschriften sind Navigationsanker für Screenreader-Nutzer — fehlerhafte Reihenfolge bricht ihre Karte.
Eine <h1> pro Seite; niemals Ebenen überspringen (h1 → h2 → h3, nicht h1 → h3).
Überschriftenebene = Dokumentstruktur. Visuelle Größe = CSS. Beides niemals verwechseln.
