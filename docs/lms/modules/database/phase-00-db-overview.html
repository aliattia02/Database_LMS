Section 1

What Is a Database?

A database is an organised collection of data that can be stored, retrieved, and modified efficiently. But so is a spreadsheet. The key difference is what happens at scale — with many users, billions of rows, and the need for guaranteed data integrity.

FeatureSpreadsheetDatabase
Data volume~1M rows before it breaksBillions of rows, no problem
Simultaneous users1 user editing safelyThousands of concurrent users
Data integrityYou can type anything anywhereEnforced types, constraints, foreign keys
Query speedSlow on large dataIndexed queries in milliseconds
RelationshipsVLOOKUP / manual linkingNative JOINs across any tables
Crash recoveryFile corruption riskWAL, transactions, ACID
Access controlFile-level onlyRow-level, column-level permissions
Mental model

Think of a spreadsheet as a personal notebook and a database as a professional filing system with staff, locks, an index, and a security desk. Both store information — they serve completely different scales.

Storage — Data persists reliably
A database stores data on disk so it survives power loss, crashes, and restarts. Unlike a variable in a running program, database data is durable.
Retrieval — Data is queryable
You don't scan every row to find what you need. Indexes and query planners let you find one record in billions in milliseconds.
Integrity — Data is correct
Rules (constraints, types, foreign keys) prevent bad data from entering in the first place. A database rejects age = "hello". A spreadsheet accepts it.
A — Atomicity: All or nothing
A transaction either completes fully or rolls back completely. Transferring money: if the debit succeeds but the credit fails, the entire transaction is undone. No partial writes.
C — Consistency: Valid state to valid state
A transaction can only bring the database from one valid state to another. Rules (constraints, foreign keys) are never violated mid-transaction.
I — Isolation: Concurrent transactions don't interfere
Multiple users running queries simultaneously see consistent results. Transaction A's in-progress work is invisible to Transaction B until A commits.
D — Durability: Committed data survives crashes
Once the database says "COMMIT succeeded," the data is on disk — even if the power goes out one millisecond later. The Write-Ahead Log (WAL) guarantees this.
Key insight

ACID is what makes databases trustworthy enough to handle money transfers, medical records, and booking systems. For healthcare or finance data, these guarantees are not optional — they are a correctness requirement.

Section 2

The Relational Model & SQL

Relational databases organise data into tables with rows and columns, linked by keys and queried with SQL. This model runs banks, hospitals, e-commerce, and almost every enterprise system on earth.

ConceptWhat it isWhy it matters
Table (Relation)Rows and columns — one real-world entity type per tableFundamental unit of storage
Primary KeyUnique identifier for each row — never null, never duplicateAlways find and reference exactly one row
Foreign KeyColumn referencing another table's primary keyLinks tables; enforces referential integrity
SchemaBlueprint — which tables, columns, types existDefined upfront; changes are controlled migrations
IndexPre-sorted lookup structure on one or more columnsTurns O(n) full scans into O(log n) lookups
SQLStructured Query Language — universal relational DB languageLearn once, works on MariaDB, PostgreSQL, MySQL, Oracle…
The core idea

Instead of one giant table with duplicated data, split into smaller focused tables and link them with keys. Update in one place; every reference stays consistent automatically.

SQL — linked tables
-- Passengers table (master data) CREATE TABLE passengers ( passenger_id INT AUTO_INCREMENT PRIMARY KEY, full_name VARCHAR(100) NOT NULL, email VARCHAR(255) UNIQUE NOT NULL ); -- Bookings table links passengers ↔ flights via foreign keys CREATE TABLE bookings ( booking_id INT AUTO_INCREMENT PRIMARY KEY, passenger_id INT NOT NULL, flight_id INT NOT NULL, FOREIGN KEY (passenger_id) REFERENCES passengers(passenger_id), FOREIGN KEY (flight_id) REFERENCES flights(flight_id) );

Why choose MariaDB?

MariaDB is a drop-in MySQL replacement (~99% syntax compatible), adds useful features like RETURNING and ed25519 auth, and ships Galera multi-master clustering as a first-class feature — all fully open source with no Oracle dependencies.

NoSQL Landscape

NoSQL databases were built to solve problems that relational databases struggle with: unstructured data, massive horizontal write scaling, and non-tabular data models. Understanding what problem each type solves is how you pick the right tool.

Document stores (MongoDB)
JSON-like documents instead of rows. Flexible schema — great for products with varying attributes. Poor at multi-table JOINs.
Key-value (Redis)
In-memory, microsecond latency. Sessions, caching, pub/sub, rate limiting. Not a replacement for a primary database — a complement.
Wide-column (Cassandra)
Massive write throughput, no single master, append-mostly. Netflix, Apple iCloud. Queries by partition key only — no ad-hoc JOINs.
Graph (Neo4j)
Relationship traversal as a first-class operation. Social graphs, fraud detection, recommendation engines. Wrong choice for tabular/transactional data.

🏥
Healthcare EHR → MariaDB + Galera
Patient data, vitals, prescriptions — highly structured, ACID required for regulatory compliance, complex JOINs, audit trails. SQL is the only sensible choice. Galera for high availability.
🚂
Transport Booking System → MariaDB + Redis
Booking and seat inventory → MariaDB (structured, transactional). Real-time departure boards → Redis (in-memory pub/sub, sub-millisecond updates). Two databases, each doing its job.
📊
Analytics Dashboard → ClickHouse / BigQuery
Aggregating billions of events — row-oriented SQL is slow here. Columnar databases scan only the columns needed, 100× faster for GROUP BY / SUM queries at scale.
⚡
API Rate Limiting & Sessions → Redis
Needs sub-millisecond response on every API call. Redis TTL and atomic INCR make this trivial. Don't use a relational database for this.
Section 6

Self-Check Questions

Explanation

The choice depends on the data model and requirements. SQL is the default — ACID guarantees and relational integrity are hard to replicate at the application level, and SQL expertise transfers across every engine. NoSQL is appropriate when data is genuinely unstructured, when horizontal write scaling beyond what sharding can provide is needed, or for a specific pattern like Redis for caching or a graph DB for relationship traversal. For a booking system — transactional, relational data with inventory integrity — a relational database like MariaDB is the correct answer.

Common misconceptions

"NoSQL scales better" is an oversimplification. SQL scales well too, with proper indexing, read replicas, and connection pooling. Never pick NoSQL "because it's modern" when the data is fundamentally relational.

Explanation

ACID is the set of properties that make database transactions reliable. Atomicity means all steps in a transaction succeed or none do — no partial writes. Consistency means the database always goes from one valid state to another. Isolation means concurrent transactions don't interfere — MariaDB's default REPEATABLE READ isolation prevents dirty reads. Durability means a committed transaction survives a crash — guaranteed by the InnoDB redo log. For systems handling financial data, medical records, or critical bookings, these properties are not optional — they are fundamental correctness requirements.

Explanation

MariaDB was forked from MySQL in 2009 by MySQL's original developer Michael Widenius after Oracle acquired Sun Microsystems, which owned MySQL. The concern was that Oracle's commercial interests might conflict with open development. Practically, MariaDB adds Galera multi-master clustering as a first-class feature, uses ed25519 by default for stronger authentication, supports the RETURNING clause, and is governed by the MariaDB Foundation independently of Oracle. It is ~99% SQL-compatible with MySQL — admin commands and config files are nearly identical.

⚑ Key Takeaways
⚑MariaDB is NOT just a MySQL rename — it has Galera multi-master clustering, ed25519 auth, the RETURNING clause, and is governed by the MariaDB Foundation independent of Oracle.
⚑ACID's Durability is guaranteed by the InnoDB redo log (WAL). innodb_flush_log_at_trx_commit=1 is the only setting that fully preserves D. Never use 0 or 2 on production systems where data loss is unacceptable.
⚑MariaDB's default isolation level is REPEATABLE READ (same as MySQL). PostgreSQL defaults to READ COMMITTED. Understanding both is essential for working across database engines.
⚑Redis is not a primary database replacement — it is an in-memory cache/queue layer. The typical pattern is a relational database for persistent data and Redis for sub-millisecond caching and pub-sub.
⚑The "NoSQL scales better" claim is misleading. SQL with proper indexes, read replicas, and connection pooling handles most workloads. Choose NoSQL for a specific, measured reason — not speculation.

Abschnitt 1

Was ist eine Datenbank?

Eine Datenbank ist eine organisierte Sammlung von Daten, die effizient gespeichert, abgerufen und geändert werden kann. Das trifft auch auf ein Tabellenblatt zu — der entscheidende Unterschied liegt im Maßstab: viele gleichzeitige Benutzer, Milliarden von Zeilen und garantierte Datenintegrität.

MerkmalTabellenkalkulationDatenbank
Datenvolumen~1 Mio. Zeilen bis zum AbsturzMilliarden Zeilen, kein Problem
Gleichzeitige Benutzer1 Benutzer bearbeitet sicherTausende gleichzeitig
DatenintegritätJede Eingabe möglichTypen, Constraints, Fremdschlüssel erzwungen
AbfragegeschwindigkeitLangsam bei großen DatenIndizierte Abfragen in Millisekunden
AbsturzsicherheitRisiko der DateibeschädigungWAL, Transaktionen, ACID
A — Atomarität: Alles oder nichts
Eine Transaktion wird entweder vollständig ausgeführt oder vollständig zurückgerollt. Kein partielles Schreiben.
C — Konsistenz: Von einem gültigen Zustand zum nächsten
Eine Transaktion überführt die Datenbank immer von einem gültigen Zustand in einen anderen. Regeln (Constraints, Fremdschlüssel) werden nie verletzt.
I — Isolation: Nebenläufige Transaktionen stören sich nicht
Mehrere Benutzer sehen konsistente Ergebnisse. Nicht abgeschlossene Änderungen von Transaktion A sind für Transaktion B unsichtbar.
D — Dauerhaftigkeit: Bestätigte Daten überleben Abstürze
Sobald die Datenbank COMMIT bestätigt, sind die Daten auf Disk — garantiert durch das Write-Ahead Log (WAL / Redo-Log in InnoDB).
Abschnitt 2

Das relationale Modell & SQL

Relationale Datenbanken organisieren Daten in Tabellen mit Zeilen und Spalten, die über Schlüssel verknüpft und mit SQL abgefragt werden. Dieses Modell betreibt Banken, Krankenhäuser, E-Commerce und nahezu jedes Unternehmenssystem der Welt.

Primärschlüssel (PRIMARY KEY)
Eindeutiger Identifikator für jede Zeile — niemals NULL, niemals doppelt. Ermöglicht das genaue Auffinden und Referenzieren genau einer Zeile.
Fremdschlüssel (FOREIGN KEY)
Eine Spalte, die auf den Primärschlüssel einer anderen Tabelle verweist. Verknüpft Tabellen und erzwingt referenzielle Integrität.
Index
Vorberechnete Suchstruktur auf einer oder mehreren Spalten. Wandelt O(n) Tabellenscans in O(log n) Suchoperationen um.
Kein Duplizieren

Statt einer riesigen Tabelle mit duplizierten Daten: kleinere, spezialisierte Tabellen, verknüpft über Schlüssel. Aktualisierung an einer Stelle — alle Referenzen bleiben automatisch konsistent.

Abschnitt 3

SQL vs. NoSQL — Entscheidungsrahmen

1. Wie strukturiert sind die Daten?
Konsistente Spalten, klar definiertes Schema → SQL. Unterschiedliche Felder pro Datensatz, verschachtelt, hierarchisch → NoSQL.
2. Werden Transaktionen benötigt?
Geld, medizinische Akten, Bestellungen → SQL/ACID. Logs, Analysen, Caches → NoSQL.
3. Wie verknüpft sind die Daten?
Viele Beziehungen, JOINs, Fremdschlüssel-Integrität → SQL. Eigenständige Datensätze pro Dokument/Schlüssel → NoSQL.
Kernentscheidung: SQL oder NoSQL?

Die Wahl hängt vom Datenmodell ab. SQL ist der Standardfall — ACID-Garantien sind schwer auf Anwendungsebene nachzubauen, und SQL-Kenntnisse übertragen sich auf jede Engine. NoSQL ist sinnvoll, wenn Daten genuint unstrukturiert sind, horizontale Schreibskalierung benötigt wird oder ein spezielles Zugriffsmuster vorliegt — etwa Redis für Caching. In den meisten Produktivsystemen kommen beide Typen zum Einsatz.

⚑ Wichtigste Erkenntnisse
⚑MariaDB ist KEIN einfaches MySQL-Umbenennungs-Projekt — es hat Galera-Clustering, ed25519-Auth, RETURNING und wird unabhängig von Oracle durch die MariaDB Foundation verwaltet.
⚑ACID's Dauerhaftigkeit (D) wird durch das InnoDB-Redo-Log (WAL) garantiert. innodb_flush_log_at_trx_commit=1 ist die einzige vollständig ACID-konforme Einstellung — niemals 0 oder 2 auf Produktivsystemen verwenden, wo Datenverlust inakzeptabel ist.
⚑MariaDB's Standard-Isolationslevel ist REPEATABLE READ. PostgreSQL's Standard ist READ COMMITTED. Beide zu kennen ist grundlegend für die Arbeit mit verschiedenen Datenbank-Engines.
⚑Redis ist kein Ersatz für eine Primärdatenbank, sondern ein In-Memory-Cache/Warteschlange. Das typische Muster: relationale Datenbank für persistente Daten und Redis für Sub-Millisekunden-Caching und Pub-Sub.
