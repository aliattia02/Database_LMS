Filesystem Navigation & Key Directories

Linux has a single unified tree rooted at /. Knowing where databases, logs, configs, and binaries live saves you minutes every incident.

PathContentsDBA relevance
/etc/my.cnf.d/MariaDB configuration fragmentsFirst stop for every tuning change
/var/lib/mysql/InnoDB data & redo logs, binlogsWhere your data actually lives
/var/log/mariadb/Error log, slow query logFirst stop during incident response
/var/log/System journals, auth.log, messagesOS-level errors, OOM events
/etc/All system configuration filesfirewalld, SELinux, network configs
/usr/bin/ /usr/sbin/Executables (mariadb, mysqladmin…)Know where binaries are, for cron & scripts
/tmp/ /var/tmp/Temp files — purged on rebootMariaDB uses for sort/tmp tables
/proc/sys/Live kernel parametersTune without reboot: swappiness, vm.dirty_*
bash — navigation
# Where am I? What's here? pwd # print working directory ls -lah /var/lib/mysql/ # list with human-readable sizes ls -lt /var/lib/mysql/ | head # newest files first # Move around cd /var/lib/mysql cd - # jump back to previous directory # Find something fast find /etc -name "*.cnf" # all MariaDB config fragments find /var/lib/mysql -name "*.ibd" -newer /tmp/ref # recently modified tables # Disk usage — critical for DBA df -h # free space on all mounts du -sh /var/lib/mysql/* # which databases use the most space du -sh /var/lib/mysql/ # total data directory size
Common DBA mistake

Filling /var/lib/mysql/ causes MariaDB to crash with ERROR 28: No space left on device. Run df -h and du -sh /var/lib/mysql/ before every mariabackup run.

Section 2

Reading Files & Logs

A DBA spends a significant portion of every incident reading logs. Know how to get to the relevant part of a large log file instantly.

bash — log reading
# tail: see the live log (like watching a live feed) tail -f /var/log/mariadb/mariadb.log # follow as new lines arrive tail -n 200 /var/log/mariadb/mariadb.log # last 200 lines # grep: find the signal in the noise grep -i "error" /var/log/mariadb/mariadb.log grep -i "galera\|wsrep\|sst" /var/log/mariadb/mariadb.log # Galera events grep -c "Aborted" /var/log/mariadb/mariadb.log # count aborted connections # Combined: live filtered stream tail -f /var/log/mariadb/mariadb.log | grep --line-buffered "ERROR\|WARN" # less: navigate a huge file less /var/log/mariadb/mariadb.log # Inside less: G=end g=top /pattern=search n=next q=quit # journald: system-level logs for MariaDB service journalctl -u mariadb --since "2 hours ago" journalctl -u mariadb -p err # errors only journalctl -u mariadb -f # follow live
Section 3

systemd — Service Management

MariaDB on RHEL 8/9 runs as a systemd unit. Every DBA command set starts and ends with systemctl.

bash — systemctl
# Status — is it running? Why did it fail? systemctl status mariadb systemctl is-active mariadb # exits 0 if active, 3 if not systemctl is-enabled mariadb # will it start on boot? # Lifecycle systemctl start mariadb systemctl stop mariadb systemctl restart mariadb # full stop + start systemctl reload mariadb # reload config without full restart (if supported) # Boot behaviour systemctl enable mariadb # start on boot systemctl disable mariadb # do not start on boot # Other units (firewall, MaxScale) systemctl status firewalld systemctl status maxscale # Logs via journalctl journalctl -u mariadb -n 50 --no-pager # last 50 lines
Incident first step

Always run systemctl status mariadb before anything else. The status output shows the last few log lines and the exit code — 90% of the time it tells you exactly what went wrong.

bash
# View the unit file systemctl cat mariadb # Override without editing the original (survives package upgrades) systemctl edit mariadb # Creates /etc/systemd/system/mariadb.service.d/override.conf # Example: add an environment variable or LimitNOFILE increase
override.conf — raise open file limit
[Service] LimitNOFILE=65536 # needed when open_files_limit is set high in my.cnf TimeoutStartSec=300 # give MariaDB time to perform SST before systemd kills it
TimeoutStartSec — Galera gotcha

During Galera SST (full state transfer), MariaDB can take several minutes to start. If systemd's TimeoutStartSec is too short (default 90s), systemd kills the node before SST finishes. Always set it to at least 300 or infinity on Galera nodes.

Section 4

Package Management — dnf & rpm

MariaDB, MaxScale, and their dependencies are installed and updated via dnf (RHEL 8/9). Knowing rpm for verification and troubleshooting is equally important.

bash — dnf
# Query and information dnf info mariadb-server # version, repo, size dnf list installed | grep maria # what's installed dnf repolist # active repositories # Install dnf install -y mariadb-server mariadb-backup dnf install -y maxscale # from MariaDB Enterprise repo # Update safely (single package) dnf update mariadb-server --setopt=protected_packages= # Hold a package at current version dnf versionlock add mariadb-server # requires dnf-plugin-versionlock # Clean up dnf clean all # clear metadata cache dnf autoremove # remove unused dependencies
bash — rpm
# What version is installed? rpm -q mariadb-server # e.g. mariadb-server-10.11.4-1.el9 # Which package owns a file? rpm -qf /usr/sbin/mysqld # useful when debugging "where did this come from" # List all files in a package rpm -ql mariadb-server # all files installed by the package # Verify file integrity (detects tampering or corruption) rpm -V mariadb-server # silent = OK; output = changed file
Section 5

Networking for DBAs

Galera uses ports 3306, 4444, 4567, and 4568. MaxScale listens on 3306 or 3307. If any of these are blocked, the cluster fails. Knowing how to inspect and configure the firewall is non-negotiable.

PortProtocolUsed byBlock this and…
3306TCPMariaDB client connectionsNo apps can connect
4444TCPGalera SST (full state transfer)New nodes can never join
4567TCP+UDPGalera replication traffic (wsrep)Cluster loses quorum immediately
4568TCPGalera IST (incremental state transfer)IST falls back to slow SST
3307TCPMaxScale listener (read/write split)Apps lose the proxy layer
6603TCPMaxScale REST API & maxctrlmaxctrl commands fail
bash — firewalld
# Check current state firewall-cmd --list-all # Open ports permanently (Galera cluster node) firewall-cmd --permanent --add-port=3306/tcp firewall-cmd --permanent --add-port=4444/tcp firewall-cmd --permanent --add-port=4567/tcp firewall-cmd --permanent --add-port=4567/udp firewall-cmd --permanent --add-port=4568/tcp firewall-cmd --reload # Verify a specific port is open firewall-cmd --query-port=4567/tcp # returns yes/no # Check what's actually listening (from OS side) ss -tlnp # all TCP listening + PID ss -tlnp | grep "3306\|4567\|4444" # Test connectivity from another node nc -zv node2.internal 4567 # is port 4567 reachable? telnet node2.internal 3306 # quick MariaDB connectivity test
Always --permanent + --reload

Without --permanent, the firewall rule is lost after the next firewall-cmd --reload or system reboot. Without --reload, the permanent rule is not active yet.

Section 6

Permissions & SELinux

SELinux on RHEL 8/9 runs in enforcing mode by default. It blocks MariaDB from accessing non-standard paths and ports silently — your database just fails with a cryptic error. Know how to check and fix SELinux denials without disabling it.

enforcing
SELinux actively blocks policy violations. The correct production state — never disable SELinux on KRITIS systems.
permissive
SELinux logs violations but does not block them. Use temporarily to diagnose — not a permanent fix.
disabled
SELinux completely off. Requires reboot to re-enable. Never acceptable on production KRITIS infrastructure.
bash — SELinux for DBAs
# Check current mode getenforce # Enforcing / Permissive / Disabled sestatus # detailed status + policy # Check for recent denials (the AVC audit log) ausearch -m avc -ts recent # recent denials grep "denied" /var/log/audit/audit.log | grep mysql # Generate a fix policy from the denials (use with care) ausearch -m avc -ts recent | audit2allow -M mariadb_custom semodule -i mariadb_custom.pp # Fix: relabel a non-standard data directory semanage fcontext -a -t mysqld_db_t '/data/mysql(/.*)?' restorecon -Rv /data/mysql/ # Temporarily switch to permissive for diagnosis only setenforce 0 # permissive (not persistent across reboot) setenforce 1 # back to enforcing
Never permanently disable SELinux

On OS GmbH KRITIS infrastructure, setenforce 0 is a temporary diagnostic step only. Setting SELINUX=disabled in /etc/selinux/config violates the security baseline. Use audit2allow to build a proper policy module instead.

Section 7

Bash Scripting Essentials for DBAs

Every DBA at OS GmbH writes scripts — health checks, backup wrappers, cluster monitoring, on-call alerts. The patterns below cover 90% of real DBA scripts.

bash
#!/usr/bin/env bash # mariadb-healthcheck.sh — checks all Galera nodes are in sync set -euo pipefail # -e exit on error, -u unset var = error, -o pipefail IFS=$'\n\t' # Constants — edit here, not scattered through the script readonly DB_HOST="127.0.0.1" readonly DB_USER="healthcheck" readonly DB_PASS_FILE="/etc/mariadb/.healthcheck.pwd" readonly LOG_FILE="/var/log/mariadb/healthcheck.log" # Logging helper log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG_FILE"; } # Read password safely (never put passwords in scripts) DB_PASS=$(<"$DB_PASS_FILE") # Run a query and capture output wsrep_state=$(mysql -h "$DB_HOST" -u "$DB_USER" -p"$DB_PASS" -sNe \ "SHOW GLOBAL STATUS LIKE 'wsrep_local_state_comment'" 2>&1 | awk '{print $2}') if [[ "$wsrep_state" != "Synced" ]]; then log "ERROR: wsrep_local_state_comment = $wsrep_state (expected Synced)" exit 1 fi log "OK: node is Synced"
bash — crontab
# Edit crontab for the mariadb system user crontab -u mysql -e # Format: minute hour day-of-month month day-of-week command # * * * * * /path/to/script # Full backup every night at 01:30 30 1 * * * /usr/local/bin/mariadb-backup-full.sh >> /var/log/mariadb/backup.log 2>&1 # Incremental backup every hour during business hours 0 8-18 * * 1-5 /usr/local/bin/mariadb-backup-incr.sh >> /var/log/mariadb/backup.log 2>&1 # Health check every 5 minutes */5 * * * * /usr/local/bin/mariadb-healthcheck.sh # List current cron jobs crontab -u mysql -l
Section 8

Interview Questions

This is the most common on-call scenario question
Sample answer

"First: systemctl status mariadb — this tells me if the service failed to start and shows the last few journal lines with the exit code. Second: journalctl -u mariadb -n 100 --no-pager to see the full startup log. Third: tail -n 200 /var/log/mariadb/mariadb.log for the MariaDB error log — this usually shows the root cause. Fourth: df -h to rule out a full disk. Fifth: ausearch -m avc -ts recent for SELinux denials if the error mentions permission denied. By step five, I've diagnosed 99% of startup failures."

Sample answer

"With firewall-cmd. I add the four Galera ports with --permanent and then reload: ports 3306 TCP (client), 4444 TCP (SST), 4567 TCP+UDP (wsrep replication), and 4568 TCP (IST). The --permanent flag makes the rule survive reboots and reloads. Without it, the next reload wipes the rule. I verify with firewall-cmd --list-all and ss -tlnp to confirm MariaDB is actually listening on those ports."

Sample answer

"I never disable SELinux. I check the audit log with ausearch -m avc -ts recent | grep mysql to confirm the denial, then I apply the correct SELinux file context to the new directory: semanage fcontext -a -t mysqld_db_t '/newpath(/.*)?' followed by restorecon -Rv /newpath/. This labels the files with the policy MariaDB expects. If a custom context is needed, I'd generate a module with audit2allow."

Never say this in the interview

"I just set SELINUX=disabled in the config." On KRITIS infrastructure this is a security violation, not a solution.

⚑ Must-Know Before the Interview
⚑Galera requires 4 ports open: 3306 (client), 4444 (SST), 4567 TCP+UDP (wsrep), 4568 (IST). Block any one and the cluster fails silently.
⚑systemctl status mariadb is always your first command during an incident. Never start guessing before reading it.
⚑SELinux: never disable it on KRITIS production. Use semanage fcontext + restorecon for path issues, audit2allow for custom policy.
⚑Add TimeoutStartSec=300 (or infinity) to the systemd MariaDB override on all Galera nodes — default 90s is not enough for SST.
⚑Always use set -euo pipefail at the top of every bash script. It turns silent failures into visible errors — essential for production automation.
⚑df -h before every backup. A full disk is the single most common cause of DBA incidents at 3am — it is also the most embarrassing.

Abschnitt 1

Dateisystem-Navigation & Schlüsselverzeichnisse

Linux hat einen einzigen, einheitlichen Verzeichnisbaum, der bei / beginnt. Wer weiß, wo Datenbankdaten, Logs, Konfigurationen und Binärdateien liegen, spart bei jedem Vorfall wertvolle Minuten.

/etc/my.cnf.d/ — Konfiguration
MariaDB-Konfigurationsfragmente. Erste Anlaufstelle für jede Tuning-Änderung.
/var/lib/mysql/ — Datendirectory
InnoDB-Datendateien, Redo-Logs, Binary-Logs. Hier liegen die eigentlichen Daten.
/var/log/mariadb/ — Logs
Fehlerprotokoll, Slow-Query-Log. Erste Anlaufstelle bei der Incident-Analyse.
/proc/sys/ — Live-Kernel-Parameter
Kernel-Parameter im Betrieb tunen (swappiness, vm.dirty_*) — ohne Neustart.
Häufiger DBA-Fehler

Wenn /var/lib/mysql/ voll läuft, bricht MariaDB mit ERROR 28: No space left on device ab. Immer df -h und du -sh /var/lib/mysql/ vor jedem Backup prüfen.

Abschnitt 2

systemd — Dienstverwaltung

MariaDB auf RHEL 8/9 läuft als systemd-Unit. Jeder DBA-Befehlssatz beginnt und endet mit systemctl.

systemctl status mariadb
Erster Befehl bei jedem Vorfall. Zeigt ob der Dienst läuft, letzte Log-Zeilen und den Exit-Code.
systemctl start / stop / restart mariadb
Dienst starten, stoppen, neu starten. restart = vollständiger Stopp + Start.
systemctl enable / disable mariadb
Festlegen, ob MariaDB beim Systemstart automatisch gestartet wird.
journalctl -u mariadb -f
Live-Stream der systemd-Logs für den MariaDB-Dienst — entspricht tail -f auf Systemebene.
Galera: TimeoutStartSec erhöhen

Während Galera SST (vollständige Zustandsübertragung) kann MariaDB mehrere Minuten zum Starten brauchen. Wenn systemd's TimeoutStartSec zu kurz ist (Standard: 90s), beendet systemd den Node vor dem Ende des SST. Auf Galera-Nodes mindestens 300 oder infinity setzen.

Abschnitt 3

Netzwerk & Firewall für DBAs

Galera benötigt die Ports 3306, 4444, 4567 und 4568. MaxScale hört auf 3306 oder 3307. Wenn einer dieser Ports blockiert ist, schlägt der Cluster stillschweigend fehl.

PortVerwendet vonWenn blockiert…
3306/tcpMariaDB-ClientsKeine App kann verbinden
4444/tcpGalera SSTNeue Nodes können nie beitreten
4567/tcp+udpwsrep-ReplikationCluster verliert sofort das Quorum
4568/tcpGalera ISTIST fällt auf langsamen SST zurück
Immer --permanent + --reload

Ohne --permanent geht die Firewallregel beim nächsten Reload oder Neustart verloren. Ohne --reload ist die permanente Regel noch nicht aktiv.

Abschnitt 4

SELinux — sicher verwalten, nie deaktivieren

Musterlösung im Vorstellungsgespräch

„Ich deaktiviere SELinux nie. Ich prüfe das Audit-Log mit ausearch -m avc -ts recent | grep mysql zur Bestätigung des Denials. Dann weise ich dem neuen Verzeichnis den richtigen SELinux-Dateikontext zu: semanage fcontext -a -t mysqld_db_t '/neupfad(/.*)?' und anschließend restorecon -Rv /neupfad/. Bei Bedarf erstelle ich ein eigenes Richtlinienmodul mit audit2allow."

Nie im Vorstellungsgespräch sagen

„Ich setze einfach SELINUX=disabled." Auf KRITIS-Infrastruktur ist das ein Sicherheitsverstoß, keine Lösung.
