Section 2

DDL — Defining the Schema

DDL (Data Definition Language) is how you create, alter, and drop tables and databases. The schema you define here determines every query, index, and join you'll write for the life of the application.

INT / BIGINT
Whole numbers. Use INT for IDs <2.1B rows. BIGINT for anything larger, or for Unix timestamps.
VARCHAR(n)
Variable-length string up to n bytes. Best for names, emails. Don't use TEXT unless you need >65,535 chars.
DECIMAL(p,s)
Exact decimal. Use for money — never FLOAT or DOUBLE for currency (floating-point rounding errors).
DATETIME / TIMESTAMP
TIMESTAMP stores in UTC and converts to session timezone. DATETIME stores literally. Use TIMESTAMP for audit fields.
BOOLEAN / TINYINT(1)
MariaDB has no native BOOL — it's TINYINT(1). TRUE=1, FALSE=0.
ENUM
Fixed set of string values. Use sparingly — adding a value requires ALTER TABLE, which is slow on large tables.
Common mistakes
  • Using FLOAT/DOUBLE for money — causes rounding errors. Use DECIMAL(10,2).
  • Using TEXT instead of VARCHAR when you know the max length — TEXT can't be fully indexed.
  • Using CHAR for variable-length data — wastes space with trailing spaces.
SQL — CREATE TABLE
CREATE TABLE patients ( patient_id INT UNSIGNED NOT NULL AUTO_INCREMENT, first_name VARCHAR(80) NOT NULL, last_name VARCHAR(80) NOT NULL, date_of_birth DATE NOT NULL, email VARCHAR(255) NULL, active TINYINT(1) NOT NULL DEFAULT 1, created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP, PRIMARY KEY (patient_id), UNIQUE KEY uq_email (email) -- email must be unique if provided ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci; CREATE TABLE appointments ( appt_id INT UNSIGNED NOT NULL AUTO_INCREMENT, patient_id INT UNSIGNED NOT NULL, doctor_id INT UNSIGNED NOT NULL, scheduled_at DATETIME NOT NULL, status ENUM('pending','confirmed','cancelled') NOT NULL DEFAULT 'pending', notes TEXT NULL, PRIMARY KEY (appt_id), KEY idx_patient (patient_id), -- index for JOIN speed KEY idx_doctor (doctor_id), KEY idx_scheduled (scheduled_at), CONSTRAINT fk_appt_patient FOREIGN KEY (patient_id) REFERENCES patients(patient_id) ) ENGINE=InnoDB;
Always add ENGINE=InnoDB

InnoDB is the default in MariaDB 10.x but state it explicitly. It's the only engine that supports foreign keys, ACID transactions, and row-level locking — all of which matter for production systems.

SQL — ALTER TABLE
-- Add a column ALTER TABLE patients ADD COLUMN phone VARCHAR(20) NULL AFTER email; -- Change a column's type ALTER TABLE patients MODIFY COLUMN first_name VARCHAR(120) NOT NULL; -- Rename a column (MariaDB 10.5+) ALTER TABLE patients RENAME COLUMN email TO email_address; -- Add an index ALTER TABLE appointments ADD INDEX idx_status (status); -- Drop a column ALTER TABLE patients DROP COLUMN phone;
ALTER TABLE locks tables on large data

On a table with millions of rows, an ALTER TABLE can lock reads and writes for minutes. In production, use pt-online-schema-change (Percona Toolkit) or MariaDB's ALGORITHM=INSTANT (for supported operations) to avoid downtime.

Section 3

SELECT — Querying Data

The SELECT statement is the most important SQL command. Understanding the logical order of execution (not the written order) is the key to writing correct, efficient queries.

Written order vs execution order

You write SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT, but MariaDB processes them in a completely different order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. This is why you can't use a column alias defined in SELECT inside a WHERE clause — WHERE runs before SELECT.

#ClauseWhat it doesCan reference…
1FROM / JOINIdentifies the source tables and builds the combined datasetTable names, aliases
2WHEREFilters rows before any groupingTable columns — NOT SELECT aliases
3GROUP BYGroups remaining rows into bucketsTable columns
4HAVINGFilters grouped results (post-aggregation WHERE)Aggregate functions + table columns
5SELECTChooses and transforms columns to returnAnything — aliases defined here
6ORDER BYSorts the final result setTable columns + SELECT aliases ✓
7LIMITTruncates result to n rows—
SQL — SELECT patterns
-- Basic: all columns, all rows SELECT * FROM patients; -- Column selection with alias SELECT patient_id AS id, CONCAT(first_name, ' ', last_name) AS full_name, TIMESTAMPDIFF(YEAR, date_of_birth, CURDATE()) AS age FROM patients WHERE active = 1 ORDER BY last_name, first_name; -- Filter with pattern matching SELECT * FROM patients WHERE email LIKE '%@fraport.de'; -- Fraport employees -- NULL handling — must use IS NULL, not = NULL SELECT * FROM patients WHERE email IS NULL; SELECT * FROM patients WHERE email IS NOT NULL; SELECT COALESCE(email, 'no email') FROM patients; -- Range and IN SELECT * FROM appointments WHERE scheduled_at BETWEEN '2025-01-01' AND '2025-01-31' AND status IN ('confirmed', 'pending'); -- Pagination SELECT * FROM patients ORDER BY created_at DESC LIMIT 20 OFFSET 40; -- page 3 (0-indexed), 20 per page
Section 4

Aggregations — GROUP BY, COUNT, SUM, AVG

Aggregation transforms many rows into summary statistics — the foundation of reports, dashboards, and analytics queries.

SQL — aggregations
-- How many appointments per doctor? SELECT doctor_id, COUNT(*) AS total_appts, COUNT(CASE WHEN status = 'confirmed' THEN 1 END) AS confirmed, COUNT(CASE WHEN status = 'cancelled' THEN 1 END) AS cancelled FROM appointments WHERE scheduled_at >= '2025-01-01' GROUP BY doctor_id HAVING total_appts > 10 -- HAVING filters groups (not rows) ORDER BY total_appts DESC; -- Daily appointment count for the last 30 days SELECT DATE(scheduled_at) AS appt_date, COUNT(*) AS daily_count FROM appointments WHERE scheduled_at >= DATE_SUB(CURDATE(), INTERVAL 30 DAY) GROUP BY DATE(scheduled_at) ORDER BY appt_date;
WHERE vs HAVING — the most tested interview distinction

WHERE filters individual rows before grouping — it can reference table columns but NOT aggregate functions. HAVING filters groups after aggregation — it can reference aggregate functions. Using WHERE COUNT(*) > 10 is a syntax error. Use HAVING.

Section 5

JOINs — Combining Tables

Joins are the core of relational database power. Most production queries involve at least one join — understanding exactly what rows each type returns prevents hours of debugging.

INNER JOIN — matching rows only
Returns rows that have a matching value in BOTH tables. If a patient has no appointment, they won't appear. Most common join type.
LEFT JOIN — all rows from left, matched from right
Returns ALL rows from the left table, with NULL for right-table columns when there's no match. Use to find patients with no appointments.
RIGHT JOIN — all rows from right, matched from left
Same as LEFT JOIN with tables swapped. Rarely used — rewrite as LEFT JOIN for readability.
SQL — JOIN examples
-- INNER JOIN: appointments with patient names (only matched rows) SELECT a.appt_id, a.scheduled_at, CONCAT(p.first_name, ' ', p.last_name) AS patient_name, a.status FROM appointments a INNER JOIN patients p ON a.patient_id = p.patient_id WHERE a.status = 'confirmed' ORDER BY a.scheduled_at; -- LEFT JOIN: all patients, even those with no appointments SELECT p.patient_id, CONCAT(p.first_name, ' ', p.last_name) AS patient_name, COUNT(a.appt_id) AS appt_count FROM patients p LEFT JOIN appointments a ON p.patient_id = a.patient_id GROUP BY p.patient_id, p.first_name, p.last_name ORDER BY appt_count DESC; -- Find patients with NO appointments (anti-join pattern) SELECT p.* FROM patients p LEFT JOIN appointments a ON p.patient_id = a.patient_id WHERE a.appt_id IS NULL; -- NULL here means no match found
Interview tip on JOINs

The NULL-checking anti-join pattern (LEFT JOIN … WHERE right.id IS NULL) comes up in almost every SQL interview. Know it cold.

SQL — three-table JOIN
-- Appointments with both patient name and doctor name SELECT a.appt_id, a.scheduled_at, CONCAT(p.first_name, ' ', p.last_name) AS patient, CONCAT(d.first_name, ' ', d.last_name) AS doctor, a.status FROM appointments a INNER JOIN patients p ON a.patient_id = p.patient_id INNER JOIN doctors d ON a.doctor_id = d.doctor_id WHERE a.scheduled_at BETWEEN '2025-03-01' AND '2025-03-31' ORDER BY a.scheduled_at;
Section 6

DML — Modifying Data

INSERT, UPDATE, and DELETE modify the data in your tables. Knowing their safe patterns — especially limiting UPDATE and DELETE — prevents production disasters.

SQL — DML patterns
-- INSERT: single row INSERT INTO patients (first_name, last_name, date_of_birth, email) VALUES ('Fatima', 'Al-Rashid', '1990-03-15', 'f.alrashid@example.com'); -- INSERT: multiple rows (more efficient than one at a time) INSERT INTO patients (first_name, last_name, date_of_birth) VALUES ('Ahmed', 'Hassan', '1985-07-22'), ('Leila', 'Müller', '1978-11-08'), ('Mohamed', 'Schmidt', '1995-02-14'); -- INSERT … ON DUPLICATE KEY UPDATE (upsert) INSERT INTO patients (patient_id, email) VALUES (1001, 'new@email.com') ON DUPLICATE KEY UPDATE email = VALUES(email); -- UPDATE: always use WHERE — limit rows affected explicitly UPDATE patients SET active = 0, email = 'archived@void.invalid' WHERE patient_id = 1001 -- ← NEVER omit WHERE on UPDATE LIMIT 1; -- safety net: never update more than 1 row -- DELETE: preview with SELECT first, then delete SELECT * FROM appointments WHERE status = 'cancelled' AND scheduled_at < '2024-01-01'; -- run SELECT first to confirm rows DELETE FROM appointments WHERE status = 'cancelled' AND scheduled_at < '2024-01-01'; -- then delete with identical WHERE
The most dangerous SQL mistake

UPDATE patients SET active = 0; without a WHERE clause deactivates EVERY patient. Always write SELECT + WHERE first, verify the row count, then convert to UPDATE/DELETE with the same WHERE clause. In a transaction if possible — so you can ROLLBACK if something's wrong.

Section 7

Transactions — The Safety Net

A transaction wraps multiple SQL statements into an atomic unit. Either all succeed together, or all are rolled back together. Essential for any operation that modifies related rows across tables.

SQL — transaction with error handling pattern
-- Safe multi-step operation: cancel appointment + log the cancellation START TRANSACTION; UPDATE appointments SET status = 'cancelled' WHERE appt_id = 5001 LIMIT 1; -- If affected rows = 0, something's wrong — ROLLBACK -- (In application code, check ROW_COUNT() here) INSERT INTO audit_log (entity, entity_id, action, performed_by, ts) VALUES ('appointment', 5001, 'cancelled', 'dr.schmidt', NOW()); COMMIT; -- both changes are written to disk atomically -- If anything above fails (or you discover a mistake): -- ROLLBACK; -- undoes everything since START TRANSACTION
MariaDB autocommit

By default, MariaDB runs in autocommit=ON — every statement is its own instant transaction. Once you issue START TRANSACTION, autocommit is suspended until you COMMIT or ROLLBACK. Forgetting COMMIT leaves the connection holding locks — other sessions block.

Galera: transaction scope matters even more

In a Galera cluster, a transaction is replicated to all nodes at COMMIT time using write-set replication. Long-running transactions or large write-sets increase certification conflict probability and can trigger flow control, stalling the entire cluster. Keep transactions short and targeted.

LevelDirty ReadNon-Repeatable ReadPhantom ReadUse case
READ UNCOMMITTEDPossiblePossiblePossibleNever in production
READ COMMITTEDProtectedPossiblePossiblePostgreSQL default; high concurrency reporting
REPEATABLE READProtectedProtectedUsually*MariaDB/MySQL default ★
SERIALIZABLEProtectedProtectedProtectedFinancial transactions; very low throughput

* InnoDB's MVCC implementation largely prevents phantoms even at REPEATABLE READ.

SQL — check and set isolation level
SELECT @@transaction_isolation; -- current session level SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED; -- change for session
Section 8

Indexes — Making Queries Fast

An index trades storage and write speed for dramatic read speed gains. Without indexes, every query scans every row — fine for 100 rows, catastrophic for 100 million.

SQL — index creation
-- Single-column index (most common) CREATE INDEX idx_appt_scheduled ON appointments (scheduled_at); -- Composite index — column ORDER matters! -- Useful for: WHERE doctor_id = ? AND status = ? CREATE INDEX idx_doctor_status ON appointments (doctor_id, status); -- Unique index (prevents duplicates) CREATE UNIQUE INDEX uq_patient_email ON patients (email); -- Check index usage with EXPLAIN EXPLAIN SELECT * FROM appointments WHERE scheduled_at BETWEEN '2025-03-01' AND '2025-03-31';
EXPLAIN output — what to look for
+----+-------+------+------+---------------------+------+-------+ | id | table | type | key | key_len | rows | Extra | +----+-------+------+------+---------------------+------+-------+ | 1 | appointments | range | idx_appt_scheduled | 5 | 247 | Using index condition | +----+-------+------+------+---------------------+------+-------+ type=range → Good. Uses index for a range scan. type=ALL → Bad. Full table scan. Add an index or fix the WHERE clause. type=ref → Good. Index lookup on equality condition. rows → Estimated rows examined. Lower = faster.
Common index killers
  • Wrapping an indexed column in a function: WHERE YEAR(scheduled_at) = 2025 forces a full scan — use WHERE scheduled_at BETWEEN '2025-01-01' AND '2025-12-31'
  • Leading wildcard: WHERE last_name LIKE '%müller' can't use a B-tree index. Trailing wildcard LIKE 'müller%' can.
  • Implicit type conversion: WHERE patient_id = '1001' (string vs INT) may prevent index use — match types explicitly.
Section 9

Interview Questions

Interview strategy
This is a screening question — answer it in one confident sentence, then give an example.
Sample answer

"WHERE filters individual rows before grouping; HAVING filters grouped results after aggregation. If I want appointments from January, that's a WHERE clause — it narrows the rows the GROUP BY sees. If I want only doctors who have more than 10 appointments in that period, that's a HAVING clause — it filters the groups that GROUP BY produced. You can't use aggregate functions in WHERE; you must use HAVING."

Classic data quality / GROUP BY interview question
Sample answer

"I'd group by email and use HAVING to filter for any count greater than 1:"

SQL — find duplicates
SELECT email, COUNT(*) AS occurrences FROM patients WHERE email IS NOT NULL GROUP BY email HAVING occurrences > 1 ORDER BY occurrences DESC;
Sample answer

"INNER JOIN returns only rows with a match in both tables. If I INNER JOIN patients to appointments, patients who have never booked an appointment are excluded. LEFT JOIN returns all rows from the left table — patients — plus appointment data where it exists, with NULLs where it doesn't. So if I want to find patients who have never had an appointment, I LEFT JOIN appointments and then filter WHERE appt_id IS NULL. That's the anti-join pattern and it comes up constantly in production data audits."

This tests your EXPLAIN knowledge and systematic thinking
Sample answer

"First I run EXPLAIN on the query to see the execution plan — specifically I look at the type column. If I see 'ALL', that's a full table scan and I need an index. The rows column tells me how many rows MariaDB estimates it needs to examine. If the right index exists but isn't being used, I check whether the WHERE clause wraps the column in a function, or there's an implicit type conversion — both prevent index use. For a query on the slow query log, I check log_queries_not_using_indexes to confirm. Then I add or adjust the index and re-run EXPLAIN to verify the plan improves."

Abschnitt 1

DDL — Das Schema definieren

DDL (Data Definition Language) wird verwendet, um Tabellen und Datenbanken zu erstellen, zu ändern und zu löschen. Das hier definierte Schema bestimmt jede Abfrage, jeden Index und jeden JOIN für die gesamte Lebensdauer der Anwendung.

INT / BIGINT — Ganzzahlen
INT für IDs <2,1 Mrd. Zeilen. BIGINT für größere Mengen oder Unix-Timestamps.
VARCHAR(n) — Zeichenketten variabler Länge
Optimal für Namen und E-Mails. TEXT nur verwenden, wenn mehr als 65.535 Zeichen benötigt werden.
DECIMAL(p,s) — exakte Dezimalzahlen
Immer für Geldbeträge verwenden — niemals FLOAT oder DOUBLE (Rundungsfehler bei Fließkomma).
DATETIME / TIMESTAMP
TIMESTAMP speichert in UTC und konvertiert in die Session-Zeitzone. DATETIME speichert den Wert literal. TIMESTAMP für Audit-Felder verwenden.
Häufige Fehler
  • FLOAT/DOUBLE für Geldbeträge — führt zu Rundungsfehlern. DECIMAL(10,2) verwenden.
  • TEXT statt VARCHAR, wenn die Maximallänge bekannt ist — TEXT kann nicht vollständig indiziert werden.
Abschnitt 2

SELECT — Daten abfragen

Die logische Ausführungsreihenfolge (nicht die Schreibreihenfolge) zu verstehen ist der Schlüssel zum Schreiben korrekter Abfragen.

Die wichtigste Regel

WHERE filtert einzelne Zeilen vor dem Gruppieren — keine Aggregatfunktionen erlaubt. HAVING filtert Gruppen nach der Aggregation — Aggregatfunktionen sind erlaubt. WHERE COUNT(*) > 10 ist ein Syntaxfehler. HAVING verwenden.

Musterantwort im Vorstellungsgespräch

„WHERE filtert Zeilen vor dem Gruppieren, HAVING filtert Gruppen nach der Aggregation. Termine aus Januar zu filtern ist ein WHERE-Ausdruck. Nur Ärzte mit mehr als 10 Terminen zu filtern ist ein HAVING-Ausdruck — es filtert die durch GROUP BY erzeugten Gruppen."

INNER JOIN — nur übereinstimmende Zeilen
Gibt Zeilen zurück, die in BEIDEN Tabellen übereinstimmen. Patienten ohne Termin erscheinen nicht.
LEFT JOIN — alle Zeilen links, übereinstimmende rechts
Gibt ALLE Zeilen der linken Tabelle zurück, mit NULL für Spalten der rechten Tabelle, wenn keine Übereinstimmung vorhanden ist.
Anti-Join-Muster

LEFT JOIN … WHERE rechte.id IS NULL findet Patienten ohne Termine. Dieses Muster kommt in fast jedem SQL-Gespräch vor — auswendig lernen.

Abschnitt 3

Transaktionen — das Sicherheitsnetz

Eine Transaktion bündelt mehrere SQL-Anweisungen zu einer atomaren Einheit. Entweder alle Schritte gelingen gemeinsam, oder alle werden zurückgerollt.

Schlüsselfakt

MariaDB's Standardisolationslevel ist REPEATABLE READ. Das bedeutet: während einer Transaktion sieht man immer dieselben Daten, auch wenn andere Transaktionen Änderungen commiten. PostgreSQL's Standard ist READ COMMITTED — ein wichtiger Unterschied, der im Gespräch getestet wird.

Galera-Besonderheit

In Galera wird eine Transaktion bei COMMIT über alle Knoten repliziert. Lange Transaktionen oder große Write-Sets erhöhen die Wahrscheinlichkeit von Zertifizierungskonflikten und können Flow Control auslösen, wodurch der gesamte Cluster blockiert wird. Transaktionen kurz und gezielt halten.

Abschnitt 4

Indizes — Abfragen beschleunigen

Musterantwort zu „Langsame Abfrage"

„Als erstes führe ich EXPLAIN auf der Abfrage aus, um den Ausführungsplan zu sehen — insbesondere die Spalte type. Sehe ich 'ALL', ist das ein vollständiger Tabellenscan und ich brauche einen Index. Die Spalte rows zeigt, wie viele Zeilen MariaDB schätzt prüfen zu müssen. Wenn der richtige Index existiert aber nicht verwendet wird, prüfe ich, ob die WHERE-Klausel die Spalte in eine Funktion einbettet oder eine implizite Typkonvertierung stattfindet — beides verhindert die Indexnutzung."

Drei Index-Killer
  • Funktion um indizierte Spalte: YEAR(scheduled_at) = 2025 → erzwingt Tabellenscan
  • Führender Wildcard: LIKE '%müller' → B-Tree-Index kann nicht verwendet werden
  • Implizite Typkonvertierung: Zeichenkette mit INTEGER-Spalte vergleichen
⚑ Vor dem Gespräch sicher wissen
⚑WHERE filtert Zeilen, HAVING filtert Gruppen. Aggregatfunktionen sind in WHERE nicht erlaubt. Dieser Unterschied wird in fast jedem SQL-Gespräch getestet.
⚑NULL-Vergleiche erfordern IS NULL / IS NOT NULL. = NULL ergibt immer NULL (unbekannt) — niemals TRUE.
⚑EXPLAIN-Spalte type: ALL = Tabellenscan (schlecht), range/ref = Index verwendet (gut). Erste Prüfung bei langsamen Abfragen.
⚑MariaDB's Standard-Isolationslevel ist REPEATABLE READ. PostgreSQL's Standard ist READ COMMITTED. Beide kennen — wird im Gespräch gefragt.
⚑Niemals FLOAT oder DOUBLE für Geldbeträge — Fließkomma-Rundungsfehler korrumpieren Finanzdaten. DECIMAL(10,2) verwenden.
⚑In Galera: Transaktionen kurz halten. Lange Transaktionen erzeugen Zertifizierungskonflikte und können Flow Control auslösen — blockiert den gesamten Cluster.
