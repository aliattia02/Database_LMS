Lesson 1 · 5h

Window Functions

ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, SUM OVER, PARTITION BY, and FRAME clauses — the single most impactful SQL feature to learn after JOINs. They appear in virtually every analytics query in a professional environment.

Window functions let you perform calculations across rows related to the current row — without collapsing them like GROUP BY does. In CareMate, you rank patients by vital readings per city without losing individual rows. In OS GmbH's booking system, you compute running revenue totals and detect price anomalies — all in a single pass over the data.

Michael Nietsch will probe whether you can read a business requirement and map it to the right window function. Knowing when to use LAG vs a self-join, or DENSE_RANK vs ROW_NUMBER, signals genuine experience.

🎯 Interview Anchor

"Window functions return a value per row while still being able to see other rows in the partition — that's their superpower over GROUP BY. I use PARTITION BY like GROUP BY for the window, ORDER BY to define ranking or running-aggregate direction, and FRAME clauses to build rolling-window analytics."

Window functions never collapse rows
Unlike GROUP BY, a window function appends a computed column to every row. You can mix them with WHERE, JOIN, and GROUP BY in the same query — they execute after all filters, in their own logical phase.

EHR: rank patients by latest weight per city without losing individual vital rows. OS/DB: show each booking's price alongside the route average in the same row.
OVER() — the window definition
Every window function requires OVER(). Empty OVER = whole result set. Add PARTITION BY to scope by group, ORDER BY for ranking or running aggregates.

EHR: OVER(PARTITION BY city ORDER BY recorded_at DESC) for per-city vital rankings. OS/DB: OVER(PARTITION BY route_id ORDER BY booked_at) for per-route cumulative bookings.
RANK vs DENSE_RANK vs ROW_NUMBER
ROW_NUMBER: unique sequential number with no ties — always 1,2,3,4. RANK: skips ranks after a tie — 1,1,3,4. DENSE_RANK: never skips — 1,1,2,3. Use ROW_NUMBER for pagination and deduplication; DENSE_RANK for leaderboards where every rank number must be represented.

EHR: DENSE_RANK on systolic BP to identify the top 3 hypertensive risk patients per clinic. OS/DB: ROW_NUMBER per route to find the single cheapest available seat.
LAG / LEAD — time-series comparisons
LAG(col, n, default) fetches the value n rows behind in the window order. LEAD(col, n, default) looks ahead. Both accept a default for when no prior or next row exists. Eliminates expensive self-joins for period-over-period comparisons.

EHR: LAG(weight_kg) to compute weight change between consecutive patient visits. OS/DB: LAG(price) to flag bookings that are more than 30% more expensive than the previous booking on the same route.
FRAME clause — rolling windows
ROWS BETWEEN n PRECEDING AND CURRENT ROW defines a moving frame for rolling aggregates. ROWS uses physical row count; RANGE uses logical value ranges (rows with the same ORDER BY value count as one unit). Critical for time-series analytics.

EHR: 7-day rolling average of patient pulse using ROWS BETWEEN 6 PRECEDING AND CURRENT ROW. OS/DB: 30-day rolling revenue per route.
🩺 Context

CareMate clinical analytics: rank patients by blood pressure risk per city, track weight trends over visits, and build a 7-day rolling pulse average for remote monitoring — all avoiding costly application-layer loops.

SQL🩺 EHR
-- 1. Latest systolic BP rank within each city (DENSE_RANK — no skipped ranks) WITH latest_vitals AS ( SELECT v.patient_id, p.full_name, p.city, v.systolic, v.diastolic, v.recorded_at, ROW_NUMBER() OVER ( PARTITION BY v.patient_id ORDER BY v.recorded_at DESC ) AS rn -- rn=1 means the most recent reading FROM vitals v JOIN patients p ON v.patient_id = p.id ) SELECT full_name, city, systolic, diastolic, DENSE_RANK() OVER ( PARTITION BY city ORDER BY systolic DESC ) AS bp_rank_in_city, ROUND(AVG(systolic) OVER (PARTITION BY city), 1) AS city_avg_systolic -- city mean on every row FROM latest_vitals WHERE rn = 1; -- only the most recent vital per patient -- 2. Weight change between consecutive visits using LAG SELECT p.full_name, v.recorded_at, v.weight_kg, LAG(v.weight_kg, 1, v.weight_kg) OVER ( PARTITION BY v.patient_id ORDER BY v.recorded_at ) AS prev_weight_kg, v.weight_kg - LAG(v.weight_kg, 1, v.weight_kg) OVER ( PARTITION BY v.patient_id ORDER BY v.recorded_at ) AS delta_kg -- positive=gained, negative=lost FROM vitals v JOIN patients p ON v.patient_id = p.id ORDER BY p.id, v.recorded_at; -- 3. 7-day rolling average pulse per patient (ROWS frame) SELECT patient_id, recorded_at, pulse, ROUND(AVG(pulse) OVER ( PARTITION BY patient_id ORDER BY recorded_at ROWS BETWEEN 6 PRECEDING AND CURRENT ROW -- up to last 7 readings ), 1) AS rolling_7_avg_pulse FROM vitals ORDER BY patient_id, recorded_at;
🚂 OS/DB Context

OS GmbH operational analytics: cumulative route revenue, top-N bookings per route for yield management, and price spike detection — all executed in single SQL passes to minimize DB load during peak hours.

SQL🚂 OS/DB
-- 1. Running revenue per route with % of total route revenue SELECT b.route_id, r.origin_station, r.destination_station, b.booked_at, b.price, SUM(b.price) OVER ( PARTITION BY b.route_id ORDER BY b.booked_at ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS running_revenue, ROUND(b.price / SUM(b.price) OVER (PARTITION BY b.route_id) * 100, 2 ) AS pct_of_route_total, -- this booking's revenue share RANK() OVER ( PARTITION BY b.route_id ORDER BY b.price DESC ) AS price_rank -- highest-value booking = 1 FROM bookings b JOIN routes r ON b.route_id = r.id WHERE b.status = 'confirmed' ORDER BY b.route_id, b.booked_at; -- 2. Price spike detection: flag bookings >40% above previous booking on same route WITH priced AS ( SELECT route_id, booked_at, price, LAG(price) OVER ( PARTITION BY route_id ORDER BY booked_at ) AS prev_price FROM bookings WHERE status = 'confirmed' ) SELECT route_id, booked_at, prev_price, price, ROUND((price - prev_price) / prev_price * 100, 1) AS pct_jump FROM priced WHERE prev_price IS NOT NULL AND price > prev_price * 1.4 -- more than 40% jump = flag for review ORDER BY pct_jump DESC;
⚠️ RANK() gaps in leaderboards

RANK() after a tie goes 1→1→3 — it skips rank 2. If you display "top 5 patients by risk" using RANK(), ties at rank 1 can mean only 3 patients appear (ranks 1,1,3,4,5 — rank 2 is never emitted). Use DENSE_RANK() for leaderboards where every position number must be represented.

⚠️ Using window functions in WHERE clauses

Window functions are computed after WHERE and GROUP BY — you cannot use them in WHERE. Wrap the query in a CTE or subquery, then filter on the window result in the outer query. Common mistake: WHERE ROW_NUMBER() OVER(...) = 1 — this causes a syntax error. Correct pattern: CTE with ROW_NUMBER(), then WHERE rn = 1 in the outer SELECT.

⚠️ ROWS vs RANGE frame — silent wrong results

RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW includes all rows with the same ORDER BY value as the current row. If two vitals have the same recorded_at, RANGE treats them as one unit and both get the same running sum. ROWS uses physical position. Use ROWS unless you explicitly need value-based grouping — RANGE is rarely what you want for time-series data.

Interview strategy — answer with confidence

RANK() assigns the same rank to ties and then skips the next rank — 1,1,3. DENSE_RANK() also assigns the same rank to ties but never skips — 1,1,2. For an OS GmbH booking dashboard ranking passengers by total spend per loyalty tier, DENSE_RANK() is the right choice: if two gold-tier passengers are tied for rank 1, you still want rank 2 to be the next passenger. RANK() would jump from 1 to 3, making the leaderboard confusing. I use ROW_NUMBER() when I need strictly unique row identifiers — for example, picking exactly one booking per passenger per route for deduplication.

🎯 Anchor

"DENSE_RANK for leaderboards — no skipped numbers. RANK when gaps after ties are intentional. ROW_NUMBER for deduplication where you need exactly one row per partition."

Interview strategy — answer with confidence

Using SUM() OVER with a ROWS or RANGE frame. Partition by route_id, order by booking month, and use ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. This accumulates revenue across all bookings in chronological order within each route's partition — no self-join required, and the query scales linearly with table size. If you need month-level granularity, first aggregate monthly totals into a CTE using GROUP BY, then apply the running SUM() OVER in the outer query on the pre-aggregated data.

🎯 Anchor

"SUM() OVER (PARTITION BY route_id ORDER BY month ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) — one pass, no self-join, scales linearly."

Interview strategy — answer with confidence

SQL has a strict logical execution order: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. Window functions are evaluated in the SELECT phase, after WHERE. When the WHERE clause is processed, the window function result doesn't exist yet. The solution is to either wrap the query in a CTE or subquery where the window function appears in the SELECT, then filter on the result in the outer query's WHERE clause. This is why the pattern WITH cte AS (SELECT ..., ROW_NUMBER() OVER(...) AS rn FROM ...) SELECT * FROM cte WHERE rn = 1 is so common for deduplication.

🎯 Anchor

"Window functions execute in the SELECT phase — after WHERE. Wrap in a CTE to filter on the window result in an outer WHERE clause."

Interview strategy — answer with confidence

The classic approach is a correlated subquery: WHERE recorded_at = (SELECT MAX(recorded_at) FROM vitals v2 WHERE v2.patient_id = v.patient_id). This works but scans the vitals table once per patient — O(n²) complexity. With window functions, ROW_NUMBER() OVER (PARTITION BY patient_id ORDER BY recorded_at DESC) in a CTE, then WHERE rn = 1, achieves the same result in a single linear scan. For OS GmbH's tables with millions of rows, the window function approach can be 10-100× faster. I always prefer the window function pattern for this class of "latest per group" queries.

🎯 Anchor

"ROW_NUMBER() OVER (PARTITION BY patient_id ORDER BY recorded_at DESC) in a CTE, then WHERE rn = 1 — linear scan vs correlated subquery's O(n²). Always the window function for production tables."

Write a query that returns the 2 patients with the highest latest systolic reading in each city. Use only window functions — no correlated subqueries allowed. Include each patient's name, city, systolic, and their rank within that city.

SQL🩺 EHR
WITH latest AS ( SELECT v.patient_id, p.full_name, p.city, v.systolic, ROW_NUMBER() OVER ( PARTITION BY v.patient_id ORDER BY v.recorded_at DESC ) AS visit_rn -- pick the most recent vital per patient FROM vitals v JOIN patients p ON v.patient_id = p.id ), ranked AS ( SELECT full_name, city, systolic, DENSE_RANK() OVER ( PARTITION BY city ORDER BY systolic DESC ) AS city_rank -- rank within city, ties share the same rank FROM latest WHERE visit_rn = 1 -- only the latest reading per patient ) SELECT full_name, city, systolic, city_rank FROM ranked WHERE city_rank <= 2 -- top 2 per city (DENSE_RANK so ties are included) ORDER BY city, city_rank;
Lesson 2 · 3h

CTEs & Recursive Queries

WITH clause, multiple CTEs, recursive CTE for hierarchy traversal — the tool that turns unreadable 12-level subqueries into readable, maintainable pipelines. Essential for org-chart and referral-chain queries.

CTEs transform deeply nested queries into readable, maintainable SQL. Recursive CTEs unlock an entire class of hierarchy and graph problems — referral chains, organisational structures, route networks — that are genuinely impossible with plain SELECT. At OS GmbH, station-to-station route planning and escalation trees rely on recursive queries.

🎯 Interview Anchor

"A CTE is a named subquery that executes once in the session and can be referenced multiple times in the same query. Recursive CTEs have an anchor member (base case) and a recursive member (the JOIN back on itself) — they terminate when the recursive member returns no rows."

CTE = named subquery, not a temp table
A CTE defined with WITH executes in-line when referenced — it is not materialized to disk unless the optimiser decides to. MariaDB's optimiser may inline it or materialise it depending on complexity. Multiple CTEs can be defined in sequence and each can reference the ones before it, creating a readable pipeline.

EHR: stage patient filtering, visit aggregation, and doctor attribution as three separate named CTEs before the final SELECT. OS/DB: CTE chain for monthly revenue → route percentiles → anomaly flags.
Recursive CTE — the anchor + recursive pattern
A recursive CTE has two parts joined by UNION ALL: the anchor member (base case — rows with no parent) and the recursive member (SELECT that JOINs the CTE back on itself). MariaDB limits depth with @@max_recursive_iterations (default 1000) — always add a depth column and a depth limit to prevent runaway recursion on bad data.

EHR: traverse a doctor referral chain. OS/DB: find all stations reachable from Frankfurt Hbf with transfer count.
Materialised vs inlined CTEs
MariaDB 10.4+ has the MATERIALIZED and NOT MATERIALIZED hints. A materialised CTE is computed once and cached as a temp table — useful if referenced multiple times in the query. An inlined CTE is substituted directly — better when referenced once or when the optimiser can push predicates through. The default behaviour depends on the CTE's content; force it with WITH cte AS MATERIALIZED (...).

EHR: materialise expensive patient-vitals aggregation used in multiple joins. OS/DB: inline a simple route filter used once.
🩺 Context

CareMate uses recursive CTEs to traverse doctor referral chains (DSGVO audit requirement) and multi-CTE pipelines to build patient risk scores in a single readable query.

SQL🩺 EHR
-- Multi-CTE pipeline: active patients → recent vitals → risk score WITH active_patients AS ( SELECT id, full_name, city, date_of_birth FROM patients WHERE created_at > NOW() - INTERVAL 2 YEAR ), latest_vitals AS ( -- One row per patient: the most recent vitals reading SELECT patient_id, systolic, diastolic, weight_kg, pulse FROM ( SELECT *, ROW_NUMBER() OVER (PARTITION BY patient_id ORDER BY recorded_at DESC) AS rn FROM vitals ) ranked WHERE rn = 1 ), risk_scored AS ( -- Simple BP-based risk flag: systolic >=140 = HIGH SELECT ap.id, ap.full_name, ap.city, lv.systolic, lv.weight_kg, CASE WHEN lv.systolic >= 180 THEN 'CRITICAL' WHEN lv.systolic >= 140 THEN 'HIGH' WHEN lv.systolic >= 120 THEN 'ELEVATED' ELSE 'NORMAL' END AS bp_risk_level FROM active_patients ap JOIN latest_vitals lv ON ap.id = lv.patient_id ) SELECT city, bp_risk_level, COUNT(*) AS patient_count, ROUND(AVG(systolic), 1) AS avg_systolic, ROUND(AVG(weight_kg), 1) AS avg_weight_kg FROM risk_scored GROUP BY city, bp_risk_level ORDER BY city, bp_risk_level;
🚂 OS/DB Context

OS GmbH route planning: recursive CTE to find all stations reachable from a given origin within N transfers — critical for disruption management when a line is cancelled and passengers need re-routing options.

SQL🚂 OS/DB
-- Recursive CTE: all reachable destinations from Frankfurt Hbf within 3 transfers WITH RECURSIVE reachable_stations AS ( -- ANCHOR: direct routes from Frankfurt Hbf SELECT r.destination_station AS station, 1 AS transfers, CAST(r.origin_station AS CHAR(200)) AS path -- track the path to detect cycles FROM routes r WHERE r.origin_station = 'Frankfurt Hbf' AND r.scheduled_departure > NOW() UNION ALL -- RECURSIVE MEMBER: extend one more transfer from each reachable station SELECT r.destination_station, rs.transfers + 1, CONCAT(rs.path, ' → ', r.origin_station) FROM routes r JOIN reachable_stations rs ON r.origin_station = rs.station WHERE rs.transfers < 3 -- STOP at 3 transfers (prevents runaway) AND rs.path NOT LIKE CONCAT('%', r.destination_station, '%') -- cycle guard: don't revisit stations ) SELECT DISTINCT station, MIN(transfers) AS min_transfers_needed FROM reachable_stations GROUP BY station ORDER BY min_transfers_needed, station;
⚠️ Infinite recursion on bad data

A recursive CTE with a missing or incorrect termination condition will loop until MariaDB hits @@max_recursive_iterations (default 1000) and throws an error. Always: (1) add an explicit depth/counter column and a WHERE depth < N condition, and (2) add a cycle-detection guard using the path or a visited-set pattern. Never deploy a recursive CTE on production data without testing it against rows with circular references.

⚠️ CTEs are not always materialised — predicate pushdown surprises

Developers sometimes expect a CTE to "run once and cache" — but MariaDB may inline it, causing it to re-execute for each reference. If your CTE is expensive and referenced multiple times, use the MATERIALIZED hint explicitly. Conversely, expecting MariaDB to push a WHERE predicate into a CTE may not happen — verify with EXPLAIN.

Interview strategy — answer with confidence

Both produce an inline result set, but CTEs are named, defined at the top of the query, and can be referenced multiple times. A derived table is anonymous and defined inline — readable only once in the query. CTEs also support recursion, which derived tables cannot do. In MariaDB, a non-recursive CTE is generally inlined by the optimizer the same way as a derived table, so performance is equivalent. The primary advantage of CTEs is readability and reusability within the same query. For OS GmbH, I use CTEs whenever a pipeline has more than two steps — the code becomes reviewable by colleagues and testable in isolation.

🎯 Anchor

"CTEs: named, reusable within query, support recursion, readable. Derived tables: anonymous, inline, referenced once. Performance equivalent for non-recursive cases — use CTEs for anything complex."

Interview strategy — answer with confidence

A recursive CTE has two mandatory parts joined by UNION ALL. The anchor member is a plain SELECT that returns the base rows — the starting point of traversal (for example, root nodes where parent_id IS NULL). The recursive member is a SELECT that JOINs the CTE back on itself to extend the result by one level per iteration. Recursion terminates when the recursive member's JOIN produces zero rows — no more children to follow. In MariaDB, you must also add an explicit depth limit (WHERE depth < N) because a data cycle would otherwise run until max_recursive_iterations is reached and throw an error. Always add a cycle-guard predicate in production.

🎯 Anchor

"Anchor (base case) UNION ALL recursive member (self-join). Terminates when the recursive member returns zero rows. Always add a depth limit and cycle guard for production data."

Interview strategy — answer with confidence

First, EXPLAIN the outer query — MariaDB shows whether the CTE is inlined or materialised. If inlined, the CTE is re-evaluated for each reference, which may be expensive. Force materialisation with the MATERIALIZED hint. If the CTE itself has a complex filter, check whether the optimiser is pushing predicates into it — if not, move the filter inside the CTE explicitly. For recursive CTEs, check the recursion depth: a large max_recursive_iterations count in EXPLAIN ANALYZE indicates slow traversal, possibly due to missing indexes on the join columns. Always index the FK columns used in the recursive JOIN.

🎯 Anchor

"EXPLAIN first: is it inlined or materialised? Force MATERIALIZED if needed. Push filters inside the CTE explicitly. For recursive CTEs, index the self-join columns."

The doctors table has a column referred_by_id pointing to another doctor. Write a recursive CTE that, starting from doctor ID 1, returns the full referral chain: doctor ID, name, depth in the chain, and the full path as a string like "Dr. Müller → Dr. Schmidt → Dr. Weber".

SQL🩺 EHR
WITH RECURSIVE referral_chain AS ( -- ANCHOR: the root doctor (starting point) SELECT id, full_name, referred_by_id, 0 AS depth, CAST(full_name AS CHAR(500)) AS chain_path FROM doctors WHERE id = 1 UNION ALL -- RECURSIVE MEMBER: follow referrals one level deeper SELECT d.id, d.full_name, d.referred_by_id, rc.depth + 1, CONCAT(rc.chain_path, ' → ', d.full_name) FROM doctors d JOIN referral_chain rc ON d.referred_by_id = rc.id WHERE rc.depth < 10 -- prevent runaway on cycles ) SELECT id AS doctor_id, full_name, depth AS chain_depth, chain_path FROM referral_chain ORDER BY depth;
Lesson 3 · 3h

EXPLAIN & Query Optimization

Reading EXPLAIN output, identifying table scans, understanding join types, and applying indexes and query rewrites to eliminate performance bottlenecks. The most practical DBA skill in day-to-day operations.

Every slow query that hits OS GmbH's 24/7 infrastructure is a potential incident. EXPLAIN is your first diagnostic tool — it tells you exactly how the query engine plans to execute a query before you run it. Understanding the output and knowing how to rewrite or re-index based on it is the core skill that separates a good DBA from a great one.

🎯 Interview Anchor

"I start with EXPLAIN FORMAT=JSON for any slow query — I look at type (ALL is a full table scan — always bad on large tables), key (which index was used), rows (estimated rows examined), and filtered (% of rows surviving the WHERE). Then I check whether a composite or covering index would eliminate the scan."

EXPLAIN type column — the access method
From worst to best: ALL (full scan) → index (full index scan) → range (index range scan) → ref (non-unique index lookup) → eq_ref (unique index, one match per join) → const/system (single row, effectively cached). For large tables in production, ALL and index are danger signs requiring immediate investigation.

EHR: type=ALL on the vitals table (millions of rows) means every query is reading the whole table. OS/DB: eq_ref on bookings.passenger_id means an efficient PK/unique-index lookup.
key vs possible_keys
possible_keys: all indexes MariaDB considered for this table access. key: the index actually chosen. If possible_keys is empty, no index exists for this access pattern — you need to create one. If key is NULL but possible_keys is populated, the optimiser decided a full scan is cheaper (often because the table is tiny or the index selectivity is very low).

EHR: no index on vitals.patient_id → possible_keys=NULL, type=ALL on a 5-million-row table — create the index immediately. OS/DB: bookings has an index on passenger_id but the query scans route_id — wrong index for this query pattern.
rows × filtered = rows actually examined
The rows column is the estimated number of rows the engine will examine for that table. The filtered column (%) is what fraction survives the WHERE clause. rows × (filtered/100) = output rows passed to the next step. A large rows count with a small filtered value means the index is not selective enough — the engine reads many rows only to discard most.

EHR: rows=50000, filtered=2.0 → the engine reads 50k vitals but only 1k survive the patient_id filter — add a composite index. OS/DB: rows=1000000, filtered=0.1 on bookings — the status='confirmed' filter is not indexed.
Covering index — the index-only scan
When a secondary index contains all columns needed by the query (SELECT list + WHERE + ORDER BY), MariaDB can answer the query from the index alone without touching the main table (clustered index). EXPLAIN shows "Using index" in the Extra column. Covering indexes can reduce I/O by 10-100× for high-frequency read queries.

EHR: CREATE INDEX idx_vitals_cover ON vitals(patient_id, recorded_at, systolic) — covers the most common clinical query pattern. OS/DB: INDEX ON bookings(route_id, status, booked_at, price) for the revenue analytics query.
EXPLAIN ANALYZE — actual vs estimated
EXPLAIN ANALYZE (MariaDB 10.9+) actually executes the query and shows both estimated and actual row counts. Discrepancies between estimated and actual rows indicate stale statistics — run ANALYZE TABLE to refresh. The output also shows actual execution time per node, revealing which join or sort step is the bottleneck.

EHR: estimated 100 rows, actual 45,000 rows — statistics are severely stale, run ANALYZE TABLE vitals. OS/DB: filesort in Extra with 2-second sort time — add an ORDER BY-matching index.
🩺 Context

A CareMate report query is slow — the clinical team reports the "patients needing follow-up" dashboard takes 8 seconds. Walk through EXPLAIN diagnosis and the fix.

SQL🩺 EHR
-- STEP 1: Run EXPLAIN on the slow query EXPLAIN FORMAT=JSON SELECT p.full_name, p.city, v.systolic, v.recorded_at, vi.reason FROM patients p JOIN vitals v ON v.patient_id = p.id JOIN visits vi ON vi.patient_id = p.id WHERE v.systolic >= 140 AND vi.status = 'scheduled' AND p.city = 'Frankfurt' ORDER BY v.recorded_at DESC; -- STEP 2: Read the output (key findings) -- patients: type=ref, key=PRIMARY ← good, filtered by city via index -- vitals: type=ALL, key=NULL ← BAD — full scan of 5M rows, no index on patient_id! -- visits: type=ALL, key=NULL ← BAD — no index on patient_id + status -- STEP 3: Fix — add targeted indexes -- Composite index on vitals: patient_id first (join), then systolic (filter), recorded_at (sort) CREATE INDEX idx_vitals_pid_sys_at ON vitals (patient_id, systolic, recorded_at); -- Composite index on visits: patient_id + status (both used in WHERE) CREATE INDEX idx_visits_pid_status ON visits (patient_id, status); -- Optional: covering index on patients if city queries are frequent CREATE INDEX idx_patients_city ON patients (city, id, full_name); -- STEP 4: Verify improvement EXPLAIN ANALYZE SELECT p.full_name, v.systolic FROM patients p JOIN vitals v ON v.patient_id = p.id WHERE v.systolic >= 140 AND p.city = 'Frankfurt'; -- Expected: type changes from ALL to range/ref, query drops from 8s → under 100ms
🚂 OS/DB Context

A revenue reporting query on the bookings table is timing out every night. EXPLAIN reveals an index choice problem and a filesort on a 50-million-row table. Diagnosis and fix using covering indexes.

SQL🚂 OS/DB
-- Slow query: nightly route revenue report EXPLAIN SELECT b.route_id, SUM(b.price) AS total_revenue, COUNT(*) AS booking_count FROM bookings b WHERE b.status = 'confirmed' AND b.booked_at >= '2024-01-01' AND b.booked_at < '2025-01-01' GROUP BY b.route_id ORDER BY total_revenue DESC; -- EXPLAIN shows: -- type=ALL on bookings (50M rows), Extra=Using filesort -- Existing index: idx_bookings_passenger — wrong index for this query! -- No index covers (status, booked_at, route_id, price) -- FIX 1: Covering index that serves WHERE + GROUP BY + SELECT aggregate CREATE INDEX idx_bookings_revenue_cover ON bookings (status, booked_at, route_id, price); -- Columns ordered: most selective filter first (status), then range (booked_at), -- then GROUP BY column (route_id), then aggregate column (price) -- This is an index-only scan — engine never touches the main table rows -- FIX 2: Verify with EXPLAIN after adding the index EXPLAIN SELECT b.route_id, SUM(b.price), COUNT(*) FROM bookings b WHERE b.status = 'confirmed' AND b.booked_at BETWEEN '2024-01-01' AND '2024-12-31' GROUP BY b.route_id; -- Expected after fix: -- type=range (index range scan on booked_at) -- key=idx_bookings_revenue_cover -- Extra=Using index (covering — no table access!) -- Query time drops from 45s → under 1s -- FIX 3: Keep index statistics fresh for the optimiser ANALYZE TABLE bookings;
⚠️ Function on indexed column kills the index

WHERE YEAR(booked_at) = 2024 prevents the index on booked_at from being used — the function is applied to every row. Always rewrite to a range: WHERE booked_at >= '2024-01-01' AND booked_at < '2025-01-01'. Same applies to UPPER(email), DATE(created_at), and any expression wrapping an indexed column.

⚠️ Implicit type conversion disables indexes

If booked_at is a VARCHAR but you compare it to a DATE literal, MariaDB must cast every row — resulting in a full scan. Always match the data type of your comparison value to the column's declared type. This is especially common with status columns stored as INT but compared with a string, or varchar dates compared with DATE functions.

⚠️ EXPLAIN rows is an estimate — verify with EXPLAIN ANALYZE

The rows column in EXPLAIN is based on index statistics — it can be wildly wrong if the table has skewed data distribution or hasn't been ANALYZEd recently. Always run EXPLAIN ANALYZE (or use EXPLAIN EXTENDED + SHOW WARNINGS) on production-representative data before concluding an optimization is working.

Interview strategy — answer with confidence

Step 1: Run EXPLAIN FORMAT=JSON on the query — look at type (ALL = danger), key (which index was chosen), rows (estimated scan size), and Extra (filesort/tmp table flags). Step 2: If type=ALL on a large table, check whether an index exists on the filtered columns. If not, design one. Step 3: Check column order — composite index columns must match the query's filter + join + sort order. Step 4: Check for function calls on indexed columns in the WHERE clause — rewrite them to range conditions. Step 5: Run EXPLAIN ANALYZE to compare estimated vs actual rows — if they diverge significantly, run ANALYZE TABLE to refresh statistics. Step 6: After adding indexes, re-run EXPLAIN to confirm type changed to range/ref and Extra shows "Using index" for a covering index scan.

🎯 Anchor

"EXPLAIN first — look at type, key, rows, Extra. ALL on a large table = immediate red flag. Design a covering index matching filter+sort columns. EXPLAIN ANALYZE to verify actual vs estimated rows. ANALYZE TABLE if statistics are stale."

Interview strategy — answer with confidence

Filesort means MariaDB must sort the result set in memory (or on disk if it exceeds sort_buffer_size) because the ORDER BY columns are not served by the index in the right order. It does not always mean disk I/O — the name is historical. To eliminate it: (1) add an index whose leftmost columns match the ORDER BY clause, in the same direction (ASC/DESC must match). (2) If the query also has a WHERE clause, the index must cover the WHERE columns first, then the ORDER BY columns. (3) If GROUP BY is present, the GROUP BY columns must also be in the index before ORDER BY. Covering the ORDER BY in the index makes the engine read already-sorted data and skip the sort step entirely.

🎯 Anchor

"Filesort = sort not served by an index. Fix: add an index whose column order matches WHERE → GROUP BY → ORDER BY. Once the data is read in index order, MariaDB skips the sort entirely."

Transactions & MVCC

ACID guarantees, isolation levels, MVCC snapshot reads, deadlock anatomy, and long-running transaction hazards. Understanding this layer is essential for diagnosing the class of bugs that only appear under concurrent load.

OS GmbH processes millions of booking transactions daily. A single misunderstood isolation level or an uncommitted long-running transaction can cascade into lock contention, replication lag, and failed backups. At 3am, you need to know exactly what isolation level a transaction is running under and why a query is blocking.

🎯 Interview Anchor

"MVCC means a plain SELECT never blocks and is never blocked — it reads a consistent snapshot from when the transaction started. Writes create a new row version; the old version stays in the undo log until no active transaction needs it. Long transactions bloat the undo log and can starve InnoDB's purge thread."

ACID — the four guarantees
Atomicity: all-or-nothing — either all statements in a transaction commit, or all roll back. Consistency: the DB moves from one valid state to another — constraints are never violated mid-transaction. Isolation: concurrent transactions do not see each other's partial changes (degree depends on isolation level). Durability: once committed, changes survive a crash — guaranteed by the redo log flush.

EHR: a CareMate visit creation inserts into visits AND updates a patient's last_visit_at atomically — no partial inserts. OS/DB: a seat booking reserves the seat AND deducts credit in one transaction.
MVCC — reads never block writes
Multi-Version Concurrency Control: when a row is updated, InnoDB does not overwrite it — it writes a new version and keeps the old version in the undo log. A plain SELECT reads the version that existed when its transaction started (consistent snapshot). This means reads never acquire locks and never block writes, and writes never block reads. The undo log is cleaned up by the purge thread once no active transaction needs the old version.

EHR: clinical reads of patient history are never blocked by concurrent vital inserts. OS/DB: reporting queries run concurrently with booking writes without locking each other.
Isolation levels — four choices
READ UNCOMMITTED: can read uncommitted changes (dirty reads) — almost never correct. READ COMMITTED: reads only committed data, but different reads within the same transaction can see different data (non-repeatable reads). REPEATABLE READ (MariaDB default): same rows seen throughout the transaction — consistent snapshot. SERIALIZABLE: full sequential isolation — every read acquires a shared lock — lowest concurrency.

EHR: REPEATABLE READ prevents nurses from seeing different patient weights in consecutive reads of the same transaction. OS/DB: READ COMMITTED is sometimes used for reporting to avoid long snapshot holding.
Deadlocks — anatomy and auto-resolution
A deadlock occurs when two transactions each hold a lock the other needs, and both wait forever. InnoDB detects cycles in the wait-for graph in ~1 second and rolls back the transaction with the least undo data. The rolled-back transaction receives error 1213. Application code must always retry on deadlock.

EHR: Nurse session A locks patient row 42, waits for visit row 100. Nurse session B holds visit row 100, waits for patient row 42 → InnoDB detects and rolls back the lighter transaction. OS/DB: two booking sessions trying to lock the same seat row in opposite order.
Long transactions — the hidden danger
A transaction that stays open for minutes or hours prevents the undo log from being purged — undo log grows unboundedly, filling disk. It also holds any row locks it has acquired, blocking other writers. Monitor with SHOW ENGINE INNODB STATUS (History list length) and information_schema.INNODB_TRX. Any transaction open longer than 60 seconds in OLTP is a warning sign.

EHR: a FHIR export script that opens a transaction and slowly reads millions of rows keeps snapshots alive for the entire export duration. OS/DB: a misconfigured ETL job holding an open transaction during a 30-minute file write.

⚠️ autocommit=1 masking missing explicit transactions

MariaDB defaults to autocommit=1 — every statement is its own transaction. Developers assume multi-statement sequences are atomic when they are not. Always wrap multi-step business operations (reserve seat + charge payment) in explicit START TRANSACTION ... COMMIT. Monitor for sessions with autocommit=0 that forget to COMMIT — they hold locks indefinitely.

⚠️ Deadlock not retried — application crashes instead

InnoDB rolls back one deadlock victim and returns error 1213 (ER_LOCK_DEADLOCK). If the application doesn't catch this and retry the transaction, the user sees an error. All write-path code must handle deadlocks with a retry loop (typically 3 attempts with exponential back-off). Deadlocks are not bugs to eliminate — they are a normal part of concurrent access. Design for retry.

⚠️ REPEATABLE READ snapshot taken at transaction start, not at first read

In MariaDB REPEATABLE READ, the snapshot is taken when the first data-reading statement executes in the transaction — not at START TRANSACTION. This means: START TRANSACTION; (wait 5 minutes); SELECT ... — the snapshot is 5 minutes old. Use START TRANSACTION WITH CONSISTENT SNAPSHOT to force an immediate snapshot if time-consistency from the exact start moment matters.

Interview strategy — answer with confidence

MVCC — Multi-Version Concurrency Control — means that when a row is updated, InnoDB writes a new version of the row rather than overwriting in place. Reads can still see the old version via the undo log, which means plain SELECT statements never need to acquire locks and are never blocked by concurrent writes. For OS GmbH's booking system this is critical: reporting queries and booking writes run simultaneously without blocking each other. The trade-off is the undo log — old versions must be kept until no active transaction needs them. A transaction that stays open for hours (like a poorly written batch job) prevents purge, causing undo log bloat and eventual disk pressure.

🎯 Anchor

"MVCC: reads see a consistent snapshot, never block. Writes create new versions, keep old in undo log. The cost: long-open transactions bloat the undo log — always monitor History list length in SHOW ENGINE INNODB STATUS."

Interview strategy — answer with confidence

InnoDB continuously maintains a wait-for graph of which transactions are waiting for locks held by other transactions. When it detects a cycle — typically within one second — it selects a victim (the transaction with the least undo data to roll back), rolls it back completely, and returns error 1213 to that session. The other transaction can then proceed and acquire the released lock. On the application side: every write path must catch error 1213 and retry the entire transaction, typically with exponential back-off and a max of 3 attempts. Deadlocks in low-contention systems suggest a lock ordering problem — consistently acquiring locks in the same order across transactions eliminates most deadlock cycles.

🎯 Anchor

"InnoDB rolls back the lighter transaction (1213 error) within ~1 second. Application must catch 1213 and retry. Prevent deadlocks by always acquiring locks in the same order across all transactions."

Interview strategy — answer with confidence

REPEATABLE READ — the InnoDB default. Within a transaction, all reads see the same consistent snapshot from when the first read occurred, regardless of concurrent commits by other sessions. I'd consider changing it in two scenarios: (1) READ COMMITTED for long-running analytics queries that don't need snapshot consistency and where holding a snapshot for 30 minutes would prevent undo log purge. (2) SERIALIZABLE for critical financial or inventory-check operations where phantom reads — new rows matching a range condition inserted by other transactions — would cause correctness problems. Lowering below READ COMMITTED (to READ UNCOMMITTED) is almost never justified in production — dirty reads corrupt any business logic that depends on committed data.

🎯 Anchor

"Default: REPEATABLE READ. Change to READ COMMITTED for long analytics sessions to prevent undo bloat. Use SERIALIZABLE for critical inventory/financial checks where phantom reads are unacceptable."

Lesson 5 · 2h

Stored Routines, Views & Events

Stored procedures, functions, triggers, views, and the event scheduler — the database-side automation layer. Know when to use them and when not to: the debate over business logic in the DB is a real interview topic.

Stored procedures, views, and events exist throughout enterprise MariaDB deployments. At OS GmbH, audit triggers and scheduled event-based cleanup jobs run continuously. Understanding these components — and their performance implications — shows operational maturity.

🎯 Interview Anchor

"I use views to abstract complexity and enforce row-level security (WHERE deleted_at IS NULL on every view). Stored procedures for multi-step operations that would otherwise require round-trips. Triggers sparingly — for audit logs where application-level consistency cannot be guaranteed. Events for scheduled cleanup that doesn't need an external cron dependency."

Views — abstraction and security layer
A view is a named stored query. It can hide columns (PII exclusion), pre-apply soft-delete filters, join complex schemas, and enforce access patterns. Simple views (single table, no DISTINCT/GROUP BY) are updatable. Complex views are read-only. Views do not store data — they re-execute the underlying query on every access.

EHR: v_active_patients view: SELECT * FROM patients WHERE deleted_at IS NULL — all queries go through the view; deleted patients are invisible. OS/DB: v_confirmed_bookings view joining bookings + routes + passengers for the reporting team.
Stored Procedures — server-side logic
Stored procedures execute multiple SQL statements in a single server round-trip, reduce network overhead, and can enforce consistent business logic across different clients. They support variables, IF/ELSE, loops, and SIGNAL for error raising. In MariaDB, procedures use the DELIMITER workaround in CLI (;; or $$) since the default ; terminates the procedure prematurely.

EHR: sp_discharge_patient(patient_id) — closes open visits, sets discharge date, sends FHIR notification trigger, logs to audit_log in one call. OS/DB: sp_cancel_booking(booking_id, refund_reason).
Triggers — automatic audit and consistency
A trigger fires automatically BEFORE or AFTER INSERT, UPDATE, or DELETE on a specific table. Inside, OLD holds the previous row values and NEW holds the new values. Triggers are invisible to application code — useful for audit logging, but dangerous because they can cause unexpected latency on writes and are often forgotten during migrations.

EHR: AFTER UPDATE trigger on patients to write every email change to audit_log automatically — DSGVO compliance regardless of which application path triggered the update. OS/DB: BEFORE INSERT on bookings to validate seat availability.
Events — the built-in scheduler
MariaDB's event scheduler runs SQL on a schedule without needing an external cron job. Events can run ONCE at a specific time, or EVERY n SECOND/MINUTE/HOUR. Must enable with SET GLOBAL event_scheduler = ON (or in my.cnf). Events run with the DEFINER's privileges and are replicated to replicas by default.

EHR: daily event to delete soft-deleted patients older than 7 years (DSGVO retention limit). OS/DB: hourly event to archive completed routes older than 90 days to the routes_archive table.

⚠️ Triggers are invisible — migrations break silently

A trigger on the patients table fires on every INSERT/UPDATE/DELETE — including bulk data migrations. A poorly timed trigger that inserts into audit_log during a migration that copies 50 million rows will write 50 million audit records, fill your disk, and crash the migration. Always check SHOW TRIGGERS before bulk data operations and consider temporarily disabling non-essential triggers.

⚠️ event_scheduler off by default after restart

Unless event_scheduler = ON is set in my.cnf (not just SET GLOBAL), it resets to OFF after a MariaDB restart. This has caused outages when a critical cleanup event silently stops running. Always add event_scheduler = ON to my.cnf and verify with SHOW VARIABLES LIKE 'event_scheduler' after any restart.

⚠️ Views don't store data — expensive views compound

A view over a view over a JOIN is not cached — it re-executes the full underlying query every time. If a view is complex or joins millions of rows, materialise it as a scheduled summary table instead. Check whether your views appear in slow query log — they won't be labelled as "view access" but will show the underlying query text.

Interview strategy — answer with confidence

Triggers are appropriate when you need guaranteed audit coverage regardless of the access path — multiple applications, direct DB access, or migrations that bypass the application layer. DSGVO audit requirements for patient email changes fit this model: you can't rely on every application team to call the audit function correctly. The trade-off is that triggers add latency to every write, are invisible in code reviews, and must be checked before migrations. Application-level audit is easier to test, debug, and version-control. My default is application-level; I add triggers when compliance requires guaranteed capture or when auditing third-party ETL access that I can't control.

🎯 Anchor

"Triggers for compliance-guaranteed audit when multiple access paths exist. Application-level for everything else — easier to test, version, and debug. Always check SHOW TRIGGERS before bulk operations."

Interview strategy — answer with confidence

Three main risks. First, testability: stored procedures are hard to unit test — you need a DB connection for every test, which slows CI pipelines. Application code with pure functions is far easier to test. Second, version control and deployment: procedure changes require DB migrations, not just code deploys, which complicates rollbacks. Third, scaling: stored procedures run on the DB server — heavy logic can max out CPU on the database tier, which is your hardest resource to scale horizontally. I use stored procedures for multi-step, high-network-round-trip operations (like sp_cancel_booking) where the performance gain is clear, and keep business rules in the application layer.

🎯 Anchor

"Stored procedures: hard to unit test, complicate deployments, add CPU to the DB tier (hardest to scale). Use them for multi-round-trip operations where the network win is real — keep business rules in the application."

type=ALL means a full table scan — MariaDB reads every row in the table to find matches. On a large table (thousands of rows or more), this is almost always a problem: it is O(n) in data volume, hits disk for every row not in the buffer pool, and doesn't scale as the table grows. The exception is very small tables — for a lookup table with 20 rows, a full scan is faster than an index lookup. In practice: type=ALL on any table with more than a few thousand rows is a red flag that needs an index. Check EXPLAIN rows to estimate impact — a table scan of 50 rows is fine; 5 million is not.

History list length is the number of undo log records that the purge thread has not yet processed. It represents old row versions that are still being kept because at least one active transaction might need them (MVCC consistency). A healthy value is typically under 1000. Values over 100,000 indicate a long-running transaction is holding an old snapshot, preventing purge. This causes the undo log to grow, consuming disk space, and slowing down queries as InnoDB must traverse longer version chains to find the correct row version for a snapshot read. The DEFINER specifies which MySQL user account's privileges are used when the routine or trigger executes. By default it's the user who created it. If the DEFINER account is dropped or renamed, the routine fails with a privilege error at runtime — not at creation time, which makes it a common silent production failure. For security: routines that access sensitive tables should use a dedicated service account with minimal privileges as the DEFINER, not a DBA or root account. 🎯 Anchor
