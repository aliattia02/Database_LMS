Core Conceptual Questions
Open every card — these are the gaps this module closes
Why this matters
This is the single most common confusion in the first week of team Git. Getting it wrong means teammates cannot see your work or accidentally think you've done nothing.
The distinction
A commit records a snapshot in your local .git history — only you can see it. Your local repo and the remote are independent copies.

A push uploads your local commits to the remote (GitHub) so your team can access them. You can have ten commits before ever pushing — and they're invisible to everyone else until you do.

Think of it as: commit = save a draft to your hard drive; push = email it to the team.
git commit saves ONLY to your local repository's history.
git push uploads those commits to the remote (GitHub).
Commits you haven't pushed are invisible to your team.
Push unfinished work to your own feature branch — never to main.
Why this matters
Solo projects: fine. Team projects: always wrong. Main is the stable baseline everyone depends on. One bad push breaks the whole team's build — no warning, no discussion.
The case for feature branches
On a team, main is the stable source of truth — every deployment, every CI pipeline, every other developer's baseline. If you push broken code to main, you break everyone's build with no safety net.

Feature branches let you work, experiment, and break things safely in isolation. They cost nothing to create (they're just pointers, not file copies). You merge back only when the work is ready and has been reviewed.

On a professional codebase, main is protected — direct pushes are blocked by GitHub's branch protection settings. You cannot push to main even if you wanted to.
Main is protected on every professional team — direct push is blocked.
Feature branches are free — create one for every logical change.
Branches enable code review before anything changes shared code.
If two people both worked on main, every push would cause conflicts.
The key distinction
A merge is a Git operation. A PR is a process. One is a command; the other is the quality control workflow that wraps it.
What a PR is
git merge is the technical act of combining two branches' commit histories — one command, done locally.

A Pull Request (PR) is the GitHub workflow that wraps that merge: it shows a diff of every changed line, opens a discussion thread where reviewers leave comments, runs automated CI/CD checks, and requires an approval before the merge happens.

You can merge without a PR (git merge locally). On a team you almost never should — the PR is where quality control, documentation, and technical discussion all happen. After a PR merges, the feature branch is typically deleted.
PR = diff + discussion thread + CI checks + approval + merge.
git merge = just the technical combining step, no workflow.
PRs are mandatory on protected branches — no approval means no merge.
PRs create a permanent record of what changed, why, and who approved it.
Reframe before reading
Conflicts are information, not errors. Git is saying: "two people changed the same lines — I need you to decide which version is correct." That's a reasonable request.
What Git does
When Git tries to merge two branches and finds the same lines edited differently in both, it cannot automatically choose. It inserts conflict markers into the file and pauses:
<<<<<<< HEAD your current branch's version of the lines ======= the incoming branch's version of the lines >>>>>>> feature/wearable-data-sync
You edit the file to produce the correct final result — which might be one side, the other, or a manual blend of both. Then you remove all markers, git add the file, and git commit to complete the merge.
Conflicts are normal — not a sign something went wrong.
Every conflict marker must be removed before you can commit.
If unsure about someone else's code — ask them before resolving it yourself.
VS Code shows conflicts with coloured one-click buttons (simpler for common cases).
The rule
Reset is for LOCAL history. Revert is for SHARED branches. This distinction is the difference between a tidy undo and a Git disaster that forces teammates to re-pull.
The two tools
git reset --soft HEAD~1 moves HEAD back one commit and unstages that commit's changes (the file modifications stay). It rewrites history. If others have already pulled your commits, their history and yours now diverge — causing chaos on the next pull.

git revert HEAD creates a new commit that does the exact opposite of the previous one. The original commit stays in history. This is safe for shared branches because it adds to history rather than rewriting it.
Branch is local only and nobody has pulled it → use git reset --soft.
Branch is on GitHub and others may have pulled → use git revert.
git reset --hard discards file changes permanently — use only with full awareness.
⚠️ Never git reset --hard on a branch your teammates have pulled. This is one of the most common Git disasters on teams. The correct tool for shared branches is always git revert.
Mental model
Stash is a temporary side pocket for incomplete work. It lets you set aside half-finished changes, do something else, and come back — without committing code that isn't ready.
How it works
git stash saves your current uncommitted changes to a hidden stack and restores your working directory to the last committed state — instantly giving you a clean tree.

git stash pop re-applies the most recently saved changes to your current branch.

Classic scenario: you're mid-feature when someone asks you to urgently review a PR. You can't commit half-finished code. Stash it, switch branches, review, pop the stash when you're back.
git stash = save uncommitted work + clean working directory.
git stash pop = restore the last stash.
git stash list = see all stashed entries (you can stash multiple times).
Don't use stash as long-term storage — commit your work to a branch instead.
First rule
Check your team's existing branch names and match that pattern. If none exists yet, propose the type-prefix system — it's the most widely adopted convention.
Type-prefix convention
Format: type/short-description-in-kebab-case

feature/wearable-data-sync — new functionality
bugfix/fhir-mapping-null-id — a specific bug fix
docs/api-integration-guide — documentation only
chore/update-dependencies — maintenance, no behaviour change
hotfix/critical-auth-crash — urgent production fix
One branch = one logical change. Don't bundle unrelated fixes.
Keep the slug short and descriptive. Lowercase, hyphens, under 50 characters.
Delete branches after merging — don't let them pile up in the remote.
The branch name shows up in every PR, CI log, and git log.
Why it matters
Your description is the context a reviewer needs before they look at a single line of code. A blank or vague description forces the reviewer to reverse-engineer your intent from the diff — slow, frustrating, and often inaccurate.
Good PR description structure
Title format: type: short description of what changed (#issue)
Good: feat: add null-check for FHIR encounter ID mapping (#87)
Bad: fix / update stuff / blank

Description should include:
1. What changed (the technical fact)
2. Why it changed (the motivation — what problem does this solve?)
3. How to test it
4. Any caveats or design decisions worth discussing
5. Closing reference: Closes #87
Write the description for the reviewer, not for yourself.
Atomic PRs (one logical change) are always easier to review than large multi-change PRs.
If a PR takes more than two hours to review, it's probably too large — split it.

Zentrale Konzeptfragen
Öffne jede Karte — das sind die Lücken, die dieses Modul schließt
Warum das wichtig ist
Das ist die häufigste Verwechslung in der ersten Woche mit Team-Git. Wer das falsch versteht, denkt entweder, die Teammates sehen Änderungen sofort — oder umgekehrt, dass nichts passiert ist.
Der Unterschied
Ein Commit speichert einen Snapshot in der lokalen .git-History — nur du kannst ihn sehen. Lokales Repository und Remote sind unabhängige Kopien.

Ein Push lädt deine lokalen Commits auf das Remote (GitHub) hoch, sodass das Team darauf zugreifen kann. Du kannst zehn Commits machen, bevor du jemals pushst — für alle anderen sind sie bis dahin unsichtbar.

Merkhilfe: Commit = Entwurf auf die Festplatte speichern; Push = dem Team mailen.
git commit speichert NUR in deiner lokalen Repository-History.
git push lädt diese Commits auf das Remote (GitHub) hoch.
Commits, die du nicht gepusht hast, sind für dein Team unsichtbar.
Unfertige Arbeit auf deinen eigenen Feature-Branch pushen — niemals auf main.
Warum das wichtig ist
Solo-Projekte: problemlos. Team-Projekte: immer falsch. Main ist die stabile Basis, von der alle abhängen. Ein schlecht gepushter Commit bricht den Build des gesamten Teams — ohne Warnung, ohne Diskussion.
Das Argument für Feature-Branches
Im Team ist main die stabile Quelle der Wahrheit — jede Deployment-Pipeline, jedes CI-System, jede andere Entwicklerin oder jeder andere Entwickler basiert darauf. Wer defekten Code auf main pusht, bricht den Build aller — ohne Sicherheitsnetz.

Feature-Branches erlauben isoliertes Arbeiten, Experimentieren und Fehler machen. Sie kosten nichts (sie sind nur Zeiger, keine Dateikopien). Gemergt wird erst, wenn die Arbeit fertig ist und gereviewed wurde.

Auf einer professionellen Codebasis ist main geschützt — direkte Pushes werden durch GitHubs Branch-Protection-Einstellungen blockiert. Du kannst schlicht nicht auf main pushen, auch wenn du wolltest.
Main ist auf jedem professionellen Team geschützt — direkter Push blockiert.
Feature-Branches sind kostenlos — erstelle einen für jede logische Änderung.
Branches ermöglichen Code-Review, bevor geteilter Code geändert wird.
Wenn zwei Personen direkt auf main arbeiten, führt jeder Push zu Konflikten.
Der zentrale Unterschied
Ein Merge ist eine Git-Operation. Ein PR ist ein Prozess. Das eine ist ein Befehl; das andere ist der Qualitätskontrollworkflow, der ihn umgibt.
Was ein PR ist
git merge ist die technische Aktion, zwei Branch-Historien zusammenzuführen — ein Befehl, lokal ausgeführt.

Ein Pull Request (PR) ist der GitHub-Workflow, der diesen Merge umhüllt: Er zeigt ein Diff jeder geänderten Zeile, öffnet einen Diskussions-Thread wo Reviewer Kommentare hinterlassen, führt automatisierte CI/CD-Checks aus und verlangt eine Genehmigung, bevor der Merge stattfindet.

Mergen ohne PR ist möglich (git merge lokal). Im Team sollte man das fast nie tun — der PR ist der Ort, wo Qualitätskontrolle, Dokumentation und technische Diskussion stattfinden.
PR = Diff + Diskussions-Thread + CI-Checks + Genehmigung + Merge.
git merge = nur der technische Zusammenführungsschritt, kein Workflow.
PRs sind auf geschützten Branches Pflicht — keine Genehmigung, kein Merge.
PRs erstellen ein dauerhaftes Protokoll darüber, was sich geändert hat, warum und wer es genehmigt hat.
Zuerst umentscheiden
Konflikte sind Informationen, keine Fehler. Git sagt: „Zwei Personen haben dieselben Zeilen geändert — ich brauche deine Entscheidung, welche Version korrekt ist." Das ist eine vernünftige Anfrage.
Was Git macht
Wenn Git zwei Branches zusammenführen will und findet, dass dieselben Zeilen in beiden unterschiedlich bearbeitet wurden, kann es nicht automatisch wählen. Es fügt Konflikt-Marker in die Datei ein und hält an:
<<<<<<< HEAD Version deines aktuellen Branches ======= Version des eingehenden Branches >>>>>>> feature/wearable-data-sync
Du bearbeitest die Datei zum korrekten Endergebnis, entfernst alle Marker, führst git add für die Datei aus und dann git commit, um den Merge abzuschließen.
Konflikte sind normal — kein Zeichen, dass etwas schiefgelaufen ist.
Jeder Konflikt-Marker muss entfernt werden, bevor du committen kannst.
Bei fremdem Code: erst fragen, dann auflösen.
VS Code zeigt Konflikte mit farbigen Ein-Klick-Buttons (einfacher für häufige Fälle).
Die Regel
Reset für LOKALE History. Revert für GETEILTE Branches. Dieser Unterschied trennt ein sauberes Rückgängigmachen von einem Git-Desaster, das Teammitglieder zum erneuten Pullen zwingt.
Die zwei Werkzeuge
git reset --soft HEAD~1 bewegt HEAD einen Commit zurück und hebt das Staging der Änderungen auf (die Dateimodifikationen bleiben erhalten). Es schreibt History um. Wenn andere deine Commits bereits gepullt haben, divergieren jetzt ihre History und deine — beim nächsten Pull gibt es Chaos.

git revert HEAD erstellt einen neuen Commit, der das genaue Gegenteil des vorherigen tut. Der Original-Commit bleibt in der History. Sicher für geteilte Branches, da History nur hinzugefügt, nicht umgeschrieben wird.
Branch nur lokal, niemand hat gepullt → git reset --soft.
Branch auf GitHub, andere könnten gepullt haben → git revert.
git reset --hard verwirft Dateiänderungen dauerhaft — nur mit vollständigem Bewusstsein verwenden.
⚠️ Niemals git reset --hard auf einem Branch, den Teammitglieder gepullt haben. Das ist eines der häufigsten Git-Desaster im Team. Das richtige Werkzeug für geteilte Branches ist immer git revert.
Mentales Modell
Stash ist eine temporäre Seitentasche für unfertige Arbeit. Er erlaubt dir, halbfertige Änderungen beiseite zu legen, etwas anderes zu tun, und zurückzukommen — ohne Code zu committen, der noch nicht fertig ist.
Wie es funktioniert
git stash speichert deine aktuellen uncommitted Änderungen in einem versteckten Stack und stellt das Arbeitsverzeichnis auf den letzten committed Zustand zurück — sofort ein sauberes Working Directory.

git stash pop wendet die zuletzt gespeicherten Änderungen auf deinen aktuellen Branch wieder an.

Klassisches Szenario: Du bist mitten im Feature, als jemand dich bittet, dringend einen PR zu reviewen. Unfertige Code committen geht nicht. Stash it, Branch wechseln, reviewen, Stash zurückholen.
git stash = uncommitted Arbeit speichern + Working Directory bereinigen.
git stash pop = letzten Stash wiederherstellen.
git stash list = alle Stash-Einträge anzeigen (mehrfaches Stashen möglich).
Stash nicht als Langzeitspeicher verwenden — Arbeit stattdessen auf einem Branch committen.
Erste Regel
Schau dir die bestehenden Branch-Namen deines Teams an und passe dein Muster daran an. Wenn noch keines existiert, schlage das Typ-Präfix-System vor — es ist die am weitesten verbreitete Konvention.
Typ-Präfix-Konvention
Format: typ/kurze-beschreibung-in-kebab-case

feature/wearable-data-sync — neue Funktionalität
bugfix/fhir-mapping-null-id — ein bestimmter Bug-Fix
docs/api-integration-guide — nur Dokumentation
chore/update-dependencies — Wartung, keine Verhaltensänderung
hotfix/critical-auth-crash — dringender Produktions-Fix
Ein Branch = eine logische Änderung. Keine unzusammenhängenden Fixes bündeln.
Slug kurz und beschreibend halten. Kleinbuchstaben, Bindestriche, unter 50 Zeichen.
Branches nach dem Mergen löschen — nicht im Remote ansammeln lassen.
Der Branch-Name erscheint in jedem PR, CI-Log und git log.
Warum das wichtig ist
Deine Beschreibung ist der Kontext, den ein Reviewer braucht, bevor er eine einzige Zeile Code liest. Eine leere oder vage Beschreibung zwingt den Reviewer, deine Absicht aus dem Diff zu rekonstruieren — langsam, frustrierend und oft ungenau.
Struktur einer guten PR-Beschreibung
Titel-Format: typ: kurze Beschreibung was sich geändert hat (#Issue)
Gut: feat: null-check für FHIR Encounter-ID-Mapping hinzugefügt (#87)
Schlecht: fix / Sachen aktualisiert / leer

Beschreibung sollte enthalten:
1. Was sich geändert hat (die technische Tatsache)
2. Warum es sich geändert hat (die Motivation — welches Problem löst das?)
3. Wie man es testet
4. Eventuelle Vorbehalte oder Designentscheidungen zur Diskussion
5. Issue-Referenz: Closes #87
Beschreibung für den Reviewer schreiben, nicht für sich selbst.
Atomare PRs (eine logische Änderung) sind immer einfacher zu reviewen als große Multi-Änderungs-PRs.
Wenn ein PR mehr als zwei Stunden zum Reviewen braucht, ist er wahrscheinlich zu groß — aufteilen.

افتح كل بطاقة — هذه هي الفجوات التي تسدها هذه الوحدة
لماذا هذا مهم
هذا هو الخلط الأكثر شيوعًا في الأسبوع الأول من استخدام Git في الفريق. الفهم الخاطئ يعني أن زملاءك لا يرون عملك، أو أنك تظن أن شيئًا لم يحدث.
الفرق
الـ commit يسجّل لقطة في سجل .git المحلي — أنت وحدك من يراها. المستودع المحلي والـ remote نسختان مستقلتان.

الـ push يرفع commits المحلية إلى الـ remote (GitHub) لكي يتمكن فريقك من الوصول إليها. يمكنك عمل عشرة commits قبل أي push — وتبقى غير مرئية للجميع حتى تقوم بالـ push.

فكّر بالأمر هكذا: commit = حفظ مسودة على القرص الصلب؛ push = إرسالها بالبريد الإلكتروني للفريق.
git commit يحفظ فقط في سجل مستودعك المحلي.
git push يرفع تلك الـ commits إلى الـ remote (GitHub).
الـ commits التي لم تقم بـ push بعد غير مرئية لفريقك.
ارفع العمل غير المكتمل إلى feature branch الخاص بك — ليس إلى main أبدًا.
لماذا هذا مهم
في المشاريع الفردية: لا مشكلة. في مشاريع الفريق: خطأ دائمًا. الـ main هو الأساس المستقر الذي يعتمد عليه الجميع. push واحد سيء يكسر build الفريق بأكمله — بلا إنذار، بلا نقاش.
حجة الـ Feature Branches
في الفريق، main هو مصدر الحقيقة المستقر — كل pipeline نشر، كل CI، كل مطور آخر يعتمد عليه. من يدفع كودًا معطوبًا إلى main يكسر build الجميع بلا شبكة أمان.

الـ Feature branches تتيح العمل والتجريب والأخطاء في عزلة. تكلفتها صفر (مجرد مؤشرات، لا نسخ ملفات). الدمج يحدث فقط عندما يكون العمل جاهزًا وتمت مراجعته.

على أي codebase احترافية، main محمي — الـ push المباشر محظور بواسطة إعدادات branch protection في GitHub. لا يمكنك Push إلى main حتى لو أردت.
Main محمي في كل فريق احترافي — الـ push المباشر محظور.
الـ Feature branches مجانية — أنشئ واحدًا لكل تغيير منطقي.
الـ Branches تتيح مراجعة الكود قبل تغيير أي كود مشترك.
إذا عمل شخصان على main مباشرة، كل push يسبب تعارضات.
الفرق الجوهري
الـ merge عملية Git. الـ PR عملية. الأول أمر؛ الثاني هو سير عمل ضبط الجودة الذي يلف ذلك الأمر.
ما هو الـ PR
git merge هو الفعل التقني لدمج سجلَي commit لـ branch-ين — أمر واحد ينفَّذ محليًا.

الـ Pull Request (PR) هو سير عمل GitHub الذي يلف ذلك الدمج: يعرض diff لكل سطر متغير، يفتح thread نقاش حيث يترك المراجعون تعليقاتهم، ينفذ فحوصات CI/CD الآلية، ويستلزم موافقة قبل إتمام الدمج.

يمكنك الدمج بلا PR (محليًا بـ git merge). في الفريق يجب ألا تفعل ذلك تقريبًا — الـ PR هو المكان الذي تتم فيه مراقبة الجودة والتوثيق والنقاش التقني.
PR = diff + thread نقاش + فحوصات CI + موافقة + دمج.
git merge = خطوة الدمج التقنية فقط، بلا سير عمل.
الـ PRs إلزامية على الـ branches المحمية — لا موافقة يعني لا دمج.
تنشئ الـ PRs سجلًا دائمًا بما تغيّر ولماذا ومن وافق.
إعادة التأطير أولاً
التعارضات معلومات لا أخطاء. Git يقول: "شخصان عدّلا نفس الأسطر — أحتاج قرارك أيّ نسخة صحيحة." هذا طلب معقول تمامًا.
ما يفعله Git
عندما يحاول Git دمج branch-ين ويجد أن نفس الأسطر عُدِّلت بشكل مختلف في كليهما، لا يستطيع الاختيار تلقائيًا. يُدرج علامات التعارض في الملف ويتوقف:
<<<<<<< HEAD نسخة الـ branch الحالي من الأسطر ======= نسخة الـ branch الوارد من الأسطر >>>>>>> feature/wearable-data-sync
تحرر الملف للوصول إلى النتيجة النهائية الصحيحة، تحذف جميع العلامات، تنفذ git add على الملف ثم git commit لإتمام الدمج.
التعارضات طبيعية — ليست علامة على وجود خطأ.
يجب إزالة كل علامة تعارض قبل إمكانية الـ commit.
إذا كنت غير متأكد من كود شخص آخر — اسأله قبل الحل.
VS Code يعرض التعارضات مع أزرار ملونة بنقرة واحدة (أسهل للحالات الشائعة).
القاعدة
Reset للسجل المحلي. Revert للـ branches المشتركة. هذا الفرق هو ما يفصل التراجع النظيف عن كارثة Git تجبر الزملاء على إعادة الـ pull.
الأداتان
git reset --soft HEAD~1 يحرك HEAD خطوة للخلف ويلغي staging التغييرات (تبقى تعديلات الملفات). إنه يعيد كتابة السجل. إذا سحب آخرون commits-ك مسبقًا، سجلهم وسجلك سيتباعدان — مما يسبب فوضى في الـ pull التالي.

git revert HEAD ينشئ commit جديدًا يفعل عكس الـ commit السابق بالضبط. الـ commit الأصلي يبقى في السجل. آمن للـ branches المشتركة لأنه يضيف إلى السجل بدلاً من إعادة كتابته.
الـ branch محلي فقط ولم يسحبه أحد → استخدم git reset --soft.
الـ branch على GitHub وقد يكون آخرون سحبوه → استخدم git revert.
git reset --hard يحذف تغييرات الملفات نهائيًا — استخدمه بوعي تام فقط.
⚠️ لا تستخدم أبدًا git reset --hard على branch سحبه زملاؤك. هذه إحدى أكثر كوارث Git شيوعًا في الفرق. الأداة الصحيحة للـ branches المشتركة دائمًا هي git revert.
النموذج الذهني
الـ stash جيب جانبي مؤقت للعمل غير المكتمل. يتيح لك تنحية التغييرات نصف المنجزة، فعل شيء آخر، ثم العودة — بدون commit لكود غير جاهز.
كيف يعمل
git stash يحفظ التغييرات غير المُنجَزة الحالية في stack مخفي ويعيد مجلد العمل إلى آخر حالة مُنجَزة — مما يمنحك مجلد عمل نظيفًا فورًا.

git stash pop يعيد تطبيق آخر التغييرات المحفوظة على branch-ك الحالي.

السيناريو الكلاسيكي: أنت في منتصف ميزة جديدة وطُلب منك مراجعة PR بشكل عاجل. لا يمكنك commit كود نصف منجز. احفظه بـ stash، بدّل الـ branch، راجع، ثم استعد الـ stash عند عودتك.
git stash = حفظ العمل غير المُنجَز + تنظيف مجلد العمل.
git stash pop = استعادة آخر stash.
git stash list = عرض جميع الـ stash entries (يمكن عمل stash أكثر من مرة).
لا تستخدم الـ stash للتخزين طويل المدى — بدلاً من ذلك اعمل commit لعملك على branch.
القاعدة الأولى
راجع أسماء الـ branches الموجودة في فريقك واتبع نفس النمط. إذا لم يكن هناك نمط بعد، اقترح نظام type-prefix — إنه الاصطلاح الأكثر انتشارًا.
اصطلاح البادئة النوعية
الصيغة: type/short-description-in-kebab-case

feature/wearable-data-sync — وظيفة جديدة
bugfix/fhir-mapping-null-id — إصلاح bug محدد
docs/api-integration-guide — توثيق فقط
chore/update-dependencies — صيانة، بلا تغيير في السلوك
hotfix/critical-auth-crash — إصلاح عاجل للإنتاج
Branch واحد = تغيير منطقي واحد. لا تجمع إصلاحات غير مترابطة.
اجعل الاسم قصيرًا ووصفيًا. حروف صغيرة، شرطات، أقل من 50 حرفًا.
احذف الـ branches بعد الدمج — لا تتركها تتراكم في الـ remote.
اسم الـ branch يظهر في كل PR وسجل CI وgit log.
لماذا هذا مهم
الوصف هو السياق الذي يحتاجه المراجع قبل النظر في سطر واحد من الكود. الوصف الفارغ أو الغامض يجبر المراجع على استنتاج نيتك من الـ diff — بطيء ومُحبِط وغالبًا غير دقيق.
هيكل وصف PR جيد
صيغة العنوان: type: short description of what changed (#issue)
جيد: feat: add null-check for FHIR encounter ID mapping (#87)
سيئ: fix / update stuff / فارغ

يجب أن يتضمن الوصف:
١. ما الذي تغيّر (الحقيقة التقنية)
٢. لماذا تغيّر (الدافع — ما المشكلة التي يحلها؟)
٣. كيفية اختباره
٤. أي تحفظات أو قرارات تصميمية تستحق النقاش
٥. مرجع الإغلاق: Closes #87
اكتب الوصف للمراجع، لا لنفسك.
الـ PRs الذرية (تغيير منطقي واحد) دائمًا أسهل في المراجعة من الـ PRs الكبيرة متعددة التغييرات.
إذا استغرق مراجعة PR أكثر من ساعتين، فهو على الأرجح كبير جدًا — قسّمه.
