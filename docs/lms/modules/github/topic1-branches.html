Git & GitHub — Topic 1 of 5

Branches — Mental Model

Branches are parallel development lines, not copies of your code — master this picture and everything else in Git becomes logical.

🌿
The Mental Model
Concept first — the picture that makes every Git command make sense

A branch is a pointer, not a copy

The most important thing to understand: Git does not copy your files when you create a branch. A branch is a lightweight label — a moveable pointer — that points to a specific commit in your repository's history. Creating one writes 41 bytes to disk. That's it.

When you commit on a branch, that pointer moves forward to the new commit. Your other branches stay exactly where they were. That's how parallel work happens: multiple pointers referencing different commits in the same history graph.

main: A ── B ── C \ feature/user-auth D ── E ── F HEAD → whichever branch you are currently working on

When you run git switch feature/user-auth, Git moves HEAD to that branch and updates your working directory to match. Your files change — but nothing is copied or lost. Switch back to main and your files return to state C.

main is the stable source-of-truth. Every teammate depends on it. It must always be in a working, deployable state.
Feature branches are isolated experiments. You break things freely here — main never sees the mess unless you merge and review first.
One branch = one logical change. Not "everything I worked on this week." One feature, one fix, one purpose.

Core Branch Commands
The seven commands you will use every day

Daily branch workflow

# See all local branches (* marks the current one) git branch # Create a new branch AND switch to it in one step git switch -c feature/user-auth # Switch to an existing branch git switch main # See the full commit graph across all branches git log --all --oneline --graph # Merge your feature branch INTO your current branch git switch main git merge feature/user-auth # Push your branch to GitHub for the first time git push -u origin feature/user-auth # Delete a branch after merging (clean up the remote) git branch -d feature/user-auth

Note on git checkout: You'll see git checkout -b branch-name everywhere online — that's the older syntax. git switch (Git 2.23+) was created specifically for branch operations and is clearer. Both work; prefer switch for new work.

Key Concepts — Open Every Card
The questions that trip up developers new to team Git
Mental model
Understand this one command deeply and the rest of Git becomes logical rather than magical. Every other operation — merge, conflict, reset — is just moving or reading these same pointers.
Step by step
1. Git creates a new branch pointer at exactly the same commit HEAD currently points to.

2. Git moves HEAD to point at the new branch instead of the old one.

3. Git updates your working directory — but since the new branch starts at the same commit, your files don't change at all.

From here, every new commit you make advances only the new branch pointer, while main stays exactly where it was. That divergence is what makes parallel work possible.
Creating a branch is instantaneous — Git writes just 41 bytes to disk.
Your files don't change at creation — you're still at the same commit.
HEAD = "where am I right now?" — it follows you as you switch branches.
Every commit only advances the branch HEAD is currently pointing to.
Think of HEAD as a cursor
HEAD is Git's "you are here" marker. Every operation — git log, git diff, git reset — uses HEAD as its reference point. Understanding HEAD turns cryptic git output into readable information.
What HEAD is
HEAD is a special pointer that points to the branch you're currently on. When you commit, HEAD moves forward with the branch. When you switch branches, HEAD moves to the new branch and your working directory updates to match.

Detached HEAD: if you check out a specific commit SHA or tag directly (not a branch name), HEAD points straight at the commit. You can explore, but any commits you make here "float" without a branch. Run git switch -c recovery-branch to attach them to a new branch before switching away.
Normal state: HEAD → branch → commit.
Detached HEAD: HEAD → commit directly (no branch tracking it).
git log starts from HEAD and walks backwards through history.
HEAD~1 = one commit before HEAD — used constantly in git reset and git revert.
Remote-tracking branches
The most common source of "why don't I see my teammate's changes?" confusion. Your local branch and the remote branch are separate — they only sync when you explicitly fetch, push, or pull.
Two separate pointers
main = your local branch. You commit to it, reset it, move it freely.

origin/main = a remote-tracking branch — a local snapshot of what GitHub looked like the last time you communicated with it. Git updates it automatically on fetch/pull/push. You cannot commit to it directly.

git pull does two things: git fetch (updates origin/main from GitHub) then git merge origin/main into your local main. Always pull before starting new work.
origin is the default name for the remote. You can have multiple remotes.
git fetch updates remote-tracking branches without touching your local branches.
git status compares local vs origin/branch and reports "ahead / behind."
Pull before you branch. If you branch off a stale main, your PR will arrive with unnecessary merge conflicts.
Practical rule for new team members
Default to merge, always. Rebase rewrites commit history and causes serious problems when used on shared branches. Once you understand Git deeply you can decide when rebase makes sense — but not yet.
The difference
Merge creates a new "merge commit" that ties two branch histories together. The original commits stay unchanged. Safe everywhere — even on shared branches.

Rebase replays your branch's commits on top of the target branch, rewriting each commit's SHA. History looks linear and clean, but it's dangerous if others have already pulled your branch — their local copy and yours now have diverged histories and the next pull creates chaos.
⚠️ Never rebase a branch that other people have already pulled. This is the rule. Rebase is only safe for your own local branch before anyone else has touched it.
Default: merge. Safe, explicit, preserves history accurately.
Many teams configure GitHub to enforce merge commits on PRs — they're protecting shared history.
Ask your team's preference before using rebase.
Common accident — easy fix if caught early
You meant to commit to your feature branch but accidentally committed to main before branch protection blocked it. The earlier you catch this, the simpler the rescue.
The fix — before you push
Scenario: you committed to main, meant to commit to feature/my-feature, and have NOT pushed yet.

# Step 1 — create the right branch at current HEAD (includes your commit) git switch -c feature/my-feature # Step 2 — go back to main git switch main # Step 3 — undo the commit on main (file changes are kept, un-staged) git reset --soft HEAD~1
Your commit now lives only on feature/my-feature. Main is back where it belongs.
⚠️ If you already pushed to main and others have pulled — do NOT use git reset. That rewrites shared history. Use git revert HEAD instead (creates a new undo commit), then git cherry-pick your original commit SHA onto the correct branch.
Your most important learning tool
Most branch confusion is invisible because people can't see the commit graph. One command makes the whole picture visible. Run it constantly while you're learning — watch how it changes after every branch, commit, and merge.
The command and its output
git log --all --oneline --graph
Sample output:

You can see where branches diverge, where HEAD is, and what has been merged — all at a glance.
Run this after every branch/commit/merge while you're learning — let the graph cement the mental model.

Das mentale Modell
Erst das Konzept — das Bild, das jeden Git-Befehl verständlich macht

Ein Branch ist ein Zeiger, keine Kopie

Das Wichtigste zuerst: Git kopiert keine Dateien, wenn du einen Branch erstellst. Ein Branch ist ein leichtgewichtiges Label — ein beweglicher Zeiger — der auf einen bestimmten Commit in der Repository-History zeigt. Einen Branch zu erstellen schreibt 41 Bytes auf die Festplatte. Das war's.

Wenn du auf einem Branch committest, bewegt sich dieser Zeiger zum neuen Commit vorwärts. Deine anderen Branches bleiben exakt dort, wo sie waren. So entsteht paralleles Arbeiten: mehrere Zeiger referenzieren verschiedene Commits im selben History-Graph.

main: A ── B ── C \ feature/user-auth D ── E ── F HEAD → der Branch, auf dem du gerade arbeitest

Wenn du git switch feature/user-auth ausführst, bewegt Git HEAD zu diesem Branch und aktualisiert dein Arbeitsverzeichnis. Deine Dateien ändern sich — aber nichts wird kopiert oder verloren. Zurück zu main, und deine Dateien kehren zu Zustand C zurück.

main ist die stabile Wahrheitsquelle. Jedes Teammitglied verlässt sich darauf. Er muss immer in einem lauffähigen Zustand sein.
Feature-Branches sind isolierte Experimente. Hier kannst du frei Fehler machen — main sieht den Chaos nicht, bis du mergst und reviewst.
Ein Branch = eine logische Änderung. Nicht "alles, woran ich diese Woche gearbeitet habe." Ein Feature, ein Fix, ein Zweck.

Schlüsselkonzepte — Jede Karte öffnen
Die Fragen, an denen neue Team-Entwickler scheitern
Mentales Modell
Verstehe diesen einen Befehl tief, und der Rest von Git wird logisch statt magisch. Jede andere Operation — Merge, Konflikt, Reset — bewegt nur diese Zeiger oder liest sie.
Schritt für Schritt
1. Git erstellt einen neuen Branch-Zeiger genau beim selben Commit, auf den HEAD aktuell zeigt.

2. Git bewegt HEAD auf den neuen Branch statt auf den alten.

3. Git aktualisiert dein Arbeitsverzeichnis — da der neue Branch beim selben Commit startet, ändern sich deine Dateien überhaupt nicht.

Von hier an bewegt jeder neue Commit nur den neuen Branch-Zeiger, während main exakt dort bleibt, wo es war.
Einen Branch erstellen dauert weniger als eine Millisekunde — Git schreibt nur 41 Bytes auf die Festplatte.
Deine Dateien ändern sich beim Erstellen nicht — du bist noch beim selben Commit.
HEAD = "Wo bin ich gerade?" — es folgt dir, wenn du Branches wechselst.
Jeder Commit bewegt nur den Branch, auf den HEAD gerade zeigt.
Stell dir HEAD als Cursor vor
HEAD ist Gits "Du bist hier"-Markierung. Jede Operation — git log, git diff, git reset — verwendet HEAD als Ausgangspunkt. HEAD zu verstehen macht Git-Ausgaben von kryptisch zu lesbar.
Was HEAD ist
HEAD ist ein spezieller Zeiger, der auf den Branch zeigt, auf dem du gerade arbeitest. Wenn du committest, bewegt sich HEAD mit dem Branch vorwärts. Beim Branch-Wechsel bewegt sich HEAD auf den neuen Branch und dein Arbeitsverzeichnis wird aktualisiert.

Detached HEAD: Wenn du direkt einen Commit-SHA oder Tag auscheckst (keinen Branch-Namen), zeigt HEAD direkt auf den Commit. Du kannst dich umsehen, aber Commits hier "schweben" ohne Branch. Führe git switch -c recovery-branch aus, um sie an einen neuen Branch zu knüpfen.
Normalzustand: HEAD → Branch → Commit.
Detached HEAD: HEAD → Commit direkt (kein Branch, der ihn trackt).
HEAD~1 = ein Commit vor HEAD — wird ständig in git reset und git revert verwendet.
Remote-Tracking-Branches
Die häufigste Quelle von "Warum sehe ich die Änderungen meines Teamkollegen nicht?"-Verwirrung. Dein lokaler Branch und der Remote-Branch sind separate Dinge — sie synchronisieren sich nur, wenn du explizit fetch, push oder pull ausführst.
Zwei separate Zeiger
main = dein lokaler Branch. Du committest darauf, resettest ihn, bewegst ihn frei.

origin/main = ein Remote-Tracking-Branch — ein lokaler Schnappschuss, wie GitHub beim letzten Kontakt aussah. Git aktualisiert ihn automatisch beim fetch/pull/push. Du kannst nicht direkt darauf committen.

git pull macht zwei Dinge: git fetch (aktualisiert origin/main von GitHub) dann git merge origin/main in dein lokales main. Immer pullen, bevor du neue Arbeit anfängst.
origin ist nur der Standardname für das Remote. Du kannst mehrere Remotes haben.
git fetch aktualisiert Remote-Tracking-Branches, ohne deine lokalen Branches anzutasten.
Vor dem Branchen pullen. Wenn du von einem veralteten main abzweigst, hat dein PR unnötige Merge-Konflikte.
Praktische Regel für neue Teammitglieder
Standardmäßig immer mergen. Rebase schreibt die Commit-History um und verursacht ernsthafte Probleme auf geteilten Branches. Sobald du Git tief verstehst, kannst du entscheiden — aber noch nicht.
Der Unterschied
Merge erstellt einen neuen "Merge-Commit", der zwei Branch-Historien verbindet. Die Original-Commits bleiben unverändert. Überall sicher — auch auf geteilten Branches.

Rebase spielt die Commits deines Branches oben auf dem Zielbranch nach und schreibt dabei die SHA jedes Commits um. Die History sieht linearer aus, ist aber gefährlich, wenn andere deinen Branch schon gepullt haben — ihre lokale Kopie und deine haben jetzt abweichende Historien.
⚠️ Niemals einen Branch rebasen, den andere bereits gepullt haben. Das ist die Regel. Rebase ist nur sicher für deinen eigenen lokalen Branch, bevor ihn jemand anderes angefasst hat.
Häufiges Versehen — einfache Lösung, wenn früh erkannt
Du wolltest auf deinem Feature-Branch committen, hast aber versehentlich auf main committed, bevor der Schutz das verhindert hat. Je früher du es bemerkst, desto einfacher die Lösung.
Die Lösung — vor dem Push
Szenario: Du hast auf main committed, wolltest aber auf feature/mein-feature — und hast noch NICHT gepusht.

# Schritt 1 — richtigen Branch beim aktuellen HEAD erstellen (enthält deinen Commit) git switch -c feature/mein-feature # Schritt 2 — zurück zu main wechseln git switch main # Schritt 3 — Commit auf main rückgängig machen (Dateiänderungen bleiben als unstaged) git reset --soft HEAD~1
Dein Commit lebt jetzt nur noch auf feature/mein-feature. Main ist zurück wo es sein soll.
⚠️ Wenn du schon auf main gepusht hast und andere gepullt haben — kein git reset. Das schreibt geteilte History um. Verwende stattdessen git revert HEAD (erstellt einen neuen Undo-Commit), dann git cherry-pick deinen ursprünglichen Commit auf den richtigen Branch.
Dein wichtigstes Lernwerkzeug
Die meisten Branch-Probleme sind unsichtbar. Ein Befehl macht das gesamte Bild sichtbar — führe ihn ständig aus, während du Git lernst.
Der Befehl und seine Ausgabe

النموذج الذهني
المفهوم أولاً — الصورة التي تجعل كل أمر في Git منطقيًا

الـ Branch مؤشرٌ لا نسخة

أهم ما يجب فهمه: Git لا ينسخ ملفاتك عند إنشاء branch. الـ branch مجرد ملصق خفيف الوزن — مؤشر متحرك — يشير إلى commit محدد في سجل المستودع. إنشاء branch يكتب 41 بايت فقط على القرص. هذا كل شيء.

عندما تعمل commit على branch، يتقدم هذا المؤشر للأمام نحو الـ commit الجديد. بقية الـ branches تبقى في مكانها بالضبط. هكذا يتحقق العمل المتوازي: مؤشرات متعددة تشير إلى commits مختلفة في نفس مخطط السجل.

main: A ── B ── C \ feature/user-auth D ── E ── F HEAD → الـ branch الذي تعمل عليه حاليًا

عندما تنفذ git switch feature/user-auth، يحرك Git الـ HEAD إلى ذلك الـ branch ويحدّث مجلد العمل ليتطابق معه. ملفاتك تتغير — لكن لا شيء ينسخ أو يُفقد. العودة إلى main وملفاتك ترجع إلى حالة C.

main هو مصدر الحقيقة المستقر. كل زميل في الفريق يعتمد عليه. يجب أن يكون دائمًا في حالة عمل وقابلية نشر.
Feature branches هي تجارب معزولة. يمكنك ارتكاب الأخطاء هنا بحرية — main لن يرى الفوضى إلا بعد المراجعة والـ merge.
Branch واحد = تغيير منطقي واحد. ليس "كل ما عملت عليه هذا الأسبوع." ميزة واحدة، إصلاح واحد، هدف واحد.

المفاهيم الأساسية — افتح كل بطاقة
الأسئلة التي يتعثر عندها المطورون الجدد في العمل الجماعي
النموذج الذهني
افهم هذا الأمر بعمق وسيصبح باقي Git منطقيًا لا سحريًا. كل عملية أخرى — merge وconflict وreset — مجرد تحريك هذه المؤشرات أو قراءتها.
خطوة بخطوة
١. ينشئ Git مؤشر branch جديد عند نفس الـ commit الذي يشير إليه HEAD حاليًا.

٢. يحرك Git الـ HEAD ليشير إلى الـ branch الجديد بدلاً من القديم.

٣. يحدّث Git مجلد العمل — لكن لأن الـ branch الجديد يبدأ عند نفس الـ commit، ملفاتك لا تتغير إطلاقًا.

من هنا فصاعدًا، كل commit جديد يحرك فقط مؤشر الـ branch الجديد، بينما يبقى main بالضبط حيث كان.
إنشاء branch فوري — Git يكتب 41 بايتًا فقط على القرص.
ملفاتك لا تتغير عند الإنشاء — لا تزال عند نفس الـ commit.
HEAD = "أين أنا الآن؟" — يتبعك عند التنقل بين الـ branches.
كل commit يحرك فقط الـ branch الذي يشير إليه HEAD حاليًا.
فكّر في HEAD كـ cursor
HEAD هو علامة "أنت هنا" في Git. كل عملية — git log وgit diff وgit reset — تستخدم HEAD كنقطة انطلاق. فهم HEAD يحول مخرجات Git من رموز مبهمة إلى معلومات مقروءة.
ما هو HEAD
HEAD مؤشر خاص يشير إلى الـ branch الذي تعمل عليه حاليًا. عند الـ commit، يتقدم HEAD مع الـ branch. عند تغيير الـ branch، ينتقل HEAD إلى الـ branch الجديد ويتحدث مجلد العمل.

Detached HEAD: إذا checkout-t مباشرة على SHA commit أو tag (وليس اسم branch)، يشير HEAD إلى الـ commit مباشرة. يمكنك الاستكشاف، لكن أي commit تعمله هنا "يطفو" بلا branch يحتضنه. نفذ git switch -c recovery-branch لإلحاقها ببranch جديد.
الحالة الطبيعية: HEAD ← Branch ← Commit.
Detached HEAD: HEAD يشير إلى Commit مباشرة (بلا branch يتتبعه).
HEAD~1 = commit واحد قبل HEAD — يُستخدم كثيرًا في git reset وgit revert.
Remote-tracking branches
أكثر مصادر الارتباك شيوعًا: "لماذا لا أرى تغييرات زميلي؟" الـ branch المحلي والـ branch البعيد شيئان منفصلان — يتزامنان فقط عند تنفيذ fetch أو push أو pull صراحةً.
مؤشران منفصلان
main = الـ branch المحلي. تعمل عليه commits وتعيد ضبطه وتحركه بحرية.

origin/main = remote-tracking branch — نسخة محلية تعكس حالة GitHub في آخر تواصل معه. Git يحدثه تلقائيًا عند fetch/pull/push. لا يمكنك الـ commit عليه مباشرة.

git pull يقوم بعمليتين: git fetch (يحدّث origin/main من GitHub) ثم git merge origin/main في main المحلي. دائمًا pull قبل بدء عمل جديد.
origin مجرد الاسم الافتراضي للـ remote. يمكنك امتلاك remotes متعددة.
git fetch يحدّث الـ remote-tracking branches دون المساس بالـ branches المحلية.
قم بالـ pull قبل الـ branch. إذا تفرعت من main قديم، سيصل الـ PR محملاً بتعارضات غير ضرورية.
القاعدة العملية للأعضاء الجدد في الفريق
استخدم merge دائمًا كخيار افتراضي. الـ rebase يعيد كتابة تاريخ الـ commits ويسبب مشاكل خطيرة على الـ branches المشتركة. بمجرد فهمك العميق لـ Git يمكنك تقرير متى يكون مناسبًا — لكن ليس الآن.
الفرق
Merge ينشئ "merge commit" جديدًا يربط تاريخَي branch معًا. الـ commits الأصلية تبقى دون تغيير. آمن في كل مكان — حتى على الـ branches المشتركة.

Rebase يعيد تشغيل commits الـ branch فوق الـ branch المستهدف، مع إعادة كتابة SHA لكل commit. السجل يبدو خطيًا ونظيفًا، لكنه خطير إذا سحب آخرون الـ branch بالفعل — نسختهم المحلية ونسختك الآن لهما تاريخان متباينان.
⚠️ لا تعمل rebase أبدًا على branch سحبه أشخاص آخرون. هذه هي القاعدة. الـ rebase آمن فقط على الـ branch المحلي الخاص بك قبل أن يلمسه أحد.
خطأ شائع — حل سهل إذا اكتُشف مبكرًا
أردت الـ commit على feature branch لكنك وضعته على main عن طريق الخطأ قبل أن يمنعه الحماية. كلما اكتشفت ذلك مبكرًا، كان الحل أبسط.
الحل — قبل الـ push
السيناريو: وضعت commit على main، أردته على feature/my-feature، ولم تقم بالـ push بعد.

# الخطوة ١ — أنشئ الـ branch الصحيح عند HEAD الحالي (يتضمن commit-ك) git switch -c feature/my-feature # الخطوة ٢ — عد إلى main git switch main # الخطوة ٣ — تراجع عن الـ commit على main (التغييرات تبقى unstaged) git reset --soft HEAD~1
الـ commit الآن موجود فقط على feature/my-feature. main عاد لمكانه الصحيح.
⚠️ إذا push-ت بالفعل إلى main وسحب آخرون — لا تستخدم git reset. ذلك يعيد كتابة السجل المشترك. استخدم git revert HEAD بدلاً منه (ينشئ commit تراجع جديد)، ثم git cherry-pick تغييرك الأصلي إلى الـ branch الصحيح.
أهم أداة تعليمية لديك
معظم مشاكل الـ branches غير مرئية. أمر واحد يجعل الصورة كاملة واضحة — نفّذه باستمرار أثناء تعلمك لـ Git وراقب كيف يتغير بعد كل branch وcommit وmerge.
الأمر ومخرجاته
git log --all --oneline --graph
مثال على المخرجات:
* f3e8a2c (HEAD → feature/user-auth) Add sync endpoint * 4b1c9d7 Refactor device model | * 8e2a31f (main) Merge PR: fix null check in FHIR mapping | * 7d9c012 Fix null ID in FHIR mapping |/ * 3a5e891 Initial commit
ترى بوضوح أين تتفرع الـ branches، وأين يقع HEAD، وما الذي دُمج — دفعة واحدة. موقع learngitbranching.js.org يحرّك هذا المخطط لحظيًا بينما تكتب أوامر Git الحقيقية — أفضل 20 دقيقة ستستثمرها في هذا الموضوع.

ثلاثة branches، مستودع واحد، مخطط كامل

من المنهج: أنشئ ثلاثة branches مسماة من main، اعمل commit على كل منها، راقب التفرق بـ git log --all --graph، ثم بدّل بينها. لا push بعد — فقط اشعر كيف تعزل الـ branches العمل.

# ١. إنشاء مستودع تدريبي git init branch-practice && cd branch-practice echo "v1" > notes.txt && git add . && git commit -m "Initial commit" # ٢. إنشاء وتعبئة ثلاثة branches git switch -c feature/alpha echo "alpha work" >> notes.txt && git commit -am "Alpha: add sync endpoint" git switch main git switch -c feature/beta echo "beta work" >> notes.txt && git commit -am "Beta: add device model" git switch main git switch -c bugfix/null-check echo "bugfix" >> notes.txt && git commit -am "Fix: null check for FHIR ID" # ٣. مشاهدة تفرق الثلاثة branches git log --all --oneline --graph # ٤. دمج أحدها ومراقبة تغيير المخطط git switch main git merge feature/alpha git log --all --oneline --graph

بعد هذا التمرين يجب أن ترى ثلاثة خطوط تتفرع من نفس نقطة البداية، ثم أحدها يعود ليندمج في main. هذه الصورة هي الأساس الكامل لما يلي في هذا المقرر.
