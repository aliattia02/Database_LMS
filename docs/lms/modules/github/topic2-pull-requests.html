Git & GitHub — Topic 2 of 5

Pull Requests as Code Review

A PR is not just a merge button — it is your team's primary tool for knowledge sharing, quality control, and catching mistakes before they hit production.

🔀
What a PR Really Is
Concept first — before you touch the GitHub UI

A Pull Request is a conversation, not a file transfer

The name is misleading. A PR does not "pull" anything — it is a request for review that says: "I have changes on my branch. Please look at them, discuss them, and merge them into the target branch if they're good."

The merge itself is the easy part. The review conversation — comments, requested changes, approvals — is the real value. It is how knowledge moves across the team, how bugs get caught before production, and how new developers learn the codebase's conventions and quality bar.

What it contains: a diff (every line changed), a title and description you write, a discussion thread, a review status, and CI check results.
What it is not: a ZIP file, an email attachment, or a finished product. It is a proposal, open for scrutiny.
Protected branches + PRs: on a real research team, main is protected — you literally cannot push directly to it. Every change goes through a PR. This is not bureaucracy; it is your safety net.
✍️
Writing a PR That Gets Reviewed Fast
Reviewers have limited time — make it easy for them to say yes

The PR description template

A blank description is the single most common reason PRs sit unreviewed. Reviewers need context to do their job. Fill in this template every time — it takes 3 minutes and saves everyone an hour.

## What does this PR do? ## Why is this change needed? ## How was it tested? ## Screenshots / output (if UI or data change) ## Checklist - [ ] I ran the full test suite locally - [ ] No debug logs / commented-out code left in - [ ] Branch is up to date with main
Keep PRs small. A diff under 400 lines gets reviewed in minutes. 1,000 lines gets skimmed or deferred.
One logical change per PR — not "I fixed the bug AND refactored the module AND updated the docs."
Add reviewers deliberately. Assign the person most familiar with the code, not everyone on the team.
Draft PRs are valid. Open one early to share context, get early feedback, and signal "this is in progress" — mark it ready when it's actually ready.

Key Concepts — Open Every Card
The PR questions that trip up developers new to team workflows
Three merge strategies — you will see all three in the wild
GitHub offers three ways to merge a PR. Understanding them prevents surprise when you look at the git log and see something unexpected. Teams usually enforce one strategy via repo settings — ask which one your team uses on day one.
The three strategies
Merge commit (default) — creates a new merge commit on main that ties both histories together. The full branch history is preserved. Best for feature work that benefits from a clear "this entire PR landed here" marker.

Squash and merge — flattens all commits on your branch into a single commit, then adds it to main. Main stays clean and linear. Individual WIP commits disappear. Common in teams that care about a tidy log.

Rebase and merge — replays each commit from your branch onto main as if they were written there directly, with new SHAs. No merge commit. Cleanest history, but loses the branch reference in the log.
For research teams at UKM: squash merge is common — it keeps the main branch log readable at a glance.
Whatever the strategy, once merged, delete your feature branch immediately. It has served its purpose.
After merging, git switch main && git pull to get the merged commit locally.
The review response workflow
Responding to review comments professionally is a core team skill. The goal is not to "win" the argument — it is to converge on the best code. Reviewers are doing you a favour.
Step by step
1. Read all comments before making any changes — some may be related or one may invalidate another.

2. For each comment: either fix it and reply "Done, updated in [SHA]", or discuss it — "I kept this as-is because X, does that make sense?"

3. Make your changes locally on the same branch and push:
git add . git commit -m "Address review: refactor null check per feedback" git push
The PR updates automatically — no need to close and reopen it.

4. Mark each resolved comment as Resolved on GitHub once addressed.

5. Re-request review from the reviewer when all feedback is addressed — don't leave them to guess you're ready.
Don't force-push to a PR branch after review has started — it rewrites history and loses the reviewer's inline comment anchors.
A second push to the PR shows as a new set of changed files on GitHub — reviewers can see exactly what changed since their last review.
Tone: short, collaborative, non-defensive. "Good catch — fixed" is better than a paragraph of justification.
Keeping your branch in sync
This happens constantly on active teams — someone else merged their PR to main while you were working. Your branch is now behind. Resolve it before asking for a merge.
The right fix
# 1. Switch to main and pull the latest git switch main git pull # 2. Go back to your feature branch git switch feature/my-feature # 3. Merge main into your branch (brings your branch up to date) git merge main # 4. Resolve any conflicts that arise (see Topic 3), then push git push
Your PR is now up to date. The "out of date" warning disappears and CI re-runs on the merged state.
⚠️ GitHub's "Update branch" button does the same thing via the UI. Use it when the merge is clean. If there are conflicts, you must resolve them locally — GitHub cannot auto-resolve code conflicts.
Use git merge main, not git rebase main, for a PR branch that others may already have pulled.
Get into the habit: git pull on main every morning before starting new work.
Continuous Integration — your automated first reviewer
CI runs automatically on every push to a PR. It is not optional and it is not personal. A failing CI check means the automated gate caught something before a human had to.
What CI typically checks
Build — does the code compile / does the app start without errors?

Tests — do all unit and integration tests pass? A single failing test blocks the merge.

Linting — does the code follow the style rules (PEP 8, ESLint, etc.)? Formatting errors fail CI.

Type checks — for typed codebases (TypeScript, mypy), are there type errors?
Click the failing check name on GitHub — it links directly to the CI log. The error is almost always in the last 20 lines.
Fix the issue locally, commit, and push — CI re-runs automatically. No manual trigger needed.
Never ask a teammate to "just merge it" past a failing CI check. That failure exists for a reason.
Commit messages are the first line of documentation
On a team, git log is a shared changelog that everyone reads. "fix stuff" tells a teammate nothing. A well-written commit message explains what changed and why — in 72 characters.
The formula
Format: type(scope): short description in imperative mood

feat(fhir): add LOINC code mapping for lab results fix(auth): resolve token refresh truncation on bcrypt hash docs(api): clarify pagination parameters in README chore(deps): upgrade FastAPI to 0.111.0 refactor(models): extract device schema into separate module
Rules: start with a verb ("add", "fix", "remove", "update"). Under 72 characters. No period at the end. The what in the subject, the why in the body if needed.
The type prefix matches branch naming: feat, fix, docs, chore, refactor, test.
If squash merge is used, your individual commit messages collapse into one — the PR title becomes the commit on main. Make the PR title a good commit message too.
Code review is a skill, not just reading code
New developers think code review is about catching bugs. It is that — but also about correctness, clarity, security, and mentoring. A good reviewer makes the author a better developer.
Reviewer checklist
Correctness — does it do what the PR description says? Are edge cases handled?

Clarity — can a new team member understand this in 6 months? Are names meaningful?

Security — any hardcoded credentials, unvalidated input, or exposed endpoints?

Tests — are there tests for the new behaviour? Do they actually test the right thing?

Scope — is the PR doing only what it claims? Or is there hidden refactoring mixed in?
Leave nitpick comments with a "nit:" prefix — "nit: rename `x` to `deviceId` for clarity". The author can ignore nits if they disagree; they don't block merge.
Approve when you're satisfied. "Looks good to me" (LGTM) with no comments is a valid review.
Respond within 24 hours. A PR sitting unreviewed for days is a team process failure, not a Git problem.

Open a PR, review it, and merge it

From the curriculum: create a feature branch, make a small change, push it, open a PR on GitHub, self-review it, and merge it. Every step is deliberate — notice what GitHub shows you at each stage.

# 1. Start from an up-to-date main git switch main && git pull # 2. Create a feature branch git switch -c feature/pr-practice # 3. Make a real change (anything — edit a README, add a comment) echo "# PR practice" >> NOTES.md git add NOTES.md git commit -m "docs: add PR practice note" # 4. Push the branch to GitHub git push -u origin feature/pr-practice # 5. Go to GitHub — it will show a banner "Compare & pull request" # Click it. Fill in the description template. Open the PR. # 6. Review your own diff in the "Files changed" tab # Leave at least one comment on your own code (seriously — practise the UI) # 7. Merge the PR via GitHub UI → "Squash and merge" → Confirm # 8. After this you should be able to open a PR end-to-end without hesitation. The first PR you open on a real codebase will feel familiar, not foreign.

Was ein PR wirklich ist
Erst das Konzept — bevor du die GitHub-Oberfläche anfasst

Ein Pull Request ist ein Gespräch, kein Dateitransfer

Der Name ist irreführend. Ein PR „pullt" nichts — er ist eine Anfrage zur Überprüfung, die sagt: „Ich habe Änderungen auf meinem Branch. Bitte schau sie dir an, diskutiere sie, und merge sie in den Zielbranch, wenn sie gut sind."

Der Merge selbst ist der einfache Teil. Das Review-Gespräch — Kommentare, angeforderte Änderungen, Genehmigungen — ist der eigentliche Wert. So wandert Wissen im Team, so werden Bugs vor der Produktion gefunden, und so lernen neue Entwickler die Konventionen der Codebase kennen.

Was er enthält: ein Diff (jede geänderte Zeile), einen Titel und eine Beschreibung, einen Diskussions-Thread, einen Review-Status und CI-Ergebnisse.
Was er nicht ist: keine ZIP-Datei, kein E-Mail-Anhang, kein fertiges Produkt. Es ist ein Vorschlag, offen für Prüfung.
Protected Branches + PRs: in einem echten Forschungsteam ist main geschützt — du kannst nicht direkt darauf pushen. Jede Änderung läuft über einen PR. Das ist kein Bürokratismus; es ist dein Sicherheitsnetz.
✍️
Einen PR schreiben, der schnell gereviewed wird
Reviewer haben wenig Zeit — mach es ihnen leicht, Ja zu sagen

Die PR-Beschreibungsvorlage

Eine leere Beschreibung ist der häufigste Grund, warum PRs unreviewed liegen bleiben. Reviewer brauchen Kontext, um ihren Job zu machen. Fülle diese Vorlage jedes Mal aus — es dauert 3 Minuten und spart allen eine Stunde.

## Was macht dieser PR? ## Warum ist diese Änderung notwendig? ## Wie wurde es getestet? ## Screenshots / Ausgabe (bei UI- oder Datenänderungen) ## Checkliste - [ ] Test-Suite lokal ausgeführt - [ ] Keine Debug-Logs / auskommentierter Code hinterlassen - [ ] Branch ist aktuell mit main
Halte PRs klein. Ein Diff unter 400 Zeilen wird in Minuten gereviewed. 1.000 Zeilen werden überflogen oder aufgeschoben.
Eine logische Änderung pro PR — nicht „Ich habe den Bug gefixt UND das Modul refaktoriert UND die Docs aktualisiert."
Draft-PRs sind valide. Öffne einen früh, um Kontext zu teilen und frühes Feedback zu erhalten — markiere ihn als bereit, wenn er es wirklich ist.

Schlüsselkonzepte — Öffne jede Karte
PR-Fragen, die neue Team-Entwickler überraschen
Drei Merge-Strategien — du wirst allen drei begegnen
GitHub bietet drei Wege, einen PR zu mergen. Das Verständnis davon verhindert Überraschungen, wenn du im git log etwas Unerwartetes siehst. Teams erzwingen meist eine Strategie über die Repository-Einstellungen — frage am ersten Tag, welche dein Team verwendet.
Die drei Strategien
Merge-Commit (Standard) — erstellt einen neuen Merge-Commit auf main, der beide Historien verbindet. Die vollständige Branch-Historie bleibt erhalten. Gut für Feature-Arbeit, bei der ein klarer "dieser PR landete hier"-Marker nützlich ist.

Squash und Merge — fasst alle Commits auf deinem Branch zu einem einzigen Commit zusammen und fügt ihn zu main hinzu. Main bleibt sauber und linear. Einzelne WIP-Commits verschwinden. Verbreitet in Teams, die auf einen aufgeräumten Log achten.

Rebase und Merge — spielt jeden Commit deines Branches auf main ab, als wären sie dort direkt geschrieben worden, mit neuen SHAs. Kein Merge-Commit. Sauberste Historie, aber verliert den Branch-Verweis im Log.
Nach dem Merge: Feature-Branch sofort löschen. Er hat seinen Zweck erfüllt.
Nach dem Merge lokal: git switch main && git pull, um den gemergten Commit zu erhalten.
Der Review-Antwort-Workflow
Professionell auf Review-Kommentare zu reagieren ist eine Kernkompetenz im Team. Das Ziel ist nicht, das Argument zu „gewinnen" — es geht darum, zum besten Code zu kommen. Reviewer tun dir einen Gefallen.
Schritt für Schritt
1. Alle Kommentare lesen, bevor du Änderungen machst — einige hängen zusammen.

2. Für jeden Kommentar: entweder beheben und antworten „Erledigt, aktualisiert in [SHA]", oder diskutieren — „Ich habe das beibehalten, weil X — macht das Sinn?"

3. Änderungen lokal auf demselben Branch vornehmen und pushen:
git add . git commit -m "fix: null-check laut Review-Feedback anpassen" git push
Der PR aktualisiert sich automatisch — kein Schließen und Wiedereröffnen nötig.

4. Jeden erledigten Kommentar auf GitHub als Resolved markieren.

5. Den Reviewer zur erneuten Überprüfung anfordern — lass ihn nicht raten, dass du bereit bist.
Kein Force-Push auf einen PR-Branch, nachdem das Review begonnen hat — das überschreibt die Kommentar-Anker des Reviewers.
Ton: kurz, kollaborativ, nicht defensiv. „Guter Hinweis — behoben" ist besser als ein Absatz voller Rechtfertigung.
Branch synchron halten
Das passiert ständig in aktiven Teams — jemand anderes hat seinen PR in main gemergt, während du gearbeitet hast. Dein Branch ist jetzt hinter main. Behebe es, bevor du um einen Merge bittest.
Die richtige Lösung
# 1. Zu main wechseln und aktualisieren git switch main git pull # 2. Zurück zum Feature-Branch git switch feature/my-feature # 3. Main in deinen Branch mergen (holt deinen Branch auf den Stand von main) git merge main # 4. Etwaige Konflikte lösen (siehe Thema 3), dann pushen git push
Dein PR ist jetzt aktuell. Die „out of date"-Warnung verschwindet und CI läuft neu.
⚠️ Githubs „Update branch"-Schaltfläche macht dasselbe per UI. Nutze sie bei sauberem Merge. Bei Konflikten musst du sie lokal lösen — GitHub kann Code-Konflikte nicht automatisch auflösen.
Continuous Integration — dein automatisierter erster Reviewer
CI läuft automatisch bei jedem Push auf einen PR. Es ist nicht optional und nicht persönlich. Ein fehlgeschlagener CI-Check bedeutet, dass die automatisierte Hürde etwas gefunden hat, bevor ein Mensch es musste.
Was CI typischerweise prüft
Build — lässt sich der Code kompilieren / startet die App fehlerfrei?

Tests — bestehen alle Unit- und Integrationstests? Ein einziger fehlgeschlagener Test blockiert den Merge.

Linting — hält der Code die Stilregeln ein (PEP 8, ESLint usw.)? Formatierungsfehler lassen CI scheitern.

Typprüfungen — gibt es bei getypten Codebasen (TypeScript, mypy) Typfehler?
Klicke auf den Namen des fehlgeschlagenen Checks auf GitHub — er verlinkt direkt zum CI-Log. Der Fehler steht fast immer in den letzten 20 Zeilen.
Niemals einen Teamkollegen bitten, „einfach zu mergen" trotz fehlgeschlagenem CI. Dieser Fehler hat einen Grund.
Commit-Messages sind die erste Zeile der Dokumentation
Im Team ist git log ein gemeinsames Changelog, das alle lesen. „fix stuff" sagt einem Teamkollegen nichts. Eine gut geschriebene Commit-Message erklärt, was und warum — in 72 Zeichen.
Die Formel
Format: type(scope): kurze Beschreibung im Imperativ

feat(fhir): LOINC-Code-Mapping für Laborbefunde hinzufügen fix(auth): Token-Refresh-Abschneidung bei bcrypt-Hash beheben docs(api): Paginierungsparameter in README klären chore(deps): FastAPI auf 0.111.0 aktualisieren refactor(models): Geräteschema in separates Modul auslagern
Regeln: mit Verb beginnen ("add"/"hinzufügen", "fix"/"beheben"). Unter 72 Zeichen. Kein Punkt am Ende. Das Was im Betreff, das Warum im Body falls nötig.
Code Review ist eine Fähigkeit, nicht nur Code lesen
Neue Entwickler denken, Code Review dient dem Bug-Finden. Das stimmt — aber es geht auch um Korrektheit, Klarheit, Sicherheit und Mentoring.
Reviewer-Checkliste
Korrektheit — tut es, was die PR-Beschreibung sagt? Sind Randfälle abgedeckt?

Klarheit — kann ein neues Teammitglied das in 6 Monaten noch verstehen?

Sicherheit — hardcodierte Zugangsdaten, unvalidierte Eingaben, exponierte Endpoints?

Tests — gibt es Tests für das neue Verhalten? Testen sie das Richtige?

Umfang — macht der PR nur das, was er behauptet?
Nitpick-Kommentare mit „nit:"-Präfix kennzeichnen — „nit: `x` in `deviceId` umbenennen für Klarheit". Der Autor kann Nits ignorieren; sie blockieren keinen Merge.
Innerhalb von 24 Stunden antworten. Ein PR, der tagelang unreviewed liegt, ist ein Teamprozess-Fehler, kein Git-Problem.

ما هو الـ Pull Request فعلاً
المفهوم أولاً — قبل أن تلمس واجهة GitHub

الـ Pull Request محادثة، لا نقل ملفات

الاسم مضلل. الـ PR لا "يسحب" شيئاً — هو طلب مراجعة يقول: "لديّ تغييرات على branch الخاص بي. من فضلك ألقِ نظرة عليها، ناقشها، وادمجها في الـ branch المستهدف إذا كانت جيدة."

الدمج نفسه هو الجزء السهل. محادثة المراجعة — التعليقات، التغييرات المطلوبة، الموافقات — هي القيمة الحقيقية. هكذا تنتقل المعرفة في الفريق، وهكذا تُكتشف الأخطاء قبل الإنتاج، وهكذا يتعلم المطورون الجدد الكود.

ما يحتويه: diff (كل سطر تغيّر)، عنوان ووصف تكتبهما، خيط نقاش، حالة مراجعة، ونتائج CI.
ما ليس هو: ليس ملف ZIP، ولا مرفق بريد إلكتروني، ولا منتجاً نهائياً. هو اقتراح مفتوح للتدقيق.
Protected Branches + PRs: في فريق بحثي حقيقي، main محمي — لا يمكنك الـ push مباشرة إليه. كل تغيير يمر عبر PR. هذا ليس بيروقراطية؛ هو شبكة الأمان الخاصة بك.
✍️
كيف تكتب PR يُراجَع بسرعة
وقت المراجعين محدود — اجعل من السهل عليهم قول نعم

قالب وصف الـ PR

الوصف الفارغ هو السبب الأكثر شيوعاً لبقاء الـ PRs دون مراجعة. المراجعون يحتاجون إلى سياق ليؤدوا عملهم. املأ هذا القالب في كل مرة — يستغرق 3 دقائق ويوفر على الجميع ساعة.

## What does this PR do? ## Why is this change needed? ## How was it tested? ## Screenshots / output (if UI or data change) ## Checklist - [ ] Ran the full test suite locally - [ ] No debug logs / commented-out code left in - [ ] Branch is up to date with main
اجعل الـ PRs صغيرة. الـ diff الذي يقل عن 400 سطر يُراجَع في دقائق. 1000 سطر يُتصفَّح أو يُؤجَّل.
تغيير منطقي واحد لكل PR — ليس "أصلحت الـ bug وأعدت الـ refactor وحدّثت الـ docs".
Draft PRs صالحة. افتح واحداً مبكراً لمشاركة السياق والحصول على تغذية راجعة مبكرة — ضعه كجاهز عندما يكون كذلك فعلاً.
💬
المفاهيم الأساسية — افتح كل بطاقة
أسئلة الـ PR التي تفاجئ المطورين الجدد في بيئات الفريق
ثلاث استراتيجيات دمج — ستصادفها كلها
يقدم GitHub ثلاث طرق لدمج PR. فهمها يمنع المفاجآت عند قراءة git log. الفرق عادةً يُجبر على استراتيجية واحدة — اسأل عنها في يومك الأول.
الاستراتيجيات الثلاث
Merge commit (الافتراضي) — ينشئ commit دمج جديد على main يربط التاريخَين معاً. تاريخ الـ branch كاملاً محفوظ. مناسب للـ feature work التي تستفيد من علامة واضحة "وصل هذا الـ PR هنا".

Squash and merge — يضغط كل commits الـ branch في commit واحد ثم يضيفه إلى main. Main يبقى نظيفاً وخطياً. Commits الـ WIP الفردية تختفي. شائع في الفرق التي تهتم بسجل مقروء.

Rebase and merge — يُعيد تشغيل كل commit من branch-ك فوق main كأنه كُتب هناك مباشرةً مع SHAs جديدة. لا merge commit. أنظف تاريخ، لكن يفقد مرجع الـ branch في السجل.
بعد الدمج: احذف الـ feature branch فوراً. قد أدّى غرضه.
بعد الدمج محلياً: git switch main && git pull للحصول على الـ commit المدموج.
سير عمل الرد على المراجعة
الرد المهني على تعليقات المراجعة مهارة أساسية في الفريق. الهدف ليس "الفوز" بالنقاش — بل الوصول إلى أفضل كود. المراجعون يقدمون لك خدمة.
خطوة بخطوة
1. اقرأ جميع التعليقات قبل إجراء أي تغييرات — بعضها مترابط.

2. لكل تعليق: إما أصلحه وردّ "تم، محدّث في [SHA]"، أو ناقشه — "أبقيت هذا كما هو لأن X، هل هذا منطقي؟"

3. أجرِ تغييراتك محلياً على نفس الـ branch وادفعها:
git add . git commit -m "fix: address null-check review feedback" git push
الـ PR يتحدث تلقائياً — لا حاجة لإغلاقه وإعادة فتحه.

4. علّم كل تعليق تم التعامل معه بـ Resolved على GitHub.

5. اطلب المراجعة مرة أخرى من المراجع — لا تتركه يخمن أنك جاهز.
لا تستخدم force-push على branch الـ PR بعد بدء المراجعة — يُعيد كتابة التاريخ ويفقد مراسي تعليقات المراجع.
النبرة: مختصرة، تعاونية، غير دفاعية. "ملاحظة جيدة — تم الإصلاح" أفضل من فقرة مليئة بالتبريرات.
إبقاء الـ branch متزامناً
يحدث هذا باستمرار في الفرق النشطة — دمج أحدهم PR-ه في main بينما كنت تعمل. الـ branch الخاص بك الآن خلف main. حله قبل طلب الدمج.
الحل الصحيح
# ١. الانتقال إلى main وسحب الأحدث git switch main git pull # ٢. العودة إلى feature branch git switch feature/my-feature # ٣. دمج main في branch-ك (يجلب branch-ك إلى مستوى main) git merge main # ٤. حل أي تعارضات تظهر (انظر الموضوع ٣)، ثم push git push
الـ PR الآن محدّث. تختفي رسالة "out of date" ويُعاد تشغيل CI.
⚠️ زر "Update branch" في GitHub يفعل نفس الشيء عبر الواجهة. استخدمه عند الدمج النظيف. عند وجود تعارضات يجب حلها محلياً — GitHub لا يستطيع حل تعارضات الكود تلقائياً.
Continuous Integration — المراجع الأول الآلي
CI يعمل تلقائياً عند كل push على PR. ليس اختيارياً وليس شخصياً. فشل فحص CI يعني أن البوابة الآلية اكتشفت شيئاً قبل أن يضطر إنسان للقيام بذلك.
ما يفحصه CI عادةً
Build — هل يُترجَم الكود؟ هل تبدأ التطبيق بدون أخطاء؟

Tests — هل تنجح جميع الاختبارات؟ اختبار واحد فاشل يمنع الدمج.

Linting — هل يتبع الكود قواعد الأسلوب (PEP 8، ESLint...)؟ أخطاء التنسيق تُفشل CI.

Type checks — في codebases المكتوبة (TypeScript، mypy)، هل هناك أخطاء نوع؟
انقر على اسم الفحص الفاشل على GitHub — يرتبط مباشرة بسجل CI. الخطأ في الغالب في الـ 20 سطراً الأخيرة.
لا تطلب من زميل "الدمج فقط" رغم فشل CI. هذا الفشل له سبب.
رسائل الـ Commit هي السطر الأول من التوثيق
في الفريق، git log سجل تغييرات مشترك يقرؤه الجميع. "fix stuff" لا يخبر أي زميل بشيء. رسالة commit جيدة تشرح ماذا وسبب — في 72 حرفاً.
الصيغة
الصياغة: type(scope): وصف مختصر بصيغة الأمر

feat(fhir): add LOINC code mapping for lab results fix(auth): resolve token refresh truncation on bcrypt hash docs(api): clarify pagination parameters in README chore(deps): upgrade FastAPI to 0.111.0 refactor(models): extract device schema into separate module
القواعد: ابدأ بفعل أمر ("add"، "fix"، "remove"). أقل من 72 حرفاً. بدون نقطة في النهاية. الـ ماذا في الموضوع، الـ لماذا في الـ body عند الحاجة.
بادئة النوع تطابق تسمية الـ branch: feat، fix، docs، chore، refactor، test.
عند squash merge، رسائل commits الفردية تتدمج في واحدة — عنوان الـ PR يصبح الـ commit على main. اجعل عنوان الـ PR رسالة commit جيدة أيضاً.
مراجعة الكود مهارة، لا مجرد قراءة كود
المطورون الجدد يظنون أن مراجعة الكود تهدف لاكتشاف الـ bugs. صحيح — لكن أيضاً الصحة، الوضوح، الأمان، والتوجيه.
قائمة فحص المراجع
الصحة — هل يفعل ما تقوله الـ PR description؟ هل الحالات الطرفية مغطاة؟

الوضوح — هل يستطيع عضو فريق جديد فهمه بعد 6 أشهر؟ هل الأسماء ذات معنى؟

الأمان — بيانات اعتماد مكتوبة ثابتة، مدخلات غير محقَّقة، endpoints مكشوفة؟

الاختبارات — هل هناك اختبارات للسلوك الجديد؟ هل تختبر الشيء الصحيح؟

النطاق — هل يفعل الـ PR فقط ما يدّعيه؟
الملاحظات الثانوية بادئة "nit:" — "nit: أعد تسمية x إلى deviceId للوضوح". يمكن للمؤلف تجاهل الـ nits؛ لا تمنع الدمج.

التطبيق العملي
افعل هذا قبل الانتقال إلى الموضوع الثالث — تجربة PR حقيقية في بيئة آمنة

افتح PR وراجعه وادمجه

من المنهج: أنشئ feature branch، أجرِ تغييراً صغيراً، ادفعه، افتح PR على GitHub، راجعه بنفسك، وادمجه. كل خطوة مقصودة — لاحظ ما يعرضه GitHub في كل مرحلة.

# ١. ابدأ من main محدّث git switch main && git pull # ٢. أنشئ feature branch git switch -c feature/pr-practice # ٣. أجرِ تغييراً حقيقياً echo "# PR practice" >> NOTES.md git add NOTES.md git commit -m "docs: add PR practice note" # ٤. ادفع الـ branch إلى GitHub git push -u origin feature/pr-practice # ٥. اذهب إلى GitHub — انقر "Compare & pull request" # املأ قالب الوصف. افتح الـ PR. # ٦. في تبويب "Files changed" راجع diff-ك # اترك تعليقاً واحداً على الأقل على كودك — مارس الواجهة # ٧. ادمج الـ PR عبر GitHub UI → "Squash and merge" → تأكيد # ٨. نظّف محلياً git switch main git pull git branch -d feature/pr-practice

بعد هذا يجب أن تتمكن من فتح PR من البداية للنهاية دون تردد. الـ PR الأول الذي تفتحه على codebase حقيقي سيكون مألوفاً، لا غريباً.
