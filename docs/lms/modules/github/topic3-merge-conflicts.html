Git & GitHub — Topic 3 of 5

Merge Conflicts — Read, Resolve, Move On

A conflict is not an error — it is Git telling you it needs a human decision. Learn to read the markers and you will resolve any conflict calmly in under five minutes.

⚡
What a Conflict Actually Is
Demystify it first — panic is optional, understanding is not

Git can't decide — so it stops and asks you

A merge conflict happens when two branches changed the same lines of the same file in different ways. Git merges everything it can automatically — line additions, deletions in different areas, new files — without any help from you. It only stops when it genuinely cannot know which version to keep.

This is not a mistake. It is Git being honest: "Both branches touched line 42. I don't know which version is correct. You do. Please decide."

What causes it: you and a teammate both edited the same lines, or one of you deleted a file the other modified, or both branches added the same filename with different content.
What Git does: it pauses the merge, marks the conflicting sections with special markers inside the file, and waits. Your working directory is now in a "mid-merge" state.
What you do: edit the file to contain the final correct version, remove the markers, stage it, and complete the merge with a commit.
🔍
Reading the Conflict Markers
Three lines — once you know what each means, the file tells you exactly what to do

Anatomy of a conflict block

When Git writes conflict markers into a file, it always uses this three-part structure:

<<<<<<< HEAD (your branch — what you currently have) product_id = record.get("product_id") category = "electronics" ======= (divider — everything above is yours, everything below is theirs) product_id = record.get("id") category = record.get("category", "general") >>>>>>> feature/product-schema (incoming branch being merged)
<<<<<<< HEAD — your version (the branch you ran git merge from). Everything between this line and ======= is what you had.
======= — the divider. Above = yours. Below = theirs.
>>>>>>> feature/product-schema — the incoming version (the branch being merged in). Everything between ======= and this line is what they had.
One file can have multiple conflict blocks — one per conflicting section. Resolve each one.

Your job: delete all three marker lines and leave only the code that should survive. That might be your version, their version, or a blend of both — you decide based on what is actually correct.

Key Concepts — Open Every Card
The conflict questions every developer faces in their first week on a team
The complete CLI workflow
This is the sequence you'll run every time. Learn it once and you will never feel lost mid-conflict again — regardless of which editor or tool you prefer, the underlying steps are always the same.
Step by step
# 1. Start the merge (conflict occurs here) git switch main git merge feature/product-schema # → CONFLICT (content): Merge conflict in src/models/product.py # → Automatic merge failed; fix conflicts and then commit the result. # 2. See which files have conflicts git status # → both modified: src/models/product.py # 3. Open the conflicting file in your editor # Find every <<<<<<< block, decide what the final code should be, # delete the three marker lines, save the file. # 4. Stage the resolved file git add src/models/product.py # 5. Confirm no conflicts remain git status # → All conflicts fixed but you are still merging. # 6. Complete the merge with a commit git commit # Git pre-fills the message: "Merge branch 'feature/product-schema'" # Save and close the editor to accept it.
The merge is now complete. The commit history shows both branches converging.
git status is your compass throughout. Run it after every step to see where you are.
Don't git add . blindly — stage only the files you've resolved. Accidentally staging an unresolved file with markers still in it will commit broken code.
If you staged a file and want to un-stage it: git restore --staged src/models/product.py
The visual merge editor — faster for most conflicts
VS Code detects conflict markers and replaces them with a visual editor that lets you click to accept versions instead of editing raw markers. For most conflicts this is faster and less error-prone than editing the markers by hand.
In VS Code
1. Open the conflicting file — VS Code highlights conflict blocks in color and shows four action buttons above each block:

  Accept Current Change — keep your version (HEAD), discard theirs.
  Accept Incoming Change — keep their version, discard yours.
  Accept Both Changes — append both versions one after the other (useful when both additions should survive).
  Compare Changes — opens a side-by-side diff view.

2. For complex conflicts, click "Resolve in Merge Editor" (VS Code 1.69+) — a three-panel view: yours on the left, theirs on the right, result in the centre. Edit the result panel directly for blended resolutions.

3. Save the file. Stage it: git add <file>. Then git commit.
The Source Control panel (Ctrl+Shift+G) lists all conflicting files under "Merge Changes" — click each one to open it.
"Accept Both Changes" stacks the two versions — you usually still need to edit the result to remove duplication.
VS Code does NOT stage or commit for you. After clicking "Accept", you still run git add and git commit.
The escape hatch — always available before you commit
You are mid-merge and realise the conflict is too complex to resolve right now — you need to talk to your teammate first, or you just want to start over. As long as you haven't run git commit yet, you can always abort.
Abort and return to the pre-merge state
git merge --abort
This rewinds everything: your working directory returns to exactly how it was before you ran git merge. All conflict markers disappear. No partial state is left behind.

After aborting, git status should show a clean working tree on your original branch.
⚠️ git merge --abort only works while a merge is in progress (i.e. before git commit). After you commit the merge, the merge is complete — use git revert to undo it if needed (covered in Topic 4).
Not sure if a merge is in progress? Run git status — it will say "You have unmerged paths" or "All conflicts fixed but you are still merging."
After aborting, pull your teammate's changes, understand what they did, then re-attempt the merge — this time you'll know what to expect.
The most common conflict mistake — easy to recover from
You staged and committed a file that still contained <<<<<<< markers. Your app now has literal conflict markers in its source code. It happens. Here's how to fix it without panic.
Recovery — not yet pushed
# Undo the commit but keep the file changes staged git reset --soft HEAD~1 # Now edit the file — remove all conflict markers properly # Then re-stage and re-commit git add src/models/product.py git commit -m "fix: resolve merge conflict in product model"
⚠️ If you already pushed the broken commit, don't use git reset (it rewrites shared history). Instead, edit the file to remove markers, then commit and push a new fix commit. Topic 4 covers the full undo toolkit.
Prevention: configure your editor or CI to search for <<<<<<< in staged files before committing. Some teams add this as a pre-commit hook.
git diff --check detects whitespace errors and conflict markers in staged/unstaged changes — run it before committing after any merge.
Conflicts are inevitable — but their size is not
You cannot eliminate conflicts on a team — but you can keep them small and fast to resolve. The techniques below are the habits that separate developers who dread merge day from those who don't notice it.
The four habits
1. Merge main into your branch regularly — not once before your PR, but every day or two. Small, frequent merges surface conflicts early while they're still trivial.

git switch main && git pull git switch feature/my-feature git merge main
2. Keep branches short-lived — a branch open for a week diverges far less than one open for a month. The longer a branch lives, the more painful the eventual merge.

3. One logical change per branch — narrow branches touch fewer files and conflict less.

4. Communicate before touching shared files — if you know a teammate is about to refactor the same module, coordinate. A 2-minute message saves a 20-minute conflict.
The conflict you dread most is always the one from the branch nobody merged from main in three weeks.
Trunk-based development (very short-lived branches, merged daily) is the extreme end of this principle — worth reading about after you're comfortable with the basics.
Reuse Recorded Resolution — Git remembers how you fixed it last time
If you regularly merge a long-lived branch and keep encountering the same conflict in the same file, rerere can remember your resolution and apply it automatically next time. It's an advanced feature — worth knowing it exists.
How to enable it
# Enable globally (recommended — set it once and forget it) git config --global rerere.enabled true
Once enabled: next time you hit a conflict, resolve it normally. Git records the resolution. The next time the same conflict appears (e.g. on a subsequent merge), Git applies your previous resolution automatically.

You'll see: Resolved 'src/models/product.py' using previous resolution.
Most useful in long-running integration branches that repeatedly merge the same feature branches.
Don't rely on it blindly — always verify the auto-applied resolution makes sense before committing.
Skip this if you're just starting out — master the manual workflow first.

Hands-On Practice
Create a real conflict, resolve it — builds the muscle memory in 15 minutes

Engineer a conflict, then fix it

The best way to be calm about conflicts is to have caused and resolved one deliberately. Run this script — it manufactures a predictable conflict so you can practise the full resolution workflow without any stakes.

# 1. Fresh practice repo git init conflict-practice && cd conflict-practice echo "product_id = record.get('id')" > model.py git add . && git commit -m "Initial model" # 2. Create a feature branch and change the same line git switch -c feature/inventory-lookup echo "product_id = record.get('product_id')" > model.py git commit -am "feat: use explicit product_id key" # 3. Back on main, make a different change to the same line git switch main echo "product_id = record.get('sku')" > model.py git commit -am "chore: align key name with API spec" # 4. Now merge — this WILL conflict git merge feature/inventory-lookup # → CONFLICT (content): Merge conflict in model.py # 5. Inspect the markers cat model.py # 6. Resolve: edit model.py to contain the correct final line, # remove all three marker lines, save # 7. Stage and commit git add model.py git commit # (accept the pre-filled merge commit message) # 8. Verify the graph git log --all --oneline --graph

After this exercise you should be able to look at any conflict file calmly, identify which version to keep (or blend), and complete the merge without hesitation.

Was ein Konflikt wirklich ist
Erst entmystifizieren — Panik ist optional, Verständnis nicht

Git kann nicht entscheiden — also hält es an und fragt dich

Ein Merge-Konflikt entsteht, wenn zwei Branches dieselben Zeilen derselben Datei auf unterschiedliche Weise geändert haben. Git merged alles, was es kann, automatisch — Zeilenergänzungen, Löschungen in verschiedenen Bereichen, neue Dateien — ganz ohne deine Hilfe. Es stoppt nur, wenn es genuinen keine Ahnung hat, welche Version zu behalten ist.

Das ist kein Fehler. Es ist Git, das ehrlich ist: "Beide Branches haben Zeile 42 angefasst. Ich weiß nicht, welche Version korrekt ist. Du weißt es. Bitte entscheide."

Was es verursacht: du und ein Teammitglied haben beide dieselben Zeilen bearbeitet, oder einer hat eine Datei gelöscht, die der andere geändert hat, oder beide Branches haben denselben Dateinamen mit unterschiedlichem Inhalt hinzugefügt.
Was Git tut: es pausiert den Merge, markiert die konfliktierenden Abschnitte mit speziellen Markierungen innerhalb der Datei und wartet. Dein Arbeitsverzeichnis befindet sich nun in einem "mitten-im-Merge"-Zustand.
Was du tust: bearbeite die Datei so, dass sie die endgültige korrekte Version enthält, entferne die Markierungen, stage sie, und schließe den Merge mit einem Commit ab.
🔍
Die Konflikt-Markierungen lesen
Drei Zeilen — sobald du weißt, was jede bedeutet, sagt dir die Datei genau, was zu tun ist

Anatomie eines Konflikt-Blocks

Wenn Git Konflikt-Markierungen in eine Datei schreibt, verwendet es immer diese dreiteilige Struktur:

<<<<<<< HEAD (dein Branch — was du aktuell hast) product_id = record.get("product_id") category = "electronics" ======= (Trennlinie — alles darüber ist deins, alles darunter ist deren Version) product_id = record.get("id") category = record.get("category", "general") >>>>>>> feature/product-schema (eingehender Branch, der gemergt wird)
<<<<<<< HEAD — deine Version (der Branch, von dem aus du git merge ausgeführt hast). Alles zwischen dieser Zeile und ======= ist dein Stand.
======= — die Trennlinie. Oben = deins. Unten = ihres.
>>>>>>> feature/product-schema — die eingehende Version (der Branch, der gemergt wird). Alles zwischen ======= und dieser Zeile ist ihr Stand.
Eine Datei kann mehrere Konflikt-Blöcke haben — einen pro konfliktierende Sektion. Löse jeden einzeln.

Deine Aufgabe: lösche alle drei Markierungszeilen und lasse nur den Code übrig, der überleben soll — deine Version, ihre Version, oder eine Mischung aus beiden.

Der vollständige CLI-Workflow
Das ist die Sequenz, die du jedes Mal ausführst. Lerne sie einmal und du wirst dich nie mehr mitten in einem Konflikt verloren fühlen.
Schritt für Schritt
# 1. Merge starten (Konflikt tritt hier auf) git switch main git merge feature/product-schema # → CONFLICT (content): Merge conflict in src/models/product.py # 2. Welche Dateien haben Konflikte? git status # → both modified: src/models/product.py # 3. Konfliktdatei im Editor öffnen # Jeden <<<<<<<-Block finden, entscheiden, was der finale Code ist, # alle drei Markierungszeilen löschen, Datei speichern. # 4. Aufgelöste Datei stagen git add src/models/product.py # 5. Bestätigen, dass keine Konflikte mehr vorhanden sind git status # → All conflicts fixed but you are still merging. # 6. Merge mit einem Commit abschließen git commit # Git füllt die Nachricht vor: "Merge branch 'feature/product-schema'"
Der Merge ist nun abgeschlossen. Der Commit-Verlauf zeigt, wie beide Branches konvergieren.
git status ist dein Kompass. Führe es nach jedem Schritt aus.
Kein blindes git add . — stage nur die Dateien, die du aufgelöst hast.
Datei wieder entstagen: git restore --staged src/models/product.py
Der visuelle Merge-Editor — für die meisten Konflikte schneller
VS Code erkennt Konflikt-Markierungen und ersetzt sie durch einen visuellen Editor, mit dem du per Klick Versionen akzeptieren kannst, statt rohe Markierungen zu bearbeiten.
In VS Code
1. Konfliktdatei öffnen — VS Code hebt Konfliktblöcke farbig hervor und zeigt vier Schaltflächen:

  Accept Current Change — deine Version behalten (HEAD), ihre verwerfen.
  Accept Incoming Change — ihre Version behalten, deine verwerfen.
  Accept Both Changes — beide Versionen hintereinander anfügen.
  Compare Changes — Seite-an-Seite-Diff-Ansicht öffnen.

2. Bei komplexen Konflikten: "Resolve in Merge Editor" (VS Code 1.69+) — drei-Panel-Ansicht: deine links, ihre rechts, Ergebnis in der Mitte.

3. Datei speichern. Stagen: git add <file>. Dann git commit.
Das Source-Control-Panel (Strg+Shift+G) listet alle Konfliktdateien unter "Merge Changes".
VS Code staged oder committet NICHT für dich. Nach dem Klick auf "Accept" noch immer git add und git commit ausführen.
Der Notausgang — immer verfügbar, bevor du committetest
Du bist mitten in einem Merge und merkst, dass der Konflikt zu komplex ist, um ihn jetzt zu lösen. Solange du noch kein git commit ausgeführt hast, kannst du jederzeit abbrechen.
Abbruch und Rückkehr zum Zustand vor dem Merge
git merge --abort
Das macht alles rückgängig: dein Arbeitsverzeichnis kehrt genau in den Zustand zurück, in dem es vor git merge war. Alle Konfliktmarkierungen verschwinden. Kein Teilzustand bleibt zurück.
⚠️ git merge --abort funktioniert nur, solange ein Merge läuft (also vor git commit). Nach dem Commit ist der Merge abgeschlossen — verwende dann git revert (behandelt in Thema 4).
Der häufigste Konflikt-Fehler — leicht zu beheben
Du hast eine Datei gestagt und committet, die noch <<<<<<<-Markierungen enthielt. Dein App-Code enthält jetzt buchstäbliche Konfliktmarkierungen. Es passiert. So behebst du es.
Behebung — noch nicht gepusht
# Commit rückgängig machen, Dateiänderungen gestagt lassen git reset --soft HEAD~1 # Datei bearbeiten — alle Konfliktmarkierungen korrekt entfernen # Dann neu stagen und committen git add src/models/product.py git commit -m "fix: Merge-Konflikt in product model auflösen"
Prävention: git diff --check erkennt Konfliktmarkierungen in staged/unstaged Änderungen — vor jedem Merge-Commit ausführen.
Falls bereits gepusht: Datei bearbeiten, Markierungen entfernen, neuen Fix-Commit pushen (kein git reset auf geteilter History).
Konflikte sind unvermeidlich — aber ihre Größe ist es nicht
Du kannst Konflikte im Team nicht eliminieren — aber du kannst sie klein und schnell lösbar halten.
Die vier Gewohnheiten
1. Main regelmäßig in deinen Branch mergen — nicht einmal kurz vor dem PR, sondern jeden Tag oder jeden zweiten Tag.

git switch main && git pull git switch feature/my-feature git merge main
2. Branches kurzlebig halten — ein Branch, der eine Woche offen ist, divergiert weit weniger als einer, der einen Monat offen bleibt.

3. Eine logische Änderung pro Branch — schmale Branches berühren weniger Dateien und kollidieren weniger.

4. Kommunizieren, bevor gemeinsame Dateien angefasst werden — eine 2-Minuten-Nachricht spart 20 Minuten Konfliktlösung.
Reuse Recorded Resolution — Git merkt sich, wie du es letztes Mal gelöst hast
Wenn du denselben Konflikt in derselben Datei immer wieder siehst, kann rerere deine Lösung speichern und automatisch anwenden. Ein fortgeschrittenes Feature — gut zu wissen, dass es existiert.
Aktivierung
# Global aktivieren (einmal einrichten und vergessen) git config --global rerere.enabled true
Einmal aktiviert: löse den Konflikt normal. Git speichert die Lösung. Beim nächsten gleichen Konflikt wendet Git deine vorherige Lösung automatisch an.

Du siehst: Resolved 'src/models/product.py' using previous resolution.
Am nützlichsten bei langlebigen Integrations-Branches, die immer wieder dieselben Feature-Branches mergen.
Überspringe das fürs Erste — meistere zuerst den manuellen Workflow.

ما هو تعارض الدمج فعلاً
أزِل الغموض أولاً — الذعر اختياري، الفهم ليس كذلك

Git لا يستطيع أن يقرر — فيتوقف ويسألك

يحدث تعارض الدمج عندما يغيّر branch-ان نفس الأسطر في نفس الملف بطرق مختلفة. Git يدمج كل ما يستطيع تلقائياً — إضافات الأسطر، الحذف في مناطق مختلفة، الملفات الجديدة — دون أي مساعدة منك. يتوقف فقط عندما لا يعرف فعلاً أي نسخة يحتفظ بها.

هذا ليس خطأ. هذا Git يكون صادقاً: "كلا الـ branch-ين لمسا السطر 42. لا أعرف أي نسخة صحيحة. أنت تعرف. من فضلك قرر."

ما يسببه: أنت وزميلك عدّلتما نفس الأسطر، أو أحدكما حذف ملفاً عدّله الآخر، أو أضاف كلاكما نفس اسم الملف بمحتوى مختلف.
ما يفعله Git: يوقف الدمج، يضع علامات خاصة داخل الملف، وينتظر. مجلد العمل الآن في حالة "منتصف الدمج".
ما تفعله أنت: عدّل الملف ليحتوي النسخة الصحيحة النهائية، احذف العلامات، stage-ه، وأكمل الدمج بـ commit.
🔍
قراءة علامات التعارض
ثلاثة أسطر — بمجرد معرفة ما يعنيه كل منها، يخبرك الملف بالضبط ماذا تفعل

تشريح كتلة التعارض

عندما يكتب Git علامات التعارض في ملف، يستخدم دائماً هذا الهيكل ثلاثي الأجزاء:

<<<<<<< HEAD (branch-ك — ما لديك حالياً) product_id = record.get("product_id") category = "electronics" ======= (الفاصل — كل ما فوقه نسختك، كل ما تحته نسختهم) product_id = record.get("id") category = record.get("category", "general") >>>>>>> feature/product-schema (الـ branch الوارد المراد دمجه)
<<<<<<< HEAD — نسختك (الـ branch الذي نفّذت منه git merge). كل ما بين هذا السطر و======= هو ما كان لديك.
======= — الفاصل. ما فوقه = نسختك. ما تحته = نسختهم.
>>>>>>> feature/product-schema — النسخة الواردة (الـ branch المراد دمجه). كل ما بين ======= وهذا السطر هو ما كان لديهم.
يمكن أن يحتوي الملف على كتل تعارض متعددة — واحدة لكل قسم متعارض. أحل كلاً منها.

مهمتك: احذف الأسطر الثلاث للعلامات واترك فقط الكود الذي يجب أن يبقى — نسختك، نسختهم، أو مزيجاً منهما.

في VS Code
1. افتح الملف المتعارض — VS Code يُبرز كتل التعارض بالألوان ويعرض أربعة أزرار:

  Accept Current Change — احتفظ بنسختك (HEAD)، تجاهل نسختهم.
  Accept Incoming Change — احتفظ بنسختهم، تجاهل نسختك.
  Accept Both Changes — أضف كلا النسختين متتاليتين.
  Compare Changes — افتح عرض diff جنباً إلى جنب.

2. للتعارضات المعقدة: "Resolve in Merge Editor" (VS Code 1.69+) — عرض ثلاثي الأقسام: نسختك يساراً، نسختهم يميناً، النتيجة في المركز.

3. احفظ الملف. Stage-ه: git add <file>. ثم git commit.
لوحة Source Control (Ctrl+Shift+G) تعرض جميع الملفات المتعارضة تحت "Merge Changes".
VS Code لا يقوم بالـ stage أو الـ commit نيابةً عنك. بعد النقر على "Accept" لا تزال تحتاج إلى git add وgit commit.
مخرج الطوارئ — متاح دائماً قبل الـ commit
أنت في منتصف الدمج وتدرك أن التعارض معقد جداً للحل الآن — تحتاج للتحدث مع زميلك أولاً، أو تريد البداية من جديد. ما دمت لم تنفّذ git commit بعد، يمكنك الإلغاء دائماً.
إلغاء والعودة لحالة ما قبل الدمج
git merge --abort
يُعيد هذا كل شيء: مجلد عملك يعود بالضبط كما كان قبل تنفيذ git merge. تختفي جميع علامات التعارض. لا يبقى أي حالة جزئية.
⚠️ git merge --abort يعمل فقط بينما الدمج جارٍ (أي قبل git commit). بعد الـ commit، استخدم git revert للتراجع (مغطى في الموضوع 4).
غير متأكد هل الدمج جارٍ؟ نفّذ git status — سيقول "You have unmerged paths" أو "All conflicts fixed but you are still merging."
الخطأ الأكثر شيوعاً في التعارضات — سهل الإصلاح
قمت بـ stage وcommit لملف لا يزال يحتوي على علامات <<<<<<<. يحدث هذا. إليك كيفية الإصلاح.
الإصلاح — قبل الـ push
# تراجع عن الـ commit مع الإبقاء على تغييرات الملف مُضافة للـ stage git reset --soft HEAD~1 # عدّل الملف — احذف جميع علامات التعارض بشكل صحيح # ثم أعد الـ stage والـ commit git add src/models/product.py git commit -m "fix: resolve merge conflict in product model"
للوقاية: git diff --check يكتشف علامات التعارض في التغييرات — نفّذه قبل كل commit بعد دمج.
إذا push-ت بالفعل: عدّل الملف، احذف العلامات، ادفع commit إصلاح جديد (لا تستخدم git reset على تاريخ مشترك).
التعارضات حتمية — لكن حجمها ليس كذلك
لا يمكنك القضاء على التعارضات في الفريق — لكن يمكنك إبقاءها صغيرة وسريعة الحل.
العادات الأربع
١. ادمج main في branch-ك بانتظام — ليس مرة واحدة قبل الـ PR، بل كل يوم أو يومين. تعارضات صغيرة ومتكررة أفضل من تعارض ضخم واحد.

git switch main && git pull git switch feature/my-feature git merge main
٢. أبقِ الـ branches قصيرة العمر — branch مفتوح أسبوعاً يتباعد أقل بكثير من branch مفتوح شهراً.

٣. تغيير منطقي واحد لكل branch — branches ضيقة تلمس ملفات أقل وتتعارض أقل.

٤. تواصل قبل لمس الملفات المشتركة — رسالة دقيقتين توفر 20 دقيقة حل تعارض.
التعارض الأكثر إرهاقاً هو دائماً الذي جاء من branch لم يُدمج فيه main منذ ثلاثة أسابيع.
Reuse Recorded Resolution — Git يتذكر كيف أصلحته في المرة الأخيرة
إذا كنت تواجه نفس التعارض في نفس الملف مراراً، يمكن لـ rerere أن يحفظ حلك ويطبقه تلقائياً في المرة القادمة. ميزة متقدمة — من الجيد معرفة وجودها.
كيفية التفعيل
# تفعيل عالمياً (اضبطه مرة وانسَه) git config --global rerere.enabled true
بمجرد التفعيل: احل التعارض بشكل طبيعي. Git يحفظ الحل. في المرة القادمة عند ظهور نفس التعارض، يطبق Git حلك السابق تلقائياً.

ستظهر: Resolved 'src/models/product.py' using previous resolution.
الأكثر فائدة في branches التكامل طويلة العمر التي تدمج نفس الـ feature branches مراراً.
تخطَّ هذا في البداية — أتقن سير العمل اليدوي أولاً.

التطبيق العملي
أنشئ تعارضاً حقيقياً وحلّه — يبني ذاكرة العضلات في 15 دقيقة

اصنع تعارضاً ثم أصلحه

أفضل طريقة للهدوء أمام التعارضات هي أن تكون قد تسببت في واحد وحللته عن قصد. نفّذ هذا السكريبت — يصنع تعارضاً متوقعاً للتدريب بدون أي عواقب.

# ١. مستودع تدريبي جديد git init conflict-practice && cd conflict-practice echo "product_id = record.get('id')" > model.py git add . && git commit -m "Initial model" # ٢. أنشئ feature branch وغيّر نفس السطر git switch -c feature/inventory-lookup echo "product_id = record.get('product_id')" > model.py git commit -am "feat: use explicit product_id key" # ٣. عد إلى main وأجرِ تغييراً مختلفاً على نفس السطر git switch main echo "product_id = record.get('sku')" > model.py git commit -am "chore: align key name with API spec" # ٤. ادمج الآن — سيحدث تعارض بالتأكيد git merge feature/inventory-lookup # → CONFLICT (content): Merge conflict in model.py # ٥. تفحّص العلامات cat model.py # ٦. الحل: عدّل model.py لتحتوي السطر الصحيح النهائي، # احذف الأسطر الثلاثة للعلامات، احفظ # ٧. Stage وcommit git add model.py git commit # ٨. تحقق من الرسم البياني git log --all --oneline --graph

بعد هذا التمرين يجب أن تتمكن من النظر إلى أي ملف تعارض بهدوء، تحديد أي نسخة تحتفظ بها (أو تمزجها)، وإكمال الدمج دون تردد.
