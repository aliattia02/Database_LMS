Undo Without Disaster

git reset, git revert, git stash, git cherry-pick — three scenarios, the right tool for each, zero accidental data loss.

🧭
Where Are You? The Three Undo Scenarios
The right undo command depends entirely on how far your change has travelled — get this picture right first

Every undo question reduces to one question: where does this change live?

Before you reach for any undo command, answer one question first: has the change been staged? Committed? Pushed? Git gives you a different, safer tool for each stage — and reaching for the wrong one is exactly how "undo" turns into "lost work" or "broken history for the whole team."

The three scenarios below map directly onto the three commands this topic covers. Identify which one you're in before you type anything.

Unstaged or staged, not committed: the change only exists in your working directory or the index. Nothing is in the commit history yet. → git restore
Committed, not pushed: the change is saved in a commit, but that commit exists only on your machine — no one else has it. → git reset
Committed and pushed: the commit exists on the shared remote. A teammate may have already pulled it onto their own machine. → git revert
⚠️ The golden rule: never rewrite history that anyone else might already have. git reset and git commit --amend rewrite commits — that's only safe on commits that exist solely on your machine. Once a commit is pushed, undo it forward with a new commit (git revert), never backward by erasing it.
🔧
restore vs reset vs revert
Same goal — undo something — but each command operates at a completely different layer

Three commands, three different layers of "undo"

git restore works on your working directory and staging area only — it never touches commit history. git reset moves your branch pointer to an earlier commit and can erase commits from local history entirely. git revert does the opposite of a commit by creating a brand-new commit that cancels it out — history only ever grows forward, which is exactly why it's the only one of the three that's safe to use after pushing.

git restore <file> — working directory only. Throws away uncommitted edits. History untouched.
git reset — moves your branch pointer backward. Can discard commits entirely. Rewrites local history — never use it on commits you've already pushed.
git revert — adds a new commit that reverses an old one. History grows forward, nothing is erased. Safe to push at any time.

You'll constantly need to say "how far back" with both reset and revert — that's what HEAD~1, HEAD~2 notation is for. HEAD always points at your current commit; HEAD~1 means "one commit before HEAD," HEAD~2 means "two commits before HEAD," and so on.

git log --oneline -3 # a1b2c3d (HEAD -> main) fix: typo in readme # e4f5g6h feat: add patient lookup # i7j8k9l chore: initial commit git reset --soft HEAD~1 # undoes only the "fix: typo" commit
reset also has three modes — --soft, --mixed, --hard — that decide what happens to your staged changes and working files. Covered in full in the Q&A below.

Key Concepts — Open Every Card
The undo questions every developer eventually needs answered, usually under mild panic
Throwing away edits you haven't committed
You edited a file, didn't like the result, and want it back exactly as it was at the last commit. Nothing has been committed yet — this is the simplest undo there is.
The fix
# Discard changes to one file git restore src/models/device.py # Discard changes to everything in the working directory git restore . # If the change is already staged (git add), restore the staged copy too git restore --staged src/models/device.py git restore src/models/device.py
git restore replaced the older, more confusing git checkout -- <file> syntax (Git 2.23+). Both still work, but restore says exactly what it does.
This is destructive and instant — there's no "are you sure?" prompt. If you need that code back later, it's only recoverable through your editor's own undo history, not Git's.
It's still only on your machine — full freedom to rewrite
You committed something locally — wrong message, wrong files, or you just changed your mind — and haven't run git push yet. Because no one else has this commit, you're free to rewrite history.
Three flavors, by how much you want to keep
Keep changes staged, just undo the commit (most common)

git reset --soft HEAD~1
Keep changes, but unstaged — edit further before re-committing

git reset HEAD~1 # --mixed is the default
Discard the commit AND its changes completely — use with care

git reset --hard HEAD~1
⚠️ git reset --hard permanently discards uncommitted work with no confirmation prompt. Run git status and git diff first to double-check — there's no built-in undo for --hard beyond git reflog (an advanced, not-guaranteed recovery path).
HEAD~1 means "one commit back." Use HEAD~2, HEAD~3, etc. to undo several commits at once.
All three flavors only move your local branch pointer — never run them on a commit someone else might have already pulled.
The commit is shared now — rewrite forward, not backward
A teammate may have already pulled this commit onto their machine. Running git reset here would rewrite history they already have, causing painful conflicts the next time they pull. The safe tool is git revert.
Undo a pushed commit
# Find the commit you want to undo git log --oneline -5 # Create a new commit that reverses it git revert a1b2c3d # Git opens an editor with a pre-filled message: # "Revert 'feat: add patient lookup'" # Save and close to accept it git push
revert never deletes a commit — it adds a new one that cancels the old one's changes. The original commit stays visible in history, which is exactly what makes this safe to share.
git revert HEAD~2..HEAD reverts a range of commits in one go, useful when several commits need undoing together.
⚠️ Never run git reset or git push --force on a branch others pull from (main, develop, etc.) to "undo" a pushed commit — that rewrites history out from under your teammates.
A shelf for unfinished work you're not ready to commit
You're mid-change but need to switch branches right now — an urgent bug, a teammate's update to pull, or just something to check — and your edits aren't commit-worthy yet. Stashing saves them aside without a commit.
Save, switch, restore
# Save your uncommitted changes (staged + unstaged) and clean the working dir git stash # Switch branches, do whatever you needed to do... git switch main git pull # Come back and reapply your stashed changes git switch feature/patient-lookup git stash pop
git stash push -m "wip: patient lookup" gives the stash a label — helpful once you have more than one saved.
git stash list shows everything you've stashed. git stash pop applies and removes the most recent one; git stash apply applies it but keeps it in the list, useful if you want to apply the same stash to more than one branch.
A stash can conflict on pop, just like a merge — resolve the markers the same way you would a merge conflict (Topic 3).
Move exactly one commit, leave the rest behind
A commit exists on another branch that you want — and only that one commit, not the whole branch's history. Cherry-pick copies a single commit's changes onto your current branch as a brand-new commit.
Pick a commit across branches
# Find the commit's hash on the other branch git log feature/device-model --oneline # Switch to the branch that should receive it git switch hotfix/release-1.4 # Apply just that one commit here git cherry-pick a1b2c3d
Common real-world case: a bugfix landed on main but also needs to ship on an older release branch that can't take all of main's history.
Cherry-picking creates a new commit with a new hash — it's a copy, not a move. The original commit still exists on its original branch too.
Conflicts can happen during a cherry-pick, just like a merge — resolve the markers, then git cherry-pick --continue (or --abort to cancel).
Same command, three very different blast radii
All three modes move your branch pointer to a target commit. What differs is what happens to the staging area and your working directory files — and that difference decides whether your edits survive.
The three modes
--soft — moves the branch pointer only. Changes from the undone commit(s) land back in the staging area, ready to re-commit immediately.

--mixed (the default) — moves the pointer and unstages the changes, but keeps them in your working directory as edits. You'll need git add again before committing.

--hard — moves the pointer and discards the changes completely, from both staging and the working directory. Nothing remains — use only when you're certain.

git reset --soft HEAD~1 # safest — keeps everything staged git reset HEAD~1 # mixed — keeps everything, unstaged git reset --hard HEAD~1 # most destructive — keeps nothing
Mental shortcut: soft keeps the most, hard keeps the least, mixed sits in between (and is the default if you omit the flag).
When unsure which to use, default to --soft or plain reset — you can always discard the changes afterward, but you can't easily get them back after --hard.

Hands-On Practice
Run every undo tool once in a throwaway repo — ten minutes, zero risk

Try all five tools in one practice repo

Reading about restore, reset, revert, stash, and cherry-pick is no substitute for typing them. Run this script in a disposable folder — it walks through every tool in this topic in sequence, with nothing real at stake.

# 1. Fresh practice repo git init undo-practice && cd undo-practice echo "v1" > notes.txt git add . && git commit -m "Initial notes" # 2. Unstaged change — discard it with restore echo "oops, bad edit" >> notes.txt git restore notes.txt cat notes.txt # → back to "v1" only # 3. Commit something, then undo the commit but keep the change staged echo "v2" >> notes.txt git commit -am "Add v2 line" git reset --soft HEAD~1 git status # → the v2 line is staged again, ready to re-commit or discard # 4. Stash a half-finished change, switch away and back, then pop it echo "wip change" >> notes.txt git stash git status # → clean working directory git stash pop # → "wip change" line is back # 5. Commit it, branch off, and cherry-pick that exact commit elsewhere git commit -am "Add wip line" git log --oneline -1 git switch -c other-branch main git cherry-pick <hash-from-the-log-above> # 6. Revert a commit as if it had already been pushed git switch undo-practice 2>/dev/null || git switch main git log --oneline git revert HEAD git log --oneline # → a brand-new commit now undoes the previous one — nothing was erased

After this you'll have personally used every command in this topic at least once — which is the only way the difference between "rewriting" and "reverting" history really sticks.

Wo befindest du dich? Die drei Undo-Szenarien
Der richtige Undo-Befehl hängt vollständig davon ab, wie weit deine Änderung schon gereist ist — verstehe dieses Bild zuerst

Jede Undo-Frage reduziert sich auf eine Frage: Wo lebt diese Änderung?

Bevor du zu irgendeinem Undo-Befehl greifst, beantworte zuerst eine Frage: Wurde die Änderung gestaged? Committet? Gepusht? Git gibt dir für jede Stufe ein anderes, sichereres Werkzeug — und das falsche zu wählen ist genau der Weg, wie aus "Rückgängig machen" "verlorene Arbeit" oder "kaputter Verlauf für das ganze Team" wird.

Die drei Szenarien unten entsprechen direkt den drei Befehlen, die dieses Thema behandelt. Identifiziere zuerst, in welchem du dich befindest, bevor du irgendetwas eingibst.

Nicht gestaged oder gestaged, aber nicht committet: die Änderung existiert nur in deinem Arbeitsverzeichnis oder im Index. Noch nichts ist im Commit-Verlauf. → git restore
Committet, nicht gepusht: die Änderung ist in einem Commit gespeichert, aber dieser Commit existiert nur auf deinem Rechner — niemand sonst hat ihn. → git reset
Committet und gepusht: der Commit existiert auf dem gemeinsamen Remote. Ein Teammitglied hat ihn vielleicht schon auf den eigenen Rechner gepullt. → git revert
⚠️ Die goldene Regel: schreibe niemals Verlauf um, den jemand anders schon haben könnte. git reset und git commit --amend schreiben Commits um — das ist nur sicher bei Commits, die ausschließlich auf deinem Rechner existieren. Sobald ein Commit gepusht ist, mache ihn mit einem neuen Commit rückgängig (git revert), niemals rückwärts durch Löschen.
🔧
restore vs. reset vs. revert
Gleiches Ziel — etwas rückgängig machen — aber jeder Befehl arbeitet auf einer völlig anderen Ebene

Drei Befehle, drei verschiedene Ebenen von "Undo"

git restore wirkt nur auf dein Arbeitsverzeichnis und den Staging-Bereich — es rührt den Commit-Verlauf nie an. git reset verschiebt deinen Branch-Zeiger auf einen früheren Commit und kann Commits vollständig aus dem lokalen Verlauf entfernen. git revert macht das Gegenteil eines Commits, indem es einen brandneuen Commit erzeugt, der ihn aufhebt — der Verlauf wächst nur nach vorne, und genau das macht es als einziges der drei sicher nach dem Push.

git restore <Datei> — nur Arbeitsverzeichnis. Verwirft nicht committete Änderungen. Verlauf unberührt.
git reset — verschiebt deinen Branch-Zeiger zurück. Kann Commits vollständig verwerfen. Schreibt lokalen Verlauf um — niemals auf bereits gepushte Commits anwenden.
git revert — fügt einen neuen Commit hinzu, der einen alten umkehrt. Der Verlauf wächst nach vorne, nichts wird gelöscht. Jederzeit sicher zu pushen.

Bei reset und revert musst du ständig sagen "wie weit zurück" — dafür ist die HEAD~1-, HEAD~2-Notation da. HEAD zeigt immer auf deinen aktuellen Commit; HEAD~1 bedeutet "ein Commit vor HEAD", HEAD~2 bedeutet "zwei Commits vor HEAD" und so weiter.

git log --oneline -3 # a1b2c3d (HEAD -> main) fix: Tippfehler im Readme # e4f5g6h feat: Patient-Lookup hinzufügen # i7j8k9l chore: initialer Commit git reset --soft HEAD~1 # macht nur den "fix: Tippfehler"-Commit rückgängig
reset hat außerdem drei Modi — --soft, --mixed, --hard — die entscheiden, was mit deinen gestagten Änderungen und Arbeitsdateien passiert. Vollständig erklärt im Q&A unten.

Schlüsselkonzepte — Öffne jede Karte
Die Undo-Fragen, die irgendwann jeder Entwickler beantworten muss — meist unter leichter Panik
Nicht committete Änderungen verwerfen
Du hast eine Datei bearbeitet, das Ergebnis gefällt dir nicht, und du willst sie genau so zurück, wie sie beim letzten Commit war. Noch nichts wurde committet — das ist das einfachste Undo, das es gibt.
Die Lösung
# Änderungen an einer Datei verwerfen git restore src/models/device.py # Änderungen am gesamten Arbeitsverzeichnis verwerfen git restore . # Wenn die Änderung schon gestaged ist (git add), auch die gestagte Kopie zurücksetzen git restore --staged src/models/device.py git restore src/models/device.py
git restore ersetzt die ältere, verwirrendere Syntax git checkout -- <Datei> (Git 2.23+). Beide funktionieren noch, aber restore sagt genau, was es tut.
Das ist destruktiv und sofort wirksam — es gibt keine "Bist du sicher?"-Abfrage. Wenn du diesen Code später brauchst, ist er nur über die eigene Undo-Historie deines Editors wiederherstellbar, nicht über Git.
Er existiert nur auf deinem Rechner — volle Freiheit zum Umschreiben
Du hast etwas lokal committet — falsche Nachricht, falsche Dateien, oder du hast einfach deine Meinung geändert — und noch nicht git push ausgeführt. Weil niemand sonst diesen Commit hat, darfst du den Verlauf frei umschreiben.
Drei Varianten, je nachdem, wie viel du behalten willst
Änderungen gestaged behalten, nur den Commit rückgängig machen (am häufigsten)

git reset --soft HEAD~1
Änderungen behalten, aber ungestaged — vor dem erneuten Committen weiter bearbeiten

git reset HEAD~1 # --mixed ist der Standard
Den Commit UND seine Änderungen vollständig verwerfen — mit Vorsicht verwenden

git reset --hard HEAD~1
⚠️ git reset --hard verwirft nicht committete Arbeit dauerhaft, ohne Bestätigung. Prüfe vorher mit git status und git diff — es gibt kein eingebautes Undo für --hard, abgesehen von git reflog (eine fortgeschrittene, nicht garantierte Rettungsmöglichkeit).
HEAD~1 bedeutet "einen Commit zurück." Nutze HEAD~2, HEAD~3 usw., um mehrere Commits gleichzeitig rückgängig zu machen.
Alle drei Varianten verschieben nur deinen lokalen Branch-Zeiger — wende sie nie auf einen Commit an, den jemand anders schon gepullt haben könnte.
Der Commit ist jetzt geteilt — vorwärts umkehren, nicht rückwärts umschreiben
Ein Teammitglied hat diesen Commit vielleicht schon auf den eigenen Rechner gepullt. git reset würde hier Verlauf umschreiben, den die Person schon hat, und beim nächsten Pull schmerzhafte Konflikte verursachen. Das sichere Werkzeug ist git revert.
Einen gepushten Commit rückgängig machen
# Den Commit finden, den du rückgängig machen willst git log --oneline -5 # Einen neuen Commit erstellen, der ihn umkehrt git revert a1b2c3d # Git öffnet einen Editor mit vorausgefüllter Nachricht: # "Revert 'feat: add patient lookup'" # Speichern und schließen, um sie zu übernehmen git push
revert löscht nie einen Commit — es fügt einen neuen hinzu, der die Änderungen des alten aufhebt. Der ursprüngliche Commit bleibt im Verlauf sichtbar, und genau das macht es sicher zu teilen.
git revert HEAD~2..HEAD macht mehrere Commits in einem Schritt rückgängig — nützlich, wenn mehrere zusammen rückgängig gemacht werden müssen.
⚠️ Führe niemals git reset oder git push --force auf einem Branch aus, von dem andere pullen (main, develop usw.), um einen gepushten Commit "rückgängig zu machen" — das schreibt deinen Teammitgliedern den Verlauf unter den Füßen weg um.
Ein Regal für unfertige Arbeit, die noch nicht commit-würdig ist
Du bist mitten in einer Änderung, musst aber jetzt sofort den Branch wechseln — ein dringender Bug, ein Update eines Teammitglieds zum Pullen, oder einfach etwas, das du nachsehen musst — und deine Änderungen sind noch nicht reif für einen Commit. Stashen legt sie ohne Commit zur Seite.
Sichern, wechseln, zurückholen
# Nicht committete Änderungen sichern (gestaged + ungestaged) und Arbeitsverzeichnis säubern git stash # Branch wechseln, das Nötige erledigen ... git switch main git pull # Zurückkehren und die gestashten Änderungen wieder anwenden git switch feature/patient-lookup git stash pop
git stash push -m "wip: Patient-Lookup" gibt dem Stash eine Beschriftung — hilfreich, sobald du mehr als einen gespeichert hast.
git stash list zeigt alles, was du gestasht hast. git stash pop wendet den neuesten Stash an und entfernt ihn aus der Liste; git stash apply wendet ihn an, lässt ihn aber in der Liste — nützlich, wenn du denselben Stash auf mehrere Branches anwenden willst.
Ein Stash kann beim pop in Konflikt geraten, genau wie ein Merge — löse die Markierungen genauso wie einen Merge-Konflikt (Thema 3).
Genau einen Commit verschieben, den Rest zurücklassen
Auf einem anderen Branch existiert ein Commit, den du willst — und nur diesen einen, nicht die gesamte Branch-Historie. Cherry-pick kopiert die Änderungen eines einzelnen Commits als brandneuen Commit auf deinen aktuellen Branch.
Einen Commit über Branches hinweg auswählen
# Den Hash des Commits auf dem anderen Branch finden git log feature/device-model --oneline # Zum Branch wechseln, der ihn erhalten soll git switch hotfix/release-1.4 # Nur diesen einen Commit hier anwenden git cherry-pick a1b2c3d
Häufiger realer Fall: ein Bugfix landet auf main, muss aber auch auf einem älteren Release-Branch ausgeliefert werden, der nicht die gesamte Historie von main übernehmen kann.
Cherry-pick erzeugt einen neuen Commit mit neuem Hash — es ist eine Kopie, keine Verschiebung. Der ursprüngliche Commit existiert weiterhin auf seinem ursprünglichen Branch.
Konflikte können bei einem Cherry-pick genauso auftreten wie bei einem Merge — Markierungen lösen, dann git cherry-pick --continue (oder --abort zum Abbrechen).
Gleicher Befehl, drei sehr unterschiedliche Wirkungsradien
Alle drei Modi verschieben deinen Branch-Zeiger auf einen Ziel-Commit. Der Unterschied liegt darin, was mit dem Staging-Bereich und deinen Arbeitsdateien passiert — und das entscheidet, ob deine Änderungen überleben.
Die drei Modi
--soft — verschiebt nur den Branch-Zeiger. Änderungen aus dem rückgängig gemachten Commit landen wieder im Staging-Bereich, bereit zum sofortigen erneuten Committen.

--mixed (der Standard) — verschiebt den Zeiger und entfernt die Änderungen aus dem Staging-Bereich, behält sie aber als Bearbeitungen im Arbeitsverzeichnis. Du musst vor dem Committen erneut git add ausführen.

--hard — verschiebt den Zeiger und verwirft die Änderungen vollständig, sowohl aus dem Staging-Bereich als auch dem Arbeitsverzeichnis. Nichts bleibt übrig — nur verwenden, wenn du dir absolut sicher bist.

git reset --soft HEAD~1 # am sichersten — behält alles gestaged git reset HEAD~1 # mixed — behält alles, ungestaged git reset --hard HEAD~1 # am destruktivsten — behält nichts
Eselsbrücke: soft behält am meisten, hard behält am wenigsten, mixed liegt dazwischen (und ist der Standard, wenn du die Option weglässt).
Im Zweifel --soft oder das einfache reset wählen — du kannst Änderungen danach immer noch verwerfen, aber nach --hard bekommst du sie nicht leicht zurück.

Praktische Übung
Jedes Undo-Werkzeug einmal in einem Wegwerf-Repo ausführen — zehn Minuten, kein Risiko

Alle fünf Werkzeuge in einem Übungs-Repo ausprobieren

Über restore, reset, revert, stash und cherry-pick zu lesen ersetzt nicht, sie selbst einzutippen. Führe dieses Skript in einem Wegwerf-Ordner aus — es geht jedes Werkzeug dieses Themas der Reihe nach durch, ohne dass etwas Echtes auf dem Spiel steht.

# 1. Frisches Übungs-Repo git init undo-practice && cd undo-practice echo "v1" > notes.txt git add . && git commit -m "Initial notes" # 2. Nicht gestagte Änderung — mit restore verwerfen echo "oops, schlechte Änderung" >> notes.txt git restore notes.txt cat notes.txt # → wieder nur "v1" # 3. Etwas committen, dann den Commit rückgängig machen, Änderung bleibt gestaged echo "v2" >> notes.txt git commit -am "v2-Zeile hinzufügen" git reset --soft HEAD~1 git status # → die v2-Zeile ist wieder gestaged, bereit zum erneuten Committen oder Verwerfen # 4. Eine halbfertige Änderung stashen, weg- und zurückwechseln, dann pop echo "wip Änderung" >> notes.txt git stash git status # → sauberes Arbeitsverzeichnis git stash pop # → "wip Änderung"-Zeile ist zurück # 5. Committen, abzweigen, und genau diesen Commit anderswo cherry-picken git commit -am "wip-Zeile hinzufügen" git log --oneline -1 git switch -c other-branch main git cherry-pick <Hash-aus-dem-Log-oben> # 6. Einen Commit rückgängig machen, als wäre er schon gepusht worden git switch main git log --oneline git revert HEAD git log --oneline # → ein brandneuer Commit macht jetzt den vorigen rückgängig — nichts wurde gelöscht

Danach hast du jeden Befehl dieses Themas mindestens einmal selbst benutzt — und nur so bleibt der Unterschied zwischen "Verlauf umschreiben" und "Verlauf rückgängig machen" wirklich hängen.

أين أنت؟ سيناريوهات التراجع الثلاثة
الأمر الصحيح للتراجع يعتمد كلياً على مدى المسافة التي قطعتها تغييرتك — افهم هذه الصورة أولاً

كل سؤال تراجع يُختصر إلى سؤال واحد: أين تعيش هذه التغيير؟

قبل أن تستخدم أي أمر تراجع، أجب أولاً عن سؤال واحد: هل تم stage للتغيير؟ هل تم commit له؟ هل تم push له؟ يمنحك Git أداة مختلفة وأكثر أماناً لكل مرحلة — واستخدام الأداة الخاطئة هو بالضبط كيف يتحول "التراجع" إلى "عمل ضائع" أو "تاريخ معطّل لكل الفريق."

السيناريوهات الثلاثة أدناه تقابل مباشرة الأوامر الثلاثة التي يغطيها هذا الموضوع. حدّد أيها ينطبق عليك قبل أن تكتب أي شيء.

غير مُهيّأ (unstaged) أو مُهيّأ، لم يُنفَّذ commit له: التغيير موجود فقط في مجلد العمل أو في الفهرس (index). لا شيء في تاريخ الـ commits بعد. ← git restore
تم commit له، لم يُدفع (push): التغيير محفوظ في commit، لكن هذا الـ commit موجود فقط على جهازك — لا أحد آخر يملكه. ← git reset
تم commit له ودُفع (push): الـ commit موجود على الـ remote المشترك. قد يكون زميلك قد سحبه (pull) بالفعل إلى جهازه. ← git revert
⚠️ القاعدة الذهبية: لا تُعِد كتابة تاريخ قد يملكه شخص آخر بالفعل. git reset وgit commit --amend يُعيدان كتابة الـ commits — وهذا آمن فقط على commits موجودة حصرياً على جهازك. بمجرد أن يُدفع commit، تراجع عنه بـ commit جديد للأمام (git revert)، أبداً للخلف بحذفه.

restore وreset وrevert
نفس الهدف — التراجع عن شيء — لكن كل أمر يعمل على طبقة مختلفة تماماً

ثلاثة أوامر، ثلاث طبقات مختلفة من "التراجع"

git restore يعمل فقط على مجلد العمل ومنطقة الـ staging — لا يلمس تاريخ الـ commits أبداً. git reset يُحرّك مؤشر الـ branch إلى commit أقدم ويمكنه حذف commits بالكامل من التاريخ المحلي. git revert يفعل عكس الـ commit بإنشاء commit جديد تماماً يُلغي تأثيره — التاريخ ينمو للأمام فقط، وهذا بالضبط ما يجعله الأداة الآمنة الوحيدة من الثلاثة بعد الـ push.

git restore <file> — مجلد العمل فقط. يتخلى عن التعديلات غير المُنفَّذ لها commit. التاريخ غير ملموس.
git reset — يُحرّك مؤشر branch-ك للخلف. يمكنه حذف commits بالكامل. يُعيد كتابة التاريخ المحلي — لا تستخدمه أبداً على commits دُفعت بالفعل.
git revert — يضيف commit جديداً يعكس commit قديماً. التاريخ ينمو للأمام، لا شيء يُحذف. آمن للـ push في أي وقت.

ستحتاج باستمرار لتحديد "إلى أي مدى للخلف" مع reset وrevert — لهذا وُجدت ترميزات HEAD~1 وHEAD~2. HEAD يشير دائماً إلى commit-ك الحالي؛ HEAD~1 تعني "commit واحد قبل HEAD"، وHEAD~2 تعني "commit-ان قبل HEAD"، وهكذا.

git log --oneline -3 # a1b2c3d (HEAD -> main) fix: typo in readme # e4f5g6h feat: add patient lookup # i7j8k9l chore: initial commit git reset --soft HEAD~1 # يتراجع فقط عن commit "fix: typo"
reset له أيضاً ثلاثة أنماط — --soft، --mixed، --hard — تُحدد ما يحدث لتغييراتك المُهيّأة (staged) وملفات مجلد عملك. مغطّاة بالتفصيل في الأسئلة أدناه.

المفاهيم الأساسية — افتح كل بطاقة
أسئلة التراجع التي يحتاج كل مطور للإجابة عنها عاجلاً أم آجلاً، عادةً تحت ذعر خفيف
التخلي عن تعديلات لم تُنفَّذ لها commit
عدّلت ملفاً، ولم تُعجبك النتيجة، وتريده تماماً كما كان عند آخر commit. لم يُنفَّذ أي commit بعد — هذا أبسط تراجع موجود.
الحل
# التراجع عن التغييرات في ملف واحد git restore src/models/device.py # التراجع عن التغييرات في كل مجلد العمل git restore . # إذا كان التغيير مُهيّأ بالفعل (git add)، أعد ضبط النسخة المُهيّأة أيضاً git restore --staged src/models/device.py git restore src/models/device.py
git restore حلّ محل الصياغة الأقدم والأكثر إرباكاً git checkout -- <file> (Git 2.23+). كلاهما يعمل، لكن restore يقول بالضبط ماذا يفعل.
هذا تدميري وفوري — لا يوجد سؤال "هل أنت متأكد؟". إذا احتجت هذا الكود لاحقاً، فهو قابل للاستعادة فقط عبر سجل التراجع الخاص بمحرّرك، لا عبر Git.
لا يزال موجوداً فقط على جهازك — حرية كاملة لإعادة الكتابة
نفّذت commit محلياً — رسالة خاطئة، ملفات خاطئة، أو غيّرت رأيك فقط — ولم تُنفّذ git push بعد. لأن لا أحد آخر يملك هذا الـ commit، أنت حر في إعادة كتابة التاريخ.
ثلاث طرق، حسب ما تريد الاحتفاظ به
الاحتفاظ بالتغييرات مُهيّأة، فقط التراجع عن الـ commit (الأكثر شيوعاً)

git reset --soft HEAD~1
الاحتفاظ بالتغييرات، لكن غير مُهيّأة — للتعديل أكثر قبل إعادة الـ commit

git reset HEAD~1 # --mixed هو الافتراضي
التخلي عن الـ commit وتغييراته بالكامل — استخدمه بحذر

git reset --hard HEAD~1
⚠️ git reset --hard يتخلى بشكل دائم عن العمل غير المُنفَّذ له commit دون أي تأكيد. نفّذ git status وgit diff أولاً للتحقق — لا يوجد تراجع مدمج عن --hard بخلاف git reflog (طريقة استعادة متقدمة وغير مضمونة).
HEAD~1 تعني "commit واحد للخلف." استخدم HEAD~2، HEAD~3 وهكذا للتراجع عن عدة commits معاً.
الطرق الثلاث جميعها تُحرّك فقط مؤشر branch-ك المحلي — لا تستخدمها أبداً على commit قد يكون شخص آخر سحبه (pull) بالفعل.
الـ commit مشترك الآن — تراجع للأمام، لا تُعِد الكتابة للخلف
قد يكون زميلك قد سحب هذا الـ commit بالفعل إلى جهازه. استخدام git reset هنا سيُعيد كتابة تاريخ يملكه بالفعل، ويسبب تعارضات مؤلمة عند سحبه التالي. الأداة الآمنة هي git revert.
التراجع عن commit مدفوع
# ابحث عن الـ commit الذي تريد التراجع عنه git log --oneline -5 # أنشئ commit جديداً يعكسه git revert a1b2c3d # يفتح Git محرراً برسالة مُعدّة مسبقاً: # "Revert 'feat: add patient lookup'" # احفظ وأغلق لقبولها git push
revert لا يحذف commit أبداً — يضيف واحداً جديداً يُلغي تغييرات القديم. يبقى الـ commit الأصلي مرئياً في التاريخ، وهذا بالضبط ما يجعله آمناً للمشاركة.
git revert HEAD~2..HEAD يتراجع عن مجموعة من الـ commits دفعة واحدة، مفيد عندما تحتاج عدة commits للتراجع معاً.
⚠️ لا تُنفّذ أبداً git reset أو git push --force على branch يسحب منه آخرون (main، develop إلخ) لـ"التراجع" عن commit مدفوع — هذا يُعيد كتابة التاريخ من تحت أقدام زملائك.
رفّ لعمل غير مكتمل لست جاهزاً لتنفيذ commit له
أنت في منتصف تغيير، لكنك تحتاج لتبديل الـ branch الآن — خطأ عاجل، تحديث من زميلك لسحبه، أو فقط شيء للتحقق منه — وتعديلاتك ليست جاهزة لـ commit بعد. الـ stash يحفظها جانباً دون commit.
حفظ، تبديل، استعادة
# حفظ تغييراتك غير المُنفَّذ لها commit (المُهيّأة + غير المُهيّأة) وتنظيف مجلد العمل git stash # بدّل الـ branch، نفّذ ما تحتاجه... git switch main git pull # عُد وأعد تطبيق تغييراتك المحفوظة git switch feature/patient-lookup git stash pop
git stash push -m "wip: patient lookup" يعطي الـ stash تسمية — مفيد بعد أن يصبح لديك أكثر من واحد محفوظ.
git stash list يُظهر كل ما حفظته. git stash pop يطبّق الأحدث ويحذفه من القائمة؛ git stash apply يطبّقه لكن يحتفظ به في القائمة — مفيد إن أردت تطبيق نفس الـ stash على أكثر من branch.
قد يتعارض الـ stash عند pop، تماماً كالدمج — حلّ العلامات بنفس طريقة حل تعارض الدمج (الموضوع 3).
نقل commit واحد بالضبط، وترك الباقي
يوجد commit على branch آخر تريده — وفقط هذا الواحد، ليس كل تاريخ الـ branch. الـ cherry-pick ينسخ تغييرات commit واحد إلى branch-ك الحالي كـ commit جديد تماماً.
اختيار commit عبر الـ branches
# ابحث عن hash الـ commit على الـ branch الآخر git log feature/device-model --oneline # بدّل إلى الـ branch الذي يجب أن يستقبله git switch hotfix/release-1.4 # طبّق فقط هذا الـ commit هنا git cherry-pick a1b2c3d
حالة شائعة فعلية: إصلاح خطأ وصل إلى main لكنه يحتاج أيضاً للإصدار في branch إصدار أقدم لا يستطيع أخذ كل تاريخ main.
الـ cherry-pick يُنشئ commit جديداً بـ hash جديد — هو نسخة، لا نقل. الـ commit الأصلي يبقى موجوداً على branch-ه الأصلي أيضاً.
قد تحدث تعارضات أثناء cherry-pick تماماً كالدمج — حلّ العلامات، ثم git cherry-pick --continue (أو --abort للإلغاء).
نفس الأمر، ثلاث دوائر تأثير مختلفة جداً
الأنماط الثلاثة جميعها تُحرّك مؤشر branch-ك إلى commit مستهدف. الفرق يكمن في ما يحدث لمنطقة الـ staging وملفات مجلد عملك — وهذا الفرق يُحدد إن كانت تعديلاتك ستنجو.
الأنماط الثلاثة
--soft — يُحرّك مؤشر الـ branch فقط. تغييرات الـ commit(s) المُتراجَع عنها تعود إلى منطقة الـ staging، جاهزة لإعادة الـ commit فوراً.

--mixed (الافتراضي) — يُحرّك المؤشر ويُلغي تهيئة (unstage) التغييرات، لكنه يحتفظ بها في مجلد عملك كتعديلات. ستحتاج git add مرة أخرى قبل الـ commit.

--hard — يُحرّك المؤشر ويتخلى عن التغييرات بالكامل، من منطقة الـ staging ومجلد العمل معاً. لا يبقى شيء — استخدمه فقط حين تكون متأكداً تماماً.

git reset --soft HEAD~1 # الأكثر أماناً — يحتفظ بكل شيء مُهيّأً git reset HEAD~1 # mixed — يحتفظ بكل شيء، غير مُهيّأ git reset --hard HEAD~1 # الأكثر تدميراً — لا يحتفظ بشيء
اختصار للتذكر: soft يحتفظ بالأكثر، hard يحتفظ بالأقل، mixed في الوسط (وهو الافتراضي إن لم تحدد الخيار).
عند الشك، استخدم --soft أو reset العادي — يمكنك دائماً التخلي عن التغييرات بعد ذلك، لكن لا يمكنك استعادتها بسهولة بعد --hard.

التطبيق العملي
شغّل كل أداة تراجع مرة واحدة في مستودع تجريبي — عشر دقائق، بلا أي خطر

جرّب الأدوات الخمس كلها في مستودع تدريبي واحد

قراءة عن restore وreset وrevert وstash وcherry-pick لا تُعوّض كتابتها بنفسك. نفّذ هذا السكريبت في مجلد يمكن التخلص منه — يمر على كل أداة في هذا الموضوع بالترتيب، دون أي شيء حقيقي على المحك.

# 1. مستودع تدريبي جديد git init undo-practice && cd undo-practice echo "v1" > notes.txt git add . && git commit -m "Initial notes" # 2. تغيير غير مُهيّأ — التخلي عنه بـ restore echo "oops, bad edit" >> notes.txt git restore notes.txt cat notes.txt # → يعود إلى "v1" فقط # 3. نفّذ commit، ثم تراجع عن الـ commit مع الإبقاء على التغيير مُهيّأ echo "v2" >> notes.txt git commit -am "Add v2 line" git reset --soft HEAD~1 git status # → سطر v2 مُهيّأ مرة أخرى، جاهز لإعادة الـ commit أو التخلي عنه # 4. احفظ تغييراً غير مكتمل بـ stash، بدّل وارجع، ثم pop echo "wip change" >> notes.txt git stash git status # → مجلد عمل نظيف git stash pop # → سطر "wip change" عاد # 5. نفّذ commit له، أنشئ branch جديداً، وطبّق نفس الـ commit عبر cherry-pick في مكان آخر git commit -am "Add wip line" git log --oneline -1 git switch -c other-branch main git cherry-pick <hash-من-السجل-أعلاه> # 6. تراجع عن commit كما لو أنه دُفع بالفعل git switch main git log --oneline git revert HEAD git log --oneline # → commit جديد تماماً يتراجع الآن عن السابق — لم يُحذف شيء

بعد هذا، ستكون استخدمت بنفسك كل أمر في هذا الموضوع مرة واحدة على الأقل — وهذه الطريقة الوحيدة التي يترسخ بها الفرق بين "إعادة كتابة" التاريخ و"التراجع" عنه.

✓ انتهيت من هذا الموضوع عندما...

✓تستطيع التعافي من commit محلي سيء باستخدام git reset --soft أو reset العادي أو --hard، وتشرح بالضبط ما يتركه كل منها.
✓تستطيع التراجع بأمان عن commit دُفع بالفعل باستخدام git revert، وتشرح لماذا لا يكون git reset هو الحل أبداً في تلك الحالة.
