Protected Branches — The Team's Safety Net
Why main is locked and why that makes everyone's life easier

The first rule of team Git: no one pushes directly to main

On a solo project you push to main freely. On a shared codebase, main (and often develop) is branch-protected on GitHub. Protection means: no direct pushes, no force-pushes, and every change must arrive through a reviewed, approved pull request. Some teams also require the branch to be up to date with main before merging, and a passing CI run before anyone can click the merge button.

This isn't bureaucracy — it's the mechanism that keeps the shared history clean and ensures no single person can accidentally (or intentionally) break the code everyone else depends on.

Branch protection rules (GitHub): set in the repository Settings → Branches. Common rules: require PR before merging, require at least one approval, require status checks to pass, disallow force-push.
What happens if you try to push directly: GitHub rejects the push with remote: error: GH006: Protected branch update failed. The fix is always "open a PR instead."
Who can bypass: repository admins can be granted bypass rights, but on most research teams even admins go through PRs — it keeps the audit trail complete.
⚠️ Never ask for (or use) force-push rights on a shared branch. Even when it feels like the fastest fix, it rewrites history out from under every teammate who has already pulled those commits — causing painful, confusing divergences that are hard to untangle.

The Team Git Loop
The repeating daily cycle every contributor follows — six steps, no variations

Every feature, fix, or experiment follows the same loop

Whether a change takes ten minutes or ten days, it follows the same path. Understanding this loop as a fixed sequence — not as a collection of arbitrary commands — is what separates developers who feel in control of Git from those who feel like they're guessing.

1. Pull the latest main — always start fresh: git switch main && git pull.
2. Create a named feature branch — branch off the freshly updated main: git switch -c feature/my-thing.
3. Make commits on your branch — small, focused commits with clear messages. Push regularly so your work is backed up: git push -u origin feature/my-thing.
4. Keep your branch up to date — if main has moved, merge or rebase it in before opening the PR (covered in Q&A below).
5. Open a PR on GitHub — write a clear description, assign reviewers, link any related issues.
6. Respond to review, get approval, merge, delete the branch — the branch's job is done the moment it's merged; keep the repo tidy.
# The full loop, as typed git switch main git pull git switch -c feature/add-export-button # ...work, work, work... git add src/components/ExportButton.tsx git commit -m "feat: add CSV export button to results panel" git push -u origin feature/add-export-button # Open the PR on GitHub, get review, get approval # After merge on GitHub — clean up locally git switch main git pull git branch -d feature/add-export-button

That last step — deleting the local branch after merge — is the one most developers skip early on. After a few months you end up with dozens of stale local branches that all look like they might still be active. Delete them as you go.

Key Concepts — Open Every Card
The team-workflow questions that trip up every developer in their first weeks on a shared codebase
The guard rails that keep shared history clean
Branch protection is a GitHub repository setting that prevents anyone from pushing directly to a branch — they must go through a PR instead. It's the most important organizational tool in team Git, and virtually every professional team uses it on main.
Typical protection ruleset
Require a pull request before merging — no direct pushes, ever.

Require at least N approvals — most teams require 1; some require 2 for production branches.

Require status checks to pass — CI (tests, linting) must be green before the merge button is clickable.

Require branch to be up to date — your branch must include the latest main before you can merge.

Disallow force pushes — no rewriting of shared history, ever.
These rules are set per branch, not per user. Even repo admins are subject to them (unless explicitly given bypass rights — which most teams don't grant).
If your push is rejected with GH006: Protected branch update failed, don't look for a workaround — open a PR. That's the intended path.
Reviewers have limited time — make their job frictionless
A PR with no description gets reviewed slowly, superficially, or not at all — reviewers don't know where to focus. A PR with a clear description of what changed, why it changed, and how to test it gets a thoughtful review in a fraction of the time.
What every PR description needs
What: one sentence summarising the change. ("Adds a CSV export button to the results panel.")

Why: link the issue/ticket this resolves, or explain the motivation. ("Closes #42 — users reported needing to export results for their own analysis.")

How to test it: concrete steps a reviewer can follow. ("1. Run the app. 2. Navigate to Results. 3. Click Export — a file named results.csv should download.")

Screenshots / before-after (for UI changes) — a picture saves five minutes of reviewer effort.

What NOT to include: don't explain line-by-line what the code does — reviewers read code; they can't always read your intent. Focus on intent, not implementation.
Many teams use a PR template (a .github/pull_request_template.md file) that pre-fills the description with the required sections. If your repo has one, always fill in every section.
The title follows the same convention as commit messages: imperative mood, under 72 characters. "Add CSV export to results panel" — not "Added export" or "Export stuff".
Your branch has diverged — bring main in before merging
You've been working on a branch for two days. In that time, three teammates merged their PRs into main. Now your branch is behind — the PR can't merge cleanly, and if your team requires an up-to-date branch, GitHub's merge button is greyed out. You need to pull main into your branch.
The standard fix — merge main into your branch
# Make sure your local main is up to date git switch main git pull # Switch back to your feature branch git switch feature/add-export-button # Bring main's changes in git merge main # If there are conflicts, resolve them (see Topic 3), then: git add . git commit # Git pre-fills the merge commit message # Push the updated branch git push
Some teams prefer git rebase main instead of git merge main — rebase rewrites your commits on top of the latest main, producing a linear history. Both approaches are valid; follow your team's convention.
If you rebase and the branch is already on GitHub, you'll need git push --force-with-lease (not --force) — this is the only legitimate use of force-push, and only on your own feature branch, never on a shared branch.
Do this regularly on long-running branches — once a day is not excessive. Small, frequent merges are much easier to resolve than one large one at the end.
⚠️ Never use git rebase on a branch that other people are also working on — it rewrites commit hashes, breaking their local copy of the branch.
Code review is a dialogue, not a judgment
Getting your first critical review comment feels personal. It isn't — reviewers are commenting on the code, not on you. How you respond to review shapes your reputation on the team more than the code itself does. Developers who engage thoughtfully are trusted with more responsibility, faster.
The right response pattern
When you agree: make the change, push the commit, and reply on the comment with something short — "Done" or "Fixed in the latest push" — so the reviewer knows to re-check. Don't just silently push.

When you disagree or need clarification: ask — don't just ignore the comment or make a passive-aggressive change. "I went with X because Y — does that address your concern?" is a perfectly professional reply. Reviewers don't have full context; you do.

When the comment is out of scope: acknowledge it and open a separate issue. "Good point — I've opened #47 to track this separately so it doesn't block this PR." Never argue that a valid concern doesn't matter.
Always resolve a comment — either by making the change or by explaining why you didn't. Leaving comments unresolved is the most common reason PRs stall.
On GitHub, the reviewer (not the author) is the one who clicks "Resolve conversation" once they're satisfied. Don't resolve your own comments — let the reviewer close them.
Four habits that mark an experienced team player from a newcomer
These aren't advanced Git topics — they're workflow habits that experienced developers follow automatically. Violating them doesn't break Git, but it creates friction for everyone on the team and signals to senior developers that the person hasn't worked on a shared codebase before.
The four rules
1. Commit to a branch, never directly to main. Even for a one-line fix. Branches are free — there's no reason to skip this step.

2. Pull before you push. git pull before starting work and before opening a PR. If you skip this, your push may be rejected, or your PR will have needless conflicts.

3. Keep commits focused and messages meaningful. "fixes" or "wip" commit messages make git log useless for everyone. The convention: imperative mood, present tense, under 72 characters. "Add timeout handling to API calls", not "added some stuff".

4. Never push secrets to any branch. API keys, passwords, tokens, and .env files don't belong in Git — not even in a "private" branch that you plan to clean up later. Once a secret is in history it should be considered compromised and rotated.
⚠️ Pushed secrets cannot be "deleted" with a normal commit. They remain in the repository's full history, visible to anyone with read access. If you accidentally push a secret, rotate it immediately and contact your team lead — do not wait.
Three merge buttons, three very different outcomes in git log
GitHub offers three merge strategies when closing a PR. Most teams pick one and enforce it consistently. Knowing what each one produces in the history helps you understand why your team made its choice — and prevents surprises when you look at git log after a merge.
The three strategies
Merge commit (default) — all branch commits are preserved as-is, plus a merge commit is added. History shows exactly what happened and when. Branch topology is visible in git log --graph. Preferred when the individual commit history of a feature is meaningful.

Squash and merge — all branch commits are squashed into one commit on main. The PR's entire work becomes a single, clean commit. Branch commits disappear from main's history. Preferred when branch commits are messy "wip" checkpoints and only the final result matters.

Rebase and merge — branch commits are replayed one by one on top of main, with no merge commit. History is perfectly linear. Preferred on teams that value a clean, readable git log above all else.
Check which strategy your team uses before you open your first PR — it affects how you should write your commits and whether your individual commit messages will appear in main's log.
Squash is the most forgiving for newcomers — messy branch commits are collapsed into one clean entry. Many research teams use squash by default for exactly this reason.

Hands-On Practice
Simulate a complete team PR cycle from scratch — two branches, one "teammate," one merge

Run the full team loop in a local practice repo

The script below simulates you and a teammate both working on the same repo — your teammate merges to main while you're mid-feature, exactly the situation that catches new developers off guard. Work through it once and the loop will feel mechanical (in a good way) from that point forward.

# ─── Setup: create a shared practice repo ─────────────────────────────────── git init team-workflow-practice && cd team-workflow-practice echo "# Team App" > README.md git add . && git commit -m "chore: initial commit" # ─── You: start a feature branch ──────────────────────────────────────────── git switch -c feature/add-dashboard echo "dashboard placeholder" > dashboard.txt git add . && git commit -m "feat: scaffold dashboard page" # ─── Simulate a teammate merging to main while you work ───────────────────── # (In real life this happens on the remote; we fake it locally) git switch main echo "utility function" > utils.txt git add . && git commit -m "feat: add shared utility (teammate's work)" # ─── You: come back to your branch and update it ──────────────────────────── git switch feature/add-dashboard git merge main # → No conflict here — different files. A merge commit is created. git log --oneline --graph # → You can see your branch history and the merge from main # ─── You: finish the feature ──────────────────────────────────────────────── echo "dashboard complete" >> dashboard.txt git add . && git commit -m "feat: complete dashboard implementation" # ─── Simulate opening a PR: merge your branch into main ───────────────────── git switch main git merge feature/add-dashboard --no-ff -m "Merge pull request: Add dashboard page" # ─── Clean up: delete the feature branch ──────────────────────────────────── git branch -d feature/add-dashboard git log --oneline --graph # → main now contains the teammate's work AND your dashboard, cleanly merged

Repeat this with a conflicting change — have both you and your "teammate" edit the same file — and resolve the conflict using the steps from Topic 3. That's the full loop with every real-world complication included.

Geschützte Branches — Das Sicherheitsnetz des Teams
Warum main gesperrt ist — und warum das das Leben aller leichter macht

Die erste Regel im Team-Git: Niemand pusht direkt auf main

Auf einem Solo-Projekt pushst du frei auf main. Auf einem gemeinsam genutzten Codebase ist main (und oft auch develop) auf GitHub branch-geschützt. Schutz bedeutet: kein direktes Pushen, kein Force-Push, und jede Änderung muss über einen geprüften, genehmigten Pull Request ankommen. Manche Teams verlangen außerdem, dass der Branch vor dem Mergen auf dem Stand von main ist, und einen bestandenen CI-Lauf, bevor jemand auf den Merge-Button klicken darf.

Das ist keine Bürokratie — es ist der Mechanismus, der die gemeinsame History sauber hält und sicherstellt, dass keine Einzelperson versehentlich (oder absichtlich) den Code kaputt machen kann, von dem alle anderen abhängen.

Branch-Protection-Regeln (GitHub): werden in den Repository-Einstellungen → Branches festgelegt. Typische Regeln: PR vor dem Mergen erforderlich, mindestens eine Genehmigung, Statusprüfungen müssen bestehen, Force-Push verboten.
Was passiert beim direkten Push: GitHub lehnt den Push mit remote: error: GH006: Protected branch update failed ab. Die Lösung ist immer: "Stattdessen einen PR öffnen."
Wer bypassen darf: Repository-Admins können Bypass-Rechte erhalten, aber auf den meisten Forschungsteams gehen selbst Admins über PRs — das hält den Audit-Trail vollständig.
⚠️ Frag niemals nach Force-Push-Rechten auf einem gemeinsamen Branch (und nutze sie nicht). Auch wenn es sich nach der schnellsten Lösung anfühlt, überschreibt es die History unter jedem Teammitglied, das diese Commits schon gepullt hat — was schmerzhafte, schwer zu entwirrende Divergenzen verursacht.

Die Team-Git-Schleife
Der täglich wiederholte Zyklus, dem jeder Contributor folgt — sechs Schritte, keine Abweichungen

Jedes Feature, jede Korrektur, jedes Experiment folgt derselben Schleife

Egal ob eine Änderung zehn Minuten oder zehn Tage dauert — sie folgt demselben Pfad. Diesen Loop als feste Abfolge zu verstehen — nicht als eine Sammlung willkürlicher Befehle — ist das, was Entwickler, die Git im Griff haben, von denen unterscheidet, die das Gefühl haben, zu raten.

1. Das aktuelle main pullen — immer frisch starten: git switch main && git pull.
2. Einen benannten Feature-Branch erstellen — vom frisch aktualisierten main abzweigen: git switch -c feature/mein-ding.
3. Commits auf deinem Branch machen — kleine, fokussierte Commits mit aussagekräftigen Nachrichten. Regelmäßig pushen, damit deine Arbeit gesichert ist: git push -u origin feature/mein-ding.
4. Den Branch aktuell halten — wenn main sich weiterentwickelt hat, vor dem Öffnen des PR einmergen oder rebasen (im Q&A unten erklärt).
5. Einen PR auf GitHub öffnen — eine klare Beschreibung schreiben, Reviewer zuweisen, verwandte Issues verlinken.
6. Review beantworten, Genehmigung einholen, mergen, Branch löschen — der Branch hat seinen Job erledigt, sobald er gemergt ist; das Repo sauber halten.
# Die vollständige Schleife, wie sie getippt wird git switch main git pull git switch -c feature/export-button-hinzufuegen # ...arbeiten, arbeiten, arbeiten... git add src/components/ExportButton.tsx git commit -m "feat: CSV-Export-Button zum Ergebnispanel hinzufügen" git push -u origin feature/export-button-hinzufuegen # PR auf GitHub öffnen, Review bekommen, Genehmigung bekommen # Nach dem Merge auf GitHub — lokal aufräumen git switch main git pull git branch -d feature/export-button-hinzufuegen

Der letzte Schritt — den lokalen Branch nach dem Merge zu löschen — ist derjenige, den die meisten Entwickler anfangs überspringen. Nach ein paar Monaten hat man Dutzende veralteter lokaler Branches, die alle so aussehen, als könnten sie noch aktiv sein. Lösche sie direkt beim Aufräumen.

Schlüsselkonzepte — Öffne jede Karte
Die Team-Workflow-Fragen, über die jeder Entwickler in den ersten Wochen auf einem gemeinsamen Codebase stolpert
Die Leitplanken, die die gemeinsame History sauber halten
Branch-Protection ist eine GitHub-Repository-Einstellung, die verhindert, dass jemand direkt auf einen Branch pusht — er muss stattdessen über einen PR gehen. Es ist das wichtigste Organisationswerkzeug im Team-Git, und praktisch jedes professionelle Team nutzt es auf main.
Typisches Regelwerk
Pull Request vor dem Mergen erforderlich — kein direktes Pushen, jemals.

Mindestens N Genehmigungen erforderlich — die meisten Teams verlangen 1; manche 2 für Produktionsbranches.

Statusprüfungen müssen bestehen — CI (Tests, Linting) muss grün sein, bevor der Merge-Button klickbar ist.

Branch muss aktuell sein — dein Branch muss das aktuelle main enthalten, bevor du mergen kannst.

Force-Pushes verboten — keine Umschreibung der gemeinsamen History, niemals.
Diese Regeln gelten pro Branch, nicht pro User. Selbst Repository-Admins unterliegen ihnen (außer ihnen wurden explizit Bypass-Rechte gewährt — was die meisten Teams nicht tun).
Wenn dein Push mit GH006: Protected branch update failed abgelehnt wird, suche nicht nach einem Workaround — öffne einen PR. Das ist der vorgesehene Weg.
Reviewer haben begrenzte Zeit — mach ihren Job reibungslos
Ein PR ohne Beschreibung wird langsam, oberflächlich oder gar nicht reviewed — Reviewer wissen nicht, worauf sie sich konzentrieren sollen. Ein PR mit einer klaren Beschreibung von Was sich geändert hat, Warum es sich geändert hat und Wie man es testet bekommt in einem Bruchteil der Zeit ein durchdachtes Review.
Was jede PR-Beschreibung braucht
Was: ein Satz, der die Änderung zusammenfasst. ("Fügt einen CSV-Export-Button zum Ergebnispanel hinzu.")

Warum: das Issue/Ticket verlinken oder die Motivation erklären. ("Schließt #42 — Nutzer berichteten, dass sie Ergebnisse für eigene Analysen exportieren müssen.")

Wie zu testen: konkrete Schritte, denen ein Reviewer folgen kann. ("1. App starten. 2. Zu Ergebnisse navigieren. 3. Auf Export klicken — eine Datei namens results.csv sollte heruntergeladen werden.")

Screenshots / Vorher-Nachher (für UI-Änderungen) — ein Bild spart fünf Minuten Reviewer-Aufwand.
Viele Teams verwenden ein PR-Template (eine .github/pull_request_template.md-Datei), das die Beschreibung mit den erforderlichen Abschnitten vorausfüllt. Wenn dein Repo eines hat, fülle immer jeden Abschnitt aus.
Der Titel folgt denselben Konventionen wie Commit-Messages: Imperativ, unter 72 Zeichen. "CSV-Export zum Ergebnispanel hinzufügen" — nicht "Export hinzugefügt" oder "Exportkram".
Dein Branch hat sich verzweigt — hole main vor dem Mergen herein
Du arbeitest seit zwei Tagen an einem Branch. In dieser Zeit haben drei Teammitglieder ihre PRs in main gemergt. Jetzt ist dein Branch hintendran — der PR kann nicht sauber gemergt werden, und wenn dein Team einen aktuellen Branch verlangt, ist Githubs Merge-Button ausgegraut. Du musst main in deinen Branch holen.
Die Standardlösung — main in deinen Branch mergen
# Sicherstellen, dass dein lokales main aktuell ist git switch main git pull # Zurück zu deinem Feature-Branch wechseln git switch feature/export-button-hinzufuegen # Die Änderungen von main hereinbringen git merge main # Bei Konflikten: auflösen (siehe Thema 3), dann: git add . git commit # Git füllt die Merge-Commit-Nachricht vor # Den aktualisierten Branch pushen git push
Manche Teams bevorzugen git rebase main statt git merge main — Rebase schreibt deine Commits auf das neueste main um und erzeugt eine lineare History. Beide Ansätze sind gültig; folge der Konvention deines Teams.
Wenn du rebased und der Branch schon auf GitHub ist, brauchst du git push --force-with-lease (nicht --force) — das ist die einzige legitime Verwendung von Force-Push, und nur auf deinem eigenen Feature-Branch.
Tu das regelmäßig bei langläufigen Branches — einmal täglich ist nicht übertrieben. Kleine, häufige Merges sind viel leichter aufzulösen als ein großer am Ende.
⚠️ Verwende git rebase niemals auf einem Branch, an dem auch andere Leute arbeiten — es überschreibt Commit-Hashes und bricht ihre lokale Kopie des Branches.
Code-Review ist ein Dialog, kein Urteil
Den ersten kritischen Review-Kommentar zu erhalten fühlt sich persönlich an. Das ist es nicht — Reviewer kommentieren den Code, nicht dich. Wie du auf Reviews reagierst, prägt deinen Ruf im Team stärker als der Code selbst. Entwickler, die durchdacht reagieren, werden schneller mit mehr Verantwortung betraut.
Das richtige Reaktionsmuster
Wenn du zustimmst: mache die Änderung, pushe den Commit und antworte kurz auf den Kommentar — "Erledigt" oder "In letztem Push behoben" — damit der Reviewer weiß, dass er nachsehen soll. Nicht einfach still pushen.

Wenn du nicht einverstanden bist oder Klärung brauchst: frag nach — ignoriere den Kommentar nicht einfach. "Ich habe X gewählt, weil Y — behebt das dein Anliegen?" ist eine völlig professionelle Antwort. Reviewer haben nicht deinen vollen Kontext; du schon.

Wenn der Kommentar außerhalb des Umfangs liegt: erkenne ihn an und öffne ein separates Issue. "Guter Punkt — ich habe #47 eröffnet, um das separat zu verfolgen, damit es diesen PR nicht blockiert."
Löse immer einen Kommentar auf — entweder durch die Änderung oder durch die Erklärung, warum du sie nicht gemacht hast. Ungelöste Kommentare sind der häufigste Grund, warum PRs ins Stocken geraten.
Auf GitHub ist es der Reviewer (nicht der Autor), der auf "Resolve conversation" klickt, sobald er zufrieden ist. Löse deine eigenen Kommentare nicht auf — lass den Reviewer sie schließen.
Vier Gewohnheiten, die einen erfahrenen Team-Player von einem Newcomer unterscheiden
Das sind keine fortgeschrittenen Git-Themen — es sind Workflow-Gewohnheiten, die erfahrene Entwickler automatisch befolgen. Ihre Verletzung bricht Git nicht, erzeugt aber Reibung für alle im Team und signalisiert erfahrenen Entwicklern, dass die Person noch nicht auf einem gemeinsamen Codebase gearbeitet hat.
Die vier Regeln
1. Commit auf einen Branch, niemals direkt auf main. Auch für eine einzeilige Korrektur. Branches sind kostenlos — es gibt keinen Grund, diesen Schritt zu überspringen.

2. Pull vor dem Push. git pull vor dem Arbeitsbeginn und vor dem Öffnen eines PR. Wenn du das überspringst, wird dein Push vielleicht abgelehnt oder dein PR hat unnötige Konflikte.

3. Commits fokussiert und Nachrichten aussagekräftig halten. "fixes" oder "wip"-Commit-Nachrichten machen git log für alle nutzlos. Die Konvention: Imperativ, Präsens, unter 72 Zeichen. "API-Aufrufen Timeout-Behandlung hinzufügen", nicht "irgendwas hinzugefügt".

4. Niemals Secrets auf einen Branch pushen. API-Schlüssel, Passwörter, Tokens und .env-Dateien gehören nicht in Git — nicht einmal auf einem "privaten" Branch, den du später bereinigen willst.
⚠️ Gepushte Secrets können nicht mit einem normalen Commit "gelöscht" werden. Sie verbleiben in der vollständigen Repository-History, sichtbar für jeden mit Lesezugriff. Falls du versehentlich ein Secret pushst, rotiere es sofort und kontaktiere deinen Teamlead — warte nicht.
Drei Merge-Buttons, drei sehr unterschiedliche Ergebnisse in git log
GitHub bietet beim Schließen eines PR drei Merge-Strategien an. Die meisten Teams wählen eine und setzen sie konsequent durch. Zu wissen, was jede in der History erzeugt, hilft dir zu verstehen, warum dein Team die Wahl getroffen hat — und verhindert Überraschungen beim Blick in git log nach einem Merge.
Die drei Strategien
Merge Commit (Standard) — alle Branch-Commits bleiben erhalten, plus ein Merge-Commit wird hinzugefügt. Die History zeigt genau, was wann passiert ist. Die Branch-Topologie ist in git log --graph sichtbar.

Squash and Merge — alle Branch-Commits werden zu einem einzigen Commit auf main zusammengefasst. Die gesamte Arbeit des PR wird zu einem sauberen Commit. Bevorzugt, wenn Branch-Commits unordentliche "wip"-Checkpoints sind.

Rebase and Merge — Branch-Commits werden einzeln auf das neueste main replayed, ohne Merge-Commit. Die History ist vollständig linear. Bevorzugt auf Teams, die ein sauberes, lesbares git log über alles stellen.
Prüfe vor deinem ersten PR, welche Strategie dein Team verwendet — das beeinflusst, wie du deine Commits schreiben solltest und ob deine Commit-Nachrichten in main's Log erscheinen.
Squash ist für Newcomer am vergebendsten — unordentliche Branch-Commits werden zu einem sauberen Eintrag zusammengefaltet. Viele Forschungsteams verwenden Squash aus genau diesem Grund als Standard.

Geh es einmal durch, und die Schleife wird sich von da an mechanisch (im guten Sinne) anfühlen.

# ─── Setup: Übungsrepo erstellen ───────────────────────────────────────────── git init team-workflow-uebung && cd team-workflow-uebung echo "# Team App" > README.md git add . && git commit -m "chore: initialer Commit" # ─── Du: Feature-Branch starten ────────────────────────────────────────────── git switch -c feature/dashboard-hinzufuegen echo "Dashboard Platzhalter" > dashboard.txt git add . && git commit -m "feat: Dashboard-Seite gerüsten" # ─── Teammitglied simulieren, das in main merged ────────────────────────────── git switch main echo "Hilfsfunktion" > utils.txt git add . && git commit -m "feat: gemeinsame Hilfsfunktion hinzufügen (Arbeit des Teammitglieds)" # ─── Du: zurück zum Branch und aktualisieren ────────────────────────────────── git switch feature/dashboard-hinzufuegen git merge main # → Kein Konflikt hier — verschiedene Dateien. Ein Merge-Commit wird erstellt. git log --oneline --graph # → Branch-History und Merge von main sind sichtbar # ─── Du: Feature fertigstellen ──────────────────────────────────────────────── echo "Dashboard komplett" >> dashboard.txt git add . && git commit -m "feat: Dashboard-Implementierung abschließen" # ─── PR simulieren: Branch in main mergen ───────────────────────────────────── git switch main git merge feature/dashboard-hinzufuegen --no-ff -m "Pull Request mergen: Dashboard-Seite hinzufügen" # ─── Aufräumen: Feature-Branch löschen ──────────────────────────────────────── git branch -d feature/dashboard-hinzufuegen git log --oneline --graph # → main enthält jetzt die Arbeit des Teammitglieds UND dein Dashboard, sauber gemergt

Wiederhole das mit einer konfliktierenden Änderung — lass sowohl dich als auch dein "Teammitglied" dieselbe Datei bearbeiten — und löse den Konflikt mit den Schritten aus Thema 3. Das ist die vollständige Schleife mit allen realen Komplikationen.

✓ Du hast dieses Thema abgeschlossen, wenn...
✓Du den vollständigen PR-Lifecycle von Anfang bis Ende beschreiben kannst: main pullen → Branch → committen → pushen → PR öffnen → Review-Zyklus → mergen → Branch löschen.
✓Du weißt, wie du einen veralteten Feature-Branch mit git merge main aktualisierst, und kannst erklären, wann (und warum) ein Team stattdessen git rebase main wählen könnte.
✓Du kannst eine PR-Beschreibung mit einer klaren Was/Warum/Wie-zu-testen-Struktur schreiben und weißt, wie du auf einen Review-Kommentar — einschließlich eines, mit dem du nicht einverstanden bist — professionell antwortest.
✓Du kannst die vier Regeln nennen, die die meisten neuen Entwickler am ersten Tag brechen, und die Konsequenz des Brechens jeder einzelnen für das restliche Team erklären.

الفروع المحمية — شبكة أمان الفريق
لماذا يكون main مقفلاً — ولماذا يجعل ذلك حياة الجميع أسهل

القاعدة الأولى في Git الجماعي: لا أحد يدفع مباشرةً إلى main

في مشروع فردي تدفع إلى main بحرية. على قاعدة كود مشتركة، يكون main (وغالباً develop أيضاً) فرعاً محمياً على GitHub. الحماية تعني: لا دفع مباشر، لا force-push، وكل تغيير يجب أن يصل عبر طلب دمج (PR) تمت مراجعته والموافقة عليه. بعض الفرق تشترط أيضاً أن يكون الفرع محدَّثاً بـ main قبل الدمج، وأن تجتاز اختبارات الـ CI قبل أن يتمكن أي شخص من النقر على زر الدمج.

هذا ليس بيروقراطية — إنه الآلية التي تحافظ على نظافة التاريخ المشترك وتضمن أنه لا يمكن لشخص واحد أن يعطّل الكود الذي يعتمد عليه الجميع.

قواعد حماية الفروع (GitHub): تُضبط في إعدادات المستودع ← Branches. قواعد شائعة: اشتراط PR قبل الدمج، اشتراط موافقة واحدة على الأقل، اشتراط اجتياز فحوصات الحالة، منع force-push.
ما يحدث عند محاولة الدفع المباشر: يرفض GitHub الدفع مع خطأ GH006: Protected branch update failed. الحل دائماً: "افتح PR بدلاً من ذلك."
من يمكنه تجاوز القواعد: يمكن منح مديري المستودعات حقوق التجاوز، لكن في معظم فرق البحث حتى المديرون يمرون عبر PRs — للحفاظ على سجل التدقيق كاملاً.
⚠️ لا تطلب أبداً حقوق force-push على فرع مشترك (ولا تستخدمها). حتى حين يبدو الأمر كأسرع حل، فإنه يعيد كتابة التاريخ من تحت كل زميل سحب تلك الـ commits بالفعل — مما يتسبب في تباينات مؤلمة يصعب فكّها.

حلقة Git الجماعية
الدورة اليومية المتكررة التي يتبعها كل مساهم — ست خطوات، بلا استثناءات

كل ميزة، كل إصلاح، كل تجربة تتبع نفس الحلقة

سواء استغرق التغيير عشر دقائق أو عشرة أيام، فإنه يسلك نفس المسار. فهم هذه الحلقة كتسلسل ثابت — لا كمجموعة أوامر اعتباطية — هو ما يميّز المطورين الذين يشعرون بالسيطرة على Git عن أولئك الذين يشعرون بأنهم يخمّنون.

1. اسحب أحدث main — ابدأ دائماً من جديد: git switch main && git pull.
2. أنشئ فرع ميزة مسمّى — تفرّع من main المحدَّث حديثاً: git switch -c feature/شيئي.
3. أنشئ commits على فرعك — commits صغيرة ومركّزة برسائل واضحة. ادفع بانتظام لحفظ عملك: git push -u origin feature/شيئي.
4. حافظ على تحديث فرعك — إن تقدّم main، ادمجه أو أعد ترتيب commits قبل فتح الـ PR (مشروح في الأسئلة والأجوبة أدناه).
5. افتح PR على GitHub — اكتب وصفاً واضحاً، عيّن مراجعين، اربط الـ issues ذات الصلة.
6. استجب للمراجعة، احصل على الموافقة، ادمج، احذف الفرع — مهمة الفرع تنتهي لحظة دمجه؛ حافظ على نظافة المستودع.
# الحلقة الكاملة كما تُكتب git switch main git pull git switch -c feature/add-export-button # ...عمل، عمل، عمل... git add src/components/ExportButton.tsx git commit -m "feat: add CSV export button to results panel" git push -u origin feature/add-export-button # افتح PR على GitHub، احصل على المراجعة والموافقة # بعد الدمج على GitHub — نظّف محلياً git switch main git pull git branch -d feature/add-export-button

الخطوة الأخيرة — حذف الفرع المحلي بعد الدمج — هي التي يتخطاها معظم المطورين في البداية. بعد بضعة أشهر تجد نفسك مع عشرات الفروع المحلية القديمة التي تبدو جميعها وكأنها لا تزال نشطة. احذفها فور انتهاء عملها.

المفاهيم الأساسية — افتح كل بطاقة
أسئلة سير العمل الجماعي التي تعثّر كل مطور في أسابيعه الأولى على قاعدة كود مشتركة
الحواجز التي تحافظ على نظافة التاريخ المشترك
حماية الفروع إعداد في مستودع GitHub يمنع أي شخص من الدفع مباشرةً إلى فرع — يجب عليه المرور عبر PR بدلاً من ذلك. إنها أهم أداة تنظيمية في Git الجماعي، وتقريباً كل فريق محترف يستخدمها على main.
مجموعة القواعد النموذجية
اشتراط pull request قبل الدمج — لا دفع مباشر أبداً.

اشتراط N موافقة على الأقل — معظم الفرق تشترط 1؛ بعضها يشترط 2 لفروع الإنتاج.

اشتراط اجتياز فحوصات الحالة — يجب أن تكون الـ CI (الاختبارات، فحص الكود) خضراء قبل أن يصبح زر الدمج قابلاً للنقر.

اشتراط تحديث الفرع — يجب أن يتضمن فرعك أحدث main قبل أن تتمكن من الدمج.

منع force push — لا إعادة كتابة للتاريخ المشترك أبداً.
هذه القواعد تُطبَّق على الفرع، لا على المستخدم. حتى مديرو المستودعات خاضعون لها (إلا إذا مُنحوا صراحةً حقوق التجاوز — وهو ما لا تمنحه معظم الفرق).
إذا رُفض دفعك بخطأ GH006: Protected branch update failed، لا تبحث عن حل بديل — افتح PR. هذا هو المسار المقصود.
وقت المراجعين محدود — اجعل عملهم خالياً من الاحتكاك
PR بلا وصف يُراجَع ببطء، أو بشكل سطحي، أو لا يُراجَع أصلاً — لا يعرف المراجعون أين يركّزون. PR بوصف واضح لـ ما تغيّر ولماذا تغيّر وكيف يُختبر يحصل على مراجعة متأنية في جزء من الوقت.
ما يحتاجه كل وصف PR
ماذا: جملة واحدة تلخّص التغيير. ("يضيف زر تصدير CSV إلى لوحة النتائج.")

لماذا: اربط الـ issue/ticket الذي يحله، أو اشرح الدافع. ("يغلق #42 — أفاد المستخدمون بحاجتهم لتصدير النتائج لتحليلهم الخاص.")

كيف تختبره: خطوات ملموسة يمكن للمراجع اتباعها. ("1. شغّل التطبيق. 2. انتقل إلى النتائج. 3. انقر على تصدير — يجب أن يُنزَّل ملف باسم results.csv.")

لقطات شاشة / قبل وبعد (لتغييرات الواجهة) — صورة واحدة توفّر خمس دقائق من جهد المراجع.
كثير من الفرق تستخدم قالب PR (ملف .github/pull_request_template.md) يملأ الوصف مسبقاً بالأقسام المطلوبة. إذا كان مستودعك يمتلك واحداً، املأ كل قسم دائماً.
يتبع العنوان نفس اتفاقية رسائل الـ commit: صيغة الأمر، أقل من 72 حرفاً. "Add CSV export to results panel" — لا "Added export" أو "Export stuff".
فرعك تباعَد — أدخل main قبل الدمج
أنت تعمل على فرع منذ يومين. في هذا الوقت، دمج ثلاثة زملاء PRs-هم في main. الآن فرعك متأخر — لا يمكن دمج الـ PR بشكل نظيف، وإذا كان فريقك يشترط تحديث الفرع، فزر الدمج في GitHub رمادي. تحتاج إلى إدخال main في فرعك.
الحل المعياري — دمج main في فرعك
# تأكد أن main المحلي محدَّث git switch main git pull # ارجع إلى فرع الميزة git switch feature/add-export-button # أدخل تغييرات main git merge main # في حال وجود تعارضات، أحلّها (راجع الموضوع 3)، ثم: git add . git commit # Git يملأ رسالة merge commit مسبقاً # ادفع الفرع المحدَّث git push
بعض الفرق تفضّل git rebase main بدلاً من git merge main — الـ rebase يعيد كتابة commits-ك فوق أحدث main، منتجاً تاريخاً خطياً. كلا النهجين صالح؛ اتبع اتفاقية فريقك.
إذا أجريت rebase والفرع موجود بالفعل على GitHub، ستحتاج git push --force-with-lease (لا --force) — هذا هو الاستخدام الوحيد المشروع لـ force-push، وفقط على فرع الميزة الخاص بك.
افعل هذا بانتظام على الفروع طويلة الأمد — مرة يومياً ليس مبالغة. عمليات الدمج الصغيرة والمتكررة أسهل بكثير من دمج كبير واحد في النهاية.
⚠️ لا تستخدم git rebase أبداً على فرع يعمل عليه أشخاص آخرون أيضاً — إنه يعيد كتابة هاشات الـ commit، مما يكسر نسختهم المحلية من الفرع.
مراجعة الكود حوار، لا حكم
تلقّي أول تعليق مراجعة نقدي يبدو شخصياً. إنه ليس كذلك — المراجعون يعلّقون على الكود، لا عليك. طريقة استجابتك للمراجعة تُشكّل سمعتك في الفريق أكثر من الكود نفسه. المطورون الذين يتعاملون بتفكّر يُوثَق بهم بمسؤوليات أكبر بشكل أسرع.
نمط الاستجابة الصحيح
حين توافق: أجرِ التغيير، ادفع الـ commit، ورد على التعليق بشيء قصير — "تم" أو "مُصلَّح في آخر دفع" — حتى يعلم المراجع بالمراجعة من جديد. لا تدفع بصمت فقط.

حين لا توافق أو تحتاج توضيحاً: اسأل — لا تتجاهل التعليق. "اخترت X لأن Y — هل هذا يعالج قلقك؟" رد احترافي تماماً. المراجعون لا يملكون سياقك الكامل؛ أنت تملكه.

حين يكون التعليق خارج نطاق PR: اعترف به وافتح issue منفصلاً. "نقطة جيدة — فتحت #47 لتتبع هذا بشكل منفصل حتى لا يُعيق هذا الـ PR."
أحلّ دائماً التعليق — إما بإجراء التغيير أو بشرح سبب عدم قيامك به. التعليقات غير المحلولة هي السبب الأكثر شيوعاً لتعثّر PRs.
على GitHub، المراجع (لا المؤلف) هو من ينقر على "Resolve conversation" حين يقتنع. لا تحلّ تعليقاتك بنفسك — دع المراجع يغلقها.
أربع عادات تميّز لاعب فريق متمرس عن وافد جديد
هذه ليست مواضيع Git متقدمة — إنها عادات سير عمل يتبعها المطورون المتمرسون تلقائياً. خرقها لا يكسر Git، لكنه يُنشئ احتكاكاً لكل شخص في الفريق ويُشير للمطورين الأقدم أن الشخص لم يعمل على قاعدة كود مشتركة من قبل.
القواعد الأربع
1. أنشئ commit على فرع، لا مباشرةً على main. حتى لإصلاح سطر واحد. الفروع مجانية — لا مبرر لتخطي هذه الخطوة.

2. اسحب قبل أن تدفع. git pull قبل بدء العمل وقبل فتح PR. إن تخطيت هذا، قد يُرفض دفعك أو يكون الـ PR به تعارضات لا داعي لها.

3. حافظ على commits مركّزة ورسائل ذات معنى. رسائل "fixes" أو "wip" تجعل git log عديم الفائدة للجميع. الاتفاقية: صيغة أمر، مضارع، أقل من 72 حرفاً.

4. لا تدفع أسراراً أبداً على أي فرع. مفاتيح API وكلمات مرور وتوكنز وملفات .env لا تنتمي لـ Git — حتى على فرع "خاص" تنوي تنظيفه لاحقاً.
⚠️ الأسرار المدفوعة لا يمكن "حذفها" بـ commit عادي. تبقى في التاريخ الكامل للمستودع، مرئية لكل من لديه صلاحية قراءة. إن دفعت سراً بالخطأ، غيّره فوراً واتصل بمسؤول فريقك — لا تنتظر.
ثلاثة أزرار دمج، ثلاث نتائج مختلفة جداً في git log
يُقدّم GitHub ثلاث استراتيجيات دمج عند إغلاق PR. تختار معظم الفرق واحدة وتُطبّقها باتساق. معرفة ما تُنتجه كل منها في التاريخ يساعدك على فهم لماذا اتخذ فريقك هذا الاختيار — ويمنع المفاجآت حين تنظر في git log بعد الدمج.
الاستراتيجيات الثلاث
Merge commit (الافتراضي) — تُحفظ جميع commits الفرع كما هي، بالإضافة إلى إضافة merge commit. يُظهر التاريخ بالضبط ما حدث ومتى. توبولوجيا الفرع مرئية في git log --graph. مفضَّل حين يكون تاريخ commits الميزة ذا معنى.

Squash and merge — تُجمَّع جميع commits الفرع في commit واحد على main. يصبح عمل الـ PR كله commit واحداً نظيفاً. مفضَّل حين تكون commits الفرع نقاط تفتيش فوضوية.

Rebase and merge — تُعاد commits الفرع واحداً تلو الآخر فوق أحدث main، بلا merge commit. التاريخ خطي تماماً. مفضَّل على الفرق التي تُولي git log النظيف الأولوية القصوى.
تحقق من الاستراتيجية التي يستخدمها فريقك قبل فتح أول PR — يؤثر ذلك على كيفية كتابة commits وما إذا كانت رسائل commits الفردية ستظهر في سجل main.
Squash هو الأكثر تسامحاً للوافدين الجدد — commits الفرع الفوضوية تُطوى في إدخال نظيف واحد. كثير من فرق البحث تستخدم squash كافتراضي لهذا السبب بالضبط.

شغّل الحلقة الجماعية الكاملة في مستودع تدريبي محلي

السكريبت التالي يحاكي عملك أنت وزميل على نفس المستودع معاً — زميلك يدمج في main بينما أنت في منتصف ميزة، بالضبط الوضع الذي يفاجئ المطورين الجدد. مرّ عليه مرة واحدة وستشعر بأن الحلقة أصبحت آلية (بالمعنى الإيجابي) من تلك اللحظة.

# ─── الإعداد: إنشاء مستودع تدريبي مشترك ───────────────────────────────────── git init team-workflow-practice && cd team-workflow-practice echo "# Team App" > README.md git add . && git commit -m "chore: initial commit" # ─── أنت: ابدأ فرع ميزة ─────────────────────────────────────────────────────── git switch -c feature/add-dashboard echo "dashboard placeholder" > dashboard.txt git add . && git commit -m "feat: scaffold dashboard page" # ─── محاكاة زميل يدمج في main بينما تعمل ───────────────────────────────────── git switch main echo "utility function" > utils.txt git add . && git commit -m "feat: add shared utility (teammate's work)" # ─── أنت: ارجع إلى فرعك وحدّثه ────────────────────────────────────────────── git switch feature/add-dashboard git merge main # → لا تعارض هنا — ملفات مختلفة. يُنشأ merge commit. git log --oneline --graph # → يمكنك رؤية تاريخ فرعك والدمج من main # ─── أنت: أكمل الميزة ──────────────────────────────────────────────────────── echo "dashboard complete" >> dashboard.txt git add . && git commit -m "feat: complete dashboard implementation" # ─── محاكاة PR: دمج فرعك في main ───────────────────────────────────────────── git switch main git merge feature/add-dashboard --no-ff -m "Merge pull request: Add dashboard page" # ─── التنظيف: حذف فرع الميزة ───────────────────────────────────────────────── git branch -d feature/add-dashboard git log --oneline --graph # → main يحتوي الآن على عمل الزميل ولوحة التحكم الخاصة بك، مدموجَين بنظافة

كرّر هذا بتغيير متعارض — دع كليكما أنت و"زميلك" يحرران نفس الملف — وأحلّ التعارض باستخدام خطوات الموضوع الثالث. تلك هي الحلقة الكاملة بكل التعقيدات الحقيقية مشمولةً.
