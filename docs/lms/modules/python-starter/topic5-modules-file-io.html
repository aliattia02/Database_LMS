Importing Modules
Reusing code from Python's standard library — and your own files

Import styles you'll see in real code

A module is just a file of reusable Python code (or a built-in component of the interpreter). The import statement loads it into your script so you can use the names it defines. There are several import styles, each with a different trade-off between convenience and clarity.

import math # import the whole module import os as operating_system # import with an alias from datetime import datetime # import one name directly from collections import defaultdict, namedtuple # import several names from math import * # ⚠️ avoid — imports everything math.sqrt(16) # → 4.0 (must use the module prefix) operating_system.getcwd() # → current working directory datetime.now() # → no prefix needed, imported directly
Python caches modules the first time they're imported — importing the same module again later in the same run does not re-run its code.
Use dir(module) to list the names a module defines, and help(module) to read its documentation from the interpreter.
from module import * works, but it makes it hard to tell where a name came from when you read the code later — avoid it outside quick experiments.

A tour of the standard library

Python ships with a large standard library, so many common tasks need no extra installation. A handful of modules cover most day-to-day needs.

import os # filesystem paths, environment variables, processes import sys # interpreter internals, command-line arguments import math # mathematical functions (sqrt, floor, pi, ...) import random # pseudo-random numbers and choices import datetime # dates, times, and durations import json # reading and writing JSON data from pathlib import Path # modern, object-oriented filesystem paths
Third-party packages (not in the standard library, like requests or pandas) are installed with pip install package_name and then imported the same way.
Never name your own script the same as a standard module (e.g. random.py) — Python will import your file instead of the real one and silently break unrelated code.

Opening and Closing Files
The open() function, file modes, and why closing matters

The open() function and file modes

open() returns a file object you can read from or write to. The second argument — the mode — controls whether you're reading, writing, or appending, and whether the file is treated as text or raw bytes.

f = open("notes.txt", "r") # open for reading (text mode is the default) content = f.read() f.close() # releases the OS file handle — don't skip this # Common modes "r" # read — file must already exist "w" # write — creates the file; OVERWRITES it if it already exists "a" # append — creates the file if missing; writes go to the end "x" # exclusive create — fails with an error if the file already exists "r+" # read AND write — file must already exist "rb" / "wb" # binary mode — for images, zip files, etc. (no text decoding)
Forgetting f.close() can leave writes un-flushed to disk and leaks a limited OS resource — every open file handle that isn't closed stays reserved until the program exits.
Pass encoding="utf-8" explicitly in text mode to avoid relying on the operating system's default encoding, which differs between platforms.

Using with to close files automatically

The with statement creates a context manager that closes the file for you automatically — even if an exception is raised inside the block. This is the standard, idiomatic way to work with files in Python.

with open("notes.txt", "r", encoding="utf-8") as f: content = f.read() # f is already closed here — no f.close() needed, even if read() had failed # Multiple files in one with statement with open("in.txt") as src, open("out.txt", "w") as dst: dst.write(src.read().upper())
⚠️ "w" mode truncates the file to zero length the instant it is opened — before you write a single byte. If you want to keep the existing content, use "a" (append) instead.
Prefer with open(...) as f: over manual open() / close() in almost all real code — it's shorter and impossible to forget to close.
As soon as execution leaves the with block — normally or via an exception — the file is closed; there is no scenario where it stays open by mistake.

Reading and Writing Files
Pulling content out of a file and pushing content into one

Reading: read(), readline(), readlines(), and iteration

Python gives you several ways to read a file, depending on whether you want it all at once, one line at a time, or as a list of lines.

with open("poem.txt") as f: whole = f.read() # entire file as a single string with open("poem.txt") as f: first_line = f.readline() # one line, including the trailing "\n" with open("poem.txt") as f: lines = f.readlines() # a list of every line, each ending in "\n" # Most memory-efficient — read one line at a time, never loading it all with open("poem.txt") as f: for line in f: print(line.strip()) # .strip() removes the trailing newline
Iterating directly over the file object (for line in f) is the preferred pattern for large files — it streams one line at a time instead of loading the whole file into memory.
.strip() removes leading and trailing whitespace, which conveniently includes the \n each line read from a file ends with.

Writing: write() and writelines()

with open("log.txt", "w") as f: f.write("First line\n") # write() does NOT add a newline for you f.writelines(["a\n", "b\n", "c\n"]) # writes each string from an iterable with open("log.txt", "a") as f: f.write("Appended later\n") # "a" mode never overwrites existing content
write() returns the number of characters written, not None — easy to assume otherwise since it's rarely used.
print("text", file=f) is also a valid way to write to a file and adds the newline for you automatically, like a normal print() call.

Modern paths with pathlib

pathlib.Path represents filesystem paths as objects instead of plain strings, and handles platform differences (like / vs \) for you.

from pathlib import Path p = Path("data") / "input.txt" # joins path segments — works on any OS p.exists() # → True / False p.is_file() # → True / False p.suffix # → ".txt" p.stem # → "input" p.read_text() # shortcut: open, read, and close in one call p.write_text("hello") # shortcut: open, write, and close in one call

Handling Files Safely
Errors, edge cases, and reading/writing structured data

Catching file errors with try / except

Files can fail to open for reasons outside your control — the file is missing, you lack permission, or the disk is full. Wrap risky file operations in try / except instead of letting an unhandled error crash the whole script.

try: with open("config.json") as f: data = f.read() except FileNotFoundError: print("Config file is missing — using defaults.") data = "{}" except PermissionError: print("No permission to read this file.") raise # Checking first is an alternative, but not airtight — # the file could still vanish between the check and the open() call from pathlib import Path if Path("config.json").exists(): ...
⚠️ Checking Path.exists() before opening reduces surprises, but it is not race-proof: another process could delete the file in the gap between the check and the open() call. try / except is the safer pattern in real systems.
FileNotFoundError and PermissionError are the two most common file exceptions — both are subclasses of the broader OSError.
Catch the most specific exception you can handle meaningfully; re-raise (raise) anything you can't actually recover from.

Reading and writing JSON

The json module pairs naturally with file I/O — it reads and writes structured data directly to and from a file object, no manual string parsing needed.

import json with open("config.json", "w") as f: json.dump({"debug": True, "retries": 3}, f, indent=2) # write JSON with open("config.json") as f: config = json.load(f) # → {'debug': True, 'retries': 3} # String versions, when you don't have a file — e.g. an API response text = json.dumps({"ok": True}) # dict → JSON string back = json.loads(text) # JSON string → dict
Remember the naming pattern: the "s" versions (dumps/loads) work on strings; the plain versions (dump/load) work on file objects.
json.dump(..., indent=2) produces human-readable, indented output — useful for config files a person might open and edit.

File modes at a glance

Mode Read? Write? If file is missing If file exists
r ✅ Yes ❌ No ❌ Raises FileNotFoundError Opens it as-is
w ❌ No ✅ Yes ✅ Creates it ⚠️ Overwrites (truncates) it
a ❌ No ✅ Yes ✅ Creates it Appends to the end
x ❌ No ✅ Yes ✅ Creates it ❌ Raises FileExistsError

Practice Questions
Typical interview and self-check questions on modules and file I/O
Mental model
import module brings in the whole toolbox and you keep reaching for it by name every time. from module import name takes one specific tool out of the box and puts it directly on your workbench — shorter to use, but the box it came from is less obvious later.
The Answer
import module loads the module and requires you to prefix every name from it with module.. from module import name imports a specific name directly into your namespace, so you use it bare, without the prefix. Both load the same underlying code — the difference is only in how the names become available afterward.
import math math.pi # → 3.14159... (prefix required) from math import pi pi # → 3.14159... (used directly, no prefix)
The prefixed form (import module) is generally easier to read in larger files — anyone can tell at a glance that math.sqrt comes from math.
You can combine both: import module for general use, plus a targeted from module import specific_thing for a name you use constantly.
Mental model
Manual open()/close() is like borrowing a library book and trusting yourself to return it — works fine until you get distracted by an error halfway through and never do. with is the library returning the book for you automatically the moment you walk out the door, no matter what happened while you were browsing.
The Answer
A with block is a context manager: it guarantees that f.close() runs when the block ends, even if an exception is raised inside it. With manual open()/close(), an exception between the two calls skips close() entirely, leaving the file handle open until the program exits.
# Risky — close() is skipped if read() raises an exception f = open("data.txt") content = f.read() # if this raises, f.close() below never runs f.close() # Safe — close() always runs, exception or not with open("data.txt") as f: content = f.read()
A try / finally around manual open()/close() achieves the same safety, but with says the same thing in less code and is the convention every Python codebase expects.
Many other objects besides files support with — locks, database connections, and network sockets all use the same pattern for guaranteed cleanup.
Mental model
Think of a file as a stack of index cards. read() hands you the entire stack glued into one long strip. readline() hands you one card at a time. readlines() hands you the whole stack, but still separated into individual cards inside a list.
The Answer
read() returns the entire file as one string. readline() returns a single line (including its trailing \n) and advances to the next one on each call. readlines() returns a list containing every line as a separate string. For large files, iterating the file object directly with for line in f: is usually best — it behaves like repeated readline() calls without building a full list in memory.
with open("a.txt") as f: f.read() # → "line1\nline2\nline3\n" (one big string) with open("a.txt") as f: f.readline() # → "line1\n" (first line only) f.readline() # → "line2\n" (next call, next line) with open("a.txt") as f: f.readlines() # → ["line1\n", "line2\n", "line3\n"]
readlines() loads the whole file into memory as a list — fine for small files, wasteful for huge ones.
All three methods continue from wherever the file's internal "cursor" currently is — calling read() twice in a row returns an empty string the second time, since the cursor is already at the end.
Mental model
Text mode is like reading a letter — Python automatically decodes the raw bytes into readable characters using an encoding, and translates line endings for you. Binary mode is like handling the sealed envelope itself — you get the exact raw bytes, untouched, with no assumptions about what they mean.
The Answer
Text mode (the default, or explicit "t") decodes bytes into str using a text encoding — usually UTF-8 — and returns str objects from read(). Binary mode ("b" suffix, e.g. "rb", "wb") returns raw bytes with no decoding at all. Binary mode is required for non-text files — images, audio, ZIP archives, and any format that isn't meant to be interpreted as characters.
with open("photo.jpg", "rb") as f: data = f.read() # → bytes object, e.g. b'\xff\xd8\xff\xe0...' with open("notes.txt", "r", encoding="utf-8") as f: text = f.read() # → str object, e.g. "Hello, world"
Opening a binary file (like an image) in text mode usually raises a UnicodeDecodeError — the bytes aren't valid text in the assumed encoding.
If you're not sure which mode you need: text files (.txt, .json, .csv, source code) → text mode; everything else → binary mode.
Mental model
There are two strategies: "look before you leap" (check first, then act) and "ask forgiveness, not permission" (just try, and handle the failure if it happens). Python culture generally favors the second for files, because checking first can never fully guarantee the file is still there a moment later.
The Answer
Wrap the open() call in try / except FileNotFoundError and provide a fallback — a default value, a friendly message, or creating the file. This handles the missing-file case without crashing, and without a separate existence check that could go stale between the check and the open.
try: with open("settings.txt") as f: settings = f.read() except FileNotFoundError: settings = "default_theme=light" # fall back to a sensible default with open("settings.txt", "w") as f: f.write(settings) # and create the file for next time
Path("file").exists() from pathlib is fine for a quick sanity check or a friendlier error message, but pair it with try/except rather than relying on it alone.
Use mode "a" (append) instead of a check-then-create dance when you simply want a file to exist and be ready to receive more data — it creates the file automatically if missing.
Mental model
JSON is just a text format for dictionaries and lists. The json module is a translator that sits between Python's native objects and that text format, so you never have to hand-write the punctuation yourself.
The Answer
Use json.dump(obj, file) to write a Python object straight to an open file as JSON, and json.load(file) to read JSON from an open file back into a Python object (typically a dict or list). The string-only siblings json.dumps() and json.loads() convert to and from a JSON string instead of a file, which is useful when the JSON came from somewhere other than a file — like a network response.
import json data = {"name": "Ada", "languages": ["Python", "SQL"]} with open("profile.json", "w") as f: json.dump(data, f, indent=2) # writes formatted JSON to the file with open("profile.json") as f: loaded = json.load(f) # → {'name': 'Ada', 'languages': [...]}
JSON only supports a subset of Python's types — dicts, lists, strings, numbers, booleans, and None. Tuples become lists, and types like set or custom classes raise a TypeError unless you supply custom encoding logic.
json.load() raises json.JSONDecodeError if the file's content isn't valid JSON — worth catching alongside FileNotFoundError when loading config files.

Module importieren
Code aus Pythons Standardbibliothek wiederverwenden — und aus eigenen Dateien

Import-Stile, die dir in echtem Code begegnen

Ein Modul ist einfach eine Datei mit wiederverwendbarem Python-Code (oder eine eingebaute Komponente des Interpreters). Die import-Anweisung lädt es in dein Skript, sodass du die darin definierten Namen nutzen kannst. Es gibt mehrere Import-Stile, jeweils mit einem anderen Verhältnis zwischen Bequemlichkeit und Klarheit.

import math # ganzes Modul importieren import os as betriebssystem # mit Alias importieren from datetime import datetime # einen Namen direkt importieren from collections import defaultdict, namedtuple # mehrere Namen importieren from math import * # ⚠️ vermeiden — importiert alles math.sqrt(16) # → 4.0 (Modul-Präfix erforderlich) betriebssystem.getcwd() # → aktuelles Arbeitsverzeichnis datetime.now() # → kein Präfix nötig, direkt importiert
Python cached Module beim ersten Import — ein erneuter Import desselben Moduls im selben Programmlauf führt seinen Code nicht noch einmal aus.
dir(modul) listet die im Modul definierten Namen auf, help(modul) zeigt seine Dokumentation direkt im Interpreter.
from modul import * funktioniert, macht es aber schwer nachzuvollziehen, woher ein Name stammt, wenn man den Code später liest — außerhalb kurzer Experimente vermeiden.

Ein Rundgang durch die Standardbibliothek

Python wird mit einer umfangreichen Standardbibliothek geliefert, sodass viele alltägliche Aufgaben keine zusätzliche Installation benötigen. Eine Handvoll Module deckt die meisten Bedürfnisse ab.

import os # Dateisystempfade, Umgebungsvariablen, Prozesse import sys # Interpreter-Interna, Kommandozeilenargumente import math # mathematische Funktionen (sqrt, floor, pi, ...) import random # Pseudozufallszahlen und -auswahl import datetime # Daten, Zeiten und Zeitspannen import json # JSON-Daten lesen und schreiben from pathlib import Path # moderne, objektorientierte Dateisystempfade
Drittanbieter-Pakete (nicht in der Standardbibliothek, z. B. requests oder pandas) werden mit pip install paketname installiert und dann genauso importiert.
Nenne dein eigenes Skript niemals wie ein Standardmodul (z. B. random.py) — Python importiert dann deine Datei statt des echten Moduls und bricht stillschweigend nicht zusammenhängenden Code.

Dateien öffnen und schließen
Die Funktion open(), Dateimodi, und warum Schließen wichtig ist

Die Funktion open() und Dateimodi

open() gibt ein Dateiobjekt zurück, von dem du lesen oder in das du schreiben kannst. Das zweite Argument — der Modus — bestimmt, ob du liest, schreibst oder anhängst, und ob die Datei als Text oder als reine Bytes behandelt wird.

datei = open("notizen.txt", "r") # zum Lesen öffnen (Textmodus ist Standard) inhalt = datei.read() datei.close() # gibt das Betriebssystem-Handle frei — nicht weglassen # Gängige Modi "r" # lesen — Datei muss bereits existieren "w" # schreiben — erstellt die Datei; ÜBERSCHREIBT sie, falls vorhanden "a" # anhängen — erstellt die Datei falls fehlend; schreibt ans Ende "x" # exklusiv erstellen — schlägt fehl, wenn die Datei schon existiert "r+" # lesen UND schreiben — Datei muss bereits existieren "rb" / "wb" # Binärmodus — für Bilder, ZIP-Dateien usw. (keine Textdekodierung)
Vergisst man datei.close(), können Schreibvorgänge ungeschrieben im Puffer bleiben, und eine begrenzte Betriebssystemressource bleibt belegt, bis das Programm endet.
Im Textmodus explizit encoding="utf-8" angeben, um sich nicht auf die Standardkodierung des Betriebssystems zu verlassen, die sich zwischen Plattformen unterscheidet.

Mit with Dateien automatisch schließen

Die with-Anweisung erstellt einen Kontextmanager, der die Datei automatisch für dich schließt — selbst wenn innerhalb des Blocks eine Ausnahme ausgelöst wird. Das ist die übliche, idiomatische Art, in Python mit Dateien zu arbeiten.

with open("notizen.txt", "r", encoding="utf-8") as datei: inhalt = datei.read() # datei ist hier bereits geschlossen — kein datei.close() nötig, selbst bei einem Fehler # Mehrere Dateien in einer with-Anweisung with open("ein.txt") as quelle, open("aus.txt", "w") as ziel: ziel.write(quelle.read().upper())
⚠️ Der Modus "w" kürzt die Datei in dem Moment, in dem sie geöffnet wird, auf null Länge — noch bevor ein einziges Byte geschrieben wurde. Soll der bestehende Inhalt erhalten bleiben, "a" (anhängen) verwenden.
with open(...) as datei: in fast allem realen Code dem manuellen open() / close() vorziehen — kürzer, und das Schließen kann nicht vergessen werden.
Sobald die Ausführung den with-Block verlässt — normal oder durch eine Ausnahme — wird die Datei geschlossen; es gibt kein Szenario, in dem sie versehentlich offen bleibt.
✍️
Dateien lesen und schreiben
Inhalte aus einer Datei holen und Inhalte in eine Datei schreiben

Lesen: read(), readline(), readlines() und Iteration

Python bietet mehrere Wege, eine Datei zu lesen — abhängig davon, ob du alles auf einmal, Zeile für Zeile, oder als Liste von Zeilen möchtest.

with open("gedicht.txt") as datei: ganzer_text = datei.read() # gesamte Datei als ein String with open("gedicht.txt") as datei: erste_zeile = datei.readline() # eine Zeile, inklusive abschließendem "\n" with open("gedicht.txt") as datei: zeilen = datei.readlines() # Liste aller Zeilen, jede endet mit "\n" # Am speichereffizientesten — eine Zeile nach der anderen, nie alles auf einmal with open("gedicht.txt") as datei: for zeile in datei: print(zeile.strip()) # .strip() entfernt den abschließenden Zeilenumbruch
Direktes Iterieren über das Dateiobjekt (for zeile in datei) ist das bevorzugte Muster für große Dateien — es liest eine Zeile nach der anderen, statt die ganze Datei in den Speicher zu laden.
.strip() entfernt führende und nachfolgende Leerzeichen, was praktischerweise auch den \n einschließt, mit dem jede gelesene Zeile endet.

Schreiben: write() und writelines()

with open("protokoll.txt", "w") as datei: datei.write("Erste Zeile\n") # write() fügt KEINEN Zeilenumbruch hinzu datei.writelines(["a\n", "b\n", "c\n"]) # schreibt jeden String aus einem Iterable with open("protokoll.txt", "a") as datei: datei.write("Später angehängt\n") # "a"-Modus überschreibt nie bestehenden Inhalt
write() gibt die Anzahl geschriebener Zeichen zurück, nicht None — leicht das Gegenteil anzunehmen, da der Rückgabewert selten genutzt wird.
print("text", file=datei) ist ebenfalls eine gültige Möglichkeit, in eine Datei zu schreiben, und fügt den Zeilenumbruch automatisch hinzu, genau wie ein normaler print()-Aufruf.

Moderne Pfade mit pathlib

pathlib.Path stellt Dateisystempfade als Objekte statt als einfache Strings dar und übernimmt Plattformunterschiede (wie / versus \) für dich.

from pathlib import Path p = Path("daten") / "eingabe.txt" # verbindet Pfadsegmente — auf jedem Betriebssystem p.exists() # → True / False p.is_file() # → True / False p.suffix # → ".txt" p.stem # → "eingabe" p.read_text() # Abkürzung: öffnen, lesen, schließen in einem Aufruf p.write_text("hallo") # Abkürzung: öffnen, schreiben, schließen in einem Aufruf

Dateien sicher handhaben
Fehler, Randfälle, und das Lesen/Schreiben strukturierter Daten

Dateifehler mit try / except abfangen

Dateien können aus Gründen außerhalb deiner Kontrolle nicht geöffnet werden — die Datei fehlt, dir fehlt die Berechtigung, oder die Festplatte ist voll. Riskante Dateioperationen in try / except einbetten, statt einen unbehandelten Fehler das ganze Skript abstürzen zu lassen.

try: with open("konfig.json") as datei: daten = datei.read() except FileNotFoundError: print("Konfigurationsdatei fehlt — verwende Standardwerte.") daten = "{}" except PermissionError: print("Keine Berechtigung, diese Datei zu lesen.") raise # Vorheriges Prüfen ist eine Alternative, aber nicht hieb- und stichfest — # die Datei könnte zwischen Prüfung und open()-Aufruf trotzdem verschwinden from pathlib import Path if Path("konfig.json").exists(): ...
⚠️ Path.exists() vor dem Öffnen zu prüfen verringert Überraschungen, ist aber nicht race-sicher: Ein anderer Prozess könnte die Datei in der Lücke zwischen Prüfung und open()-Aufruf löschen. try / except ist das sicherere Muster in echten Systemen.
FileNotFoundError und PermissionError sind die zwei häufigsten Dateifehler — beide sind Unterklassen des allgemeineren OSError.
Die spezifischste Ausnahme abfangen, die man sinnvoll behandeln kann; alles andere mit raise weiterreichen, wenn man sich nicht wirklich davon erholen kann.

JSON lesen und schreiben

Das json-Modul passt natürlich zur Datei-I/O — es liest und schreibt strukturierte Daten direkt von und zu einem Dateiobjekt, ohne manuelles String-Parsing.

import json with open("konfig.json", "w") as datei: json.dump({"debug": True, "versuche": 3}, datei, indent=2) # JSON schreiben with open("konfig.json") as datei: konfig = json.load(datei) # → {'debug': True, 'versuche': 3} # String-Varianten, wenn keine Datei vorliegt — z. B. eine API-Antwort text = json.dumps({"ok": True}) # dict → JSON-String zurueck = json.loads(text) # JSON-String → dict
Merke das Namensmuster: die Varianten mit "s" (dumps/loads) arbeiten mit Strings; die einfachen Varianten (dump/load) arbeiten mit Dateiobjekten.
json.dump(..., indent=2) erzeugt lesbare, eingerückte Ausgabe — nützlich für Konfigurationsdateien, die man von Hand öffnen und bearbeiten könnte.

Übungsfragen
Typische Interview- und Selbstkontrollfragen zu Modulen und Datei-I/O
Mentales Modell
import modul bringt die ganze Werkzeugkiste mit, und du greifst jedes Mal namentlich darauf zu. from modul import name nimmt ein bestimmtes Werkzeug aus der Kiste und legt es direkt auf deine Werkbank — kürzer in der Nutzung, aber später weniger offensichtlich, aus welcher Kiste es stammt.
Die Antwort
import modul lädt das Modul und verlangt, dass jeder Name daraus mit modul. vorangestellt wird. from modul import name importiert einen bestimmten Namen direkt in deinen Namensraum, sodass du ihn ohne Präfix verwendest. Beide laden denselben zugrunde liegenden Code — der Unterschied liegt nur darin, wie die Namen danach verfügbar werden.
import math math.pi # → 3.14159... (Präfix erforderlich) from math import pi pi # → 3.14159... (direkt verwendet, kein Präfix)
Die Form mit Präfix (import modul) ist in größeren Dateien meist leichter zu lesen — jeder erkennt auf einen Blick, dass math.sqrt aus math stammt.
Beides lässt sich kombinieren: import modul für den allgemeinen Gebrauch, plus ein gezieltes from modul import bestimmtes_ding für einen Namen, den man ständig verwendet.
Mentales Modell
Manuelles open()/close() ist, als würde man ein Bibliotheksbuch ausleihen und sich selbst zutrauen, es zurückzubringen — funktioniert gut, bis man mitten im Lesen von einem Fehler abgelenkt wird und es nie zurückgibt. with ist die Bibliothek, die das Buch automatisch zurückgibt, sobald man die Tür verlässt — egal was während des Stöberns passiert ist.
Die Antwort
Ein with-Block ist ein Kontextmanager: Er garantiert, dass datei.close() ausgeführt wird, wenn der Block endet — selbst wenn darin eine Ausnahme auftritt. Bei manuellem open()/close() überspringt eine Ausnahme zwischen den beiden Aufrufen close() komplett, sodass das Datei-Handle bis zum Programmende offen bleibt.
# Riskant — close() wird übersprungen, wenn read() eine Ausnahme auslöst datei = open("daten.txt") inhalt = datei.read() # löst dies aus, läuft das close() unten nie datei.close() # Sicher — close() läuft immer, mit oder ohne Ausnahme with open("daten.txt") as datei: inhalt = datei.read()
Ein try / finally um manuelles open()/close() erreicht dieselbe Sicherheit, aber with sagt dasselbe in weniger Code und ist die Konvention, die jede Python-Codebasis erwartet.
Viele andere Objekte außer Dateien unterstützen with — Locks, Datenbankverbindungen und Netzwerk-Sockets verwenden alle das gleiche Muster für garantiertes Aufräumen.
Mentales Modell
Stell dir eine Datei als Stapel Karteikarten vor. read() gibt dir den ganzen Stapel, zu einem langen Streifen zusammengeklebt. readline() gibt dir eine Karte nach der anderen. readlines() gibt dir den ganzen Stapel, aber weiterhin als einzelne Karten in einer Liste.
Die Antwort
read() gibt die gesamte Datei als einen String zurück. readline() gibt eine einzelne Zeile zurück (inklusive abschließendem \n) und rückt bei jedem Aufruf zur nächsten vor. readlines() gibt eine Liste zurück, die jede Zeile als eigenen String enthält. Bei großen Dateien ist direktes Iterieren über das Dateiobjekt mit for zeile in datei: meist am besten — es verhält sich wie wiederholte readline()-Aufrufe, ohne eine vollständige Liste im Speicher aufzubauen.
with open("a.txt") as datei: datei.read() # → "zeile1\nzeile2\nzeile3\n" (ein großer String) with open("a.txt") as datei: datei.readline() # → "zeile1\n" (nur erste Zeile) datei.readline() # → "zeile2\n" (nächster Aufruf, nächste Zeile) with open("a.txt") as datei: datei.readlines() # → ["zeile1\n", "zeile2\n", "zeile3\n"]
readlines() lädt die gesamte Datei als Liste in den Speicher — bei kleinen Dateien unproblematisch, bei sehr großen verschwenderisch.
Alle drei Methoden setzen dort fort, wo der interne "Cursor" der Datei aktuell steht — ruft man read() zweimal hintereinander auf, kommt beim zweiten Mal ein leerer String zurück, da der Cursor schon am Ende steht.
Mentales Modell
Textmodus ist wie das Lesen eines Briefs — Python dekodiert die rohen Bytes automatisch mithilfe einer Kodierung in lesbare Zeichen und übersetzt Zeilenenden für dich. Binärmodus ist wie der Umgang mit dem versiegelten Umschlag selbst — man bekommt die exakten rohen Bytes, unangetastet, ohne Annahmen über ihre Bedeutung.
Die Antwort
Textmodus (der Standard, oder explizit "t") dekodiert Bytes mithilfe einer Textkodierung — meist UTF-8 — in str und liefert str-Objekte von read(). Binärmodus (Suffix "b", z. B. "rb", "wb") liefert rohe bytes ohne jegliche Dekodierung. Binärmodus ist für Nicht-Text-Dateien erforderlich — Bilder, Audio, ZIP-Archive, und jedes Format, das nicht als Zeichen interpretiert werden soll.
with open("foto.jpg", "rb") as datei: daten = datei.read() # → bytes-Objekt, z. B. b'\xff\xd8\xff\xe0...' with open("notizen.txt", "r", encoding="utf-8") as datei: text = datei.read() # → str-Objekt, z. B. "Hallo, Welt"
Eine Binärdatei (wie ein Bild) im Textmodus zu öffnen löst meist einen UnicodeDecodeError aus — die Bytes sind kein gültiger Text in der angenommenen Kodierung.
Im Zweifel: Textdateien (.txt, .json, .csv, Quellcode) → Textmodus; alles andere → Binärmodus.
Mentales Modell
Es gibt zwei Strategien: "erst schauen, dann handeln" (zuerst prüfen, dann agieren) und "lieber um Verzeihung bitten als um Erlaubnis fragen" (einfach versuchen und bei Fehlschlag reagieren). Die Python-Kultur bevorzugt bei Dateien meist die zweite, weil eine vorherige Prüfung nie vollständig garantieren kann, dass die Datei einen Moment später noch da ist.
Die Antwort
Den open()-Aufruf in try / except FileNotFoundError einbetten und einen Fallback bereitstellen — einen Standardwert, eine freundliche Meldung, oder das Erstellen der Datei. Das behandelt den Fall einer fehlenden Datei, ohne abzustürzen, und ohne eine separate Existenzprüfung, die zwischen Prüfung und Öffnen veralten könnte.
try: with open("einstellungen.txt") as datei: einstellungen = datei.read() except FileNotFoundError: einstellungen = "standard_theme=hell" # auf einen sinnvollen Standard zurückfallen with open("einstellungen.txt", "w") as datei: datei.write(einstellungen) # und die Datei fürs nächste Mal anlegen
Path("datei").exists() aus pathlib eignet sich für eine schnelle Plausibilitätsprüfung oder eine freundlichere Fehlermeldung, sollte aber mit try/except kombiniert werden, statt sich allein darauf zu verlassen.
Modus "a" (anhängen) statt eines "prüfen-dann-erstellen"-Tanzes verwenden, wenn eine Datei einfach existieren und bereit sein soll, weitere Daten aufzunehmen — er erstellt die Datei automatisch, falls sie fehlt.
Mentales Modell
JSON ist einfach ein Textformat für Dictionaries und Listen. Das json-Modul ist ein Übersetzer zwischen Pythons nativen Objekten und diesem Textformat, sodass man die Zeichensetzung nie selbst von Hand schreiben muss.
Die Antwort
json.dump(objekt, datei) verwenden, um ein Python-Objekt direkt als JSON in eine geöffnete Datei zu schreiben, und json.load(datei), um JSON aus einer geöffneten Datei zurück in ein Python-Objekt zu lesen (typischerweise ein dict oder eine list). Die reinen String-Varianten json.dumps() und json.loads() konvertieren zu und von einem JSON-String statt einer Datei — nützlich, wenn das JSON nicht aus einer Datei stammt, sondern z. B. aus einer Netzwerkantwort.
import json daten = {"name": "Ada", "sprachen": ["Python", "SQL"]} with open("profil.json", "w") as datei: json.dump(daten, datei, indent=2) # formatiertes JSON in die Datei schreiben with open("profil.json") as datei: geladen = json.load(datei) # → {'name': 'Ada', 'sprachen': [...]}
Merke das Namensmuster: die Varianten mit "s" (dumps/loads) arbeiten mit Strings; die einfachen Varianten (dump/load) arbeiten mit Dateiobjekten.
json.load() löst json.JSONDecodeError aus, wenn der Dateiinhalt kein gültiges JSON ist — lohnt sich, neben FileNotFoundError beim Laden von Konfigurationsdateien abzufangen.
