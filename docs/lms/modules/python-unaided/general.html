Typing logic from scratch, not just recognising correct code when you see it.

Core Conceptual Questions
Click any question to expand — focus on understanding, not memorising
Mental model
AI generates code; it doesn't read a traceback at 2am, defend a design choice in a review, or write live in front of an interviewer.
The Answer
Generating code and being able to produce, explain, and fix it are different skills. The moments that actually test you — live coding interviews, code review questions, production incidents with hospital data, debugging at odd hours — are exactly the moments where AI assistance is slow, unavailable, or simply not appropriate to lean on. This module builds the underlying skill that AI use was masking.
AI-assisted output is a ceiling on what you can produce, not a floor on what you understand.
Unaided fluency is what gets evaluated in interviews and incident response.
Mental model
Recognition is passive — you can tell good code from bad. Production is active — you can build it under a blank cursor.
The Answer
Recognising correct code only requires pattern-matching against examples you've seen. Writing it from memory requires holding the problem, the data types, the edge cases, and the syntax in your head simultaneously — there's no autocomplete suggesting the next line. That's why this module starts with small, boring functions: they isolate production ability from recognition ability.
Small functions (3–10 lines) are deliberately chosen to make gaps obvious.
If you can type merge_dicts(*dicts) correctly first try, that's automaticity — not luck.
Mental model
Writing code is forward motion. Reading an error is reverse engineering — starting from a symptom and working back to a cause.
The Answer
A traceback like TypeError on line 45 tells you where the program gave up, not necessarily where the bug was introduced. Reading code under failure means tracing data backwards: what was passed in, what type was expected, where did the mismatch start. That's a distinct, trainable skill from writing forward logic — and it's the one AI tools are worst at replacing, because it requires holding the specific failing state in your head.
⚠️ Common mistake: treating the line number in the traceback as "the bug," instead of as the symptom's last known location.
The error line is where Python noticed the problem, not always where it started.
Trace data backwards from the crash to the first place the wrong value appeared.
Mental model
Hospital data feeds (HL7, FHIR, PACS) fail constantly in small, expected ways — timeouts, missing fields, malformed messages.
The Answer
Code that raises an unhandled exception on the first malformed message stops processing everything behind it — one bad record takes down a whole batch. Professional integration code expects failure as a normal event: it catches the specific exception, logs enough detail to diagnose later, and falls back to a sensible default or retries — without crashing the rest of the pipeline.
Failure is the default expectation when integrating external systems, not an edge case.
"Fails gracefully" beats "never fails" because the second one is impossible.
Mental model
A comprehension says "build a new list by transforming/filtering an existing one." A loop says "do several different things while iterating."
The Answer
Use a comprehension when the entire body is one expression producing one new collection, e.g. [x.strip() for x in lines if x]. Switch to a plain loop the moment you need multiple statements, side effects (logging, mutating something external), early termination logic, or more than one level of conditional branching — readability drops fast past that point.
One expression, one new collection → comprehension.
Side effects, multiple statements, or branching → loop.
Mental model
Prints answer "what happened over time, across many runs." A debugger answers "what exactly is the state right now, at this one line."
The Answer
Print statements are fast to add and good for tracing a value across loop iterations or multiple function calls without stopping execution. A breakpoint is better when you need to inspect many variables at once, step line-by-line through unfamiliar logic, or explore the live call stack — things that would take a wall of print statements to approximate. Knowing which tool fits the bug in front of you is itself the skill, not a preference for one over the other.
Use prints for "trace this value across many iterations."
Use a breakpoint for "inspect everything at this exact moment."

Mentales Modell
KI generiert Code; sie liest keinen Traceback um 2 Uhr nachts, verteidigt keine Designentscheidung im Review und schreibt nicht live vor einem Interviewer.
Die Antwort
Code zu generieren und ihn produzieren, erklären und reparieren zu können, sind unterschiedliche Fähigkeiten. Die Momente, die dich wirklich testen — Live-Coding- Interviews, Fragen im Code-Review, Produktionsvorfälle mit Krankenhausdaten, Debugging zu ungünstigen Zeiten — sind genau die Momente, in denen KI-Unterstützung langsam, nicht verfügbar oder schlicht unpassend ist. Dieses Modul baut die zugrunde liegende Fähigkeit auf, die der KI-Einsatz bisher verdeckt hat.
KI-unterstützte Ausgabe ist eine Obergrenze für das, was du produzieren kannst — keine Untergrenze für das, was du verstehst.
Eigenständige Flüssigkeit ist das, was in Interviews und bei Vorfällen bewertet wird.
Mentales Modell
Erkennen ist passiv — du unterscheidest guten von schlechtem Code. Produzieren ist aktiv — du baust ihn vor einem leeren Cursor.
Die Antwort
Korrekten Code zu erkennen erfordert nur Mustervergleich mit bereits gesehenen Beispielen. Ihn aus dem Kopf zu schreiben erfordert, das Problem, die Datentypen, die Randfälle und die Syntax gleichzeitig im Kopf zu halten — es gibt keine Autovervollständigung, die die nächste Zeile vorschlägt. Deshalb beginnt dieses Modul mit kleinen, unspektakulären Funktionen: Sie machen Lücken zwischen Produktions- und Erkennungsfähigkeit sichtbar.
Kleine Funktionen (3–10 Zeilen) sind bewusst gewählt, um Lücken sichtbar zu machen.
Wenn du merge_dicts(*dicts) im ersten Versuch korrekt tippst, ist das Automatisierung — kein Zufall.
Mentales Modell
Code schreiben ist Vorwärtsbewegung. Einen Fehler lesen ist Reverse Engineering — von einem Symptom rückwärts zur Ursache.
Die Antwort
Ein Traceback wie TypeError on line 45 zeigt, wo das Programm aufgegeben hat — nicht unbedingt, wo der Bug entstanden ist. Code unter Fehlern zu lesen bedeutet, Daten rückwärts zu verfolgen: Was wurde übergeben, welcher Typ wurde erwartet, wo begann die Abweichung. Das ist eine eigenständige, trainierbare Fähigkeit, getrennt vom Schreiben vorwärtsgerichteter Logik — und genau die, die KI- Tools am schlechtesten ersetzen können, weil sie verlangt, den konkreten fehlerhaften Zustand im Kopf zu halten.
⚠️ Häufiger Fehler: die Zeilennummer im Traceback als „den Bug" zu behandeln, statt als die letzte bekannte Stelle des Symptoms.
Die Fehlerzeile ist, wo Python das Problem bemerkt hat — nicht immer, wo es begann.
Verfolge die Daten rückwärts vom Absturz zur ersten Stelle mit dem falschen Wert.
Mentales Modell
Krankenhaus-Datenfeeds (HL7, FHIR, PACS) scheitern ständig in kleinen, erwartbaren Weisen — Timeouts, fehlende Felder, fehlerhafte Nachrichten.
Die Antwort
Code, der bei der ersten fehlerhaften Nachricht eine unbehandelte Exception wirft, stoppt die Verarbeitung alles Nachfolgenden — ein schlechter Datensatz reißt einen ganzen Batch mit sich. Professioneller Integrationscode erwartet Fehler als normales Ereignis: Er fängt die konkrete Exception ab, loggt genug Details für die spätere Diagnose und fällt auf einen sinnvollen Standard zurück oder wiederholt den Versuch — ohne den Rest der Pipeline zu zerstören.
Fehler sind beim Anbinden externer Systeme die normale Erwartung, kein Sonderfall.
„Scheitert sauber" ist besser als „scheitert nie", weil Letzteres unmöglich ist.
Mentales Modell
Eine Comprehension sagt: „baue eine neue Liste durch Transformation/Filterung einer bestehenden". Eine Schleife sagt: „tue mehrere verschiedene Dinge während der Iteration".
Die Antwort
Nutze eine Comprehension, wenn der gesamte Körper ein Ausdruck ist, der eine neue Sammlung erzeugt, z. B. [x.strip() for x in lines if x]. Wechsle zur einfachen Schleife, sobald du mehrere Anweisungen, Seiteneffekte (Logging, Verändern von etwas Externem), vorzeitige Abbruchlogik oder mehr als eine Ebene von Bedingungen brauchst — die Lesbarkeit sinkt danach schnell.
Ein Ausdruck, eine neue Sammlung → Comprehension.
Seiteneffekte, mehrere Anweisungen oder Verzweigungen → Schleife.
Mentales Modell
Prints beantworten „was über die Zeit, über viele Durchläufe hinweg passiert ist". Ein Debugger beantwortet „was genau ist der Zustand jetzt, an dieser einen Zeile".
Die Antwort
Print-Statements sind schnell hinzuzufügen und gut, um einen Wert über Schleifendurchläufe oder mehrere Funktionsaufrufe hinweg zu verfolgen, ohne die Ausführung zu stoppen. Ein Breakpoint ist besser, wenn du viele Variablen gleichzeitig inspizieren, unbekannte Logik Zeile für Zeile durchgehen oder den aktiven Call-Stack erkunden musst — Dinge, die mit Print-Statements nur mühsam nachzubilden wären. Zu wissen, welches Werkzeug zum vorliegenden Bug passt, ist selbst die Fähigkeit — nicht eine Präferenz für das eine oder andere.
Nutze Prints, um „diesen Wert über viele Durchläufe verfolgen".
Nutze einen Breakpoint, um „alles in genau diesem Moment inspizieren".

الإجابة
توليد الكود والقدرة على إنتاجه وشرحه وإصلاحه مهارتان مختلفتان. اللحظات التي تختبرك فعليًا — مقابلات الترميز المباشر، أسئلة مراجعة الكود، حوادث الإنتاج المتعلقة ببيانات المستشفى، التصحيح في أوقات غير مناسبة — هي تحديدًا اللحظات التي تكون فيها مساعدة الذكاء الاصطناعي بطيئة أو غير متاحة أو غير مناسبة للاعتماد عليها. يبني هذا الموديول المهارة الأساسية التي كان استخدام الذكاء الاصطناعي يخفيها.
المخرجات المعتمدة على الذكاء الاصطناعي سقف لما تستطيع إنتاجه، لا قاعدة لما تفهمه.
الإتقان المستقل هو ما يُقيَّم في المقابلات والاستجابة للحوادث.

النموذج الذهني
كتابة الكود حركة إلى الأمام. قراءة الخطأ هندسة عكسية — تبدأ من عرض وتعمل بالعكس للوصول إلى السبب.
الإجابة
تتبّع مثل TypeError on line 45 يخبرك أين توقف البرنامج، وليس بالضرورة أين نشأ الخطأ. قراءة الكود تحت الفشل تعني تتبّع البيانات بالعكس: ما الذي تم تمريره، وأي نوع كان متوقعًا، وأين بدأ التعارض. هذه مهارة مستقلة وقابلة للتدريب، مختلفة عن كتابة منطق إلى الأمام — وهي الأضعف بالنسبة لأدوات الذكاء الاصطناعي لأنها تتطلب الاحتفاظ بالحالة الفاشلة المحددة في ذهنك.
⚠️ خطأ شائع: التعامل مع رقم السطر في التتبّع كـ "الخطأ" نفسه، بدلًا من آخر موضع معروف للعرض.
سطر الخطأ هو حيث لاحظ بايثون المشكلة، ليس دائمًا حيث بدأت.

النموذج الذهني
تغذيات بيانات المستشفى (HL7، FHIR، PACS) تفشل باستمرار بطرق صغيرة ومتوقعة — انتهاء المهلة، حقول مفقودة، رسائل مشوّهة.
الإجابة
الكود الذي يرفع استثناءً غير معالج عند أول رسالة مشوّهة يوقف معالجة كل ما بعدها — سجل واحد سيء يُسقط دفعة كاملة. كود التكامل المحترف يتوقع الفشل كحدث طبيعي: يعترض الاستثناء المحدد، ويسجّل تفاصيل كافية للتشخيص لاحقًا، ويرجع إلى قيمة افتراضية منطقية أو يعيد المحاولة — دون إيقاف بقية خط المعالجة.
الفشل هو التوقّع الافتراضي عند ربط أنظمة خارجية، لا حالة استثنائية.
"يفشل بسلاسة" أفضل من "لا يفشل أبدًا" لأن الثاني مستحيل.
النموذج الذهني
تعبير القائمة يقول "أنشئ قائمة جديدة بتحويل/تصفية قائمة موجودة". الحلقة تقول "نفّذ عدة أشياء مختلفة خلال التكرار".
الإجابة
استخدم تعبير القائمة عندما يكون كل المحتوى تعبيرًا واحدًا ينتج مجموعة جديدة واحدة، مثل [x.strip() for x in lines if x]. انتقل إلى حلقة عادية بمجرد احتياجك لعدة جمل، أو آثار جانبية (تسجيل، تعديل شيء خارجي)، أو منطق إنهاء مبكر، أو أكثر من مستوى واحد من الشروط — تنخفض القابلية للقراءة بسرعة بعد ذلك.
تعبير واحد، مجموعة جديدة واحدة ← تعبير قائمة.
آثار جانبية، جمل متعددة، أو تفرّع ← حلقة.
النموذج الذهني

الإجابة
جمل الطباعة سريعة الإضافة وجيدة لتتبّع قيمة عبر تكرارات حلقة أو استدعاءات دوال متعددة دون إيقاف التنفيذ. نقطة التوقف أفضل عندما تحتاج لفحص متغيرات عديدة دفعة واحدة، أو المرور سطرًا بسطر خلال منطق غير مألوف، أو استكشاف مكدّس الاستدعاءات الحي — أشياء يصعب تقريبها بجدار من جمل الطباعة. معرفة الأداة المناسبة للخطأ الذي أمامك هي نفسها المهارة، لا تفضيل أحدهما على الآخر.
استخدم الطباعة لتتبّع قيمة عبر تكرارات عديدة.
استخدم نقطة توقف لفحص كل شيء في هذه اللحظة بالضبط.
