The Automaticity Gap
Why recognition is not the same as production — and why that matters

You can read it. Can you write it?

You've seen hundreds of Python functions. You can look at a function and immediately know whether it's correct. That skill — recognition — is real, and it's built from all the Flask, healthcare-backend, and past production code you've worked with.

But recognition and production are different cognitive skills. Recognition says "yes, that's right." Production starts with a blank file and a function signature and generates correct code from nothing. In a live technical review, on a whiteboard, or when debugging at 2am without an IDE, production is the skill that matters.

Recognition: "I see this is wrong because the return type is a tuple, not a dict."
Production: Opens PyCharm, blank file, types def parse_icd10_code(code: str) -> dict: from memory, writes the body correctly on the first attempt.
This topic targets the second skill — not by reading about it, but by doing it, repeatedly, until it's automatic.

Why these four functions specifically?

Each function is 3–10 lines, contains no I/O or external calls, and tests a specific logical pattern your real code already uses:

Small enough to finish in 5 minutes — you get immediate feedback on whether you actually knew it or just thought you did.
Logic, not I/O — no file reading, no API calls, no mocking required. Pure input-in, output-out functions.
Directly useful — string validation, code parsing, dict merging, and duplicate detection are patterns in your own production code.
They force edge-case thinking — what if the string is empty? What if the list has one element? What if all dicts are empty?

01
validate_email(email: str) → bool
Returns True if the string looks like a valid email. No regex library — use in, .split(), and len() only.
def validate_email(email: str) -> bool: """Return True if email contains exactly one '@' and a '.' after it.""" if "@" not in email: return False parts = email.split("@") if len(parts) != 2: return False local, domain = parts return len(local) > 0 and "." in domain and len(domain) > 2
Check len(parts) != 2 — not just len(parts) < 2 — because a@b@c should also fail.
This does not validate that the domain exists or that the TLD is real. That's fine — say so in your docstring.
02
parse_icd10_code(code: str) → dict
Splits a code like "I10" or "E11.9" into its letter chapter and numeric block. Returns a dict, not a tuple — dict keys are self-documenting.
def parse_icd10_code(code: str) -> dict: """Split an ICD-10 code into its chapter letter and numeric block. Args: code: An ICD-10 code string, e.g. 'I10' or 'E11.9'. Returns: dict with keys 'chapter' (str) and 'block' (str). Raises: ValueError: if code is empty or has no leading letter. """ if not code or not code[0].isalpha(): raise ValueError(f"Invalid ICD-10 code: {code!r}") return { "chapter": code[0].upper(), "block": code[1:] }
code[0].upper() normalises lowercase input like "i10" without crashing.
Raising ValueError — not returning None — makes the caller handle bad input explicitly rather than silently propagating None.
03
merge_dicts(*dicts) → dict
Combines any number of dicts into one. Later dicts override earlier ones on key conflicts. The *dicts signature is the key thing to type from memory.
def merge_dicts(*dicts: dict) -> dict: """Merge multiple dicts; later dicts override earlier ones on key conflicts. Args: *dicts: Any number of dict arguments. Returns: A new dict containing all keys. No input dict is mutated. """ result = {} for d in dicts: result.update(d) return result
Returns a new dict — none of the inputs are mutated. This is why result = {} then result.update(d) rather than modifying dicts[0] directly.
In Python 3.9+ you could write result = {} | d1 | d2, but the loop is more readable and handles an arbitrary number of inputs.
04
find_first_duplicate(items: list) → any
Returns the first element that appears more than once. Returns None if there are no duplicates. The O(n) approach uses a set to track what has been seen.
def find_first_duplicate(items: list): """Return the first element that appears more than once, or None. Args: items: A list of hashable elements. Returns: The first duplicate element encountered, or None if none exist. """ seen = set() for item in items: if item in seen: return item seen.add(item) return None
if item in seen is O(1) for a set, vs O(n) for a list — important when items list is large (e.g. CGM readings over a week).
"First duplicate" means first element that we've already seen, not the first element that has a copy somewhere later.

Practice Questions
Close the functions above before answering — the point is to produce, not read
Mental model
Think about the difference between a surgeon who can read an anatomy textbook and one who has performed 200 appendectomies. The textbook reader has recognition. The surgeon has automaticity.
The Answer
AI generates code faster, but that speed belongs to AI, not to you. Manual practice builds the skill of producing correct Python independently — which matters in three real situations: a whiteboard technical interview, a live review where you're expected to type, and a 2am production incident where you need to patch something immediately without waiting for an AI response. The goal isn't to compete with AI — it's to not be helpless when AI isn't available.
AI generates; you explain, debug, and own. Manual practice makes the ownership real.
Automaticity also makes you a better AI user — you spot wrong AI output immediately because you know what right looks like.
Mental model
A function's docstring is a contract. Be precise about what you promise — because whatever you promise, callers will rely on.
The Answer
Must check: exactly one @ sign; a non-empty local part before @; a domain after @ that contains a dot and has length greater than 2.

Must NOT promise: that the domain exists, that the mailbox is reachable, or that the TLD is valid. Those require a DNS lookup or SMTP connection — far beyond a simple string check. Say this in the docstring: "Returns True if the string is structurally email-like. Does not verify deliverability."
⚠️ Common mistake: checking len(parts) < 2 instead of != 2. The string a@b@c produces 3 parts, passes the < 2 check, and gets incorrectly validated as True.
Precise scope in a docstring prevents callers from depending on behaviour you didn't implement.
A small, honest function beats a large function that silently over-promises.
Mental model
None is silent. ValueError is loud. In a pipeline that processes 10,000 ICD-10 codes, you want loud failures on bad data, not silent propagation of None through 20 downstream functions.
The Answer
Returning None on bad input makes the bug invisible: the caller gets None, accesses result["chapter"], and gets TypeError: 'NoneType' object is not subscriptable — 5 function calls later. Raising ValueError immediately names the problem at its source. The caller can then catch and handle it explicitly (except ValueError → logger.error → skip this record) rather than hunting a confusing downstream crash.
Raise where you know what went wrong. The caller can decide how to handle it.
Returning None is appropriate when "no result" is a valid, expected outcome — not when the input is broken.
Mental model
update() mutates in place. merge_dicts() creates something new. In medical code where a patient dict might be shared across threads or references, mutation-by-surprise is a real bug source.
The Answer
dict.update() mutates the dict it's called on — the original is changed. merge_dicts() starts with a fresh empty dict (result = {}) and copies into it, so no input is modified. This matters because: (1) if the caller still needs the original dict unchanged, mutation is a bug; (2) *dicts handles any number of arguments, not just two; (3) the function's intent is explicit in its name and signature rather than buried in a method call.
Functions that don't mutate their inputs are easier to test and reason about.
In Python 3.9+ d1 | d2 does the same thing for exactly two dicts — but the loop handles N dicts generically.
Mental model
Most beginners conflate "find an element that has a copy" with "find an element that appears more than once after sorting." They produce a different result.
The Answer
Two common mistakes: (1) sorting first and comparing adjacent elements — this finds duplicates but not the first one in the original order; (2) using items.count(item) > 1 inside the loop — this is O(n²) because count() scans the whole list on every iteration. The correct approach is a seen = set() and checking if item in seen before adding it — O(n) total, and preserves original order.
"First duplicate" means first element we encounter for the second time — original list order, not sorted order.
Set membership test is O(1). List membership test (item in list) is O(n). These compose differently inside a loop.
Mental model
A function is testable when you can call it with known inputs and check its output without setting up a database, server, or external service first.
The Answer
A testable function has three properties: it takes inputs as parameters (not from global state), it returns its result (not by mutating a shared variable), and it has no hidden dependencies (no database calls, no datetime.now() without injection, no file reads). All four functions here are testable because they are pure input-in / output-out: assert validate_email("a@b.com") == True works with zero setup. This is the shape your own utility functions should eventually have.
If you need to spin up MongoDB to test a function, the function is doing too much.
Testable ≠ tested. These four are testable; adding assert statements makes them tested.

Die Automatisierungslücke
Warum Wiedererkennen nicht dasselbe ist wie Produzieren — und warum das wichtig ist

Du kannst es lesen. Kannst du es schreiben?

Du hast Hunderte von Python-Funktionen gesehen. Du kannst eine Funktion ansehen und sofort erkennen, ob sie korrekt ist. Diese Fähigkeit — Wiedererkennen — ist real und wurde durch den gesamten Flask-, Gesundheits-Backend- und früheren Produktionscode aufgebaut, mit dem du gearbeitet hast.

Aber Wiedererkennen und Produzieren sind unterschiedliche kognitive Fähigkeiten. Wiedererkennen sagt „ja, das ist richtig." Produzieren beginnt mit einer leeren Datei und einer Funktionssignatur und generiert korrekten Code aus dem Nichts. In einem Live-Technical-Review, an einem Whiteboard oder um 2 Uhr nachts ohne IDE ist Produzieren die Fähigkeit, die zählt.

Wiedererkennen: „Ich sehe, das ist falsch, weil der Rückgabetyp ein Tupel ist, kein Dict."
Produzieren: PyCharm öffnen, leere Datei, def parse_icd10_code(code: str) -> dict: aus dem Gedächtnis tippen, den Körper beim ersten Versuch korrekt schreiben.
Dieses Thema zielt auf die zweite Fähigkeit ab — nicht durch Lesen, sondern durch Tun, wiederholt, bis es automatisch ist.

01
validate_email(email: str) → bool
Gibt True zurück wenn der String wie eine gültige E-Mail aussieht. Keine Regex-Bibliothek — nur in, .split() und len().
def validate_email(email: str) -> bool: """Gibt True zurück wenn email genau ein '@' und danach einen '.' enthält.""" if "@" not in email: return False parts = email.split("@") if len(parts) != 2: return False local, domain = parts return len(local) > 0 and "." in domain and len(domain) > 2
Prüfe len(parts) != 2 — nicht nur len(parts) < 2 — denn a@b@c soll ebenfalls fehlschlagen.
Das validiert nicht ob die Domain existiert oder die TLD real ist. Das ist korrekt so — sag es im Docstring.
02
parse_icd10_code(code: str) → dict
Zerlegt einen Code wie "I10" oder "E11.9" in seinen Buchstaben-Kapitel und numerischen Block. Gibt ein Dict zurück, kein Tupel — Dict-Schlüssel sind selbstdokumentierend.
def parse_icd10_code(code: str) -> dict: """Zerlegt einen ICD-10-Code in seinen Kapitel-Buchstaben und numerischen Block. Args: code: Ein ICD-10-Code-String, z.B. 'I10' oder 'E11.9'. Returns: dict mit Schlüsseln 'chapter' (str) und 'block' (str). Raises: ValueError: wenn code leer ist oder keinen führenden Buchstaben hat. """ if not code or not code[0].isalpha(): raise ValueError(f"Ungültiger ICD-10-Code: {code!r}") return { "chapter": code[0].upper(), "block": code[1:] }
code[0].upper() normalisiert Kleinbuchstaben-Eingaben wie "i10" ohne Absturz.
ValueError zu raisen — nicht None zurückzugeben — zwingt den Aufrufer, fehlerhafte Eingaben explizit zu behandeln.
03
merge_dicts(*dicts) → dict
Kombiniert beliebig viele Dicts zu einem. Spätere Dicts überschreiben frühere bei Schlüsselkonflikten. Die *dicts Signatur ist die wichtigste Sache, die man aus dem Gedächtnis tippen muss.
def merge_dicts(*dicts: dict) -> dict: """Zusammenführen mehrerer Dicts; spätere überschreiben frühere bei Konflikten. Args: *dicts: Beliebige Anzahl von Dict-Argumenten. Returns: Ein neues Dict mit allen Schlüsseln. Kein Eingabe-Dict wird verändert. """ result = {} for d in dicts: result.update(d) return result
Gibt ein neues Dict zurück — keine der Eingaben wird verändert. Deshalb result = {} dann result.update(d) statt das erste Argument direkt zu modifizieren.
In Python 3.9+ könnte man result = {} | d1 | d2 schreiben, aber die Schleife ist lesbarer und verarbeitet N Dicts generisch.
04
find_first_duplicate(items: list) → any
Gibt das erste Element zurück, das mehr als einmal vorkommt. Gibt None zurück, wenn keine Duplikate vorhanden sind. Der O(n)-Ansatz verwendet ein set zur Verfolgung bereits gesehener Elemente.
def find_first_duplicate(items: list): """Gibt das erste Element zurück, das mehr als einmal vorkommt, oder None. Args: items: Eine Liste von hashbaren Elementen. Returns: Das erste angetroffene doppelte Element, oder None wenn keins existiert. """ seen = set() for item in items: if item in seen: return item seen.add(item) return None
if item in seen ist O(1) für ein Set, vs O(n) für eine Liste — wichtig wenn items-Liste groß ist.
"Erstes Duplikat" bedeutet erstes Element, das wir ein zweites Mal antreffen — originale Listenreihenfolge, nicht sortierte Reihenfolge.

Mentales Modell
Der Docstring einer Funktion ist ein Vertrag. Sei präzise darüber, was du versprichst — denn was auch immer du versprichst, Aufrufer werden sich darauf verlassen.
Die Antwort
Muss prüfen: genau ein @-Zeichen; einen nicht-leeren lokalen Teil vor @; eine Domain nach @, die einen Punkt enthält und länger als 2 Zeichen ist.

Darf nicht versprechen: dass die Domain existiert, dass das Postfach erreichbar ist, oder dass die TLD gültig ist. Das erfordert einen DNS-Lookup oder SMTP-Verbindung — weit über eine einfache String-Prüfung hinaus. Sage dies im Docstring: "Gibt True zurück wenn der String strukturell E-Mail-ähnlich ist. Überprüft nicht die Zustellbarkeit."
⚠️ Häufiger Fehler: len(parts) < 2 statt != 2 prüfen. Der String a@b@c ergibt 3 Teile, besteht die < 2 Prüfung und wird fälschlicherweise als True validiert.
Präziser Umfang im Docstring verhindert, dass Aufrufer sich auf nicht implementiertes Verhalten verlassen.
Eine kleine, ehrliche Funktion schlägt eine große Funktion, die still über-verspricht.
Mentales Modell
None ist still. ValueError ist laut. In einer Pipeline, die 10.000 ICD-10-Codes verarbeitet, willst du laute Fehler bei schlechten Daten, keine stille None-Weitergabe durch 20 nachgelagerte Funktionen.
Die Antwort
None bei fehlerhafter Eingabe zurückzugeben macht den Bug unsichtbar: Der Aufrufer erhält None, greift auf result["chapter"] zu, und erhält TypeError: 'NoneType' object is not subscriptable — 5 Funktionsaufrufe später. ValueError zu raisen benennt das Problem sofort an seiner Quelle. Der Aufrufer kann es dann explizit abfangen und behandeln (except ValueError → logger.error → Datensatz überspringen) statt einen verwirrenden nachgelagerten Absturz zu debuggen.
Raise wo du weißt was falsch gelaufen ist. Der Aufrufer entscheidet wie damit umzugehen ist.
None zurückgeben ist angemessen wenn "kein Ergebnis" ein gültiges, erwartetes Ergebnis ist — nicht wenn die Eingabe fehlerhaft ist.
Mentales Modell
update() mutiert an Ort und Stelle. merge_dicts() erstellt etwas Neues. In medizinischem Code wo ein Patienten-Dict über Threads oder Referenzen geteilt werden kann, ist unerwartete Mutation eine echte Bug-Quelle.
Die Antwort
dict.update() mutiert das Dict auf dem es aufgerufen wird — das Original wird geändert. merge_dicts() beginnt mit einem frischen leeren Dict (result = {}) und kopiert hinein, sodass keine Eingabe verändert wird. Das ist wichtig weil: (1) wenn der Aufrufer das originale Dict noch unverändert braucht, ist Mutation ein Bug; (2) *dicts verarbeitet beliebig viele Argumente, nicht nur zwei; (3) die Funktion macht ihre Absicht explizit in Name und Signatur statt in einem Methodenaufruf zu vergraben.
Funktionen die ihre Eingaben nicht mutieren sind einfacher zu testen und zu verstehen.
In Python 3.9+ macht d1 | d2 dasselbe für genau zwei Dicts — die Schleife verarbeitet N Dicts generisch.
Mentales Modell
Die meisten Anfänger verwechseln "ein Element finden, das eine Kopie hat" mit "ein Element finden, das nach dem Sortieren mehr als einmal vorkommt." Sie produzieren unterschiedliche Ergebnisse.
Die Antwort
Zwei häufige Fehler: (1) Zuerst sortieren und angrenzende Elemente vergleichen — das findet Duplikate, aber nicht das erste in der originalen Reihenfolge; (2) items.count(item) > 1 in der Schleife verwenden — das ist O(n²) weil count() bei jeder Iteration die ganze Liste scannt. Der korrekte Ansatz ist ein seen = set() und Prüfen von if item in seen vor dem Hinzufügen — O(n) gesamt und bewahrt die originale Reihenfolge.
"Erstes Duplikat" bedeutet erstes Element das wir zum zweiten Mal begegnen — originale Listenreihenfolge, nicht sortierte.
Set-Mitgliedschaftstest ist O(1). Listen-Mitgliedschaftstest (item in list) ist O(n). Diese kombinieren sich unterschiedlich innerhalb einer Schleife.
Mentales Modell
Eine Funktion ist testbar wenn man sie mit bekannten Eingaben aufrufen und ihre Ausgabe prüfen kann, ohne zuerst eine Datenbank, einen Server oder einen externen Dienst einzurichten.
Die Antwort
Eine testbare Funktion hat drei Eigenschaften: sie nimmt Eingaben als Parameter entgegen (nicht aus globalem Zustand), sie gibt ihr Ergebnis zurück (nicht durch Mutation einer gemeinsamen Variable), und sie hat keine versteckten Abhängigkeiten (keine Datenbankaufrufe, kein datetime.now() ohne Injektion, kein Dateilesen). Alle vier Funktionen hier sind testbar weil sie reines Eingabe-rein / Ausgabe-raus sind: assert validate_email("a@b.com") == True funktioniert ohne Setup. Das ist die Form, die deine eigenen Hilfsfunktionen letztendlich haben sollten.
Wenn du MongoDB hochfahren musst um eine Funktion zu testen, macht die Funktion zu viel.
Testbar ≠ getestet. Diese vier sind testbar; assert-Anweisungen hinzufügen macht sie getestet.

فجوة الأتمتة
لماذا التعرف على الكود ليس نفس إنتاجه — ولماذا هذا مهم

تستطيع قراءته. هل تستطيع كتابته؟

رأيت مئات من دوال بايثون. تستطيع النظر إلى دالة وتعرف فوراً إن كانت صحيحة. هذه المهارة — التعرف — حقيقية، وبُنيت من كل كود Flask وخلفية صحية وعمل سابق الذي عملت معه.

لكن التعرف والإنتاج مهارتان معرفيتان مختلفتان. التعرف يقول "نعم، هذا صحيح." الإنتاج يبدأ بملف فارغ وتوقيع دالة ويُنشئ كوداً صحيحاً من لا شيء. في مراجعة تقنية مباشرة، على السبورة، أو في الساعة 2 صباحاً بدون IDE، الإنتاج هو المهارة المهمة.

التعرف: "أرى أن هذا خطأ لأن نوع الإرجاع tuple وليس dict."
الإنتاج: فتح PyCharm، ملف فارغ، كتابة def parse_icd10_code(code: str) -> dict: من الذاكرة، وكتابة الجسم بشكل صحيح من المحاولة الأولى.
هذا الموضوع يستهدف المهارة الثانية — ليس بالقراءة عنها، بل بالممارسة الفعلية، بشكل متكرر، حتى تصبح تلقائية.

validate_email(email: str) → bool
ترجع True إذا بدا النص كبريد إلكتروني صالح. لا مكتبة regex — استخدم in و.split() وlen() فقط.
def validate_email(email: str) -> bool: """Return True if email contains exactly one '@' and a '.' after it.""" if "@" not in email: return False parts = email.split("@") if len(parts) != 2: return False local, domain = parts return len(local) > 0 and "." in domain and len(domain) > 2
تحقق من len(parts) != 2 — لا فقط len(parts) < 2 — لأن a@b@c يجب أن يفشل أيضاً.
هذا لا يتحقق من أن النطاق موجود أو أن TLD حقيقي. هذا صحيح — اذكر ذلك في الـ docstring.
02
parse_icd10_code(code: str) → dict
يقسّم كوداً مثل "I10" أو "E11.9" إلى حرف الفصل والكتلة الرقمية. يرجع dict لا tuple — مفاتيح dict تشرح نفسها بنفسها.
def parse_icd10_code(code: str) -> dict: """Split an ICD-10 code into its chapter letter and numeric block. Args: code: An ICD-10 code string, e.g. 'I10' or 'E11.9'. Returns: dict with keys 'chapter' (str) and 'block' (str). Raises: ValueError: if code is empty or has no leading letter. """ if not code or not code[0].isalpha(): raise ValueError(f"Invalid ICD-10 code: {code!r}") return { "chapter": code[0].upper(), "block": code[1:] }
code[0].upper() ينظّم المدخلات بحروف صغيرة مثل "i10" دون أخطاء.
رفع ValueError — لا إرجاع None — يجبر المستدعي على معالجة المدخلات الخاطئة بشكل صريح.
03
merge_dicts(*dicts) → dict
يجمع أي عدد من القواميس في قاموس واحد. القواميس اللاحقة تستبدل القواميس السابقة عند تعارض المفاتيح. توقيع *dicts هو الشيء الأهم للكتابة من الذاكرة.
def merge_dicts(*dicts: dict) -> dict: """Merge multiple dicts; later dicts override earlier ones on key conflicts. Args: *dicts: Any number of dict arguments. Returns: A new dict containing all keys. No input dict is mutated. """ result = {} for d in dicts: result.update(d) return result
يرجع dict جديداً — لا يُعدّل أي من المدخلات. لهذا السبب result = {} ثم result.update(d) بدلاً من تعديل الوسيطة الأولى مباشرة.
في Python 3.9+ يمكن كتابة result = {} | d1 | d2، لكن الحلقة أوضح وتتعامل مع N قواميس بشكل عام.
04
find_first_duplicate(items: list) → any
يرجع أول عنصر يظهر أكثر من مرة. يرجع None إن لم تكن هناك تكرارات. النهج O(n) يستخدم set لتتبع ما تم رؤيته.
def find_first_duplicate(items: list): """Return the first element that appears more than once, or None. Args: items: A list of hashable elements. Returns: The first duplicate element encountered, or None if none exist. """ seen = set() for item in items: if item in seen: return item seen.add(item) return None
if item in seen بتعقيد O(1) للـ set، مقابل O(n) للـ list — مهم عندما تكون قائمة العناصر كبيرة.
"أول تكرار" يعني أول عنصر نصادفه للمرة الثانية — ترتيب القائمة الأصلي لا الترتيب المرتب.

النموذج الذهني
docstring الدالة عقد. كن دقيقاً في ما تعِد به — لأن ما تعد به سيعتمد عليه المستدعون.
الإجابة
يجب التحقق من: وجود @ واحدة بالضبط؛ جزء محلي غير فارغ قبل @؛ نطاق بعد @ يحتوي نقطة وطوله أكبر من 2.

لا يجب الوعد بـ: أن النطاق موجود، أن صندوق البريد يمكن الوصول إليه، أو أن TLD صالح. هذا يتطلب DNS lookup أو اتصال SMTP — أبعد بكثير من فحص بسيط للنص. اذكر ذلك في docstring: "يرجع True إذا بدا النص هيكلياً كبريد إلكتروني. لا يتحقق من إمكانية التسليم."
⚠️ خطأ شائع: التحقق من len(parts) < 2 بدلاً من != 2. النص a@b@c ينتج 3 أجزاء، يجتاز فحص < 2، ويُتحقق منه خطأً على أنه True.
النطاق الدقيق في docstring يمنع المستدعين من الاعتماد على سلوك لم تُنفّذه.
دالة صغيرة صادقة أفضل من دالة كبيرة تعِد بصمت بأكثر مما تستطيع.
النموذج الذهني
None صامت. ValueError صاخب. في خط معالجة يعالج 10,000 كود ICD-10، تريد فشلاً صاخباً على البيانات السيئة، لا نشراً صامتاً لـ None عبر 20 دالة لاحقة.
الإجابة
إرجاع None عند مدخل خاطئ يجعل الخطأ غير مرئي: المستدعي يحصل على None، يصل إلى result["chapter"]، ويحصل على TypeError: 'NoneType' object is not subscriptable — بعد 5 استدعاءات دوال. رفع ValueError فوراً يُسمّي المشكلة في مصدرها. المستدعي يمكنه حينها اعتراضه والتعامل معه بشكل صريح (except ValueError → logger.error → تخطي هذا السجل) بدلاً من تتبع خطأ مُربك لاحقاً.
ارفع الاستثناء حيث تعرف ما حدث. المستدعي يقرر كيف يتعامل معه.
إرجاع None مناسب عندما "لا توجد نتيجة" هي نتيجة صالحة ومتوقعة — لا حين يكون المدخل معطوباً.
النموذج الذهني
update() يعدّل في مكانه. merge_dicts() ينشئ شيئاً جديداً. في الكود الطبي حيث قد يُشارَك dict المريض عبر خيوط أو مراجع، التعديل المفاجئ مصدر حقيقي للأخطاء.
الإجابة
dict.update() يعدّل الـ dict الذي يُستدعى عليه — الأصل يتغير. merge_dicts() يبدأ بـ dict فارغ جديد (result = {}) وينسخ إليه، لذا لا يُعدَّل أي مدخل. هذا مهم لأن: (1) إذا كان المستدعي لا يزال يحتاج الـ dict الأصلي دون تغيير، فالتعديل خطأ؛ (2) *dicts يتعامل مع أي عدد من الوسيطات لا اثنتين فقط؛ (3) القصد واضح في اسم الدالة وتوقيعها.
الدوال التي لا تعدّل مدخلاتها أسهل في الاختبار والفهم.
في Python 3.9+ d1 | d2 يفعل نفس الشيء لاثنين بالضبط — لكن الحلقة تتعامل مع N قواميس بشكل عام.
النموذج الذهني
معظم المبتدئين يخلطون بين "إيجاد عنصر له نسخة" و"إيجاد عنصر يظهر أكثر من مرة بعد الترتيب". هذان ينتجان نتائج مختلفة.
الإجابة
خطآن شائعان: (1) الترتيب أولاً ومقارنة العناصر المتجاورة — هذا يجد التكرارات لكن ليس الأول في الترتيب الأصلي؛ (2) استخدام items.count(item) > 1 داخل الحلقة — هذا O(n²) لأن count() يمسح القائمة كلها في كل تكرار. النهج الصحيح هو seen = set() والتحقق من if item in seen قبل إضافته — O(n) إجمالاً ويحافظ على الترتيب الأصلي.
"أول تكرار" يعني أول عنصر نصادفه للمرة الثانية — ترتيب القائمة الأصلي، لا الترتيب المرتب.
اختبار عضوية الـ set هو O(1). اختبار عضوية الـ list هو O(n). هذان يتركبان بشكل مختلف داخل حلقة.
النموذج الذهني
دالة قابلة للاختبار عندما يمكنك استدعاؤها بمدخلات معروفة والتحقق من مخرجاتها دون إعداد قاعدة بيانات أو خادم أو خدمة خارجية أولاً.
الإجابة
الدالة القابلة للاختبار لها ثلاث خصائص: تأخذ المدخلات كمعاملات (لا من حالة عالمية)، ترجع نتيجتها (لا بتعديل متغير مشترك)، وليس لها تبعيات خفية (لا استدعاءات قاعدة بيانات، لا datetime.now() بدون حقن، لا قراءة ملفات). الدوال الأربع هنا قابلة للاختبار لأنها نقية مدخل-داخل / مخرج-خارج: assert validate_email("a@b.com") == True تعمل بدون أي إعداد. هذا الشكل الذي يجب أن تكون عليه دوال الأدوات الخاصة بك في نهاية المطاف.
إذا احتجت لتشغيل MongoDB لاختبار دالة، فالدالة تفعل أكثر مما يجب.
قابلة للاختبار ≠ مُختبَرة. هذه الأربع قابلة للاختبار؛ إضافة جمل assert يجعلها مُختبَرة.
