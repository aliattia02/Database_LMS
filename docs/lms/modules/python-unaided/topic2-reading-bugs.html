PYTHON UNAIDED CODING — Topic 2 of 5

Reading Code & Spotting Bugs

Three broken scenarios, three root causes. The goal is not just to find the fix — it is to trace the error back to where it was born and explain why it happened.

🔍
Reading is a Separate Skill
Writing and reading code are not the same cognitive act — this topic trains the second one

Why reading broken code is harder than it looks

When you write code, you know your intent. When you read someone else's broken code — or your own from three weeks ago — you have only the text and the error message. The gap between what the code says and what it was meant to do is where bugs live.

The three scenarios below cover the most common categories of Python bugs you will encounter in real projects: a type mismatch that crashes at serialisation, a mutation side-effect that corrupts shared state, and a logic error where the code runs cleanly but the output is wrong. Each one has been taken from patterns in DiaTwin / Morafek code.

Do not just read the fix. Copy each broken snippet into a file, run it, and read the error message or wrong output before looking at the explanation.
Write the explanation, not just the fix. "It crashed because X, which happened because Y" is more valuable than a one-line patch.
Trace to the source. The line that crashes is rarely the line where the mistake was made.

Three Scenarios
Find the root cause, not just the symptom

Work through them in order

Each scenario shows broken code and the error or wrong output it produces. Read the code, form a hypothesis, then expand the explanation.

01
Type Mismatch — datetime not JSON-serialisable
The function runs cleanly and returns a dict. The crash only appears later, when the caller tries to serialise it. This is a classic deferred type error.
from datetime import datetime def add_patient(db, patient_name: str, age: int) -> dict: return { "name": patient_name, "age": age, "created": datetime.now() # ← problem } # Caller — crashes here, not in add_patient(): import json record = add_patient(None, "Omar", 34) json.dumps(record) # TypeError: Object of type datetime is not JSON serializable
Root cause: datetime.now() returns a datetime object. json.dumps() does not know how to convert it.
The mistake is in add_patient(), not in the caller. The function returns a dict it knows will be serialised — it is its responsibility to return serialisable values.
Fix
"created": datetime.now().isoformat() # str, always JSON-safe
Alternative: pass a default=str argument to json.dumps() — but this silently stringifies everything, which can hide other type errors.
In a FHIR pipeline, this exact pattern breaks when you store a datetime in a patient resource dict and then POST it as JSON. Fix it at the source.
02
Mutation Side-Effect — shared mutable default
Both patients end up with the same readings list. This is one of the most surprising bugs in Python and it comes from how default mutable arguments work.
def make_patient(name: str, readings: list = []) -> dict: return {"name": name, "readings": readings} patient_a = make_patient("Amira") patient_b = make_patient("Bashir") patient_a["readings"].append(120) patient_b["readings"].append(105) print(patient_a["readings"]) # Expected: [120] print(patient_b["readings"]) # Expected: [105] # Actual: both print [120, 105] !
Root cause: Default argument values are evaluated once at function definition time. The same list object is reused on every call where readings is not passed.
Both patients share the same list object — appending to one appends to both.
Fix
def make_patient(name: str, readings: list = None) -> dict: if readings is None: readings = [] # new list created on each call return {"name": name, "readings": readings}
The rule: never use a mutable value (list, dict, set) as a default argument. Use None and create the object inside the function body.
This exact pattern appeared in the original Morafek session initialiser — session state was being shared across different users' requests.
03
Logic Error — off-by-one in range(), silent wrong answer
No exception is raised. The function returns a number. But the number is wrong. This is the hardest category to catch: silent wrong output.
def calculate_iob(doses: list, minutes_ago: list) -> float: """Insulin on board: sum of each dose decayed over time (120 min half-life).""" total = 0.0 for i in range(len(doses) - 1): # ← off-by-one remaining = doses[i] * (1 - minutes_ago[i] / 120) total += remaining return total # Call: result = calculate_iob([5.0, 3.0], [30, 90]) # Expected: 5*(1-30/120) + 3*(1-90/120) = 3.75 + 0.75 = 4.5 # Actual: 3.75 (last dose silently skipped)
Root cause: range(len(doses) - 1) stops one index early. With 2 doses, it only processes index 0.
Why it is dangerous: the return value looks plausible. In a real insulin dosing system, underestimating IOB leads to a double-dose recommendation.
Fix
for i in range(len(doses)): # correct: process every index # Or more Pythonically: for dose, t in zip(doses, minutes_ago): total += dose * (1 - t / 120)
Prefer zip() over index-based iteration when you need parallel lists — it eliminates off-by-one errors entirely.
When a function returns a numeric result, always test it manually with known inputs before trusting it in a pipeline.

Practice Questions
Answer these before looking — the explanation is the skill, not the answer
Mental model
Functions have a contract: their return value should be usable in the context the caller expects. If add_patient() is documented to return a serialisable record, then returning a datetime object is a violation of that contract — even if the function itself doesn't crash.
The Answer
The function's entire purpose is to produce a patient record. The caller uses it with json.dumps() — a completely reasonable thing to do with a data record. The function knows it is returning a dict; it knows datetime.now() is not JSON-serialisable. The responsibility to return a serialisable value sits with the function, not with every caller that uses it. Blaming the caller would mean every caller has to remember to convert datetime objects — that is fragile and duplicated. Fix it once, in the function, with .isoformat().
"Where it crashed" and "where the mistake was made" are often different lines. Always trace upstream.
This is why code review asks "what assumptions does this function make about its callers?" — those assumptions are hidden contracts.
Mental model
Python evaluates default argument values once, when the def statement is executed. It is a deliberate design choice — the list lives in the function object itself. It only surprises people who expect "default" to mean "create fresh on each call."
The Answer
It is not a bug — it is specified behaviour. When Python sees def f(x=[]):, it creates one list object and stores it as part of the function definition. Every call that does not pass a value for x receives a reference to that same object. If you mutate it (e.g. .append()), the mutation persists into the next call. The fix — None sentinel + if x is None: x = [] — creates a new list on each call because [] is now evaluated inside the function body, not in the def line.
Immutable defaults (int, str, tuple) are safe because you cannot mutate them in place. Only mutable defaults (list, dict, set) cause this problem.
You can inspect the current value of a default with f.__defaults__ — try it to see the list growing.
Mental model
Exceptions tell you something crashed. They do not tell you the result is wrong. For logic errors, the only reliable tool is a known-answer test — you must have at least one input where you can compute the expected output by hand and compare.
The Answer
Three systematic defences: (1) manual spot-check — pick one input, calculate the expected output by hand, and assert result == expected. For IOB: calculate_iob([4.0], [60]) == 4.0 * (1 - 60/120) == 2.0 — a single-element test immediately catches the off-by-one. (2) boundary tests — test with one dose, zero doses, and many doses. Off-by-one bugs always show up at the boundaries. (3) read the loop — range(len(x) - 1) should always raise a flag: why is it minus one?
If a function computes a medical value (insulin, glucose, dosage), always test it with at least one calculation you have done by hand.
range(len(x) - 1) is almost always a bug unless you are explicitly comparing adjacent pairs, where you would use range(len(x) - 1) intentionally.
Mental model
NoneType errors mean something returned None when you expected a dict or list. Line 45 is subscripting it — so the source is the function that returned None, not line 45.
The Answer
Do not look at line 45 first. Line 45 is a victim — it subscripted result["key"] and result was None. Walk upward in the stack trace: what function produced result? Did it hit an exception path that silently returns None? Did it hit a conditional that falls off the end without a return? The most common causes: a function returns None on a failed branch instead of raising an exception; or a database query returned no result and the code didn't check before using it.
Read the full stack trace from the bottom up: the bottom is where it crashed, the top is where it started. The root cause is usually closer to the top.
Search your codebase for return None in the function that produced the value — that is where to add a guard or a raise.
Mental model
Exceptions are loud failures — they tell you where and what. Silent wrong answers are quiet failures — they tell you nothing. In medical software, quiet failures are more dangerous.
The Answer
An exception crashes the program and gives you a stack trace pointing roughly at the problem. A silent wrong answer lets the program continue — potentially for hours — while every downstream calculation uses the corrupted value. In a CGM alert pipeline, an exception means alerts stop; a silent wrong answer means alerts fire at the wrong threshold, which is worse. This is why explicit assertions and test cases are not optional for medical code. The function must be verified against known-good output before it runs on real data.
"Fail fast and loud" is a principle: raise an exception on bad input so the error is caught early rather than silently corrupting downstream results.
Silent wrong answers in Scenario 3 are caught only by tests or manual calculation — not by running the code.
Mental model
Using an index variable to walk two lists in parallel is an anti-pattern in Python. You introduce an index that doesn't need to exist, and every off-by-one, off-by-two, or wrong-length error happens at the index level — not at the logic level.
The Answer
zip(a, b) pairs elements from both lists simultaneously and stops at the shorter one — you cannot accidentally use the wrong index, you cannot go out of bounds, and there is no - 1 to mistype. The iteration variable names ( dose, t) are also more expressive than i. The body dose * (1 - t / 120) reads like the formula in the medical spec. Index-based code reads like an implementation detail.
zip() is lazy (returns an iterator) — safe for large datasets like week-long CGM readings without loading everything into memory.
If you need the index too, use enumerate(zip(a, b)) — never rebuild it with range(len(...)).

Lesen ist eine eigenständige Fähigkeit
Code schreiben und Code lesen sind nicht dasselbe — dieses Thema trainiert die zweite Fähigkeit

Warum das Lesen von fehlerhaftem Code schwieriger ist, als es aussieht

Wenn du Code schreibst, kennst du deine Absicht. Wenn du den Code von jemand anderem liest — oder deinen eigenen von vor drei Wochen — hast du nur den Text und die Fehlermeldung. Die Lücke zwischen dem, was der Code sagt, und dem, was er tun sollte, ist der Ort, wo Bugs leben.

Die drei Szenarien unten decken die häufigsten Kategorien von Python-Bugs ab, die du in echten Projekten antreffen wirst: ein Typfehler, der bei der Serialisierung abstürzt, ein Mutations-Nebeneffekt, der gemeinsamen Zustand korrumpiert, und ein Logikfehler, bei dem der Code sauber läuft, die Ausgabe aber falsch ist.

Lies nicht einfach die Lösung. Kopiere jeden kaputten Ausschnitt in eine Datei, führe ihn aus und lies die Fehlermeldung oder falsche Ausgabe, bevor du dir die Erklärung anschaust.
Schreibe die Erklärung, nicht nur die Lösung. „Es ist abgestürzt, weil X, was wegen Y passiert ist" ist wertvoller als ein einzeiliger Patch.
Verfolge den Fehler zur Quelle. Die Zeile, die abstürzt, ist selten die Zeile, wo der Fehler gemacht wurde.

Drei Szenarien
Finde die Grundursache, nicht nur das Symptom

Arbeite sie der Reihe nach durch

Jedes Szenario zeigt fehlerhaften Code und den Fehler oder die falsche Ausgabe, die er erzeugt. Lies den Code, bilde eine Hypothese, dann lies die Erklärung.

01
Typfehler — datetime nicht JSON-serialisierbar
Die Funktion läuft sauber und gibt ein Dict zurück. Der Absturz tritt erst auf, wenn der Aufrufer versucht, es zu serialisieren. Dies ist ein klassischer verzögerter Typfehler.
from datetime import datetime def add_patient(db, patient_name: str, age: int) -> dict: return { "name": patient_name, "age": age, "created": datetime.now() # ← Problem } # Aufrufer — stürzt hier ab, nicht in add_patient(): import json record = add_patient(None, "Omar", 34) json.dumps(record) # TypeError: Object of type datetime is not JSON serializable
Grundursache: datetime.now() gibt ein datetime-Objekt zurück. json.dumps() weiß nicht, wie es konvertiert werden soll.
Der Fehler liegt in add_patient(), nicht beim Aufrufer.
Lösung
"created": datetime.now().isoformat() # str, immer JSON-sicher
02
Mutations-Nebeneffekt — gemeinsames veränderbares Standardargument
Beide Patienten enden mit derselben Readings-Liste. Dies ist einer der überraschendsten Bugs in Python.
def make_patient(name: str, readings: list = []) -> dict: return {"name": name, "readings": readings} patient_a = make_patient("Amira") patient_b = make_patient("Bashir") patient_a["readings"].append(120) patient_b["readings"].append(105) print(patient_a["readings"]) # Erwartet: [120] print(patient_b["readings"]) # Erwartet: [105] # Tatsächlich: beide geben [120, 105] aus!
Grundursache: Standardargumentwerte werden einmal beim Definieren der Funktion ausgewertet. Dasselbe Listenobjekt wird bei jedem Aufruf wiederverwendet.
Beide Patienten teilen dasselbe Listenobjekt.
Lösung
def make_patient(name: str, readings: list = None) -> dict: if readings is None: readings = [] # neue Liste bei jedem Aufruf return {"name": name, "readings": readings}
03
Logikfehler — Off-by-One in range(), stille falsche Ausgabe
Kein Exception wird geworfen. Die Funktion gibt eine Zahl zurück. Aber die Zahl ist falsch. Dies ist die schwierigste Kategorie.
def calculate_iob(doses: list, minutes_ago: list) -> float: total = 0.0 for i in range(len(doses) - 1): # ← Off-by-One remaining = doses[i] * (1 - minutes_ago[i] / 120) total += remaining return total result = calculate_iob([5.0, 3.0], [30, 90]) # Erwartet: 3.75 + 0.75 = 4.5 # Tatsächlich: 3.75 (letzte Dosis stillschweigend übersprungen)
Grundursache: range(len(doses) - 1) endet einen Index zu früh.
Gefährlich: Der Rückgabewert sieht plausibel aus.
Lösung
for dose, t in zip(doses, minutes_ago): total += dose * (1 - t / 120)
❓
Übungsfragen
Beantworte diese, bevor du nachschaust — die Erklärung ist die Fähigkeit, nicht die Antwort
Mentales Modell
Funktionen haben einen Vertrag: ihr Rückgabewert muss im erwarteten Kontext des Aufrufers verwendbar sein. Wenn add_patient() dokumentiert ist, einen serialisierbaren Datensatz zurückzugeben, ist das Zurückgeben eines datetime-Objekts ein Vertragsbruch.
Die Antwort
Der gesamte Zweck der Funktion ist es, einen Patientendatensatz zu erzeugen. Der Aufrufer verwendet ihn mit json.dumps() — eine völlig vernünftige Sache mit einem Datensatz zu tun. Die Funktion weiß, dass sie ein Dict zurückgibt; sie weiß, dass datetime.now() nicht JSON-serialisierbar ist. Die Verantwortung liegt bei der Funktion, nicht bei jedem Aufrufer. Behebe es einmal mit .isoformat().
„Wo es abgestürzt ist" und „wo der Fehler gemacht wurde" sind oft verschiedene Zeilen. Immer rückwärts verfolgen.
Mentales Modell
Python wertet Standardargumentwerte einmal aus, wenn die def-Anweisung ausgeführt wird. Dies ist ein bewusstes Design — die Liste lebt im Funktionsobjekt selbst.
Die Antwort
Es ist kein Bug — es ist spezifiziertes Verhalten. Python erstellt ein Listenobjekt und speichert es als Teil der Funktionsdefinition. Jeder Aufruf, der keinen Wert für readings übergibt, erhält eine Referenz auf dasselbe Objekt. Die Lösung mit None als Sentinel erstellt bei jedem Aufruf eine neue Liste, da [] jetzt im Funktionskörper ausgewertet wird.
Unveränderliche Standardwerte (int, str, tuple) sind sicher. Nur veränderliche Standardwerte (list, dict, set) verursachen dieses Problem.
Mentales Modell
Exceptions sagen dir, dass etwas abgestürzt ist. Sie sagen dir nicht, dass das Ergebnis falsch ist. Für Logikfehler ist das einzige zuverlässige Werkzeug ein Test mit bekannter Antwort.
Die Antwort
Drei systematische Abwehrmechanismen: (1) Manuelle Überprüfung — wähle einen Input, berechne das erwartete Ergebnis von Hand und vergleiche mit assert. (2) Grenzfalltests — teste mit einer Dosis, null Dosen und vielen Dosen. Off-by-One-Fehler tauchen immer an den Grenzen auf. (3) Lies die Schleife — range(len(x) - 1) sollte immer ein Warnsignal sein: warum minus eins?
zip() statt Index-basierter Iteration eliminiert Off-by-One-Fehler vollständig.
Mentales Modell
NoneType-Fehler bedeuten, dass etwas None zurückgegeben hat, wo du ein Dict oder eine Liste erwartet hast. Zeile 45 ist das Opfer — suche die Ursache in der Funktion, die None zurückgegeben hat.
Die Antwort
Schau nicht zuerst auf Zeile 45. Zeile 45 ist ein Opfer — es hat result["key"] subscriptiert und result war None. Gehe im Stack-Trace aufwärts: Welche Funktion hat result erzeugt? Hat sie einen Fehlerpfad getroffen, der stillschweigend None zurückgibt? Lese den gesamten Stack-Trace von unten nach oben: unten ist, wo es abgestürzt ist; oben ist, wo es begann.
Suche in deiner Codebase nach return None in der Funktion, die den Wert produziert hat — dort füge einen Guard oder ein Raise hinzu.
Mentales Modell
Exceptions sind laute Fehler. Stille falsche Antworten sind gefährlichere leise Fehler — in medizinischer Software besonders kritisch.
Die Antwort
Eine Exception stoppt das Programm und gibt dir einen Stack-Trace. Eine stille falsche Antwort lässt das Programm weiterlaufen — möglicherweise stundenlang — während jede nachgelagerte Berechnung den korrumpierten Wert verwendet. In einer CGM-Alarm-Pipeline bedeutet eine Exception, dass Alarme stoppen; eine stille falsche Antwort bedeutet, dass Alarme beim falschen Schwellenwert auslösen — was schlimmer ist.
„Fail fast and loud" ist ein Prinzip: Wirf bei schlechten Eingaben eine Exception, damit der Fehler früh erkannt wird.
Mentales Modell
Eine Indexvariable zum Durchlaufen zweier Listen zu verwenden ist ein Anti-Pattern in Python. Alle Off-by-One-Fehler passieren auf der Indexebene, nicht auf der Logikebene.
Die Antwort
zip(a, b) paart Elemente aus beiden Listen gleichzeitig — du kannst nicht versehentlich den falschen Index verwenden, nicht außerhalb der Grenzen gehen, und es gibt kein - 1, das du falsch tippen könntest. Die Iterationsvariablen (dose, t) sind auch aussagekräftiger als i.
zip() ist lazy (gibt einen Iterator zurück) — sicher für große Datensätze wie wochenlange CGM-Readings.

القراءة مهارة منفصلة
كتابة الكود وقراءته ليسا نفس الفعل المعرفي — هذا الموضوع يدرّب المهارة الثانية

لماذا قراءة الكود المعطوب أصعب مما تبدو عليه

حين تكتب الكود تعرف قصدك. حين تقرأ كود شخص آخر — أو كودك من ثلاثة أسابيع مضت — لديك فقط النص ورسالة الخطأ. الفجوة بين ما يقوله الكود وما كان يُفترض أن يفعله هي حيث تعيش الأخطاء.

السيناريوهات الثلاثة أدناه تغطي أكثر فئات الأخطاء شيوعاً في Python: عدم تطابق الأنواع الذي يتعطل عند التسلسل، والأثر الجانبي للتعديل الذي يفسد الحالة المشتركة، وخطأ منطقي حيث يعمل الكود بنظافة لكن المخرج خاطئ.

لا تقرأ الحل فقط. انسخ كل مقتطف معطوب في ملف، شغّله، واقرأ رسالة الخطأ أو المخرج الخاطئ قبل الاطلاع على الشرح.
اكتب الشرح، ليس فقط الحل. "تعطّل لأن X، الذي حدث بسبب Y" أكثر قيمة من تصحيح في سطر واحد.
تتبع إلى المصدر. السطر الذي يتعطل نادراً ما يكون السطر الذي ارتُكب فيه الخطأ.

اعمل عليها بالترتيب

كل سيناريو يُظهر كوداً معطوباً والخطأ أو المخرج الخاطئ الذي ينتجه. اقرأ الكود، كوّن فرضية، ثم افتح الشرح.

01
عدم تطابق الأنواع — datetime غير قابل للتسلسل إلى JSON
الدالة تعمل بنظافة وتعيد dict. يحدث التعطل لاحقاً حين يحاول المُستدعي تسلسله. هذا خطأ نوع مؤجل كلاسيكي.
from datetime import datetime def add_patient(db, patient_name: str, age: int) -> dict: return { "name": patient_name, "age": age, "created": datetime.now() # ← المشكلة } import json record = add_patient(None, "Omar", 34) json.dumps(record) # TypeError: Object of type datetime is not JSON serializable
السبب الجذري: datetime.now() يعيد كائن datetime. json.dumps() لا يعرف كيف يحوّله.
الخطأ في add_patient()، ليس في المُستدعي.
الحل
"created": datetime.now().isoformat() # str، آمن دائماً مع JSON
02
أثر جانبي للتعديل — وسيطة افتراضية قابلة للتغيير مشتركة
ينتهي الأمر بكلا المريضين بنفس قائمة القراءات. هذا أحد أكثر الأخطاء مفاجأةً في Python.
def make_patient(name: str, readings: list = []) -> dict: return {"name": name, "readings": readings} patient_a = make_patient("Amira") patient_b = make_patient("Bashir") patient_a["readings"].append(120) patient_b["readings"].append(105) print(patient_a["readings"]) # متوقع: [120] print(patient_b["readings"]) # متوقع: [105] # الفعلي: كلاهما يطبع [120, 105]!
السبب الجذري: قيم الوسيطات الافتراضية تُقيَّم مرة واحدة عند تعريف الدالة. نفس كائن القائمة يُعاد استخدامه في كل استدعاء.
كلا المريضين يشتركان في نفس كائن القائمة.
الحل
def make_patient(name: str, readings: list = None) -> dict: if readings is None: readings = [] # قائمة جديدة في كل استدعاء return {"name": name, "readings": readings}
03
خطأ منطقي — خطأ بمقدار واحد في range()، مخرج خاطئ صامت
لا استثناء يُرفع. الدالة تعيد رقماً. لكن الرقم خاطئ. هذه أصعب فئة يُكتشف فيها الخطأ.
def calculate_iob(doses: list, minutes_ago: list) -> float: total = 0.0 for i in range(len(doses) - 1): # ← خطأ بمقدار واحد remaining = doses[i] * (1 - minutes_ago[i] / 120) total += remaining return total result = calculate_iob([5.0, 3.0], [30, 90]) # متوقع: 3.75 + 0.75 = 4.5 # الفعلي: 3.75 (الجرعة الأخيرة تجاهلت صامتة)
السبب الجذري: range(len(doses) - 1) يتوقف فهرساً واحداً مبكراً.
خطير: قيمة الإرجاع تبدو معقولة.
الحل
for dose, t in zip(doses, minutes_ago): total += dose * (1 - t / 120)

أسئلة للتدريب
أجب عليها قبل الاطلاع — الشرح هو المهارة، ليس الإجابة
النموذج الذهني
الدوال لها عقد: قيمة الإرجاع يجب أن تكون قابلة للاستخدام في السياق الذي يتوقعه المُستدعي. إذا كانت add_patient() موثقة لإرجاع سجل قابل للتسلسل، فإرجاع كائن datetime هو انتهاك لهذا العقد.
الإجابة
الغرض الكامل من الدالة هو إنتاج سجل مريض. المُستدعي يستخدمه مع json.dumps() — شيء معقول تماماً للقيام به مع سجل بيانات. الدالة تعرف أنها تعيد dict؛ تعرف أن datetime.now() غير قابل للتسلسل إلى JSON. المسؤولية تقع على الدالة لإرجاع قيم قابلة للتسلسل، وليس على كل مُستدعٍ. أصلحها مرة واحدة في الدالة بـ .isoformat().
"أين تعطّل" و"أين ارتُكب الخطأ" غالباً ما يكونان سطرين مختلفين. دائماً تتبع للخلف.
النموذج الذهني
Python تقيّم قيم الوسيطات الافتراضية مرة واحدة عند تنفيذ جملة def. هذا اختيار تصميمي متعمد — القائمة تعيش في كائن الدالة نفسه.
الإجابة
ليس خطأً — إنه سلوك محدد. Python تنشئ كائن قائمة وتخزنه كجزء من تعريف الدالة. كل استدعاء لا يمرر قيمة لـ readings يحصل على مرجع لنفس الكائن. الحل بـ None كحارس ينشئ قائمة جديدة في كل استدعاء لأن [] الآن تُقيَّم داخل جسم الدالة.
القيم الافتراضية غير القابلة للتغيير (int، str، tuple) آمنة. فقط القيم الافتراضية القابلة للتغيير (list، dict، set) تسبب هذه المشكلة.
النموذج الذهني
الاستثناءات تخبرك أن شيئاً تعطّل. لا تخبرك أن النتيجة خاطئة. للأخطاء المنطقية، الأداة الوحيدة الموثوقة هي اختبار بإجابة معروفة.
الإجابة
ثلاثة دفاعات منهجية: (1) فحص يدوي — اختر مدخلاً، احسب المخرج المتوقع يدوياً وقارنه بـ assert. (2) اختبارات الحدود — اختبر بجرعة واحدة، صفر جرعات، وجرعات كثيرة. أخطاء off-by-one تظهر دائماً عند الحدود. (3) اقرأ الحلقة — range(len(x) - 1) يجب أن يثير تساؤلاً دائماً: لماذا ناقص واحد؟
استخدام zip() بدلاً من التكرار المستند إلى الفهرس يقضي على أخطاء off-by-one تماماً.
النموذج الذهني
أخطاء NoneType تعني أن شيئاً أعاد None حيث توقعت dict أو list. السطر 45 ضحية — ابحث عن السبب في الدالة التي أعادت None.
الإجابة
لا تنظر إلى السطر 45 أولاً. اصعد في stack trace: أي دالة أنتجت result؟ هل ضربت مساراً لاستثناء يعيد None صامتاً؟ هل ضربت شرطاً ينتهي بدون return؟ اقرأ stack trace الكامل من الأسفل للأعلى: الأسفل هو أين تعطّل، الأعلى هو أين بدأ.
ابحث في الكود عن return None في الدالة التي أنتجت القيمة — هناك أضف حارساً أو raise.
النموذج الذهني
الاستثناءات أخطاء صاخبة — تخبرك بأين وماذا. الإجابات الخاطئة الصامتة أخطاء هادئة — لا تخبرك بشيء. في البرمجيات الطبية، الأخطاء الهادئة أشد خطورة.
الإجابة
الاستثناء يوقف البرنامج ويعطيك stack trace. الإجابة الخاطئة الصامتة تدع البرنامج يستمر — ربما لساعات — بينما كل حساب لاحق يستخدم القيمة الفاسدة. في خط معالجة تنبيهات CGM، الاستثناء يعني توقف التنبيهات؛ الإجابة الخاطئة الصامتة تعني تشغيل التنبيهات عند حد خاطئ — وهذا أسوأ.
"افشل بسرعة وبصوت عالٍ" مبدأ: ارفع استثناء عند المدخلات السيئة لاكتشاف الخطأ مبكراً.
النموذج الذهني
استخدام متغير فهرس لاجتياز قائمتين بالتوازي نمط مضاد في Python. كل أخطاء off-by-one تحدث على مستوى الفهرس، لا مستوى المنطق.
الإجابة
zip(a, b) يقرن العناصر من كلا القائمتين في آن واحد — لا يمكنك استخدام فهرس خاطئ عن طريق الخطأ، لا يمكنك الخروج عن الحدود، ولا يوجد - 1 يمكن خطأ كتابته. أسماء متغيرات التكرار (dose, t) أكثر تعبيراً من i. والجسم dose * (1 - t / 120) يقرأ كالصيغة في المواصفات الطبية.
zip() كسول (يعيد مكرراً) — آمن لمجموعات البيانات الكبيرة مثل قراءات CGM لأسبوع.
