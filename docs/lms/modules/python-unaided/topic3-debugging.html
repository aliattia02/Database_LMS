PYTHON UNAIDED CODING — Topic 3 of 5

Debugging with Print Statements & Breakpoints

Code that runs but gives the wrong answer. No crash, no stack trace, no obvious clue. This topic gives you two systematic techniques — and the judgement to know which one to reach for.

🛠️
When the Code Runs but Lies
Exceptions tell you where something broke — debugging tells you why the answer is wrong

The two modes of debugging

When code crashes with an exception, you have a stack trace: a location, a type, and a message. That is useful. The harder problem is when code runs to completion but the output is wrong — a dosage is 0.5 units too high, a patient lookup returns the wrong record, a total is off by one entry.

Two tools exist for this. They are complementary, not competing. Use print debugging for quick, targeted checks when you already have a hypothesis about where the problem is. Use the PyCharm debugger when you need to inspect a running program's full state — multiple variables, data structures, call stack — at a specific moment.

Anti-pattern to avoid: Adding print statements everywhere, then removing them, then adding them back. Strategic placement — entry, exit, decision points — gives you the signal without the noise.
🖨️
Technique 1 — Print Debugging
Fast, portable, works anywhere — if you place prints strategically

Where to put print statements

Think in terms of checkpoints, not coverage. You want to know: what went in, what came out, and what was true at the key decision.

01
Function entry — what did it actually receive?
Print every argument at the top of the function. Type matters as much as value.
def calculate_iob(doses: list, minutes_ago: list) -> float: print(f"[DEBUG] calculate_iob called: doses={doses}, minutes_ago={minutes_ago}") print(f"[DEBUG] types: doses={type(doses)}, minutes_ago={type(minutes_ago)}") total = 0.0 ...
Prefix with [DEBUG] or a tag — makes it easy to grep and remove later.
Print the type alongside the value — a list of strings looks like a list of floats until you print type(doses[0]).
02
Decision points — which branch did it take?
Print inside each branch of an if/else or at the start of a loop body. You want to know which path was executed.
for i, (dose, t) in enumerate(zip(doses, minutes_ago)): remaining = dose * (1 - t / 120) print(f"[DEBUG] i={i}: dose={dose}, t={t}, remaining={remaining:.3f}") total += remaining
Loop-internal prints confirm: is the loop running the right number of times? Are intermediate values correct?
Use enumerate() to get the index alongside the value — crucial for spotting the off-by-one immediately.
03
Function exit — what is it actually returning?
Print immediately before the return statement. You can also compare the intermediate total versus the expected value.
print(f"[DEBUG] calculate_iob returning: total={total:.4f}") return total
If the entry print looks right but the exit print is wrong, the bug is inside the function. If the exit print looks right but the caller gets the wrong value, something changed it after the return.

Full worked example

Take the buggy calculate_iob() from Topic 2 and debug it using print statements. The range(len(doses) - 1) bug becomes immediately visible:

def calculate_iob_buggy(doses, minutes_ago): total = 0.0 for i in range(len(doses) - 1): # bug here remaining = doses[i] * (1 - minutes_ago[i] / 120) print(f"[DEBUG] i={i}, dose={doses[i]}, remaining={remaining:.3f}") total += remaining print(f"[DEBUG] final total={total:.4f}, expected ~4.5") return total calculate_iob_buggy([5.0, 3.0], [30, 90]) # Output: # [DEBUG] i=0, dose=5.0, remaining=3.750 # [DEBUG] final total=3.7500, expected ~4.5 # ↑ only one iteration ran — the bug is in range()
The loop body printed only once (i=0), immediately showing that range(len(doses) - 1) yielded only one index for a 2-element list.
The final total (3.75) versus the expected (~4.5) confirmed the wrong answer without any additional guesswork.

Technique 2 — PyCharm Debugger (Breakpoints)
Stop the program mid-execution and inspect everything

How to use a breakpoint effectively

A breakpoint pauses execution at a specific line. While paused, you can inspect any variable, step line by line, or jump into called functions. No print statements needed — but you need to know where to stop.

01
Set a breakpoint at the first suspicious line
Click the gutter (left margin) next to the line in PyCharm. A red dot appears. Run the file in Debug mode (Shift+F9 on Windows/Linux, ⌃D on macOS).
Set the breakpoint before the line you suspect — you want to see the state just before execution reaches it.
For loop bugs, set the breakpoint inside the loop body so it triggers on each iteration.
02
Inspect the Variables panel
When execution pauses, the Variables panel (bottom-left in debug mode) shows every local variable. Expand lists and dicts to see their contents. No print needed.
For the calculate_iob bug: at the first pause inside the loop, check i. After the loop body executes once and you step again, the loop has ended — you see the bug immediately.
Hover over a variable name in the editor to see its current value inline — faster than searching the panel.
03
Step through with F8 (Step Over) and F7 (Step Into)
F8 executes the current line and moves to the next — stays at the same call level. F7 steps into a function call — lets you follow execution inside a helper function.
F8 (Step Over): use this most of the time. Moves line by line without diving into library internals.
F7 (Step Into): use when you suspect the bug is inside a function you called. Follows execution into that function's body.
Shift+F8 (Step Out): run the rest of the current function and pause at its caller — useful when you've stepped into something and want to come back up.

Print vs Breakpoint — when to use each

Situation Reach for print Reach for breakpoint
Quick spot-check on one value ✓ Faster to add and remove
Bug inside a loop (many iterations) ✓ Print every iteration — see the pattern
Need to inspect a large nested dict ✓ Variables panel expands interactively
Bug might be across multiple functions ✓ Call stack panel shows the full chain
Bug only triggers with specific input ✓ Add a conditional print ✓ Conditional breakpoint (right-click the dot)
Async code / Flask request handler ✓ Print to stdout — simpler

Practice Questions
These cover the judgement layer — not just the mechanics
Mental model
Debugging is a process of bisection. You are narrowing the search space — not looking for the bug directly, but ruling out where it cannot be.
The Answer
If x at line 12 is correct, you know the bug is introduced somewhere between line 12 and the wrong output. You've eliminated everything above line 12. Add another print at line 28. If that is also correct, the bug is between 28 and 45. This binary search approach means you find bugs in log(n) steps rather than reading every line. The print at line 12 does not have to be near the bug — it is narrowing the search space.
A correct intermediate print eliminates the code before it. An incorrect print points to the code before it as the source.
Start with a print halfway through the function, then bisect again based on what you see.
Mental model
If a loop runs 10,000 times and the bug only appears on iteration 9,847, stepping through with F8 is not viable. A conditional breakpoint pauses only when a specific expression is true.
The Answer
A conditional breakpoint is a breakpoint with an attached condition expression. PyCharm only pauses when that expression evaluates to True. Right-click on a breakpoint dot → "Edit breakpoint" → type your condition (e.g. patient_id == "P-00923" or iob_value > 10.0). In DiaTwin, this is useful when processing a batch of 500 CGM records and the wrong result only appears for one specific patient — you can pause exactly on that patient without stepping through the other 499.
Equivalent to adding if condition: breakpoint() in code — but non-destructive, no code changes needed.
The built-in breakpoint() function (Python 3.7+) is a portable alternative — it triggers the debugger even outside an IDE.
Mental model
Python's dynamic typing means a variable can look like one thing and be another. "5.0" and 5.0 print identically if you forget the quotes. The type is what matters for arithmetic and serialisation.
The Answer
print(x) shows the value. print(type(x)) shows its type. In Python, "120" * 2 == "120120" (string repetition) while 120 * 2 == 240 (arithmetic). If CGM readings come in as strings from a JSON payload and you do not convert them, arithmetic will either fail with a TypeError or produce nonsense silently. Printing both value and type together — print(f"dose={dose}, type={type(dose)}") — immediately catches type confusion.
A common source: json.loads() returns numbers as floats by default, but if your JSON has "age": "34" (quoted), it comes in as a string.
Use isinstance(x, (int, float)) as a guard rather than trusting the type at all times in a data pipeline.
Mental model
The entry print and the exit print act as a bracket. If what's inside the bracket is wrong, the bug is in the function body. You have already eliminated the caller.
The Answer
The bug is inside the function body — between the entry and the exit print. The inputs were correct; the transformation was not. Next step: add a print after each logical operation inside the function body to narrow down exactly which line corrupts the value. This is the bisection approach applied inside the function. The entry/exit bracket has localised the bug to one function; the internal bisection localises it to one operation.
Entry correct + exit wrong → bug is inside. Entry wrong → bug is in the caller or above. Neither of these requires reading every line; they give you direction.
Mental model
When paused at a breakpoint, you know what is happening now. The Call Stack tells you how execution got there — every function call in the chain, with their local variables, still alive in memory.
The Answer
The Call Stack panel lists every active function call frame above the current one. Clicking a frame in the panel switches the Variables panel to show that frame's local variables. This is useful when you are paused deep inside a helper function and want to inspect what the calling function passed — without having to add prints to the caller. In a Flask request handler, the call stack shows: your route handler → the calculation service → the utility function where the breakpoint is. You can click back up to the route handler and inspect the request object, all without restarting.
The Call Stack is the debugger equivalent of reading a stack trace — but interactive, not post-mortem.
Use it when the bug might have been introduced by the caller passing a wrong argument — check the caller's local variables to confirm what it sent.
Mental model
Debug prints are temporary scaffolding. They have noise, they reveal internal state, and they slow output. Professional code uses structured logging — which is configured to show or suppress based on the deployment environment.
The Answer
No — remove them before committing. The standard alternative is Python's logging module. Replace print(f"[DEBUG] x={x}") with logger.debug(f"x={x}"). In production, the log level is set to INFO or higher, so debug messages are suppressed. In development, set it to DEBUG and they reappear. This means you never have to add and remove the same print again — the logging system decides whether to show it based on the environment.
In DiaTwin / Morafek, every service should have a module-level logger: logger = logging.getLogger(__name__).
Committed print() statements in server-side code will appear in production logs and may leak patient data. Use logging with level guards instead.

Wenn der Code läuft, aber lügt
Exceptions sagen dir, wo etwas kaputt ist — Debugging sagt dir, warum die Antwort falsch ist

Die zwei Modi des Debuggings

Wenn Code mit einer Exception abstürzt, hast du einen Stack-Trace: einen Ort, einen Typ und eine Meldung. Das ist nützlich. Das schwierigere Problem ist, wenn Code bis zum Ende läuft, die Ausgabe aber falsch ist — eine Dosierung ist 0,5 Einheiten zu hoch, eine Patientensuche gibt den falschen Datensatz zurück, eine Summe fehlt einen Eintrag.

Es gibt zwei Werkzeuge dafür. Verwende Print-Debugging für schnelle, gezielte Checks, wenn du bereits eine Hypothese hast. Verwende den PyCharm Debugger, wenn du den vollständigen Zustand eines laufenden Programms inspizieren musst — mehrere Variablen, Datenstrukturen, Aufrufstapel.

Anti-Pattern zu vermeiden: Print-Statements überall hinzufügen, dann entfernen, dann wieder hinzufügen. Strategische Platzierung — Einstieg, Ausstieg, Entscheidungspunkte — gibt dir das Signal ohne das Rauschen.
🖨️
Technik 1 — Print-Debugging
Schnell, portabel, funktioniert überall — wenn du Prints strategisch platzierst

Wo Print-Statements platzieren

Denke in Checkpoints, nicht in Abdeckung. Du willst wissen: Was kam rein, was kam raus, und was war an der Schlüsselentscheidung wahr.

01
Funktionseinstieg — was hat sie tatsächlich empfangen?
Drucke jedes Argument am Anfang der Funktion. Typ ist genauso wichtig wie Wert.
def calculate_iob(doses: list, minutes_ago: list) -> float: print(f"[DEBUG] calculate_iob aufgerufen: doses={doses}, minutes_ago={minutes_ago}") print(f"[DEBUG] Typen: doses={type(doses)}, minutes_ago={type(minutes_ago)}") total = 0.0 ...
Mit [DEBUG] präfixen — macht es einfach, sie später zu finden und zu entfernen.
Drucke den Typ neben dem Wert — eine Liste von Strings sieht aus wie eine Liste von Floats, bis du type(doses[0]) druckst.
02
Entscheidungspunkte — welchen Zweig hat es genommen?
Drucke innerhalb jedes Zweigs eines if/else oder am Anfang eines Schleifenkörpers.
for i, (dose, t) in enumerate(zip(doses, minutes_ago)): remaining = dose * (1 - t / 120) print(f"[DEBUG] i={i}: dose={dose}, t={t}, remaining={remaining:.3f}") total += remaining
Schleifeninterner Print bestätigt: Läuft die Schleife die richtige Anzahl von Malen? Sind die Zwischenwerte korrekt?
03
Funktionsausstieg — was gibt sie tatsächlich zurück?
Drucke unmittelbar vor dem return-Statement.
print(f"[DEBUG] calculate_iob gibt zurück: total={total:.4f}") return total
Wenn der Einstiegs-Print korrekt aussieht, aber der Ausstiegs-Print falsch ist, liegt der Bug innerhalb der Funktion.

Technik 2 — PyCharm Debugger (Breakpoints)
Das Programm in der Mitte der Ausführung stoppen und alles inspizieren

Wie man einen Breakpoint effektiv einsetzt

Ein Breakpoint pausiert die Ausführung an einer bestimmten Zeile. Während der Pause kannst du jede Variable inspizieren, Zeile für Zeile vorgehen oder in aufgerufene Funktionen einsteigen.

01
Breakpoint an der ersten verdächtigen Zeile setzen
Klicke auf den Randbereich (linker Rand) neben der Zeile in PyCharm. Ein roter Punkt erscheint. Führe die Datei im Debug-Modus aus (Shift+F9 unter Windows/Linux, ⌃D unter macOS).
02
Variables-Panel inspizieren
Wenn die Ausführung pausiert, zeigt das Variables-Panel (unten links im Debug-Modus) jede lokale Variable. Listen und Dicts aufklappen um ihren Inhalt zu sehen.
03
Mit F8 (Step Over) und F7 (Step Into) durchgehen
F8 führt die aktuelle Zeile aus und geht zur nächsten — bleibt auf derselben Aufrufebene. F7 springt in einen Funktionsaufruf hinein.
F8 (Step Over): meistens verwenden. Bewegt sich zeilenweise ohne in Bibliotheksinterna einzutauchen.
F7 (Step Into): verwenden, wenn du vermutest, dass der Bug in einer aufgerufenen Funktion ist.
Shift+F8 (Step Out): den Rest der aktuellen Funktion ausführen und beim Aufrufer pausieren.

Übungsfragen
Diese decken die Urteilsebene ab — nicht nur die Mechanik
Mentales Modell
Debugging ist ein Bisektionsprozess. Du engst den Suchraum ein — suchst nicht direkt den Bug, sondern schließt aus, wo er nicht sein kann.
Die Antwort
Wenn x in Zeile 12 korrekt ist, weißt du, dass der Bug irgendwo zwischen Zeile 12 und der falschen Ausgabe eingeführt wird. Du hast alles oberhalb von Zeile 12 eliminiert. Füge einen weiteren Print in Zeile 28 hinzu. Wenn dieser auch korrekt ist, liegt der Bug zwischen 28 und 45. Dieser Binärsuchansatz bedeutet, dass du Bugs in log(n) Schritten findest, anstatt jede Zeile zu lesen.
Ein korrekter Zwischen-Print eliminiert den Code davor. Ein inkorrekter Print zeigt auf den Code davor als Quelle.
Mentales Modell
Wenn eine Schleife 10.000 Mal läuft und der Bug nur in Iteration 9.847 auftaucht, ist schrittweises Durchgehen mit F8 nicht praktikabel.
Die Antwort
Ein bedingter Breakpoint ist ein Breakpoint mit einer angehängten Bedingungsausdruck. PyCharm pausiert nur, wenn dieser Ausdruck True ergibt. Rechtsklick auf den Breakpoint-Punkt → "Breakpoint bearbeiten" → Bedingung eingeben (z.B. patient_id == "P-00923"). In DiaTwin ist das nützlich, wenn du einen Batch von 500 CGM-Datensätzen verarbeitest und das falsche Ergebnis nur für einen bestimmten Patienten auftaucht.
Entspricht dem Hinzufügen von if bedingung: breakpoint() im Code — aber zerstörungsfrei, keine Codeänderungen nötig.
Mentales Modell
Pythons dynamische Typisierung bedeutet, dass eine Variable wie eines aussehen und etwas anderes sein kann. "5.0" und 5.0 sehen beim Drucken identisch aus, wenn du die Anführungszeichen vergisst.
Die Antwort
print(x) zeigt den Wert. print(type(x)) zeigt seinen Typ. In Python ist "120" * 2 == "120120" (String-Wiederholung), während 120 * 2 == 240 (Arithmetik). Wenn CGM-Readings als Strings aus einer JSON-Payload kommen und du sie nicht konvertierst, schlägt die Arithmetik fehl oder produziert Unsinn. Beides zusammen zu drucken — Wert und Typ — erkennt Typverwechslungen sofort.
Häufige Ursache: json.loads() gibt Zahlen standardmäßig als Floats zurück, aber wenn JSON "age": "34" hat (mit Anführungszeichen), kommt es als String.
Mentales Modell
Der Einstiegs- und Ausstiegs-Print wirken wie eine Klammer. Wenn was innerhalb der Klammer falsch ist, liegt der Bug im Funktionskörper.
Die Antwort
Der Bug liegt im Funktionskörper — zwischen dem Einstiegs- und dem Ausstiegs-Print. Die Eingaben waren korrekt; die Transformation war es nicht. Nächster Schritt: einen Print nach jeder logischen Operation im Funktionskörper hinzufügen, um genau herauszufinden, welche Zeile den Wert korrumpiert. Dies ist der Bisektionsansatz, der innerhalb der Funktion angewendet wird.
Einstieg korrekt + Ausstieg falsch → Bug ist drinnen. Einstieg falsch → Bug ist im Aufrufer oder darüber.
Mentales Modell
An einem Breakpoint pausiert weißt du, was jetzt passiert. Der Call-Stack sagt dir, wie die Ausführung dorthin gekommen ist.
Die Antwort
Das Call-Stack-Panel listet jeden aktiven Funktionsaufruf-Frame über dem aktuellen auf. Das Anklicken eines Frames wechselt das Variables-Panel, um die lokalen Variablen dieses Frames zu zeigen. In einem Flask-Request-Handler zeigt der Call-Stack: dein Route-Handler → der Berechnungsservice → die Hilfsfunktion, wo der Breakpoint ist. Du kannst nach oben zum Route-Handler klicken und das Request-Objekt inspizieren, ohne neu zu starten.
Der Call-Stack ist das Debugger-Äquivalent des Lesens eines Stack-Traces — aber interaktiv, nicht post-mortem.
Mentales Modell
Debug-Prints sind temporäres Gerüst. Professioneller Code verwendet strukturiertes Logging — das je nach Deployment-Umgebung ein- oder ausgeblendet werden kann.
Die Antwort
Nein — vor dem Commit entfernen. Die Standardalternative ist Pythons logging-Modul. Ersetze print(f"[DEBUG] x={x}") durch logger.debug(f"x={x}"). In der Produktion wird der Log-Level auf INFO oder höher gesetzt, sodass Debug-Meldungen unterdrückt werden. In der Entwicklung auf DEBUG setzen und sie erscheinen wieder.
In DiaTwin/Morafek sollte jeder Service einen Logger auf Modulebene haben: logger = logging.getLogger(__name__).
Committete print()-Statements in serverseitigem Code erscheinen in Produktionsprotokollen und können Patientendaten leaken. Stattdessen logging mit Level-Guards verwenden.

حين يعمل الكود لكنه يكذب
الاستثناءات تخبرك أين انكسر شيء ما — التصحيح يخبرك لماذا الإجابة خاطئة

النمطان للتصحيح

حين ينهار الكود باستثناء، لديك stack trace: موقع، نوع، ورسالة. ذلك مفيد. المشكلة الأصعب هي حين يكتمل الكود لكن المخرج خاطئ — جرعة أعلى بـ 0.5 وحدة، بحث عن مريض يعيد السجل الخاطئ، مجموع يفتقد إدخالاً.

أداتان موجودتان لهذا. استخدم تصحيح الطباعة للفحوصات السريعة والمستهدفة حين يكون لديك فرضية. استخدم مصحّح PyCharm حين تحتاج إلى فحص الحالة الكاملة للبرنامج — متغيرات متعددة، هياكل البيانات، مكدس الاستدعاء.

نمط مضاد يجب تجنبه: إضافة عبارات print في كل مكان، ثم إزالتها، ثم إضافتها مجدداً. التوضيع الاستراتيجي — الدخول، الخروج، نقاط القرار — يعطيك الإشارة دون الضوضاء.
🖨️
الأسلوب 1 — تصحيح الطباعة
سريع، محمول، يعمل في أي مكان — إذا وضعت عبارات print باستراتيجية

أين توضع عبارات print

فكّر من حيث نقاط التفتيش، لا التغطية. تريد أن تعرف: ما الذي دخل، ما الذي خرج، وما الذي كان صحيحاً عند القرار الرئيسي.

01
دخول الدالة — ماذا تلقّت فعلاً؟
اطبع كل وسيطة في أعلى الدالة. النوع مهم بقدر القيمة.
def calculate_iob(doses: list, minutes_ago: list) -> float: print(f"[DEBUG] calculate_iob: doses={doses}, minutes_ago={minutes_ago}") print(f"[DEBUG] أنواع: doses={type(doses)}, minutes_ago={type(minutes_ago)}") total = 0.0 ...
ابدأ بـ [DEBUG] — يجعل البحث عنها والتخلص منها لاحقاً أسهل.
اطبع النوع جانب القيمة — قائمة من السلاسل تبدو كقائمة من الأعداد حتى تطبع type(doses[0]).
02
نقاط القرار — أي فرع أخذه؟
اطبع داخل كل فرع من if/else أو في بداية جسم الحلقة.
for i, (dose, t) in enumerate(zip(doses, minutes_ago)): remaining = dose * (1 - t / 120) print(f"[DEBUG] i={i}: dose={dose}, t={t}, remaining={remaining:.3f}") total += remaining
الطباعة داخل الحلقة تؤكد: هل تعمل الحلقة العدد الصحيح من المرات؟ هل القيم الوسيطة صحيحة؟
03
خروج الدالة — ماذا تعيد فعلاً؟
اطبع مباشرةً قبل جملة return.
print(f"[DEBUG] calculate_iob تعيد: total={total:.4f}") return total
إذا كان طباعة الدخول صحيحة لكن طباعة الخروج خاطئة، فالخطأ داخل الدالة.

الأسلوب 2 — مصحّح PyCharm (نقاط التوقف)
إيقاف البرنامج في منتصف التنفيذ وفحص كل شيء

كيفية استخدام نقطة توقف بفعالية

نقطة التوقف توقف التنفيذ عند سطر محدد. أثناء التوقف يمكنك فحص أي متغير، التقدم سطراً بسطر، أو الدخول إلى الدوال المستدعاة.

01
ضع نقطة توقف عند السطر الأول المشبوه
انقر على هامش اليسار بجانب السطر في PyCharm. تظهر نقطة حمراء. شغّل الملف في وضع التصحيح (Shift+F9 في Windows/Linux، ⌃D في macOS).
02
افحص لوحة Variables
حين يتوقف التنفيذ، تُظهر لوحة Variables (أسفل يسار في وضع التصحيح) كل متغير محلي. افتح القوائم والقواميس لرؤية محتواها. لا طباعة مطلوبة.
03
تقدم بـ F8 (تخطي) وF7 (دخول)
F8 ينفذ السطر الحالي وينتقل للتالي — يبقى على نفس مستوى الاستدعاء. F7 يدخل في استدعاء دالة — يتبع التنفيذ داخل جسم دالة المساعدة.
F8 (تخطي): استخدمه معظم الوقت. يتحرك سطراً بسطر دون الدخول في التفاصيل الداخلية للمكتبات.
F7 (دخول): استخدمه حين تشك أن الخطأ داخل دالة استدعيتها.

الإجابة
الخطأ في جسم الدالة — بين طباعة الدخول والخروج. المدخلات كانت صحيحة؛ التحويل لم يكن. الخطوة التالية: أضف طباعة بعد كل عملية منطقية داخل جسم الدالة لتحديد بالضبط أي سطر يُفسد القيمة. هذا هو نهج الثنائية المطبق داخل الدالة.
دخول صحيح + خروج خاطئ → الخطأ بالداخل. دخول خاطئ → الخطأ في المستدعي أو أعلى. كلاهما يعطيك اتجاهاً.

عند التوقف في نقطة توقف، تعرف ما يحدث الآن. Call Stack يخبرك كيف وصل التنفيذ إلى هناك.
الإجابة
لوحة Call Stack تُدرج كل إطار استدعاء دالة نشط فوق الإطار الحالي. النقر على إطار في اللوحة يُحوّل لوحة Variables لإظهار المتغيرات المحلية لذلك الإطار. في Flask، يُظهر Call Stack: معالج المسار → خدمة الحساب → الدالة المساعدة حيث توجد نقطة التوقف. يمكنك النقر للأعلى لمعالج المسار وفحص كائن الطلب، كل ذلك دون إعادة التشغيل.
Call Stack هو مكافئ المصحّح لقراءة stack trace — لكن تفاعلي، لا ما بعد الوفاة.
