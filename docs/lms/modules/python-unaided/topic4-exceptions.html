PYTHON UNAIDED CODING — Topic 4 of 5

Exception Handling

Real code fails. Network timeouts, missing fields, malformed payloads. This topic gives you three production patterns for handling failure gracefully — so your code logs and recovers instead of crashing hard.

🛡️
Why Code Must Fail Gracefully
A crash in a hospital integration is not just a bug — it is a missing result for a real patient

The cost of crashing hard

Code that raises an unhandled exception and terminates is useless in production. In a hospital-data pipeline, a CGM reading pipeline that crashes on one malformed sensor payload will stop processing all subsequent readings for every patient on the same batch. The one bad record poisons the entire run.

Exception handling is not about suppressing errors — it is about deciding what should happen when something you did not control goes wrong. You have three choices: log and continue, retry, or fall back to a safe default. Picking the right one depends on whether the failure is transient, permanent, or recoverable.

The worst anti-pattern: except Exception: pass. This silently swallows every error, including bugs in your own code. You will never know the pipeline is broken until a patient report is wrong.
⚙️
Three Production Patterns
Each serves a different failure mode — choose deliberately

Patterns you will use in every backend project

These are not academic — all three appear in real hospital-data pipelines and FHIR integration layers.

01
Catch, Log, and Continue
Use when: one bad item should not stop processing the rest. Permanent failure — retrying won't help.
import logging logger = logging.getLogger(__name__) def process_cgm_reading(raw_value: str) -> float | None: try: return float(raw_value) except ValueError: logger.error(f"Invalid CGM reading — could not convert to float: {raw_value!r}") return None # Caller skips None results and continues the batch readings = [process_cgm_reading(v) for v in raw_sensor_batch] valid = [r for r in readings if r is not None]
Return None (or a sentinel value) to signal that this item failed without raising. The caller decides what to do with it.
Use logger.error(), not print(). Logs persist; prints disappear when the terminal closes.
Include the bad value in the message ({raw_value!r}) so you can reproduce the failure without digging through raw data.
02
Catch and Retry
Use when: the failure is transient — a network blip, a temporarily overloaded service. Retrying with a short delay will likely succeed.
import time def fetch_fhir_patient(patient_id: str, retries: int = 3) -> dict: last_error = None for attempt in range(retries): try: response = requests.get(f"{FHIR_BASE}/Patient/{patient_id}", timeout=5) response.raise_for_status() return response.json() except (requests.ConnectionError, requests.Timeout) as e: last_error = e if attempt < retries - 1: time.sleep(2 ** attempt) # exponential back-off: 1s, 2s, 4s raise last_error # re-raise after all retries exhausted
Exponential back-off (2 ** attempt) respects the server — it does not hammer a service that is already struggling.
Re-raise after the last attempt. The caller needs to know the operation finally failed — do not silently return None for a network error, because the caller cannot distinguish "not found" from "unreachable".
Only retry on transient exceptions. A 404 Not Found will not become a 200 OK after three retries.
03
Catch and Fall Back to a Safe Default
Use when: the system can function with a degraded but safe value. Missing personalised data → use a conservative population baseline.
from datetime import datetime def get_insulin_sensitivity(patient_id: str) -> float: try: profile = load_patient_profile(patient_id) return profile["isf"] # insulin sensitivity factor (mmol/L per unit) except (KeyError, PatientNotFoundError): logger.warning( f"No ISF profile for {patient_id} — using population default 2.0" ) return 2.0 # conservative population default def get_baseline_bg(patient_id: str) -> float: try: return dynamic_baseline_from_cgm(patient_id) except InsufficientDataError: hour = datetime.now().hour return preset_baseline_for_hour(hour)
In safety-critical code, the fallback must be conservative, not optimistic. A high ISF default underestimates required insulin; a low one overestimates it. Know the clinical implication of your default.
Log at WARNING not ERROR — the system is degraded, but it did not fail.
Return the fallback explicitly from the function. Don't let the caller silently use a None — that leads to TypeError: unsupported operand type(s) for *: 'NoneType' and 'float' three call levels later.
📐
The Full try / except / else / finally Structure
Most code uses two clauses — knowing all four makes you precise

When to use else and finally

The else clause runs only if no exception was raised — useful for code that should only execute on success. The finally clause always runs — used for cleanup that must happen regardless of success or failure.

def write_cgm_to_db(session, reading: dict) -> bool: try: session.add(CGMReading(**reading)) session.commit() except IntegrityError as e: session.rollback() logger.error(f"Duplicate CGM record — skipping: {e}") return False else: logger.info(f"CGM reading committed: {reading['patient_id']}") return True finally: session.close() # always runs — session is always released
else: put success-path logic here instead of in the try block. This makes it clear that this code only runs when nothing went wrong — not just that it hasn't raised yet.
finally: file handles, database sessions, network connections. Anything that must be released regardless of what happened. This is the correct place — not after the try/except block, because an unhandled exception would skip code after the block.
For file I/O and database sessions, prefer with statements over finally — they do the same thing with less code and can't be forgotten.

Practice Questions
These test judgement, not just syntax recall
Mental model
Catching Exception is like saying "I know how to handle literally anything that can go wrong." You almost certainly do not.
The Answer
ValueError means "this specific kind of conversion or validation failed — I expected a number, I got a string." Exception catches everything: network errors, memory errors, your own typos (NameError, AttributeError). When you catch Exception, bugs in your own code are silently eaten — you will see wrong output without knowing why. Always catch the most specific exception that covers the failure you anticipate. If you need to catch multiple specific types, list them: except (ValueError, TypeError).
except Exception: pass is a smell. It means "I don't know what can go wrong but I'm going to hide it." Use it only at the very top of a program as a last-resort logger, never inside a function.
In a hospital-data pipeline: except ValueError for CGM parsing, except requests.Timeout for FHIR calls, except KeyError for missing patient profile fields. Each one signals a different root cause and a different response.
Mental model
Returning None silently passes the problem to the caller. Re-raising loudly tells the caller: "this operation failed, and you need to decide what to do about it."
The Answer
Return None (or a sentinel) when the failure is local and recoverable — one bad item in a batch that the caller can skip. Re-raise when the failure means the whole operation cannot succeed — a database that won't connect, a required config file that doesn't exist. A function that returns None for an unrecoverable failure forces its caller to check for None on every call, and if it forgets, the error surfaces three levels away as a TypeError with no useful context. Use raise (bare re-raise) to preserve the original traceback, or raise NewError("message") from original_error to add context while chaining.
Rule of thumb: if the caller has a meaningful way to recover (skip this item, use a default), return a sentinel. If the caller cannot continue without this result, re-raise.
raise inside an except block re-raises the current exception unchanged, including its original traceback. Never do raise e — it resets the traceback to this line.
Mental model
Code after a try/except block is not guaranteed to run if an unhandled exception escapes. finally runs even then — it is a contract with the runtime, not with your own control flow.
The Answer
finally executes regardless of whether the try block succeeded, raised a caught exception, raised an uncaught exception, or hit a return statement. Code placed after the try/except block only runs if no exception escaped. If your database session is not in a finally and an unexpected error is raised, the session is left open — holding a transaction lock until the connection times out. In a web server handling many concurrent requests, this exhausts the connection pool. Use with statements (context managers) for the same guarantee with less code.
with open("file.txt") as f: is equivalent to try/finally with f.close() — the file is closed even if an exception is raised inside the with block.
SQLAlchemy sessions, file handles, socket connections, and locks all need cleanup. Prefer with; use finally for cases where a context manager isn't available.
Mental model
The right level to catch an exception is the level that has enough context to respond meaningfully. A low-level HTTP fetch doesn't know whether a retry is safe. The orchestrating caller does.
The Answer
It depends on whether the three calls are independent. If one API failing should not stop the others, catch internally per-call and return partial results. If all three must succeed for the result to be valid, let exceptions bubble up — the caller decides whether to retry the whole operation, return a degraded result, or fail the request. A useful heuristic: catch exceptions at the boundary between layers (HTTP client → service layer, service layer → route handler). Do not catch at every function — that produces deeply nested try/except trees that are hard to read and harder to test.
In a typical HL7 pipeline: the per-message parse can fail silently (log + skip); the database write must succeed (retry + eventually fail loudly); the audit log must always run (finally).
Exception handling architecture is part of the system design. Think about it at the same time as the happy path, not as an afterthought.
Mental model
Built-in exceptions are generic. ValueError from a CGM parser and ValueError from a JSON deserialiser look identical to the caller. Custom exceptions make the failure type part of the public API.
The Answer
Subclass Exception (or a more specific built-in):
class InsufficientCGMDataError(ValueError): """Raised when fewer CGM readings exist than the algorithm requires.""" pass class PatientNotFoundError(LookupError): """Raised when a patient ID does not exist in the database.""" def __init__(self, patient_id: str): super().__init__(f"Patient '{patient_id}' not found") self.patient_id = patient_id
Worth it when: callers need to distinguish this failure from other ValueErrors; when the exception carries domain-specific data (patient_id); when you want to document the failure mode as part of the function signature.
Subclass an appropriate built-in so callers that catch the broader type still work: except LookupError will catch PatientNotFoundError without modification.
Keep custom exception classes small — they are signal types, not business logic containers.
Mental model
You caught a low-level exception and want to raise a higher-level one. Without chaining, the original cause disappears from the traceback.
The Answer
raise HighLevelError("context message") from original_exception links the two exceptions. Python prints both in the traceback: the original cause and the new error, connected with "The above exception was the direct cause of...". Without chaining (raise HighLevelError("...") from None), you explicitly suppress the original — useful when the low-level error leaks implementation details you don't want in a public API.
try: row = db.query(Patient).filter_by(id=patient_id).one() except NoResultFound as e: raise PatientNotFoundError(patient_id) from e # Traceback shows both — you see the SQLAlchemy cause AND the domain error
Wrapping SQLAlchemy exceptions in domain exceptions (PatientNotFoundError) keeps the service layer independent of the ORM. The caller sees domain errors, not database internals.

Warum Code sicher scheitern muss
Ein Absturz in einer Krankenhausintegration ist nicht nur ein Bug — es ist ein fehlendes Ergebnis für einen echten Patienten

Die Kosten eines harten Absturzes

Code, der eine unbehandelte Exception auslöst und terminiert, ist im Produktionsbetrieb wertlos. In einer Krankenhaus-Datenpipeline würde eine CGM-Lesepipeline, die bei einem fehlerhaften Sensor-Payload abstürzt, die Verarbeitung aller nachfolgenden Readings für jeden Patienten im selben Batch stoppen. Ein schlechter Datensatz vergiftet den gesamten Durchlauf.

Exception-Handling bedeutet nicht, Fehler zu unterdrücken — es geht darum zu entscheiden, was passieren soll, wenn etwas außerhalb deiner Kontrolle schiefgeht. Du hast drei Optionen: loggen und weitermachen, wiederholen, oder auf einen sicheren Standardwert zurückfallen. Die richtige Wahl hängt davon ab, ob der Fehler vorübergehend, dauerhaft oder behebbar ist.

Das schlimmste Anti-Pattern: except Exception: pass. Das verschluckt jeden Fehler stillschweigend — auch Bugs in deinem eigenen Code. Du wirst nie wissen, dass die Pipeline kaputt ist, bis ein Patientenbericht falsch ist.
⚙️
Drei Produktionsmuster
Jedes dient einem anderen Fehlermodus — wähle bewusst

Muster, die du in jedem Backend-Projekt verwenden wirst

Diese sind nicht akademisch — alle drei kommen in echten Krankenhaus-Datenpipelines und FHIR-Integrationsschichten vor.

01
Fangen, Loggen und Weitermachen
Verwenden wenn: ein schlechtes Element den Rest der Verarbeitung nicht aufhalten soll. Dauerhafter Fehler — Wiederholen hilft nicht.
import logging logger = logging.getLogger(__name__) def process_cgm_reading(raw_value: str) -> float | None: try: return float(raw_value) except ValueError: logger.error(f"Ungültiger CGM-Wert — Konvertierung fehlgeschlagen: {raw_value!r}") return None readings = [process_cgm_reading(v) for v in raw_sensor_batch] valid = [r for r in readings if r is not None]
Gib None zurück um zu signalisieren, dass dieses Element fehlgeschlagen ist, ohne eine Exception auszulösen. Der Aufrufer entscheidet, was damit zu tun ist.
Verwende logger.error(), nicht print(). Logs bleiben erhalten; Prints verschwinden wenn das Terminal geschlossen wird.
02
Fangen und Wiederholen
Verwenden wenn: der Fehler vorübergehend ist — ein Netzwerkausfall, ein zeitweise überlasteter Dienst. Wiederholen mit kurzer Verzögerung wird wahrscheinlich erfolgreich sein.
import time def fetch_fhir_patient(patient_id: str, retries: int = 3) -> dict: last_error = None for attempt in range(retries): try: response = requests.get(f"{FHIR_BASE}/Patient/{patient_id}", timeout=5) response.raise_for_status() return response.json() except (requests.ConnectionError, requests.Timeout) as e: last_error = e if attempt < retries - 1: time.sleep(2 ** attempt) # exponentielles Backoff: 1s, 2s, 4s raise last_error
Exponentielles Backoff (2 ** attempt) respektiert den Server — er bombardiert keinen Dienst, der bereits kämpft.
Nur bei vorübergehenden Exceptions wiederholen. Ein 404 Not Found wird nach drei Versuchen kein 200 OK.
03
Fangen und auf einen sicheren Standardwert zurückfallen
Verwenden wenn: das System mit einem degravierten aber sicheren Wert funktionieren kann. Fehlende personalisierte Daten → konservativen Populationsstandard verwenden.
def get_insulin_sensitivity(patient_id: str) -> float: try: profile = load_patient_profile(patient_id) return profile["isf"] except (KeyError, PatientNotFoundError): logger.warning( f"Kein ISF-Profil für {patient_id} — Populationsstandard 2.0 verwendet" ) return 2.0
In sicherheitskritischem Code muss der Fallback konservativ sein, nicht optimistisch. Kenne die klinische Implikation deines Standards.
Logge mit WARNING, nicht ERROR — das System ist degradiert, aber es hat nicht versagt.
❓
Übungsfragen
Diese testen Urteilsvermögen, nicht nur Syntaxkenntnisse
Mentales Modell
Exception zu fangen bedeutet zu sagen: "Ich weiß, wie ich mit buchstäblich allem umgehen soll, was schiefgehen kann." Das tue ich mit Sicherheit nicht.
Die Antwort
ValueError bedeutet "diese spezifische Konvertierung oder Validierung ist fehlgeschlagen." Exception fängt alles: Netzwerkfehler, Speicherfehler, eigene Tippfehler (NameError, AttributeError). Wenn du Exception fängst, werden Bugs in deinem eigenen Code stillschweigend verschluckt — du siehst falsche Ausgaben ohne zu wissen warum. Fange immer die spezifischste Exception, die den zu erwartenden Fehler abdeckt.
except Exception: pass ist ein Code-Smell. Verwende es nur ganz oben in einem Programm als letzten Rettungsanker-Logger, niemals innerhalb einer Funktion.
Mentales Modell
None zurückgeben gibt das Problem stillschweigend an den Aufrufer weiter. Neu auslösen sagt dem Aufrufer laut: "Diese Operation ist fehlgeschlagen."
Die Antwort
Gib None zurück, wenn der Fehler lokal und behebbar ist — ein schlechtes Element in einem Batch, das der Aufrufer überspringen kann. Löse neu aus, wenn der Fehler bedeutet, dass die gesamte Operation nicht erfolgreich sein kann — eine Datenbank die sich nicht verbindet, eine erforderliche Konfigurationsdatei die nicht existiert. Verwende raise (nacktes Neu-Auslösen) um den ursprünglichen Traceback zu erhalten, oder raise NeuerFehler("Nachricht") from ursprünglicher_fehler um Kontext hinzuzufügen.
Faustregel: Wenn der Aufrufer einen sinnvollen Weg hat sich zu erholen (Element überspringen, Standard verwenden), Sentinel zurückgeben. Wenn der Aufrufer ohne dieses Ergebnis nicht weitermachen kann, neu auslösen.
Mentales Modell
Code nach einem try/except-Block läuft nicht garantiert, wenn eine unbehandelte Exception entkommt. finally läuft selbst dann — es ist ein Vertrag mit der Laufzeit.
Die Antwort
finally wird unabhängig davon ausgeführt, ob der try-Block erfolgreich war, eine gefangene Exception auslöste, eine ungefangene Exception auslöste, oder ein return-Statement erreichte. Wenn deine Datenbanksitzung nicht in einem finally ist und ein unerwarteter Fehler ausgelöst wird, bleibt die Sitzung offen — hält eine Transaktionssperre bis die Verbindung abbricht. Verwende with-Statements (Kontextmanager) für dieselbe Garantie mit weniger Code.
with open("file.txt") as f: ist äquivalent zu try/finally mit f.close() — die Datei wird geschlossen, auch wenn eine Exception im with-Block ausgelöst wird.
Mentales Modell
Die richtige Ebene zum Fangen einer Exception ist die Ebene, die genug Kontext hat, um sinnvoll zu reagieren.
Die Antwort
Es hängt davon ab, ob die drei Aufrufe unabhängig sind. Wenn ein API-Fehler die anderen nicht stoppen soll, intern pro Aufruf fangen und Teilergebnisse zurückgeben. Wenn alle drei für ein gültiges Ergebnis erfolgreich sein müssen, Exceptions hochblubbern lassen — der Aufrufer entscheidet ob er die gesamte Operation wiederholt, ein degradiertes Ergebnis zurückgibt oder die Anfrage fehlschlägt. Nützliche Heuristik: Exceptions an Schichtgrenzen fangen, nicht in jeder Funktion.
Exception-Handling-Architektur ist Teil des Systemdesigns. Denke daran gleichzeitig mit dem Happy Path, nicht nachträglich.
Mentales Modell
Eingebaute Exceptions sind generisch. ValueError aus einem CGM-Parser und ValueError aus einem JSON-Deserialisierer sehen für den Aufrufer identisch aus.
Die Antwort
Von Exception oder einer spezifischeren eingebauten Klasse ableiten:
class InsufficientCGMDataError(ValueError): """Wird ausgelöst wenn weniger CGM-Readings vorhanden sind als der Algorithmus braucht.""" pass class PatientNotFoundError(LookupError): def __init__(self, patient_id: str): super().__init__(f"Patient '{patient_id}' nicht gefunden") self.patient_id = patient_id
Lohnt sich wenn: Aufrufer diese Fehlerart von anderen ValueErrors unterscheiden müssen; wenn die Exception domänenspezifische Daten trägt; wenn du den Fehlermodus als Teil der Funktionssignatur dokumentieren möchtest.
Mentales Modell
Du hast eine Ausnahme auf niedriger Ebene gefangen und möchtest eine Ausnahme auf höherer Ebene auslösen. Ohne Chaining verschwindet die ursprüngliche Ursache aus dem Traceback.
Die Antwort
raise HochstufigeFehler("Kontext") from ursprüngliche_exception verknüpft die beiden Exceptions. Python druckt beide im Traceback aus. Ohne Chaining siehst du nur den neuen Fehler — du verlierst die SQLAlchemy-Ursache und weißt nicht warum der Datenbankaufruf wirklich gescheitert ist. Verwende from None um die ursprüngliche Exception explizit zu unterdrücken, wenn sie Implementierungsdetails enthüllen würde.
SQLAlchemy-Exceptions in Domain-Exceptions einwickeln hält die Service-Schicht unabhängig vom ORM. Der Aufrufer sieht Domain-Fehler, keine Datenbankinternas.

لماذا يجب أن يفشل الكود بأمان
الانهيار في تكامل مستشفى ليس مجرد خطأ برمجي — بل نتيجة مفقودة لمريض حقيقي

تكلفة الانهيار القاسي

الكود الذي يطلق استثناءً غير معالَج وينهي تنفيذه لا قيمة له في الإنتاج. في خط بيانات المستشفى، خط أنابيب قراءات CGM الذي ينهار عند مواجهة بيانات مستشعر تالفة سيوقف معالجة جميع القراءات اللاحقة لكل المرضى في نفس الدفعة. سجل واحد سيء يفسد التشغيل بأكمله.

معالجة الاستثناءات لا تعني قمع الأخطاء — بل تعني تحديد ما يجب حدوثه حين يسوء شيء خارج سيطرتك. لديك ثلاثة خيارات: تسجيل والاستمرار، إعادة المحاولة، أو العودة إلى قيمة افتراضية آمنة.

أسوأ نمط مضاد: except Exception: pass. هذا يبتلع كل خطأ بصمت — بما في ذلك الأخطاء في كودك الخاص. لن تعرف أبداً أن الأنبوب معطوب حتى يكون تقرير المريض خاطئاً.

ثلاثة أنماط إنتاجية
كل نمط يخدم نوعاً مختلفاً من الفشل — اختر بوعي

أنماط ستستخدمها في كل مشروع خلفي

01
الإمساك والتسجيل والاستمرار
استخدم حين: عنصر واحد سيء لا يجب أن يوقف معالجة الباقي. فشل دائم — إعادة المحاولة لن تساعد.
def process_cgm_reading(raw_value: str) -> float | None: try: return float(raw_value) except ValueError: logger.error(f"قراءة CGM غير صالحة: {raw_value!r}") return None
أعد None للإشارة إلى فشل هذا العنصر دون إطلاق استثناء. المستدعي يقرر ماذا يفعل به.
استخدم logger.error() وليس print(). السجلات تبقى؛ الطباعة تختفي حين يُغلق الطرفية.
02
الإمساك وإعادة المحاولة
استخدم حين: الفشل مؤقت — انقطاع الشبكة، خدمة مثقلة مؤقتاً. إعادة المحاولة مع تأخير قصير ستنجح على الأرجح.
def fetch_fhir_patient(patient_id: str, retries: int = 3) -> dict: last_error = None for attempt in range(retries): try: response = requests.get(f"{FHIR_BASE}/Patient/{patient_id}", timeout=5) response.raise_for_status() return response.json() except (requests.ConnectionError, requests.Timeout) as e: last_error = e if attempt < retries - 1: time.sleep(2 ** attempt) # تراجع أسي: 1s, 2s, 4s raise last_error
التراجع الأسي (2 ** attempt) يحترم الخادم — لا يقصف خدمة تعاني بالفعل.
أعد المحاولة فقط للاستثناءات المؤقتة. خطأ 404 Not Found لن يصبح 200 OK بعد ثلاث محاولات.
03
الإمساك والعودة إلى قيمة افتراضية آمنة
استخدم حين: يمكن للنظام العمل بقيمة منخفضة لكن آمنة. بيانات شخصية مفقودة → استخدم خطاً أساسياً محافظاً للسكان.
def get_insulin_sensitivity(patient_id: str) -> float: try: profile = load_patient_profile(patient_id) return profile["isf"] except (KeyError, PatientNotFoundError): logger.warning(f"لا يوجد ملف ISF لـ {patient_id} — استخدام الافتراضي 2.0") return 2.0
في الكود الحرج أمنياً، يجب أن يكون الاحتياطي محافظاً، وليس متفائلاً. اعرف الآثار السريرية للقيمة الافتراضية.
سجّل بمستوى WARNING وليس ERROR — النظام متدهور لكنه لم يفشل.

أسئلة تدريبية
تختبر الحكم وليس مجرد استذكار الصياغة
النموذج الذهني
إمساك Exception يعني القول: "أعرف كيف أتعامل مع حرفياً كل ما يمكن أن يسوء." على الأرجح لا تعرف.
الإجابة
ValueError يعني "هذا التحويل أو التحقق المحدد فشل." أما Exception فيمسك كل شيء: أخطاء الشبكة، أخطاء الذاكرة، أخطاءك الإملائية. حين تمسك Exception، تُبتلع الأخطاء في كودك الخاص بصمت — سترى نتائج خاطئة دون معرفة السبب. امسك دائماً الاستثناء الأكثر تحديداً الذي يغطي الفشل المتوقع.
except Exception: pass رائحة كود. استخدمه فقط في أعلى البرنامج كمسجل ملاذ أخير، ليس داخل دالة أبداً.
النموذج الذهني
إعادة None تمرر المشكلة بصمت للمستدعي. إعادة الإطلاق تخبر المستدعي بصوت عالٍ: "هذه العملية فشلت."
الإجابة
أعد None حين يكون الفشل محلياً وقابلاً للإصلاح — عنصر واحد سيء في دفعة يمكن للمستدعي تخطيه. أعد الإطلاق حين يعني الفشل أن العملية بأكملها لا يمكن أن تنجح — قاعدة بيانات لا تتصل، ملف إعداد مطلوب غير موجود. استخدم raise للحفاظ على الـ traceback الأصلي، أو raise خطأ_جديد from خطأ_أصلي لإضافة سياق.
قاعدة عامة: إذا كان للمستدعي طريقة ذات معنى للتعافي، أعد sentinel. إذا لم يكن بإمكانه الاستمرار بدون هذه النتيجة، أعد الإطلاق.
النموذج الذهني
الكود بعد كتلة try/except غير مضمون التنفيذ إذا هرب استثناء غير معالَج. finally يعمل حتى عندها — إنه عقد مع وقت التشغيل.
الإجابة
يُنفَّذ finally بصرف النظر عما إذا كانت كتلة try نجحت، أو أطلقت استثناءً ممسوكاً، أو استثناءً غير ممسوك، أو وصلت لـ return. إذا لم تكن جلسة قاعدة البيانات في finally وأُطلق خطأ غير متوقع، تبقى الجلسة مفتوحة — محتجزةً قفل معاملة حتى انتهاء مهلة الاتصال. استخدم جمل with (مديري السياق) لنفس الضمان بكود أقل.
with open("file.txt") as f: يعادل try/finally مع f.close() — الملف يُغلق حتى لو أُطلق استثناء داخل كتلة with.
النموذج الذهني
المستوى الصحيح لإمساك استثناء هو المستوى الذي لديه سياق كافٍ للاستجابة بشكل هادف.
الإجابة
يعتمد على ما إذا كانت الاستدعاءات الثلاثة مستقلة. إذا كان فشل API واحد لا يجب أن يوقف الآخرين، امسك داخلياً لكل استدعاء وأعد نتائج جزئية. إذا كان يجب أن تنجح الثلاثة لتكون النتيجة صالحة، دع الاستثناءات تتصاعد — المستدعي يقرر إعادة المحاولة أو إعادة نتيجة منخفضة أو إخفاق الطلب. امسك الاستثناءات عند حدود الطبقات، وليس في كل دالة.
معمارية معالجة الاستثناءات جزء من تصميم النظام. فكر فيها في نفس وقت المسار السعيد، وليس لاحقاً.
النموذج الذهني
الاستثناءات المدمجة عامة. ValueError من محلل CGM وValueError من مُفرز JSON يبدوان متطابقين للمستدعي.
الإجابة
اشتق من Exception أو من مدمج أكثر تحديداً:
class InsufficientCGMDataError(ValueError): """يُطلق حين تكون قراءات CGM أقل مما يحتاجه الخوارزمية.""" pass class PatientNotFoundError(LookupError): def __init__(self, patient_id: str): super().__init__(f"المريض '{patient_id}' غير موجود") self.patient_id = patient_id
يستحق حين: يحتاج المستدعون للتمييز بين هذا الفشل وغيره من ValueError؛ حين يحمل الاستثناء بيانات خاصة بالنطاق؛ حين تريد توثيق نمط الفشل.
النموذج الذهني
أمسكت استثناءً منخفض المستوى وتريد إطلاق آخر عالي المستوى. بدون الربط تختفي السبب الأصلي من الـ traceback.
الإجابة
raise خطأ_عالي("سياق") from استثناء_أصلي يربط الاستثناءين. Python يطبع كليهما في الـ traceback مع "السبب المباشر لما سبق هو..." بدون الربط تفقد سبب SQLAlchemy ولا تعرف لماذا فشل استدعاء قاعدة البيانات فعلاً. استخدم from None لإخفاء الاستثناء الأصلي صراحةً حين يكشف تفاصيل تنفيذ.
لف استثناءات SQLAlchemy في استثناءات النطاق يبقي طبقة الخدمة مستقلة عن الـ ORM. المستدعي يرى أخطاء النطاق، وليس داخليات قاعدة البيانات.
