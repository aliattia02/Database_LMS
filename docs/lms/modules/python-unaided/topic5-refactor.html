What Refactoring Actually Means
Same behaviour, better structure — not rewriting, not adding features

Refactoring is not rewriting

Refactoring means changing the internal structure of code without changing its external behaviour. The tests pass before and after. The function returns the same result. What changes is how the code is organised — how easy it is to read, to test, and to extend.

The target for this topic is your own insulin calculation pipeline from past work. You know this code: you designed the logic, you understand the edge cases, and you know what the inputs and outputs mean. That domain knowledge makes it the perfect subject for an unaided refactor. You are not guessing at intent — you are expressing something you already understand in better Python.

The goal is not elegance for its own sake. Every change you make must answer the question: "Is this easier to read, test, or extend?" If you can't answer yes, revert it.

Five Refactoring Moves
Apply these in order to a real function from your codebase

Work through them on the actual code

Take your insulin calculation service (or a Python equivalent from your own codebase) and apply each move. Write the before and after side-by-side in a comment.

01
Add type hints to every function signature
Type hints do not change runtime behaviour, but they make the function's contract explicit. They also let PyCharm warn you when you pass the wrong type.
Before
def calculate_meal_dose(carbs, icr, bg, target): correction = (bg - target) / isf return carbs / icr + correction
After
def calculate_meal_dose( carbs: float, # grams icr: float, # g/unit bg: float, # mmol/L target: float, # mmol/L isf: float, # mmol/L per unit ) -> float: # units correction = (bg - target) / isf return carbs / icr + correction
Include units in the comment next to each parameter — mmol/L vs mg/dL is a safety-critical distinction.
The return type annotation tells the caller what to expect — and forces you to think about what the function actually guarantees.
02
Write a docstring: one sentence, parameters, return, raises
The docstring is the contract between the function and its callers. Write it before you refactor the body — it forces you to articulate what the function is supposed to do.
def calculate_meal_dose( carbs: float, icr: float, bg: float, target: float, isf: float ) -> float: """ Calculate total insulin dose for a meal, including a blood glucose correction. Parameters ---------- carbs : grams of carbohydrate in the meal icr : insulin-to-carb ratio (grams of carb covered by 1 unit of insulin) bg : current blood glucose (mmol/L) target : target blood glucose (mmol/L) isf : insulin sensitivity factor (mmol/L drop per 1 unit of insulin) Returns ------- float Total insulin dose in units. May be negative if bg is well below target (caller must clamp to 0 before delivery). Raises ------ ValueError If icr or isf are zero or negative (would cause division by zero or clinically nonsensical correction direction). """ ...
The "Raises" section is the most important part — it makes the exception handling contract explicit, so callers know what they must handle.
Notice the note about negative return values: documenting edge-case behaviour prevents the caller from silently delivering a negative insulin dose.
03
Split: one function, one concept
If a function does two things, split it. The test for "one thing": can you name it with a single verb and noun, without using "and"?
Before — two concepts in one
def get_dose(patient_id, meal_carbs): profile = db.get_patient_profile(patient_id) bg = cgm.get_latest_reading(patient_id) correction = (bg - profile["target"]) / profile["isf"] meal = meal_carbs / profile["icr"] total = correction + meal return max(0, total)
After — three pure functions
def correction_dose(bg: float, target: float, isf: float) -> float: return (bg - target) / isf def meal_dose(carbs: float, icr: float) -> float: return carbs / icr def total_dose(carbs: float, profile: dict, bg: float) -> float: raw = (meal_dose(carbs, profile["icr"]) + correction_dose(bg, profile["target"], profile["isf"])) return max(0.0, raw)
correction_dose and meal_dose are now pure functions (no side effects, no database calls). You can test them with a single assert statement — no mocking needed.
The clamping (max(0.0, raw)) is a business rule. By putting it in total_dose, you document that negative doses are possible mathematically but impossible to deliver.
04
Add guard clauses for invalid inputs
Fail loudly at the entry point, not silently in the middle of a calculation. A negative ICR would give a negative dose; a zero ISF causes a division-by-zero crash. Catch these before any arithmetic runs.
def calculate_meal_dose( carbs: float, icr: float, bg: float, target: float, isf: float ) -> float: """...""" if icr <= 0: raise ValueError(f"ICR must be positive, got {icr}") if isf <= 0: raise ValueError(f"ISF must be positive, got {isf}") if carbs < 0: raise ValueError(f"Carbs cannot be negative, got {carbs}") correction = (bg - target) / isf return carbs / icr + correction
Guard clauses at the top read as a specification: "this function requires these preconditions." They are documentation that executes.
Raise ValueError for caller errors (wrong input type/value), not RuntimeError. The caller passed something invalid — that is a value problem.
Include the actual value in the message (got {icr}). When a test fails at 3 AM you will thank yourself.
05
Write assert-based tests for each function
Not pytest yet — just assert statements at the bottom of the file. The point is to verify your refactored functions produce the same output as the original for known inputs.
if __name__ == "__main__": # Correction only: bg=9.0, target=5.5, isf=2.0 → (9-5.5)/2 = 1.75 units assert correction_dose(bg=9.0, target=5.5, isf=2.0) == 1.75 # Meal only: 60g carbs, ICR=10 g/unit → 6 units assert meal_dose(carbs=60.0, icr=10.0) == 6.0 # Total: 60g meal + bg=9, target=5.5, isf=2, icr=10 → 7.75 units profile = {"icr": 10.0, "isf": 2.0, "target": 5.5} result = total_dose(carbs=60.0, profile=profile, bg=9.0) assert abs(result - 7.75) < 1e-9, f"Expected 7.75, got {result}" # Clamp: negative raw dose → 0.0 assert total_dose(carbs=0.0, profile=profile, bg=4.0) == 0.0 # Guard: invalid ICR try: meal_dose(carbs=60.0, icr=0.0) assert False, "Should have raised ValueError" except ValueError: pass print("All assertions passed.")
The floating-point test uses abs(result - expected) < 1e-9 instead of == because floating-point arithmetic is not exact. Never use == for float comparisons in tests.
Test the guard clauses too — confirm that passing invalid inputs raises the exception you documented. If the test doesn't verify the guard, the guard isn't tested.
Once these pass, you have verified that refactoring preserved the original behaviour. You can now extract these to pytest without writing new test logic.

Practice Questions
These test your understanding of why clean code is worth writing
Mental model
A testable function takes inputs and returns outputs. Given the same inputs, it always returns the same output. A function that depends on global state or calls a database might return different results each time — and you cannot test it without setting up the entire system.
The Answer
A function is testable when its output depends only on its inputs — no database reads, no global variables, no datetime.now(), no random(). This is called a pure function. You can test it with one line: assert f(x) == expected. Global state breaks this because the function's output changes depending on what happened before it was called — the test environment must replicate that state, or the test is unreliable. In practice: separate the data-fetching from the calculation. get_dose(patient_id) cannot be tested without a database. calculate_meal_dose(carbs, icr, bg, target, isf) can be tested anywhere.
The refactoring move in this topic (splitting database calls from calculations) is not just aesthetics — it is what makes the calculation independently testable.
In a typical backend service: functions that compute doses, check thresholds, or format messages should all be pure. Functions that read from MongoDB or call FHIR endpoints are the thin I/O boundary — test those with mocks.
Mental model
A function should do one thing. If you can name it with a verb and a noun without using "and", it probably does one thing. "Calculate and validate and log" is three things.
The Answer
SRP at the function level means: one function, one reason to change. get_dose() that fetches patient data, calculates the dose, and logs the result will change for three independent reasons — if the database schema changes, if the calculation logic changes, or if the logging format changes. Split it into fetch_patient_profile(), calculate_meal_dose(), and log_dose_event(). Now each function has exactly one reason to change — and each can be tested independently.
The "one reason to change" test is more useful than "does one thing" — it ties the principle to real maintenance costs, not just aesthetics.
Long functions are often a sign of multiple responsibilities, but length alone is not the test. A 40-line pure calculation function is fine. A 10-line function that fetches, calculates, and logs is not.
Mental model
Python's dynamic typing means bugs surface at runtime, often in production, not at write time. Type hints move those errors earlier — to the moment you write the call.
The Answer
Type hints serve three purposes. First, they are documentation that executes — when you read def f(bg: float) -> float you know exactly what the function expects and returns, without reading the body. Second, PyCharm's static analysis uses them to flag mismatches before you run the code — passing a str where a float is expected becomes a warning, not a runtime error. Third, they make refactoring safer — if you change a parameter type, PyCharm shows every call site that would break. In a medical system where passing mg/dL to a function expecting mmol/L causes incorrect dosing, type hints are not optional aesthetics.
Use from __future__ import annotations in Python <3.10 to use newer type syntax (float | None) in older versions.
Optional[float] (pre-3.10) and float | None (3.10+) are equivalent. Both mean the function might return a value or nothing — the caller must check.
Mental model
A guard clause rejects bad input at the entry. Defensive programming wraps every operation in checks "just in case" — it produces code that is harder to read and hides bugs by silently correcting inputs the caller should have validated.
The Answer
A guard clause (if icr <= 0: raise ValueError(...)) is explicit: it states the precondition, fails immediately with a clear message, and keeps the happy path unindented. Defensive programming tries to handle every possible bad input with silent corrections — clamping negatives to zero, converting strings to floats, using default values when keys are missing — resulting in code that never clearly fails and makes it hard to know whether your inputs were even correct. The right approach: validate at boundaries (API endpoints, database reads) and then trust the data inside the system. Guard clauses are the boundary check, not internal paranoia.
Guard clauses at the top keep the function body readable — the main logic is not buried in nested if/else.
Defensive programming that silently corrects inputs is harder to debug: the function succeeds, but with wrong data, and you find the error three steps later with no useful context.
Mental model
Tests can only tell you what they test. A passing test suite is evidence of correctness for the cases it covers — not proof for all cases.
The Answer
The refactor is correct for the cases your tests cover. Whether it is correct in general depends on how complete your tests are. If your tests only cover the happy path (valid inputs, expected outputs), the refactored function might silently change behaviour for edge cases — zero carbs, a very high blood glucose, a patient with no profile. Good refactoring practice: write tests for edge cases before refactoring, so you have a safety net. If you didn't, add them now and confirm they pass on the original too. The point of tests is not to show the code works — it is to tell you when it stops working.
Edge cases to always test for calculations: zero inputs, negative inputs, boundary values (exactly at target BG), very large inputs (100+ units).
The refactor added guard clauses. If the original didn't raise on invalid input and the refactored version does, the behaviour changed — that is a breaking change, not a refactor. Document the decision.
Mental model
A docstring is the function's public contract. It describes what the function does, what it needs, and what it promises. It never describes how — that is what the code is for.
The Answer
A docstring should answer: what does this function do, what are the inputs (types and units), what does it return, and what exceptions can it raise. It should not contain implementation details — "divides carbs by ICR and adds correction" is narrating the code. A reader can read the code; the docstring should give them the context they need to use the function correctly without reading the body. Docstrings that describe implementation rot when the code changes but the docstring doesn't — they become active misinformation.
"May return negative if BG is below target (caller must clamp)" is valuable docstring content — it is a contract detail the caller must act on, not visible from the type signature.
Use NumPy-style or Google-style docstrings consistently within a project. PyCharm renders them in the Quick Documentation panel (hover + ⌃J or Ctrl+Q).

Was Refactoring wirklich bedeutet
Gleiches Verhalten, bessere Struktur — kein Neuschreiben, keine neuen Features

Refactoring ist kein Neuschreiben

Refactoring bedeutet, die interne Struktur von Code zu ändern, ohne sein externes Verhalten zu ändern. Die Tests bestehen vor und nach dem Refactoring. Die Funktion gibt dasselbe Ergebnis zurück. Was sich ändert, ist die Organisation — wie leicht der Code zu lesen, zu testen und zu erweitern ist.

Das Ziel für dieses Thema ist deine eigene Insulin-Berechnungspipeline aus früherer Arbeit. Du kennst diesen Code: du hast die Logik entworfen, du verstehst die Randfälle, und du weißt, was die Eingaben und Ausgaben bedeuten. Dieses Domänenwissen macht es zum perfekten Gegenstand für ein unbegleitetes Refactoring.

Das Ziel ist nicht Eleganz um ihrer selbst willen. Jede Änderung muss die Frage beantworten: "Ist das leichter zu lesen, zu testen oder zu erweitern?" Wenn du nicht Ja sagen kannst, mache sie rückgängig.

Fünf Refactoring-Schritte
Wende diese in dieser Reihenfolge auf eine echte Funktion aus deiner Codebasis an

Arbeite sie mit echtem Code durch

01
Type-Hints zu jeder Funktionssignatur hinzufügen
Type-Hints ändern das Laufzeitverhalten nicht, aber sie machen den Vertrag der Funktion explizit und erlauben PyCharm, falsche Typen zu warnen.
def calculate_meal_dose( carbs: float, # Gramm Kohlenhydrate icr: float, # g/Einheit bg: float, # mmol/L target: float, # mmol/L isf: float, # mmol/L pro Einheit ) -> float: # Insulineinheiten ...
Einheiten im Kommentar neben jedem Parameter angeben — mmol/L vs mg/dL ist ein sicherheitskritischer Unterschied.
02
Docstring schreiben: ein Satz, Parameter, Rückgabe, Raises
Der Docstring ist der Vertrag zwischen der Funktion und ihren Aufrufern. Schreibe ihn, bevor du den Rumpf refaktorierst — er zwingt dich, zu artikulieren, was die Funktion tun soll.
def calculate_meal_dose(...) -> float: """ Berechnet die gesamte Insulindosis für eine Mahlzeit, einschließlich einer Blutzuckerkorrektur. Parameters ---------- carbs : Gramm Kohlenhydrate in der Mahlzeit icr : Insulin-Kohlenhydrat-Verhältnis (g Kohlenhydrate pro Insulineinheit) ... Raises ------ ValueError Wenn icr oder isf null oder negativ sind. """
Der "Raises"-Abschnitt ist der wichtigste Teil — er macht den Exception-Vertrag explizit, damit Aufrufer wissen, was sie behandeln müssen.
03
Aufteilen: eine Funktion, ein Konzept
Wenn eine Funktion zwei Dinge tut, teile sie auf. Der Test für "eine Sache": kannst du sie mit einem Verb und einem Substantiv benennen, ohne "und" zu verwenden?
def correction_dose(bg: float, target: float, isf: float) -> float: return (bg - target) / isf def meal_dose(carbs: float, icr: float) -> float: return carbs / icr def total_dose(carbs: float, profile: dict, bg: float) -> float: raw = meal_dose(carbs, profile["icr"]) + correction_dose( bg, profile["target"], profile["isf"]) return max(0.0, raw)
correction_dose und meal_dose sind jetzt reine Funktionen. Du kannst sie mit einer einzigen assert-Anweisung testen — kein Mocking nötig.
04
Guard-Klauseln für ungültige Eingaben hinzufügen
Fail laut am Eintrittspunkt, nicht still in der Mitte einer Berechnung. Fange diese Fehler ab, bevor eine Arithmetik ausgeführt wird.
def calculate_meal_dose(...) -> float: if icr <= 0: raise ValueError(f"ICR muss positiv sein, erhalten: {icr}") if isf <= 0: raise ValueError(f"ISF muss positiv sein, erhalten: {isf}") if carbs < 0: raise ValueError(f"Kohlenhydrate können nicht negativ sein, erhalten: {carbs}") ...
Guard-Klauseln am Anfang lesen sich wie eine Spezifikation: "Diese Funktion setzt diese Vorbedingungen voraus."
Tatsächlichen Wert in der Nachricht angeben (erhalten: {icr}). Wenn ein Test um 3 Uhr nachts fehlschlägt, wirst du dir selbst dankbar sein.
05
Assert-basierte Tests für jede Funktion schreiben
Noch kein pytest — nur assert-Anweisungen am Ende der Datei. Der Punkt ist, zu verifizieren, dass deine refaktorierten Funktionen dieselbe Ausgabe wie das Original für bekannte Eingaben produzieren.
if __name__ == "__main__": assert correction_dose(bg=9.0, target=5.5, isf=2.0) == 1.75 assert meal_dose(carbs=60.0, icr=10.0) == 6.0 profile = {"icr": 10.0, "isf": 2.0, "target": 5.5} result = total_dose(carbs=60.0, profile=profile, bg=9.0) assert abs(result - 7.75) < 1e-9, f"Erwartet 7.75, erhalten {result}" print("Alle Assertions bestanden.")
Verwende abs(result - expected) < 1e-9 anstatt == für Float-Vergleiche — Fließkomma-Arithmetik ist nicht exakt.

Übungsfragen
Testen dein Verständnis, warum sauberer Code es wert ist ihn zu schreiben
Mentales Modell
Eine testbare Funktion nimmt Eingaben und gibt Ausgaben zurück. Bei denselben Eingaben gibt sie immer dasselbe Ergebnis zurück.
Die Antwort
Eine Funktion ist testbar, wenn ihre Ausgabe nur von ihren Eingaben abhängt — keine Datenbanklesevorgänge, keine globalen Variablen, kein datetime.now(). Dies nennt sich reine Funktion. Du kannst sie mit einer Zeile testen: assert f(x) == erwartet. Globaler Zustand bricht das, weil die Ausgabe der Funktion sich je nach vorherigen Ereignissen ändert — die Testumgebung muss diesen Zustand replizieren. In der Praxis: trenne die Datenbeschaffung von der Berechnung. get_dose(patient_id) kann ohne Datenbank nicht getestet werden. calculate_meal_dose(carbs, icr, bg, target, isf) kann überall getestet werden.
Die Refactoring-Schritte in diesem Thema (Trennung von Datenbankaufrufen und Berechnungen) sind nicht nur Ästhetik — sie machen die Berechnung unabhängig testbar.
Mentales Modell
Eine Funktion sollte eine Sache tun. Wenn du sie mit einem Verb und einem Substantiv benennen kannst, ohne "und" zu verwenden, tut sie wahrscheinlich eine Sache.
Die Antwort
SRP auf Funktionsebene bedeutet: eine Funktion, ein Grund zum Ändern. Eine get_dose() die Patientendaten abruft, die Dosis berechnet und das Ergebnis loggt, wird sich aus drei unabhängigen Gründen ändern — wenn sich das Datenbankschema ändert, wenn sich die Berechnungslogik ändert, oder wenn sich das Log-Format ändert. Teile sie in fetch_patient_profile(), calculate_meal_dose() und log_dose_event() auf. Jetzt hat jede Funktion genau einen Grund zum Ändern — und jede kann unabhängig getestet werden.
Der "ein Grund zum Ändern"-Test ist nützlicher als "eine Sache tun" — er verknüpft das Prinzip mit echten Wartungskosten.
Mentales Modell
Pythons dynamische Typisierung bedeutet, dass Fehler zur Laufzeit auftauchen, oft in der Produktion. Type-Hints verschieben diese Fehler früher — zum Zeitpunkt des Schreibens.
Die Antwort
Type-Hints erfüllen drei Zwecke. Erstens sind sie Dokumentation die ausgeführt wird — def f(bg: float) -> float teilt sofort mit, was die Funktion erwartet und zurückgibt. Zweitens verwendet PyCharms statische Analyse sie, um Fehler vor der Codeausführung zu markieren. Drittens machen sie Refactoring sicherer — wenn du einen Parametertyp änderst, zeigt PyCharm jede Aufrufstelle, die kaputt gehen würde. In einem medizinischen System, wo mg/dL statt mmol/L zu falschem Dosieren führt, sind Type-Hints keine optionale Ästhetik.
Optional[float] und float | None (3.10+) sind äquivalent. Beide bedeuten, dass die Funktion möglicherweise nichts zurückgibt — der Aufrufer muss das prüfen.
Mentales Modell
Eine Guard-Klausel weist schlechte Eingaben am Eintrittspunkt zurück. Defensive Programmierung wickelt jede Operation in Prüfungen ein "für alle Fälle" — sie versteckt Fehler durch stilles Korrigieren.
Die Antwort
Eine Guard-Klausel (if icr <= 0: raise ValueError(...)) ist explizit: sie benennt die Vorbedingung, schlägt sofort mit einer klaren Nachricht fehl, und hält den Happy Path uneingerückt. Defensive Programmierung versucht jeden möglichen schlechten Input mit stillen Korrekturen zu behandeln — Negative auf Null klemmen, Strings in Floats umwandeln — was Code produziert, der nie klar scheitert und es schwer macht zu wissen, ob die Eingaben überhaupt korrekt waren. Validiere an Grenzen; vertraue den Daten danach.
Guard-Klauseln oben halten den Funktionskörper lesbar — die Hauptlogik ist nicht in verschachtelten if/else vergraben.
Mentales Modell
Tests können dir nur sagen, was sie testen. Eine bestandene Testsuite ist Beweis für die Korrektheit der abgedeckten Fälle — kein Beweis für alle Fälle.
Die Antwort
Das Refactoring ist für die von deinen Tests abgedeckten Fälle korrekt. Ob es generell korrekt ist, hängt von der Vollständigkeit deiner Tests ab. Wenn deine Tests nur den Happy Path abdecken, könnte die refaktorierte Funktion bei Randfällen still das Verhalten ändern — null Kohlenhydrate, sehr hoher Blutzucker, ein Patient ohne Profil. Gute Refactoring-Praxis: schreibe Tests für Randfälle bevor du refaktorierst, damit du ein Sicherheitsnetz hast. Der Zweck von Tests ist nicht zu zeigen, dass Code funktioniert — sondern zu merken, wenn er aufhört zu funktionieren.
Randfälle für Berechnungen immer testen: null Eingaben, negative Eingaben, Grenzwerte, sehr große Eingaben.
Mentales Modell
Ein Docstring ist der öffentliche Vertrag der Funktion. Er beschreibt was sie tut, was sie benötigt und was sie verspricht. Er beschreibt nie wie — das ist Sache des Codes.
Die Antwort
Ein Docstring sollte antworten: was tut diese Funktion, was sind die Eingaben (Typen und Einheiten), was gibt sie zurück, und welche Exceptions kann sie auslösen. Er sollte keine Implementierungsdetails enthalten — "teilt Kohlenhydrate durch ICR und addiert Korrektur" erzählt den Code nach. Docstrings die Implementierung beschreiben werden falsch, wenn sich der Code ändert aber der Docstring nicht — sie werden aktive Fehlinformationen.
"Kann negativ sein wenn BG unter Zielwert (Aufrufer muss auf 0 klemmen)" ist wertvoller Docstring-Inhalt — es ist ein Vertragsdetail, das nicht aus der Typsignatur ersichtlich ist.

إعادة الهيكلة ليست إعادة كتابة

إعادة الهيكلة تعني تغيير البنية الداخلية للكود دون تغيير سلوكه الخارجي. الاختبارات تنجح قبل وبعد. الدالة تعيد نفس النتيجة. ما يتغير هو كيفية تنظيم الكود — كيف يسهل قراءته واختباره وتوسيعه.

الهدف لهذا الموضوع هو خط أنابيب حساب الإنسولين الخاص بك من عمل سابق. أنت تعرف هذا الكود: صممت المنطق، تفهم الحالات الحدية، وتعرف ما تعنيه المدخلات والمخرجات. هذه المعرفة بالنطاق تجعله الموضوع المثالي لإعادة هيكلة مستقلة.

الهدف ليس الأناقة لذاتها. كل تغيير يجب أن يجيب على السؤال: "هل هذا أسهل للقراءة أو الاختبار أو التوسيع؟" إذا لم تستطع الإجابة بنعم، تراجع عنه.

01
أضف تلميحات النوع لكل توقيع دالة
تلميحات النوع لا تغير سلوك وقت التشغيل، لكنها تجعل عقد الدالة صريحاً وتتيح لـ PyCharm تحذيرك عند تمرير نوع خاطئ.
def calculate_meal_dose( carbs: float, # غرام كربوهيدرات icr: float, # غ/وحدة bg: float, # mmol/L target: float, # mmol/L isf: float, # mmol/L لكل وحدة ) -> float: # وحدات إنسولين ...
أدرج الوحدات في التعليق بجانب كل معامل — mmol/L مقابل mg/dL تمييز حرج للسلامة.
02
اكتب docstring: جملة واحدة، المعاملات، الإرجاع، الاستثناءات
الـ docstring هو العقد بين الدالة ومستدعيها. اكتبه قبل إعادة هيكلة الجسم — يجبرك على التعبير عما تفترض أن تفعله الدالة.
def calculate_meal_dose(...) -> float: """ احسب إجمالي جرعة الإنسولين لوجبة، مع تصحيح السكر في الدم. Raises ------ ValueError إذا كان icr أو isf صفراً أو سالباً. """
قسم "Raises" هو الأهم — يجعل عقد معالجة الاستثناءات صريحاً حتى يعرف المستدعون ما يجب عليهم معالجته.
03
قسّم: دالة واحدة، مفهوم واحد
إذا كانت الدالة تفعل شيئين، قسّمها. اختبار "شيء واحد": هل يمكنك تسميتها بفعل واسم واحد دون استخدام "و"؟
def correction_dose(bg: float, target: float, isf: float) -> float: return (bg - target) / isf def meal_dose(carbs: float, icr: float) -> float: return carbs / icr def total_dose(carbs: float, profile: dict, bg: float) -> float: raw = meal_dose(carbs, profile["icr"]) + correction_dose( bg, profile["target"], profile["isf"]) return max(0.0, raw)
correction_dose وmeal_dose الآن دوال خالصة. يمكنك اختبارهما بجملة assert واحدة — لا حاجة لـ mocking.
04
أضف guard clauses للمدخلات غير الصالحة
افشل بصوت عالٍ عند نقطة الدخول، وليس بصمت في منتصف حساب. التقط هذه الأخطاء قبل تشغيل أي عمليات حسابية.
def calculate_meal_dose(...) -> float: if icr <= 0: raise ValueError(f"ICR يجب أن يكون موجباً، حصلنا على: {icr}") if isf <= 0: raise ValueError(f"ISF يجب أن يكون موجباً، حصلنا على: {isf}") if carbs < 0: raise ValueError(f"الكربوهيدرات لا يمكن أن تكون سالبة، حصلنا على: {carbs}") ...
Guard clauses في الأعلى تُقرأ كمواصفات: "هذه الدالة تتطلب هذه الشروط المسبقة."
05
اكتب اختبارات assert لكل دالة
ليس pytest بعد — فقط جمل assert في نهاية الملف. الهدف هو التحقق من أن دوالك المُعاد هيكلتها تنتج نفس المخرجات كالأصل للمدخلات المعروفة.
if __name__ == "__main__": assert correction_dose(bg=9.0, target=5.5, isf=2.0) == 1.75 assert meal_dose(carbs=60.0, icr=10.0) == 6.0 profile = {"icr": 10.0, "isf": 2.0, "target": 5.5} result = total_dose(carbs=60.0, profile=profile, bg=9.0) assert abs(result - 7.75) < 1e-9, f"متوقع 7.75، حصلنا على {result}" print("نجحت جميع الاختبارات.")
استخدم abs(result - expected) < 1e-9 بدلاً من == لمقارنات الـ float — الحساب الحاسوبي للأعداد العشرية ليس دقيقاً تماماً.

النموذج الذهني
الدالة القابلة للاختبار تأخذ مدخلات وتعيد مخرجات. بنفس المدخلات، تعيد دائماً نفس النتيجة.
الإجابة
الدالة قابلة للاختبار حين تعتمد مخرجاتها فقط على مدخلاتها — لا قراءات قاعدة بيانات، لا متغيرات عامة، لا datetime.now(). هذا يُسمى دالة خالصة. يمكنك اختبارها بسطر واحد: assert f(x) == متوقع. الحالة العامة تكسر هذا لأن مخرجات الدالة تتغير حسب ما حدث قبل استدعائها. عملياً: افصل جلب البيانات عن الحساب. get_dose(patient_id) لا يمكن اختباره بدون قاعدة بيانات. calculate_meal_dose(carbs, icr, bg, target, isf) يمكن اختباره في أي مكان.
خطوات إعادة الهيكلة في هذا الموضوع (فصل استدعاءات قاعدة البيانات عن الحسابات) ليست مجرد جماليات — هذا ما يجعل الحساب قابلاً للاختبار بشكل مستقل.
النموذج الذهني
يجب أن تفعل الدالة شيئاً واحداً. إذا كان بإمكانك تسميتها بفعل واسم واحد دون استخدام "و"، فعلى الأرجح تفعل شيئاً واحداً.
الإجابة
SRP على مستوى الدالة يعني: دالة واحدة، سبب واحد للتغيير. دالة get_dose() تجلب بيانات المريض وتحسب الجرعة وتسجل النتيجة ستتغير لثلاثة أسباب مستقلة. قسّمها إلى fetch_patient_profile() وcalculate_meal_dose() وlog_dose_event(). الآن لكل دالة سبب واحد بالضبط للتغيير — وكل منها يمكن اختبارها بشكل مستقل.
اختبار "سبب واحد للتغيير" أكثر فائدة من "شيء واحد" — يربط المبدأ بتكاليف الصيانة الحقيقية.
النموذج الذهني
الكتابة الديناميكية في Python تعني ظهور الأخطاء في وقت التشغيل، غالباً في الإنتاج. تلميحات النوع تنقل هذه الأخطاء في وقت أبكر — لحظة كتابة الاستدعاء.
الإجابة
تلميحات النوع تخدم ثلاثة أغراض. أولاً، هي توثيق قابل للتنفيذ — def f(bg: float) -> float يخبرك بالضبط بما تتوقعه الدالة وتعيده. ثانياً، التحليل الساكن في PyCharm يستخدمها للإشارة إلى التناقضات قبل تشغيل الكود. ثالثاً، تجعل إعادة الهيكلة أأمن — إذا غيرت نوع معامل، يُظهر PyCharm كل موقع استدعاء سيتكسر. في نظام طبي حيث تمرير mg/dL لدالة تتوقع mmol/L يسبب جرعة خاطئة، تلميحات النوع ليست جماليات اختيارية.
Optional[float] وfloat | None (3.10+) متكافئان. كلاهما يعني أن الدالة قد لا تعيد شيئاً — المستدعي يجب أن يتحقق.
النموذج الذهني
guard clause يرفض المدخلات السيئة عند نقطة الدخول. البرمجة الدفاعية تلف كل عملية في فحوصات "على كل حال" — تخفي الأخطاء بتصحيح المدخلات بصمت.
الإجابة
guard clause (if icr <= 0: raise ValueError(...)) صريح: يذكر الشرط المسبق، يفشل فوراً برسالة واضحة، ويبقي المسار السعيد غير مسنّن. البرمجة الدفاعية تحاول معالجة كل مدخل سيء بتصحيحات صامتة — تثبيت السالبة على الصفر، تحويل الـ strings إلى floats — مما ينتج كوداً لا يفشل أبداً بوضوح ويجعل من الصعب معرفة ما إذا كانت المدخلات صحيحة أصلاً. اعمل على التحقق عند الحدود ثم ثق بالبيانات داخلياً.
Guard clauses في الأعلى تحافظ على قابلية قراءة جسم الدالة — المنطق الرئيسي ليس مدفوناً في if/else متداخلة.
النموذج الذهني
الاختبارات يمكنها فقط إخبارك بما تختبره. مجموعة اختبارات ناجحة هي دليل على صحة الحالات المغطاة — وليس دليلاً على جميع الحالات.
الإجابة
إعادة الهيكلة صحيحة للحالات التي تغطيها اختباراتك. إذا كانت اختباراتك تغطي فقط المسار السعيد، فقد تغير الدالة المُعاد هيكلتها بصمت السلوك لحالات حدية — صفر كربوهيدرات، سكر دم مرتفع جداً، مريض بدون ملف. ممارسة إعادة الهيكلة الجيدة: اكتب اختبارات للحالات الحدية قبل إعادة الهيكلة. الهدف من الاختبارات ليس إظهار أن الكود يعمل — بل معرفة متى يتوقف عن العمل.
حالات حدية يجب اختبارها دائماً للحسابات: مدخلات صفر، مدخلات سالبة، قيم حدودية، مدخلات كبيرة جداً.
النموذج الذهني
الـ docstring هو العقد العام للدالة. يصف ماذا تفعل، ماذا تحتاج، وماذا تعد. لا يصف أبداً كيف — ذلك هو دور الكود.
الإجابة
يجب أن يجيب الـ docstring على: ماذا تفعل هذه الدالة، ما هي المدخلات (أنواع ووحدات)، ماذا تعيد، وأي استثناءات يمكنها إطلاق. يجب ألا يحتوي على تفاصيل التنفيذ — "يقسم الكربوهيدرات على ICR ويضيف التصحيح" يسرد الكود. docstrings التي تصف التنفيذ تصبح خاطئة حين يتغير الكود لكن الـ docstring لا — تصبح معلومات مغلوطة نشطة.
"قد يعيد سالباً إذا كان السكر تحت الهدف (المستدعي يجب التثبيت على صفر)" محتوى قيّم في الـ docstring — تفصيل عقدي غير مرئي من توقيع النوع.
