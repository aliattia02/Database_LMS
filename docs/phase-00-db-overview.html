Section 01 · Foundation
What Is a Database?
A database is an organised collection of data that can be stored, retrieved, and modified efficiently. But that definition fits a lot of things — including a spreadsheet. So what makes a database different?
Why Not Just Use a Spreadsheet? ▾
FeatureSpreadsheet (Excel / Sheets)Database
Data volume~1M rows before it breaksBillions of rows, no problem
Simultaneous users1 user editing safelyThousands of concurrent users
Data integrityYou can type anything anywhereEnforced types, constraints, foreign keys
Speed of queriesSlow on large data (filters, VLOOKUPs)Indexed queries in milliseconds
RelationshipsVLOOKUP / manual linkingNative JOINs across any tables
Automation / API accessFragile / not designed for itBuilt for apps, APIs, scripts
Crash recoveryFile corruption riskWAL, transactions, ACID
Security / access controlFile-level onlyRow-level, column-level permissions
Mental Model

Think of a spreadsheet as a personal notebook and a database as a professional filing system with staff, locks, an index, and a security desk. Both store information — but they serve completely different scales and purposes.

The Three Core Database Guarantees ▾
Storage — Data persists reliably
A database stores data on disk so it survives power loss, crashes, and restarts. Unlike a variable in a running program, database data is durable.
Retrieval — Data is queryable
You don't scan every row to find what you need. Databases use indexes, query planners, and structured query languages so you can find one record in billions in milliseconds.
Integrity — Data is correct
Rules (constraints, types, foreign keys) prevent bad data from entering in the first place. A database rejects age = "hello". A spreadsheet accepts it.
A Brief History — Why So Many Database Types? ▾
1960s
Hierarchical & Network DBs — IBM's IMS. Data stored as trees or graphs. Fast reads, rigid structure. Used in banks and airlines. Still running in some places today.
1970s
The Relational Model (Edgar Codd, IBM) — data in tables, rows, columns. Any data related by keys. The foundation of everything you'll learn in this track.
1979–1995
SQL becomes the standard — Oracle, IBM DB2, Sybase, MySQL (1995). Relational databases dominate enterprise computing.
2000s
Web scale breaks relational assumptions — Google, Amazon, Facebook hit billions of users and petabytes of unstructured data. Relational databases struggle to scale horizontally.
2009
NoSQL movement — MongoDB, Cassandra, Redis emerge. "Not Only SQL" — flexible schemas, horizontal scaling, new data models for new problems.
2010s–Now
Convergence — SQL databases add JSON support. NoSQL databases add SQL-like query languages. NewSQL (CockroachDB, TiDB) combines both. The borders blur — but the fundamentals remain.
The Takeaway

Different database types weren't invented to be confusing — each solved a real problem that existing systems couldn't handle. Understanding what problem each type solves is how you pick the right tool.

Section 02 · Core Track
Relational Databases & SQL
Relational databases organise data into tables with rows and columns. They are linked by keys and queried with SQL. This is the dominant database model — it runs banks, hospitals, e-commerce, and almost every enterprise system on earth.
The Relational Model — Core Concepts ▾
ConceptWhat It IsWhy It Matters
Table (Relation)Rows and columns, like a spreadsheet sheetThe fundamental unit of storage
Row (Tuple)One record — one flight, one customer, one orderEach row is one real-world entity
Column (Attribute)One property with an enforced data typePrevents bad data at the schema level
Primary KeyUnique identifier for each row — never null, never duplicateYou can always find and reference exactly one row
Foreign KeyA column referencing another table's primary keyLinks tables together — enforces referential integrity
SchemaThe blueprint — which tables exist, what columns, what typesDefined upfront; changes are controlled migrations
ACIDAtomicity, Consistency, Isolation, DurabilityGuarantees that transactions are safe and reliable
SQLStructured Query Language — the language of all relational DBsLearn once, works on MySQL, MariaDB, PostgreSQL, Oracle…
How Tables Link Together ▾
The Core Idea

Instead of one giant table with duplicated data, you split data into smaller, focused tables and link them with keys. This is the "relational" in relational database.

SQL — three linked tables
-- Table 1: passengers (master data) CREATE TABLE passengers ( passenger_id INT AUTO_INCREMENT PRIMARY KEY, full_name VARCHAR(100) NOT NULL, email VARCHAR(255) UNIQUE NOT NULL ); -- Table 2: flights (master data) CREATE TABLE flights ( flight_id INT AUTO_INCREMENT PRIMARY KEY, flight_number CHAR(8) NOT NULL, departure_at DATETIME NOT NULL ); -- Table 3: bookings (links passengers ↔ flights) CREATE TABLE bookings ( booking_id INT AUTO_INCREMENT PRIMARY KEY, passenger_id INT NOT NULL, flight_id INT NOT NULL, seat CHAR(4), FOREIGN KEY (passenger_id) REFERENCES passengers(passenger_id), FOREIGN KEY (flight_id) REFERENCES flights(flight_id) );
Why split? The No-Duplication Rule

If you stored the passenger's name and email on every booking row, updating an email means updating potentially thousands of rows — and one missed update causes corrupted data. Split into separate tables and you update in exactly one place.

ACID — The Four Guarantees That Make Databases Trustworthy ▾
A — Atomicity: All or nothing
A transaction either completes fully or rolls back completely. If you're transferring money and the debit succeeds but the credit fails — the entire transaction is undone. No partial writes.
C — Consistency: Valid state to valid state
A transaction can only bring the database from one valid state to another. Rules (constraints, foreign keys) are never violated. You can never end up with a booking that references a non-existent passenger.
I — Isolation: Concurrent transactions don't interfere
Multiple users running queries simultaneously see consistent results. Transaction A's in-progress work is invisible to Transaction B until A commits. No dirty reads.
D — Durability: Committed data survives crashes
Once a transaction commits, that data is written to disk (via WAL — write-ahead log). Power failure, server crash — committed data is not lost.
ACID vs NoSQL

Most NoSQL databases sacrifice some ACID guarantees in exchange for speed and horizontal scalability. This is a deliberate trade-off — but it means NoSQL databases are not suitable for financial transactions, healthcare records, or any system where partial writes are catastrophic.

SQL — One Language, Many Databases ▾
What SQL is

SQL (Structured Query Language) is the standard language for interacting with relational databases. It was standardised by ANSI in 1986. Every major relational database speaks SQL — MySQL, MariaDB, PostgreSQL, Oracle, SQL Server. Learn it once; the core transfers everywhere.

SQL GroupCommandsWhat They Do
DML Data ManipulationSELECT, INSERT, UPDATE, DELETERead and write actual data rows
DDL Data DefinitionCREATE, ALTER, DROP, TRUNCATEDefine and change the structure (schema)
DCL Data ControlGRANT, REVOKEManage user permissions
TCL Transaction ControlBEGIN, COMMIT, ROLLBACK, SAVEPOINTControl multi-step transactions
Relational DB Strengths & Limitations ▾
✓ Strengths
Strong data integrity via constraints and foreign keys
ACID transactions — safe for financial and medical data
Powerful querying with JOINs, aggregations, subqueries
Mature — 50+ years of tools, optimisation, documentation
Universal SQL standard — skills are highly portable
✗ Limitations
Rigid schema — adding columns requires migrations
Vertical scaling by default (bigger server, not more servers)
Poor fit for unstructured data (documents, blobs, graphs)
Complex JOINs can be slow without careful index design
Not optimal for real-time or massive write throughput
Section 03 · Awareness
NoSQL Database Types
"NoSQL" means "Not Only SQL" — it's not one thing but a family of very different database models, each designed for a specific kind of problem. You need to know they exist and what each is good at, even if this track focuses on SQL.
The Four Main NoSQL Families ▾
TypeData ModelBest ForLeading Databases
Document JSON/BSON documents — nested, flexible structure Content, user profiles, product catalogues, APIs MongoDB, CouchDB, Firestore
Key-Value Simple key → value pairs, like a giant dictionary Caching, sessions, real-time leaderboards, queues Redis, DynamoDB, Memcached
Wide-Column Tables but columns can vary per row, optimised for writes Time-series, IoT data, analytics at massive scale Cassandra, HBase, ScyllaDB
Graph Nodes and edges — relationships are first-class citizens Social networks, fraud detection, recommendation engines Neo4j, Amazon Neptune, ArangoDB
Document Databases — MongoDB, Firestore ▾
The Model

Data is stored as documents — JSON-like objects with nested arrays and sub-documents. No fixed schema. Two documents in the same "collection" can have completely different fields.

SQL — Relational
-- Three separate tables, linked by IDs passengers: id, name, email bookings: id, passenger_id, flight_id flights: id, number, departure -- To get a booking with passenger + flight: SELECT * FROM bookings JOIN passengers USING (passenger_id) JOIN flights USING (flight_id);
JSON — MongoDB Document
{ "_id": "booking_xyz", "passenger": { "name": "Ali Schmidt", "email": "ali@example.com" }, "flight": { "number": "LH441", "departure": "2025-06-01T14:00Z" }, "seat": "12A", "extras": ["meal", "wifi"] }
When to choose Document DBs
  • Data structure varies per record
  • You read one entity's full data together (no JOINs needed)
  • Schema needs to evolve rapidly (startup, MVP, content platform)
  • Hierarchical or nested data is natural (order with line items)
When NOT to choose Document DBs

Financial systems, healthcare records, anything where ACID across multiple documents is critical. MongoDB has multi-document transactions, but they're slower and still secondary to the document model.

Key-Value Databases — Redis, DynamoDB ▾
The Model

The simplest database model: a giant in-memory dictionary. You SET a key to a value. You GET a key. Extremely fast. Redis keeps data in RAM — reads and writes in microseconds.

Redis commands
# Store a session SET session:user_1042 "{'name':'Ali','role':'admin'}" EX 3600 # Retrieve it GET session:user_1042 # Increment a counter (e.g. page views) INCR pageviews:landing # Leaderboard — sorted set ZADD leaderboard 95.5 "Ali" ZRANGE leaderboard 0 9 REV WITHSCORES
Use CaseWhy Redis
Session storageMicrosecond reads, TTL expiry built in
API rate limitingAtomic INCR + TTL per key
Caching DB query resultsServe from RAM, skip the database entirely for hot data
Pub/Sub messagingReal-time event broadcasting between services
Leaderboards / rankingsSorted sets with O(log n) operations
Wide-Column — Cassandra ▾
The Model

Looks like a table but each row can have different columns. Designed for massive write throughput across many servers (no single master). Apache Cassandra powers Netflix, Apple iCloud, Discord (trillions of messages).

FeatureDetail
Write throughputMillions of writes/second across a cluster
Horizontal scalingAdd nodes — linear performance increase
No master nodeNo single point of failure
CQLCassandra Query Language — SQL-like syntax, very limited JOINs
Best forTime-series (IoT sensors, logs), analytics append-only workloads
Graph Databases — Neo4j ▾
The Model

Data is stored as nodes (entities) and edges (relationships). Relationships are first-class — traversing millions of connections is as fast as accessing a single row. SQL JOINs get exponentially slower with depth; graph queries don't.

ProblemWhy Graph Wins
Social networks (friends of friends)Relationship traversal is O(1) per hop in Neo4j; SQL JOINs degrade exponentially
Fraud detectionFind patterns in transaction networks in real-time
"People who bought X also bought Y"Recommendation graphs — natural model
Knowledge graphsGoogle Knowledge Graph, drug interaction databases

The Real World — They're Often Used Together ▾
Polyglot Persistence

Most production systems at scale use multiple database types — each doing what it does best. This is called polyglot persistence. Choosing SQL vs NoSQL is a false dilemma in most serious architectures.

Core business data (orders, users, products)MySQL / MariaDB / PostgreSQLACID, JOINs, integrity — the source of truth

CAP Theorem — The Fundamental Trade-off ▾
What Is CAP?

The CAP theorem states that a distributed database can only guarantee two of three properties simultaneously: Consistency (all nodes see the same data), Availability (every request gets a response), and Partition Tolerance (the system keeps working if network splits occur). Since partition tolerance is non-negotiable in any real network, the real trade-off is C vs A.

DatabaseCAP ChoiceImplication
MySQL / MariaDB (single node)CA — Consistent + AvailableNot partition tolerant — single server, data is always consistent
MySQL Galera ClusterCA with some PA featuresSynchronous replication — strong consistency across nodes
MongoDBCP — Consistent + Partition-tolerantIn a partition, primary refuses writes to preserve consistency
CassandraAP — Available + Partition-tolerantAlways accepts writes; nodes may briefly diverge (eventual consistency)
Redis (single)CASingle master — no partitions, extremely fast
Section 05 · Market
Database Engines
The major database products — who made them, what they're good at, and how they relate to each other. Focus is on the SQL family since that's your track, with enough NoSQL awareness to have an intelligent conversation.
The SQL Family Tree ▾
1979 — Oracle
Oracle Database — the original commercial SQL database. Dominant in enterprise, banking, and Oracle ERP ecosystems. Very powerful, very expensive. Still #1 in enterprise revenue.
1983 — IBM DB2
DB2 — IBM's flagship. Huge in mainframe/banking environments. Still running critical financial infrastructure worldwide.
1989 — Microsoft SQL Server
SQL Server — Microsoft's relational DB. Dominant in Windows/.NET enterprise environments. T-SQL dialect. Often used with Azure.
1995 — MySQL
MySQL — open-source, the LAMP stack (Linux, Apache, MySQL, PHP) that powered the early web. Acquired by Sun (2008), then Oracle (2010). Still the most-deployed SQL database on the internet.
1996 — PostgreSQL
PostgreSQL — open-source, standards-compliant, feature-rich. Strong JSON support, custom types, window functions. The developer favourite. Growing rapidly.
2009 — MariaDB
MariaDB — fork of MySQL created after Oracle acquisition. Drop-in replacement, mostly MySQL-compatible. Adds sequences, better clustering (Galera), and the RETURNING clause. The engine used in this training track.

Section 06 · Practical
Choose Your Database
A practical decision framework. These are the questions you ask when starting any project. There is rarely one objectively correct answer — but there are clearly wrong choices. Here's how to avoid them.
The Decision Framework — 6 Questions ▾
#QuestionSuggests SQL if…Suggests NoSQL if…
1How structured is your data?Consistent columns, well-defined schema, tabularVaried fields per record, nested, hierarchical
2Do you need transactions?Yes — money, medical records, ordersNo — logs, analytics, caches, recommendations
3How related is your data?Many relationships, JOINs needed, FK integrity mattersData is self-contained per document / key
4What scale do you need?Millions of rows, one strong serverBillions of writes/day, global distribution
5How often does schema change?Rarely — mature domain, stable structureFrequently — MVP, evolving product, unknowns
6What does your team know?SQL is universal — everyone on the team can write itTeam has specific NoSQL expertise for the use case
Scenario Verdicts ▾
🏥
Healthcare EHR — Morafek CareMate → MariaDB
Patient data, vitals, prescriptions — highly structured, ACID required for DSGVO compliance, complex JOINs across patients/doctors/visits, audit trails. SQL is the only sensible choice. Galera for HA.
🚂
Railway Operations — OS GmbH → MariaDB + Redis
Booking system → MariaDB (structured, transactional, seat inventory). Real-time departure boards → Redis (in-memory pub/sub, sub-millisecond updates). Two databases, each doing its job.
🛒
E-commerce Product Catalogue → MongoDB or PostgreSQL
Products have wildly different attributes (a shirt has size/colour, a laptop has RAM/CPU). Either PostgreSQL with JSONB or MongoDB for flexible schema. Orders and payments still benefit from SQL.
📱
Social Network (followers, feed) → Graph + Cassandra
Relationship graph → Neo4j (who follows who, mutual friends, recommendations). Activity feed → Cassandra (billions of append-only events, time-partitioned, massive write throughput).
📊
Analytics / Reporting Dashboard → ClickHouse / BigQuery
Aggregating billions of events (GROUP BY, SUM, percentile) — row-oriented SQL is inefficient. Columnar databases (ClickHouse, BigQuery, Redshift) scan only the columns needed, 100× faster for analytics.
⚡
API Rate Limiting & Session Store → Redis
Needs sub-millisecond response on every single API call. Redis TTL, atomic INCR, and in-memory speed make this trivially easy. Don't touch a relational DB for this.
The Interview-Ready Answer ▾
🎯 What to say when asked "SQL or NoSQL?"

"It depends on the data model and requirements. I start with SQL by default — ACID guarantees and relational integrity are hard to get right at the application level, and SQL skills are fully transferable. I'd reach for NoSQL when the data is genuinely unstructured, when I need horizontal write scaling beyond what sharding SQL can provide, or when the access pattern is a perfect fit — like Redis for caching or a graph DB for relationship traversal. In most real systems I'd expect to use both."

Common Mistakes to Avoid
  • Choosing NoSQL because "it scales better" — SQL also scales with proper indexing, read replicas, and connection pooling for most use cases
  • Choosing SQL because "it's familiar" when the data is fundamentally document-like or graph-like
  • Treating it as a permanent decision — databases can be migrated; getting started correctly matters more than perfection
  • Ignoring your team's expertise — a perfect database choice run poorly is worse than a good database choice run well
Default Rule of Thumb

Start with PostgreSQL or MariaDB. Add Redis for caching. Switch to NoSQL only when you have a specific, measured reason — not upfront speculation. Most applications that chose NoSQL "for scale" never hit the scale that required it, but they paid the schema flexibility and tooling cost every day.
