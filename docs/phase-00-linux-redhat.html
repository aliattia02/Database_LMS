Lesson 01 · ~2 hours
Filesystem Navigation

Before you touch a database, you must be fluent in the shell. Every DBA task — editing my.cnf, reading logs, running mariabackup — starts with navigating the filesystem and knowing where everything lives.

Click any section to expand
The Linux filesystem — where everything lives▾
/etc
Configuration files. MariaDB lives here at /etc/my.cnf or /etc/my.cnf.d/. MaxScale at /etc/maxscale.cnf. You will spend a lot of time here.
/var/lib
Application data. MariaDB data files live at /var/lib/mysql/. Each database is a subfolder. This is where your actual database rows are stored on disk.
/var/log
Log files. MariaDB error log at /var/log/mariadb/mariadb.log. The first place you look at 3am when something breaks.
/usr/bin and /usr/sbin
Executables. mysql, mysqldump, mariabackup, maxctrl all live here. /usr/sbin = admin tools (run as root). /usr/bin = regular user tools.
/tmp
Temporary files. Cleared on reboot. Fine for scratch work. Never store backups here — they disappear.
/home
User home directories. Your scripts and working files go here. MariaDB itself runs as the mysql system user whose home is /var/lib/mysql.
🎩 RHEL specific

On RHEL/CentOS/AlmaLinux, MariaDB config is at /etc/my.cnf.d/mariadb-server.cnf — not a single file. Each drop-in file in /etc/my.cnf.d/ gets loaded. On Debian/Ubuntu it is /etc/mysql/mariadb.conf.d/. Know both.

Core navigation commands▾
bash
# Where am I right now? pwd # List files — l=long format, a=include hidden, h=human-readable sizes ls -lah # Move into a directory cd /etc/my.cnf.d # Go up one level cd .. # Go back to previous directory (like undo for cd) cd - # Go to your home directory cd ~ # Jump to MariaDB data directory cd /var/lib/mysql # Jump to log directory cd /var/log/mariadb
bash
# Show directory tree (install first: dnf install tree) tree /etc/my.cnf.d tree /var/lib/mysql -L 1 # Find a file by name anywhere on the system find / -name "my.cnf" 2>/dev/null # Find files modified in the last 60 minutes (useful after an incident) find /var/lib/mysql -mmin -60 # Check disk usage — critical for a DBA df -h # disk free per mount point du -sh /var/lib/mysql # how much space MariaDB data uses du -sh /var/lib/mysql/* # per-database breakdown
💡 DBA habit

Always run df -h before and after a backup. Running out of disk mid-backup corrupts the backup silently on some versions. Set an alert at 80% disk usage on your data volume.

Reading files — logs and configs▾
bash
# Print entire file (dangerous on large log files) cat /etc/my.cnf.d/mariadb-server.cnf # First 20 lines of a file head -20 /var/log/mariadb/mariadb.log # Last 50 lines — what you actually want for logs tail -50 /var/log/mariadb/mariadb.log # Follow the log live as MariaDB writes to it (Ctrl+C to stop) tail -f /var/log/mariadb/mariadb.log # Search inside a file grep "ERROR" /var/log/mariadb/mariadb.log grep -i "galera" /var/log/mariadb/mariadb.log # -i = case insensitive grep -n "wsrep" /var/log/mariadb/mariadb.log # -n = show line numbers # Search and show 3 lines of context around each match grep -C 3 "FATAL" /var/log/mariadb/mariadb.log # Scroll through a large file (q to quit, /search to find) less /var/log/mariadb/mariadb.log
⚠️ Never do this

Never cat /var/log/mariadb/mariadb.log on a busy production server — the log file can be gigabytes. Always use tail -f for live monitoring or grep for specific errors.

Editing config files with vim▾
ℹ️ Why vim?

vim is always available on any Linux server — no GUI, no nano sometimes, but vim is always there. You need exactly enough vim to open a file, make a change, and save it.

Open a file
vim /etc/my.cnf.d/mariadb-server.cnf — you start in Normal mode. You cannot type yet.
Enter insert mode
Press i — you can now type and edit. The bottom shows -- INSERT --.
Save and quit
Press Esc to leave insert mode, then type :wq and Enter. w = write, q = quit.
Quit without saving
Press Esc then :q! — the ! forces quit even with unsaved changes.
Search in file
In Normal mode, type /innodb_buffer_pool then Enter. Press n for next match.
Jump to end of file
In Normal mode, press G (capital G). Press gg to go back to the top.
bash
# DBA workflow: edit a config, then restart the service sudo vim /etc/my.cnf.d/mariadb-server.cnf # After saving: validate config before restarting mysqld --help --verbose 2>/dev/null | grep -A1 "innodb-buffer-pool-size" # Then restart sudo systemctl restart mariadb
💡 Safer alternative

If you hate vim: sudo nano /etc/my.cnf.d/mariadb-server.cnf. Nano shows its shortcuts at the bottom (^X = Ctrl+X to exit). Install it first: sudo dnf install -y nano.

Lesson 02 · ~2 hours
Files & Permissions

MariaDB won't start if its data directory has wrong ownership. Galera SST will silently fail if the receiving node can't write files. Permissions are not optional knowledge — they are the most common cause of "why won't it start" incidents.

Click any section to expand
Reading permission strings▾
bash
ls -lah /var/lib/mysql
output
drwxr-x--x 9 mysql mysql 4.0K Apr 28 09:12 . -rw-rw---- 1 mysql mysql 16M Apr 28 09:12 ibdata1 -rw-rw---- 1 mysql mysql 512M Apr 28 09:12 ib_logfile0
d rwx r-x --x
First character: d=directory, -=file, l=symlink. Then 3 groups of 3: owner permissions, group permissions, everyone else.
r w x
r=read (4), w=write (2), x=execute (1). A - means that permission is absent. So rwx=7, rw-=6, r--=4.
mysql mysql
Owner user and owner group. MariaDB data files must be owned by the mysql user and mysql group. If they are owned by root, MariaDB cannot read or write them.
Changing ownership and permissions▾
bash
# chown: change owner sudo chown mysql:mysql /var/lib/mysql/caremate_db # -R = recursive (apply to everything inside) sudo chown -R mysql:mysql /var/lib/mysql/ # chmod: change permissions # 750 = owner:rwx group:r-x others:--- sudo chmod 750 /var/lib/mysql # 640 = owner:rw- group:r-- others:--- sudo chmod 640 /etc/my.cnf.d/mariadb-server.cnf # Check who owns a running process ps aux | grep mariadbd
⚠️ Most common DBA mistake

After manually copying data files or restoring a backup as root, the files end up owned by root instead of mysql. MariaDB then refuses to start with a cryptic permission error. Always run sudo chown -R mysql:mysql /var/lib/mysql/ after any manual file operation on the data directory.

Symlinks and hard links▾
Symlink (soft link)
A pointer to another file or directory. Used in MariaDB to point /var/lib/mysql to a different disk. If the target is deleted, the symlink breaks.
Hard link
A second name for the same file data. Deleting one name leaves the other intact. Used by mariabackup internally — understanding this explains why incremental backups are efficient.
bash
# Create a symlink — useful when moving data dir to a larger disk sudo ln -s /mnt/data/mysql /var/lib/mysql # Verify it's a symlink (shows l in ls -l) ls -lah /var/lib/mysql # Check where a symlink points readlink -f /var/lib/mysql
3am scenario — MariaDB won't start after backup restore▾
🚨 3am scenario

You restored a mariabackup to /var/lib/mysql. MariaDB refuses to start. The error log says "Can't create/write to file". Walk through your diagnosis:

1
Check the error log first
tail -50 /var/log/mariadb/mariadb.log — look for "permission denied" or "can't open".
2
Check ownership
ls -lah /var/lib/mysql/ — if you see "root root" instead of "mysql mysql", that's your problem.
3
Fix ownership recursively
sudo chown -R mysql:mysql /var/lib/mysql/
4
Check disk space
df -h /var/lib/mysql — a full disk also produces "can't write" errors that look identical.
5
Restart and verify
sudo systemctl start mariadb && systemctl status mariadb
Lesson 03 · ~1 hour
Users & Groups

Linux users and MariaDB users are completely separate. A Linux system user called mysql runs the database process. Understanding this distinction prevents a whole class of authentication confusion.

Click any section to expand
System users vs MariaDB users▾
ℹ️ Key distinction

The mysql Linux user owns MariaDB's files and process — it never logs into the shell. MariaDB has its own internal user system (root@localhost, app_user@%) stored in the mysql database. These are completely unrelated.

bash
# See who the mysql Linux user is id mysql # output: uid=27(mysql) gid=27(mysql) groups=27(mysql) # See all users on the system cat /etc/passwd # See current Linux user whoami # Switch to root (needed for most DBA operations) sudo -i # full root shell sudo su - # same thing # Run a single command as root without switching sudo systemctl restart mariadb
Creating and managing Linux users▾
bash
# Create a new user (for a new DBA colleague) sudo useradd -m -s /bin/bash dba_ali # Set their password sudo passwd dba_ali # Add user to the wheel group (gives sudo access on RHEL) sudo usermod -aG wheel dba_ali # Verify groups groups dba_ali # Delete a user (keep their home: -r removes it) sudo userdel dba_ali # Lock an account (disable without deleting) sudo usermod -L dba_ali
🎩 RHEL specific

On RHEL, the sudo group is called wheel — not sudo like on Ubuntu/Debian. Adding a user to wheel gives full sudo access. Verify with grep wheel /etc/sudoers.

sudo and the sudoers file▾
bash
# ALWAYS use visudo to edit sudoers — it validates syntax before saving # A syntax error in /etc/sudoers can lock you out of sudo entirely sudo visudo # Grant a user passwordless sudo for just mariadb restart (production pattern) # Add this line in visudo: # dba_ali ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart mariadb # Check who has sudo access sudo cat /etc/sudoers ls /etc/sudoers.d/
⚠️ Never edit /etc/sudoers directly

Always use sudo visudo. It locks the file and validates syntax before saving. Direct editing can produce a corrupt sudoers file which locks every user out of sudo, requiring console access to fix.

Lesson 04 · ~1 hour
DNF & Package Management

DNF is how you install, update, and remove software on RHEL. MariaDB, MaxScale, and every tool you use as a DBA is installed through it. You also need to understand repositories — packages don't appear by magic.

Click any section to expand
Essential DNF commands▾
bash
# Update all packages (do this right after install) sudo dnf update -y # Install a package sudo dnf install -y mariadb-server sudo dnf install -y nano tree wget curl # Search for a package dnf search mariadb dnf search maxscale # Get info about a package (version, size, description) dnf info mariadb-server # Remove a package sudo dnf remove mariadb-server # List installed packages dnf list installed dnf list installed | grep mariadb # What package does a file belong to? dnf provides /usr/bin/mysql
Adding repositories — MariaDB and MaxScale▾
ℹ️ Why repos matter

RHEL's default repositories contain older versions of MariaDB. To get MariaDB 10.11 or 11.x (the versions Galera works best with), you must add the official MariaDB repository.

bash
# Add official MariaDB 10.11 LTS repo curl -LsS https://downloads.mariadb.com/MariaDB/mariadb_repo_setup \ | sudo bash -s -- --mariadb-server-version="mariadb-10.11" # Verify the repo was added ls /etc/yum.repos.d/ cat /etc/yum.repos.d/mariadb.repo # Now install MariaDB from the official repo sudo dnf install -y MariaDB-server MariaDB-client # List all enabled repos dnf repolist # Disable a repo temporarily sudo dnf install --disablerepo=appstream mariadb-server
bash
# Add MaxScale repo (after adding MariaDB repo above) # MaxScale repo is added by the same setup script with --mariadb-maxscale-version curl -LsS https://downloads.mariadb.com/MariaDB/mariadb_repo_setup \ | sudo bash -s -- --mariadb-maxscale-version="latest" sudo dnf install -y maxscale
RPM — working with packages directly▾
bash
# Install a downloaded .rpm file sudo dnf install -y ./dbeaver-ce-latest-stable.x86_64.rpm # List all files installed by a package rpm -ql mariadb-server # Check if a package is installed rpm -q mariadb-server # Verify package files haven't been tampered with rpm -V mariadb-server
🎩 RHEL vs Debian

RHEL uses dnf / rpm and .rpm files. Debian/Ubuntu uses apt / dpkg and .deb files. The job posting mentions both. The concepts are identical — only the commands differ. On Ubuntu: apt update && apt install mariadb-server.

Lesson 05 · ~2 hours
systemd & Services

systemd manages every long-running process on RHEL — including MariaDB and MaxScale. Starting, stopping, monitoring, and auto-starting services on boot are daily DBA tasks. You also need to read journald logs, which replaced the old syslog.

Click any section to expand
systemctl — controlling services▾
bash
# Start, stop, restart a service sudo systemctl start mariadb sudo systemctl stop mariadb sudo systemctl restart mariadb # Reload config without full restart (not all services support this) sudo systemctl reload mariadb # Check status — first thing you do when something is wrong systemctl status mariadb systemctl status maxscale # Enable auto-start on boot / disable it sudo systemctl enable mariadb sudo systemctl disable mariadb # Enable AND start in one command sudo systemctl enable --now mariadb # Is it running right now? systemctl is-active mariadb systemctl is-enabled mariadb # List all running services systemctl list-units --type=service --state=running
💡 Production pattern

Always run systemctl enable --now mariadb after installation — not just start. Without enable, MariaDB will not restart after a reboot, causing an outage that looks mysterious until you realize the service was never enabled.

journalctl — reading service logs▾
ℹ️ Two log sources

MariaDB writes to its own log file at /var/log/mariadb/mariadb.log AND to the systemd journal. The journal is useful for startup errors that happen before MariaDB can open its own log file.

bash
# Show all logs for MariaDB sudo journalctl -u mariadb # Last 50 lines sudo journalctl -u mariadb -n 50 # Follow live (like tail -f) sudo journalctl -u mariadb -f # Logs since last boot sudo journalctl -u mariadb -b # Logs from a time range sudo journalctl -u mariadb --since "2025-04-28 03:00" --until "2025-04-28 04:00" # See why a service failed to start sudo journalctl -u mariadb -b --no-pager | tail -30 # Check all services that crashed since boot systemctl --failed
Service unit files — understanding MariaDB's service▾
bash
# Where is MariaDB's service unit file? systemctl cat mariadb # Show full path systemctl show mariadb --property=FragmentPath
unit file excerpt
[Unit] Description=MariaDB database server After=network.target [Service] Type=notify User=mysql Group=mysql ExecStart=/usr/sbin/mariadbd --basedir=/usr LimitNOFILE=32768 # max open files — important for busy DBs Restart=on-failure # auto-restart if it crashes [Install] WantedBy=multi-user.target
bash
# Override unit file without editing the original # Creates /etc/systemd/system/mariadb.service.d/override.conf sudo systemctl edit mariadb # After editing any unit file, reload daemon sudo systemctl daemon-reload
Podman + systemd — running containers as services▾
🎩 RHEL uses Podman not Docker

Docker requires a daemon running as root. Podman is daemonless and rootless — each container is just a process. To run Podman containers as systemd services that survive reboots, you generate a unit file.

bash
# Start your phr-db container first podman run --name phr-db \ -e MYSQL_ROOT_PASSWORD=caremate123 \ -e MYSQL_DATABASE=caremate_db \ -p 3306:3306 \ -d mysql:8 # Generate a systemd unit file for it podman generate systemd --name phr-db --files --new # Install as a user service (no root needed) mkdir -p ~/.config/systemd/user mv container-phr-db.service ~/.config/systemd/user/ systemctl --user enable --now container-phr-db.service # Allow it to run when you're not logged in loginctl enable-linger $USER # Check it systemctl --user status container-phr-db.service
Lesson 06 · ~2 hours
Networking & Firewall

Galera nodes communicate on ports 3306, 4444, 4567, and 4568. MaxScale listens on 3306 or 4006. If the firewall blocks these, the cluster simply doesn't form — and the error messages are confusing. Networking is a core DBA skill.

Click any section to expand
Network diagnostics▾
bash
# Show all network interfaces and IP addresses ip addr show ip a # shorthand # Show routing table ip route show # Test connectivity to another Galera node ping -c 4 192.168.1.102 # Test if a specific port is reachable (is MariaDB listening?) nc -zv 192.168.1.102 3306 nc -zv 192.168.1.102 4567 # Galera replication port # What ports is this server listening on? ss -tlnp ss -tlnp | grep 3306 # is MariaDB listening? ss -tlnp | grep 4006 # is MaxScale listening? # Trace route to another node traceroute 192.168.1.102
💡 Galera port map

3306 = MySQL/MariaDB client connections. 4444 = SST (State Snapshot Transfer — full sync). 4567 = Galera replication traffic (group communication). 4568 = IST (Incremental State Transfer). All four must be open between Galera nodes.

firewalld — opening ports for MariaDB and Galera▾
🎩 RHEL uses firewalld

RHEL uses firewalld instead of raw iptables. It manages zones — groups of rules. By default, your network interface is in the "public" zone which blocks almost everything.

bash
# Check firewall status sudo firewall-cmd --state sudo firewall-cmd --list-all # Open MariaDB port (3306) sudo firewall-cmd --add-port=3306/tcp --permanent # Open all Galera ports sudo firewall-cmd --add-port=4444/tcp --permanent # SST sudo firewall-cmd --add-port=4567/tcp --permanent # Galera replication sudo firewall-cmd --add-port=4567/udp --permanent # Galera multicast sudo firewall-cmd --add-port=4568/tcp --permanent # IST # Open MaxScale port sudo firewall-cmd --add-port=4006/tcp --permanent # Apply all --permanent changes sudo firewall-cmd --reload # Verify ports are open sudo firewall-cmd --list-ports # Emergency: completely disable firewall (never in production) sudo systemctl stop firewalld
Hostname and /etc/hosts configuration▾
bash
# Check current hostname hostname hostnamectl # Set hostname permanently (important for Galera node identification) sudo hostnamectl set-hostname galera-node-01 # Edit /etc/hosts — map hostnames to IPs for Galera cluster sudo vim /etc/hosts
/etc/hosts example for Galera cluster
127.0.0.1 localhost 192.168.1.101 galera-node-01 192.168.1.102 galera-node-02 192.168.1.103 galera-node-03
💡 Why /etc/hosts matters for Galera

Galera uses the hostname in wsrep_node_address and wsrep_node_name. If DNS is slow or unavailable, Galera falls back to /etc/hosts. Always populate it on all nodes — DNS failures have caused Galera split-brain scenarios in production.

SSH — accessing remote servers▾
bash
# Connect to a remote server ssh dba_ali@192.168.1.102 ssh dba_ali@galera-node-02 # Generate SSH key pair (do this once) ssh-keygen -t ed25519 -C "dba_ali@galera-node-01" # Copy your public key to a remote server (enables passwordless login) ssh-copy-id dba_ali@galera-node-02 # Copy a file to remote server (backup transfer) scp backup.tar.gz dba_ali@galera-node-02:/tmp/ # Copy a directory recursively scp -r /var/lib/mysql/caremate_db dba_ali@galera-node-02:/tmp/
Lesson 07 · ~3 hours
Bash Scripting for DBAs

You will write scripts to automate backups, rotate logs, check cluster health, and alert on disk usage. These are not optional — they are the difference between a DBA who reacts to problems and one who prevents them.

Click any section to expand
Bash fundamentals▾
bash
#!/bin/bash # First line (shebang) — tells the OS this is a bash script # Variables — no spaces around = DB_NAME="caremate_db" DB_USER="root" BACKUP_DIR="/var/backups/mysql" # Use variables with $ echo "Backing up $DB_NAME" # Command output into a variable TODAY=$(date +%Y-%m-%d) DISK_USAGE=$(df -h /var/lib/mysql | awk 'NR==2 {print $5}') # Arithmetic COUNT=$((5 + 3)) # If statement if [ "$DISK_USAGE" -gt 80 ]; then echo "WARNING: disk at $DISK_USAGE%" fi # For loop for NODE in galera-node-01 galera-node-02 galera-node-03; do echo "Checking $NODE..." ssh $NODE "systemctl is-active mariadb" done
bash
# Make a script executable chmod +x /usr/local/bin/check_cluster.sh # Run it /usr/local/bin/check_cluster.sh # Debug mode — shows every command as it runs bash -x /usr/local/bin/check_cluster.sh
Real script — automated MariaDB backup with rotation▾
bash — /usr/local/bin/mariadb_backup.sh
#!/bin/bash # MariaDB daily backup with 7-day rotation # Run via cron: 0 2 * * * /usr/local/bin/mariadb_backup.sh BACKUP_DIR="/var/backups/mysql" LOG_FILE="/var/log/mariadb_backup.log" KEEP_DAYS=7 DATE=$(date +%Y-%m-%d_%H-%M-%S) BACKUP_PATH="$BACKUP_DIR/full_$DATE" # Create backup dir if it doesn't exist mkdir -p "$BACKUP_DIR" echo "[$DATE] Starting backup..." >> "$LOG_FILE" # Run mariabackup mariabackup --backup \ --target-dir="$BACKUP_PATH" \ --user=root \ --password=caremate123 2>> "$LOG_FILE" if [ $? -eq 0 ]; then echo "[$DATE] Backup succeeded: $BACKUP_PATH" >> "$LOG_FILE" # Prepare the backup (required before restore) mariabackup --prepare --target-dir="$BACKUP_PATH" 2>> "$LOG_FILE" # Remove backups older than KEEP_DAYS find "$BACKUP_DIR" -maxdepth 1 -name "full_*" -mtime +$KEEP_DAYS -exec rm -rf {} \; echo "[$DATE] Rotation done. Kept last $KEEP_DAYS days." >> "$LOG_FILE" else echo "[$DATE] BACKUP FAILED — check logs!" >> "$LOG_FILE" # Send alert (replace with your alerting method) echo "MariaDB backup failed on $(hostname)" | mail -s "BACKUP ALERT" dba@company.com fi
Real script — Galera cluster health check▾
bash — /usr/local/bin/check_galera.sh
#!/bin/bash # Quick Galera health check — run this before any maintenance MYSQL_CMD="mysql -u root -pcaremate123 -e" echo "=== Galera Cluster Health ===" # Cluster size — should equal number of nodes SIZE=$($MYSQL_CMD "SHOW STATUS LIKE 'wsrep_cluster_size';" | awk 'NR==2 {print $2}') echo "Cluster size: $SIZE" # This node's status STATE=$($MYSQL_CMD "SHOW STATUS LIKE 'wsrep_local_state_comment';" | awk 'NR==2 {print $2}') echo "Local state: $STATE" # Is this node synced? READY=$($MYSQL_CMD "SHOW STATUS LIKE 'wsrep_ready';" | awk 'NR==2 {print $2}') echo "Ready: $READY" # Flow control (should be 0 or very low) FC=$($MYSQL_CMD "SHOW STATUS LIKE 'wsrep_flow_control_paused';" | awk 'NR==2 {print $2}') echo "Flow control paused: $FC" # Alert if cluster is degraded if [ "$SIZE" -lt 3 ]; then echo "WARNING: Cluster has only $SIZE nodes — quorum may be at risk!" fi if [ "$STATE" != "Synced" ]; then echo "WARNING: Node is not Synced — state is $STATE" fi
cron — scheduling scripts▾
bash
# Edit cron jobs for the current user crontab -e # View existing cron jobs crontab -l # Cron syntax: minute hour day month weekday command # Backup at 2am every day 0 2 * * * /usr/local/bin/mariadb_backup.sh # Cluster health check every 5 minutes */5 * * * * /usr/local/bin/check_galera.sh >> /var/log/galera_health.log 2>&1 # Disk check every hour 0 * * * * df -h /var/lib/mysql >> /var/log/disk_usage.log # System-wide cron files (run as root) ls /etc/cron.d/ ls /etc/cron.daily/
💡 Cron cheat sheet

* = every. */5 = every 5. 0 2 * * * = at 02:00 every day. 0 2 * * 0 = at 02:00 every Sunday. Always redirect both stdout and stderr: >> /var/log/myscript.log 2>&1 or you'll never know if it failed.

Lesson 08 · ~2 hours
SELinux

SELinux is enabled by default on RHEL and it will silently block MariaDB from accessing files, ports, or directories that you haven't explicitly labelled. It is the single most common cause of "it works on Debian but not RHEL" problems.

Click any section to expand
What SELinux is and why it matters▾
ℹ️ The concept

Normal Linux permissions say "can user X access file Y?" SELinux adds a second layer: "is process Z even allowed to access files of type T, regardless of Unix permissions?" MariaDB (the process) has an SELinux label. Your data directory has a label. If they don't match the SELinux policy, access is denied — even if chown and chmod look correct.

Enforcing mode
Default on RHEL production. SELinux actively blocks policy violations. An unexpected denial = MariaDB refuses to start or can't write files.
Permissive mode
SELinux logs violations but does not block them. Used for debugging and building new policies. Safe for a dev VM, never for production.
Disabled mode
SELinux completely off. Requires reboot. Never disable in production. If you're disabling SELinux to fix a problem, you're not fixing the problem.
bash
# Check current SELinux mode getenforce sestatus # Temporarily switch to permissive (survives until reboot) sudo setenforce 0 # 0=permissive, 1=enforcing # Permanently change mode — edit this file sudo vim /etc/selinux/config # Change SELINUX=enforcing to SELINUX=permissive (requires reboot)
SELinux contexts — fixing MariaDB data directory labels▾
⚠️ Most common scenario

You moved the MariaDB data directory to a new disk. Now MariaDB won't start even though chown and chmod are correct. The new path doesn't have the right SELinux context (label). MariaDB expects files labelled mysqld_db_t.

bash
# See SELinux context of files (-Z flag) ls -laZ /var/lib/mysql # should show: system_u:object_r:mysqld_db_t:s0 ls -laZ /mnt/newdisk/mysql # shows: system_u:object_r:unlabeled_t:s0 <-- this is the problem # Fix: tell SELinux this new path should have the mysql data context sudo semanage fcontext -a -t mysqld_db_t "/mnt/newdisk/mysql(/.*)?" # Apply the context to existing files sudo restorecon -Rv /mnt/newdisk/mysql # Verify ls -laZ /mnt/newdisk/mysql
bash
# Allow MariaDB to use a non-standard port sudo semanage port -a -t mysqld_port_t -p tcp 3307 # Check what ports MariaDB is allowed to use semanage port -l | grep mysqld
Diagnosing SELinux denials with audit2why▾
bash
# SELinux denials are logged to the audit log sudo tail -50 /var/log/audit/audit.log | grep denied # audit2why explains a denial in plain English sudo audit2why < /var/log/audit/audit.log # audit2allow generates a policy module to fix it sudo audit2allow -a -M my_mariadb_fix sudo semodule -i my_mariadb_fix.pp # Install tools if missing sudo dnf install -y policycoreutils-python-utils
💡 Diagnosis workflow

When MariaDB fails on RHEL and permissions look correct: (1) temporarily set setenforce 0, (2) try starting MariaDB, (3) if it works now, SELinux is the problem, (4) check /var/log/audit/audit.log for denials, (5) use audit2why to understand them, (6) fix contexts with semanage fcontext + restorecon, (7) set setenforce 1 again.

3am scenario — MariaDB refuses to start on RHEL▾
🚨 3am scenario

You get paged. MariaDB is down on a RHEL production server. systemctl status mariadb shows "failed". Walk through your exact steps.

1
Read the systemd status
systemctl status mariadb -l — the -l flag shows the full error without truncation.
2
Read the journal
sudo journalctl -u mariadb -b --no-pager | tail -50 — covers startup errors before the MariaDB log opens.
3
Read MariaDB's own error log
sudo tail -100 /var/log/mariadb/mariadb.log
4
Check disk
df -h /var/lib/mysql — a full disk causes cryptic errors. This takes 2 seconds to rule out.
5
Check permissions
ls -laZ /var/lib/mysql — ownership should be mysql:mysql and SELinux context should be mysqld_db_t.
6
Check SELinux
sudo tail -20 /var/log/audit/audit.log | grep denied — if you see denials, SELinux is blocking it.
7
Check config syntax
mysqld --help --verbose 2>&1 | head -5 — a syntax error in my.cnf produces an immediate parse error here.
3am scenario — disk filling up on the data volume▾
🚨 3am scenario

Monitoring alerts: /var/lib/mysql is at 95% disk. Writes will fail at 100%. You have maybe 30 minutes.

1
Find what's using space
du -sh /var/lib/mysql/* | sort -rh | head -20 — identifies the largest databases or files.
2
Check binary logs
Binary logs are often the culprit. ls -lah /var/lib/mysql/mysql-bin.*. They accumulate if not purged.
3
Purge old binary logs safely
In MariaDB: PURGE BINARY LOGS BEFORE DATE_SUB(NOW(), INTERVAL 3 DAY); — never delete the files directly.
4
Check for large temporary files
find /tmp /var/tmp -size +500M 2>/dev/null
5
Longer term: add a volume or move data dir
Create a symlink from /var/lib/mysql to a larger mount point, update SELinux context, restart MariaDB.
3am scenario — Galera node won't rejoin after RHEL reboot▾
🚨 3am scenario

A Galera node was rebooted for a kernel update. It won't rejoin the cluster. The other two nodes are healthy. The error log shows "failed to reach primary component".

1
Check the other nodes first
SSH to a healthy node. mysql -e "SHOW STATUS LIKE 'wsrep_cluster_size';" — should show 2 (the two healthy nodes).
2
Check firewall on the rebooted node
sudo firewall-cmd --list-ports — after reboot, firewall rules are restored from permanent config. Verify Galera ports 4567, 4568, 4444 are still open.
3
Check SELinux denials
sudo audit2why < /var/log/audit/audit.log — a kernel update can reset SELinux policy.
4
Check network connectivity
nc -zv galera-node-01 4567 from the rebooted node — can it reach the cluster?
5
Restart MariaDB on the rebooted node
sudo systemctl restart mariadb — if the node was cleanly shut down, it should trigger IST and sync without SST.
Interview Q&A — Linux for DBAs▾
What is the difference between Podman and Docker on RHEL?

Docker requires a daemon running as root, which is a security risk. Podman is daemonless — each container is a direct child process of the user who started it. On RHEL, Podman is the default and preferred tool. Commands are nearly identical (mostly a 1:1 swap of docker for podman). The main operational difference: Podman containers need a systemd unit file generated via podman generate systemd to survive reboots, whereas Docker has --restart always.

MariaDB won't start on RHEL even though permissions look correct. What do you check?

SELinux. Normal Unix permissions and SELinux contexts are independent. Check ls -laZ /var/lib/mysql — the SELinux type must be mysqld_db_t. If you moved the data directory, the new path won't have this label automatically. Fix with semanage fcontext -a -t mysqld_db_t "/new/path(/.*)?" then restorecon -Rv /new/path. Also check /var/log/audit/audit.log for denied entries and use audit2why to interpret them.

How do you make a service start automatically after a reboot on RHEL?

sudo systemctl enable mariadb — this creates a symlink in the appropriate systemd target directory so the service starts at boot. systemctl start alone only starts it for the current session. Best practice: use sudo systemctl enable --now mariadb which enables and starts in one command. Verify with systemctl is-enabled mariadb.

How do you open Galera cluster ports on RHEL's firewall?

RHEL uses firewalld. Open ports with sudo firewall-cmd --add-port=4567/tcp --permanent for each required port (3306, 4444, 4567/tcp, 4567/udp, 4568). Then sudo firewall-cmd --reload to apply. The --permanent flag is critical — without it, rules disappear on the next reload or reboot. Verify with sudo firewall-cmd --list-ports.

What is the difference between dnf and apt?

Both are package managers — dnf is for RHEL/CentOS/Fedora, apt is for Debian/Ubuntu. They manage .rpm and .deb packages respectively. The concepts are identical: update package lists, install, remove, search. The syntax differs: dnf install mariadb-server vs apt install mariadb-server. Repository files live in /etc/yum.repos.d/ on RHEL and /etc/apt/sources.list.d/ on Debian.

A MariaDB backup script is failing silently. How would you debug it?

First, check if cron is even running the script: grep CRON /var/log/cron. Then run the script manually in debug mode: bash -x /usr/local/bin/mariadb_backup.sh — this prints every command before executing it. Check the script's exit codes with echo $? after each command. Make sure cron output is redirected to a log: 0 2 * * * /usr/local/bin/mariadb_backup.sh >> /var/log/backup.log 2>&1. The 2>&1 redirects stderr alongside stdout — without it, error messages are silently discarded.

How do you find which process is using port 3306?

ss -tlnp | grep 3306 — shows the process name and PID. Or lsof -i :3306 if lsof is installed. This is useful when MariaDB fails to start with "address already in use" — another MariaDB instance or a different service might have grabbed the port.
