Phase 04 · Lesson 01 · 3h
Storage: Pages, Tablespaces & InnoDB Architecture

InnoDB doesn't think in rows — it thinks in 16KB pages. Understanding pages, the buffer pool, tablespaces, and the clustered index is the difference between guessing why a query is slow and knowing. This is the foundation every other internal concept builds on.

Click any section to expand
⚡ Why This Matters▾

When Nietsch asks "why is your buffer pool hit ratio low?" or "why is this query doing a full table scan despite an index?" — both answers live in InnoDB's page architecture. At OS GmbH, where 24/7 DB infrastructure means any performance regression pages the on-call DBA at 3am, understanding the buffer pool and clustered index is non-negotiable. For Morafek CareMate EHR, the DSGVO audit trail must be readable under load — only possible if hot pages stay in RAM.

⚓ Interview Anchor

"InnoDB stores data in 16KB pages — the clustered index IS the table, ordered by primary key on disk. A secondary index lookup costs two B-tree traversals: secondary index leaf → PK value, then PK B-tree → data row. The buffer pool is a page cache — we tune it to keep hit ratio above 99%."

🧠 Core Concepts▾
Pages — the fundamental unit of I/O
InnoDB reads and writes data in fixed-size 16KB pages. A single disk read always retrieves a full page, regardless of how many rows you need from it. Rows smaller than half a page are stored fully inline; larger rows overflow to separate pages (off-page columns).

EHR: A vitals row (~80 bytes) means ~200 rows per 16KB page. Fetching one patient's 50 vitals touches ≈1 page (sequential reads) when data is on the clustered index.
OS/DB: The ticket_scans table accumulates millions of rows/day — page density matters enormously for buffer pool efficiency.
Buffer Pool — the most important knob you have
The buffer pool is an in-memory cache of InnoDB pages. On every read, InnoDB first checks the buffer pool (cache hit = no disk I/O). On a cache miss, it reads the page from disk and loads it into the pool. The hit ratio should exceed 99% in production. Setting innodb_buffer_pool_size to ~70–80% of RAM is standard.

EHR: CareMate's 16GB server: buffer pool = 12G. Patient records and vital sign pages stay warm — API response times drop from 80ms to 4ms.
OS/DB: Fraport 128GB nodes: buffer pool = 96G. At peak operations, 99.97% hit ratio keeps query latency under 2ms for the ticketing engine.
Clustered Index — the table IS the index
InnoDB organises the table data itself as a B-tree keyed on the primary key. There is no separate "data file" — the PK B-tree leaves contain the full row data. This means: PK lookups are single-traversal. Range scans on PK are sequential on disk (fast). Secondary index lookups require two traversals. Choosing a bad PK causes random I/O and page fragmentation.

EHR: patients(id INT AUTO_INCREMENT) — sequential PK inserts never fragment the clustered index.
OS/DB: Using a UUID as PK on bookings causes random page writes and 50% buffer pool waste — always use sequential IDs or ULIDs on high-write tables.
Tablespaces — where data physically lives
By default (innodb_file_per_table=ON), each table gets its own .ibd file. The system tablespace (ibdata1) still holds the data dictionary and undo segments (until MariaDB 10.6+ where undo is separate). File-per-table enables: reclaiming disk space with OPTIMIZE TABLE, faster DROP TABLE, partial backups.

EHR: Archiving old audit records from visits_audit to cold storage is simpler with file-per-table — OPTIMIZE TABLE reclaims the freed space.
OS/DB: With shared tablespace, DROP TABLE on a 500GB historical table doesn't release disk space — file-per-table is mandatory on OS's infrastructure.
Row Formats — DYNAMIC vs COMPACT
MariaDB InnoDB default is DYNAMIC (from MariaDB 10.1+). DYNAMIC stores overflow columns off-page with only a 20-byte pointer inline. COMPACT stores the first 768 bytes inline. DYNAMIC is better for tables with TEXT/BLOB/JSON columns — avoids double-reading large fields. Row format affects: row size, overflow threshold, and maximum index key length (3072 bytes with DYNAMIC).

EHR: visits.notes is TEXT — DYNAMIC stores it off-page, keeping the main B-tree pages lean and cache-friendly.
OS/DB: Ticket JSON payloads in ticket_scans (if stored as JSON column) similarly benefit from DYNAMIC off-page storage.
🔬 Deep Dive — InnoDB Architecture▾
ℹ️ Architecture

Understanding the internal memory and disk components of InnoDB helps you diagnose performance issues before they become 3am incidents.

 ┌──────────────────── Buffer Pool (RAM) ─────────────────────┐
 │                                                             │
 │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
 │  │  Data Pages │  │ Index Pages │  │ Undo Pages  │        │
 │  │  (B-tree    │  │  (Secondary │  │  (old row   │        │
 │  │   leaves)   │  │   indexes)  │  │  versions)  │        │
 │  └─────────────┘  └─────────────┘  └─────────────┘        │
 │                                                             │
 │  LRU List: young (hot) ──────────────────► old (cold)      │
 │  Flush List: dirty pages waiting to be written to disk     │
 └──────────────────────────────────────┬────────────────────-┘
                                        │ background flush
                                        ▼
 ┌──── InnoDB Redo Log (iblogfile0/1) ──────┐ ← WAL: every
 │  Sequential circular buffer on disk      │   commit written
 │  innodb_log_file_size × 2 files          │   here first
 └─────────────────────────────┬────────────┘
                               │ checkpoint
                               ▼
 ┌──── Data Files (.ibd tablespace files) ──────────────────┐
 │  Clustered index B-tree (PK → row data)                  │
 │  Secondary indexes B-tree (col → PK ref)                 │
 │  ibdata1: system tablespace, undo logs (pre-10.6)        │
 └──────────────────────────────────────────────────────────┘

The critical insight: a committed write only needs to hit the redo log, not the data file. Pages in the buffer pool are marked "dirty" and flushed asynchronously. This is why InnoDB is fast — sequential log writes are orders of magnitude faster than random data file writes.

🩺 EHR Example — Morafek CareMate▾
🩺 Context

Diagnosing buffer pool efficiency on the CareMate production server — monitoring hit ratio, finding fragmented tables, and checking tablespace sizes to plan archival.

SQL🩺 EHR
-- 1. Buffer pool hit ratio — should be above 99% SELECT VARIABLE_VALUE INTO @reads FROM information_schema.GLOBAL_STATUS WHERE VARIABLE_NAME = 'Innodb_buffer_pool_reads'; SELECT VARIABLE_VALUE INTO @read_requests FROM information_schema.GLOBAL_STATUS WHERE VARIABLE_NAME = 'Innodb_buffer_pool_read_requests'; SELECT ROUND((1 - (@reads / @read_requests)) * 100, 2) AS hit_ratio_pct; -- Target: > 99.0%. Below 97% = buffer pool too small or data too large -- 2. Find fragmented CareMate tables (high data_free = fragmentation) SELECT table_name, ROUND(data_length / 1024 / 1024, 1) AS data_mb, ROUND(data_free / 1024 / 1024, 1) AS free_mb, ROUND(data_free / data_length * 100, 1) AS fragmentation_pct FROM information_schema.TABLES WHERE table_schema = 'carematephr' AND engine = 'InnoDB' AND data_free > 50 * 1024 * 1024 -- only tables with >50MB free space ORDER BY fragmentation_pct DESC; -- 3. Rebuild fragmented table (reclaims disk space, re-sorts rows) -- Warning: acquires MDL lock. Use pt-online-schema-change on production! OPTIMIZE TABLE visits; -- 4. Verify row format on all CareMate tables SELECT table_name, row_format, create_options FROM information_schema.TABLES WHERE table_schema = 'carematephr' AND row_format != 'Dynamic'; -- should return 0 rows on modern MariaDB

⚠️ Common Pitfalls▾
⚠ Pitfall — UUID Primary Keys

Using UUID (version 4) as a primary key on a high-write InnoDB table causes random-order page writes. Because the clustered index is sorted by PK, every insert lands in a random position — blowing the buffer pool with pages that are only 50% full and causing constant random I/O. Use AUTO_INCREMENT or UUID v7 / ULID (which are time-ordered) instead.

⚠ Pitfall — Buffer Pool Size Too Small

Setting innodb_buffer_pool_size to the default 128MB is catastrophic on any real workload. Every query that touches hot data hits disk. Rule of thumb: 70–80% of available RAM. On a 16GB server, set 12G. Also set innodb_buffer_pool_instances = 8 (or 1 per GB) to reduce internal mutex contention with multiple threads.

⚠ Pitfall — OPTIMIZE TABLE on Large Production Tables

OPTIMIZE TABLE rebuilds the table and acquires a metadata lock (MDL) for the duration — blocking all reads and writes. On a 50GB table this takes 30+ minutes. In production, use pt-online-schema-change from Percona Toolkit or MariaDB's own online DDL. Never run OPTIMIZE TABLE directly on an OS GmbH production table without a maintenance window.

Phase 04 · Lesson 02 · 3h
MVCC & Concurrency Control

Multi-Version Concurrency Control is why MariaDB reads never block writes. Instead of locking rows for reads, InnoDB keeps old versions in the undo log and serves them to readers via snapshot isolation. When Nietsch asks about the "history list length" alarm — this is the mechanism behind it.

Click any section to expand
⏪ Backward Reference — Phase 03, Lesson 04

Phase 03 (Lesson 04) covered transaction isolation levels and FOR UPDATE from the SQL developer's perspective. This lesson explains the engine mechanism underneath those behaviors — undo logs, read views, and the history list. The two lessons complement each other; neither is a substitute for the other.

⚡ Why This Matters▾

On OS GmbH's 24/7 railway operations database, thousands of simultaneous reads and writes happen every second. MVCC is why readers (reporting queries, FHIR export, analytics) don't block writers (new bookings, ticket scans). Understanding MVCC also explains the most common 3am incident at high-throughput databases: a long-running transaction causing the undo log and history list length to balloon, triggering "InnoDB table full" errors and degrading every write on the server.

⚓ Interview Anchor

"MVCC means readers never block writers. When a row is updated, InnoDB writes a new version and keeps the old in the undo log. Readers with an older snapshot transparently see the old version. The danger is long-running transactions — they prevent the purge thread from cleaning old versions, causing history list length to grow, disk to fill, and performance to degrade."

🧠 Core Concepts▾
MVCC — the core promise
Multi-Version Concurrency Control keeps multiple versions of a row simultaneously. When Transaction A updates a row, InnoDB does NOT overwrite it in place — it creates a new version and chains the old version into the undo log. Transaction B that started before A's commit reads the old version. This means SELECT never waits for an UPDATE to commit, and UPDATE never waits for a SELECT to finish.

EHR: A CareMate reporting query reading 6 months of vitals runs for 30 seconds. During those 30 seconds, nurses continue entering new vitals without blocking.
OS/DB: Nightly revenue reports run alongside real-time ticket sales. MVCC makes this possible without application-layer coordination.
Undo Log — where old versions live
The undo log is a sequential log of "how to undo" each change. For MVCC, it stores the before-image of modified rows. Each row version has a DB_TRX_ID (the transaction that created it) and a DB_ROLL_PTR (pointer to the previous version in the undo log). A read view (snapshot) is established at transaction start — InnoDB walks the undo log chain backward until it finds a version visible to the snapshot.

EHR: If a nurse corrects a patient's blood pressure reading (UPDATE), the original value stays in the undo log for any concurrent reads that started before the update.
OS/DB: Undo log is stored in the system tablespace (ibdata1) on older MariaDB — it cannot shrink. Keep transactions short to prevent unbounded growth.
Read View — your transaction's snapshot
Under REPEATABLE READ (MariaDB default), a read view is created at the first statement of a transaction. It captures: the current transaction ID, the list of active transaction IDs, and the maximum committed transaction ID. Any row version created by a transaction in the "active" list is invisible — the reader walks the undo log to find the last committed version visible before its snapshot.

EHR: A FHIR export transaction opened at 14:00 sees a consistent snapshot as of 14:00, even if dozens of vitals are updated during the export. The FHIR document is internally consistent.
OS/DB: Under READ COMMITTED (non-default), a new read view is created per statement — different statements in the same transaction can see different committed data. This is sometimes desired for reporting but can cause phantom rows.
History List Length — the MVCC danger metric
The purge thread periodically removes undo log entries for row versions that are no longer needed by any active transaction. If a long-running transaction holds a read view from the past, the purge thread cannot advance — all undo entries since that snapshot must be kept. This causes history list length to grow. When it exceeds ~10,000 it signals serious trouble: undo log filling disk, queries slowing as version chains lengthen.

EHR: A developer accidentally left a transaction open via a debug session at 09:00. By 15:00 history list length is 50,000. Every write to CareMate is slower — a 6-hour open transaction holds the purge back.
OS/DB: On KRITIS infrastructure, history list > 1,000 triggers an alert. The fix: find the oldest transaction (INNODB_TRX) and kill it.
InnoDB vs PostgreSQL MVCC — a key comparison
Both databases implement MVCC but with very different mechanics. InnoDB stores old versions in the undo log (separate from the table) — cleanup is handled by the purge thread, transparent to users. PostgreSQL stores old row versions (dead tuples) inline in the heap table — cleanup requires explicit VACUUM. InnoDB advantage: no manual VACUUM, no table bloat from dead tuples. PostgreSQL advantage: undo log size doesn't affect table size or index reads.

EHR: CareMate uses MariaDB — no VACUUM to schedule. The equivalent concern is monitoring history list length and purge thread lag.
OS/DB: If OS were on PostgreSQL, autovacuum would be a critical operational concern — especially for high-UPDATE tables like booking status.
🔬 Deep Dive — MVCC Version Chain▾
MVCC version chain — what InnoDB stores for a single row

Current row in clustered index B-tree (most recent committed version):
┌─────────────────────────────────────────────────────────┐
│ id=42 | systolic=145 | DB_TRX_ID=1050 | DB_ROLL_PTR ───┼──┐
└─────────────────────────────────────────────────────────┘  │
                                                             │
Undo log (previous versions, chained backward):             │
◄─────────────────────────────────────────────────────────  │
┌───────────────────────────────────────────┐               │
│ id=42 | systolic=138 | DB_TRX_ID=1040  ◄─┼───────────────┘
│                       | DB_ROLL_PTR ────┼──┐
└───────────────────────────────────────────┘ │
┌───────────────────────────────────────────┐ │
│ id=42 | systolic=130 | DB_TRX_ID=1020  ◄─┼─┘
└───────────────────────────────────────────┘
(end of chain — this is the original insert version)

A SELECT in transaction with read_view trx_min=1045 sees:
  - TRX_ID=1050 ≥ 1045 → invisible (committed after our snapshot)
  - TRX_ID=1040 < 1045 → VISIBLE → returns systolic=138 ✓

The longer the version chain, the slower old-version reads become — InnoDB must walk more undo log pointers. This is another reason long-running transactions hurt even read performance: they force other readers to walk longer chains.

🚂 OS/DB Example — Deutsche Bahn▾
🚂 OS/DB Context

On OS GmbH's railway operations cluster, monitoring MVCC under high concurrency: the bookings table sees thousands of UPDATE statements per second. Understanding how isolation levels interact with MaxScale read/write split is critical.

SQL🚂 OS/DB
-- 1. Show all active transactions with their lock info SELECT t.trx_id, t.trx_state, t.trx_started, TIMESTAMPDIFF(MINUTE, t.trx_started, NOW()) AS age_min, t.trx_rows_locked, t.trx_rows_modified, t.trx_query -- current or last SQL statement FROM information_schema.INNODB_TRX t ORDER BY t.trx_started ASC; -- 2. Demonstrate MVCC in action — two sessions simultaneously -- SESSION A (booking system — updates seat status): START TRANSACTION; UPDATE bookings SET status = 'boarded' WHERE id = 88341; -- NOT COMMITTED YET -- SESSION B (conductor app — reads same booking): SELECT status FROM bookings WHERE id = 88341; -- Returns: 'confirmed' (old version via MVCC) — no blocking! -- SESSION A commits: COMMIT; -- SESSION B (if REPEATABLE READ) still sees 'confirmed' — snapshot locked at start -- SESSION B (if READ COMMITTED) would now see 'boarded' on next SELECT -- 3. Set per-session isolation for analytics (READ COMMITTED = less locking overhead) SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED; -- Used by MaxScale when routing reporting queries to read replicas
⚠️ Common Pitfalls▾
⚠ Pitfall — BEGIN but never COMMIT

Application code that opens a transaction and crashes (or the developer closes their SQL client) leaves an idle open transaction. In REPEATABLE READ, this holds a read view from the moment it started — preventing the purge thread from advancing. On a busy system, the history list can reach tens of thousands within minutes. Always set wait_timeout and interactive_timeout to kill idle connections, and use connection poolers (MaxScale) that enforce timeouts.

⚠ Pitfall — Long Reads Block Purge

A read-only analytics query that runs for 2 hours holds a read view for 2 hours — even though it modifies no data. The purge thread cannot advance past the start of that query. During those 2 hours, undo log grows, every write to any table gets a longer version chain, and disk may fill. On OS GmbH infrastructure, run long reports on a dedicated read replica with its own purge thread, not on the primary.

⚠ Pitfall — Checking History List Length Reactively

By the time you notice symptoms (slow writes, disk filling), the history list may already be at 100,000+. Monitor it proactively — add Innodb_history_list_length to Prometheus/Grafana with a warning threshold of 1,000 and a critical threshold of 10,000. Alert before it becomes a 3am incident.

Phase 04 · Lesson 03 · 3h
WAL, Binlog & Crash Recovery

How MariaDB guarantees your data survives a power cut. The Write-Ahead Log (InnoDB redo log) is the engine-level durability guarantee. The binary log is the server-level replay log for replication and PITR. Understanding both — and especially the single variable that controls the trade-off between safety and speed — is core OS GmbH interview material.

Click any section to expand
⚡ Why This Matters▾

OS GmbH operates KRITIS infrastructure — critical national infrastructure for Germany's railway network. Zero data loss is not a nice-to-have; it is a regulatory requirement. The WAL and binary log are the mechanisms that make this possible. Knowing what innodb_flush_log_at_trx_commit=1 means and why the alternatives are dangerous is the kind of nuance Nietsch will probe for. For Morafek CareMate EHR, DSGVO mandates data integrity — a crash that loses committed patient data is a regulatory violation, not just a technical problem.

⚓ Interview Anchor

"innodb_flush_log_at_trx_commit=1 means every COMMIT flushes the redo log to disk — that is the only fully ACID-safe setting. Value 0 flushes once per second (lose up to 1 second of commits on crash). Value 2 writes to OS buffer but not disk (lose up to 1 second on OS crash). For KRITIS infrastructure at OS GmbH, only value 1 is acceptable."

🧠 Core Concepts▾
Write-Ahead Log (WAL) — durability before speed
Before any data page is modified in the buffer pool, the change is written to the InnoDB redo log (a sequential file on disk). On COMMIT, the log entry is flushed. On crash, InnoDB replays the redo log to re-apply changes that were in the buffer pool but not yet written to data files. This is called crash recovery and happens automatically on next startup.

EHR: If the CareMate server loses power mid-transaction, on restart InnoDB reads the redo log: committed transactions are replayed, uncommitted ones are rolled back using the undo log. No manual intervention needed.
OS/DB: OS's Galera cluster writes redo log on every node — crash recovery on a restarted node replays the log before rejoining the cluster (IST path).
innodb_flush_log_at_trx_commit — the safety dial
1 (default, safe): Log buffer → redo log file → fsync to disk on every COMMIT. Fully ACID-compliant. Slowest. 2 (fast, some risk): Log buffer → OS buffer on COMMIT. OS flushes to disk once per second. Survives MariaDB crash but loses up to 1 second if OS crashes. 0 (fastest, dangerous): Log buffer flushed once per second regardless of COMMIT. Can lose up to 1 second of commits even on clean shutdown. Never use 0 or 2 on KRITIS infrastructure.

EHR: CareMate MUST use =1. A committed vital sign reading must survive a server crash — DSGVO auditing requires it.
OS/DB: =1 with sync_binlog=1 is the combination for zero data loss on OS GmbH infrastructure.
Binary Log — server-level replay log
The binary log is separate from the InnoDB redo log. It operates at the MariaDB server level, above the storage engine. It records every DDL and DML (in ROW format: the exact before/after row images). Uses: (1) Replication — primary sends binlog events to replicas. (2) PITR — replay binlog from last backup to target time. The binlog captures what changes were made and to which rows, not the internal page operations.

EHR: CareMate runs with log_bin=ON. If a developer runs DELETE FROM vitals WHERE 1=1, we restore the last backup and replay the binlog up to T-1 second to recover all data.
OS/DB: Galera cluster requires binlog_format=ROW — STATEMENT format is non-deterministic across nodes and will cause data divergence.
Checkpoints — when WAL gets applied to disk
A checkpoint marks the point up to which dirty pages have been flushed from the buffer pool to disk. The redo log is circular — once a checkpoint advances, the log space before it can be reused. If dirty pages accumulate faster than InnoDB flushes them, the redo log fills up and MariaDB throttles writes to force a checkpoint. Tuning: innodb_log_file_size controls how long recovery takes (larger = slower restart) but how much burst write capacity you have.

EHR: innodb_log_file_size=512M is typical for CareMate. Larger values mean more data can be written before a checkpoint is forced — smoother throughput at the cost of longer crash recovery.
OS/DB: On Fraport's high-throughput cluster: innodb_log_file_size=2G. Recovery after a crash takes ~5 minutes but avoids write stalls during peak operations.
sync_binlog — the other durability knob
sync_binlog=1 means the binary log is synced to disk on every transaction commit (via fsync). sync_binlog=0 lets the OS decide when to flush — fastest but risks losing binlog events on OS crash. For zero data loss replication and PITR integrity, use both innodb_flush_log_at_trx_commit=1 AND sync_binlog=1. These two together are called "the gold standard" for durable MariaDB configuration.

EHR: Without sync_binlog=1, a crash could mean the binlog doesn't contain the last few committed transactions — PITR would miss them.
OS/DB: OS GmbH production: always sync_binlog=1. The performance cost (one extra fsync per commit) is accepted because zero data loss is a regulatory requirement.
🔬 Deep Dive — Write Path & Crash Recovery▾
THE WRITE PATH — what happens on a single UPDATE + COMMIT

Application: UPDATE vitals SET systolic=145 WHERE id=1001

Step 1: InnoDB writes undo log entry (old value for MVCC + rollback)
        undo log: "id=1001 had systolic=138"

Step 2: Modify page in buffer pool (dirty page — NOT yet on disk)
        buffer pool: page containing id=1001 updated to systolic=145

Step 3: Write to InnoDB redo log buffer (in RAM)
        redo log buffer: "page X, offset Y: systolic=138→145, trx=1050"

COMMIT:
Step 4a: innodb_flush_log_at_trx_commit=1 → fsync redo log to disk ✓
Step 4b: sync_binlog=1 → write binlog event → fsync binlog to disk ✓
Step 4c: Return "Query OK" to client

Later (asynchronous):
Step 5: Background thread flushes dirty page from buffer pool to .ibd file

CRASH between Step 3 and Step 5:

⚠️ Common Pitfalls▾
⚠ Pitfall — innodb_flush_log_at_trx_commit=2

Value 2 is often presented as a "safe" compromise — it writes to the OS buffer on every commit but only fsyncs to disk once per second. This is NOT safe for KRITIS infrastructure. An OS kernel crash or power failure can lose up to 1 second of commits even with value=2. On OS GmbH infrastructure, the answer to "why not value 2?" is: "a DB admin crash can be survived; an OS crash, which is more likely during a hardware failure, cannot."

⚠ Pitfall — Deleting Binlog Files Manually

Never use rm to delete binary log files. MariaDB maintains an index file (mariadb-bin.index) that lists active log files. Manual deletion causes index corruption, replication breaks, and PITR gaps. Always use: PURGE BINARY LOGS BEFORE '2024-06-01 00:00:00'; or set expire_logs_days.

⚠ Pitfall — Not Verifying PITR Before a Crisis

The worst time to discover your PITR procedure doesn't work is during a production data loss event at 3am. Test the full procedure (backup restore + binlog replay) monthly on a staging server. Keep a printed runbook. Verify that your backup user's binlog position is recorded in the backup metadata (xtrabackup_binlog_info file inside the backup directory).

TRY ITWhat's the difference between the redo log and the binary log?
▾

Explain to a colleague in one paragraph why MariaDB has BOTH a redo log (InnoDB) AND a binary log (server level). What does each do, and why can't one replace the other?

Show answer ↓
✓ Answer

The redo log is engine-internal: it records page-level changes needed to redo work that was in the buffer pool but not yet written to disk files. It is circular, overwritten when checkpointed, and used only for crash recovery on restart — you cannot replay it across servers or to specific timestamps. The binary log operates at the server level above all storage engines: it records logical SQL operations (or row images in ROW format) and is retained for days. It enables replication (streaming events to replicas) and point-in-time recovery (replaying to a specific timestamp after restoring a backup). Neither can replace the other: the redo log doesn't know about replication topology; the binary log doesn't help if you crash before a checkpoint is written.

Phase 04 · Lesson 04 · 3h
Locking: Row Locks, Table Locks & Deadlocks

Deadlocks are inevitable on any sufficiently concurrent system. What matters is detecting them fast, understanding their anatomy, and designing the application to minimise them. On OS GmbH's ticketing infrastructure, a deadlock that cascades across MaxScale connections can take down a booking service. This lesson covers the full locking hierarchy — from shared row locks to gap locks — and gives you the diagnostic queries to resolve them under pressure.

Click any section to expand
⚡ Why This Matters▾

"Lock wait timeout exceeded" and "Deadlock found when trying to get lock" are among the most common errors in production MariaDB deployments. At OS GmbH, where thousands of simultaneous seat reservations hit the bookings table every minute during peak travel, lock contention is a design challenge, not an edge case. Morafek CareMate's multi-user EHR has concurrent nurse sessions updating the same patient records — without proper locking understanding, data corruption or performance collapse is a matter of when, not if.

⚓ Interview Anchor

"InnoDB auto-detects deadlocks in ~1 second and rolls back the transaction with fewer undo records. The key to preventing them: always acquire locks in a consistent order across all transactions. Use SELECT ... FOR UPDATE to explicitly acquire row locks before writing. Gap locks in REPEATABLE READ prevent phantom reads but can cause surprising contention — know when to switch to READ COMMITTED."

🧠 Core Concepts▾
Shared (S) vs Exclusive (X) Locks
Shared lock (S): multiple transactions can hold it on the same row simultaneously. Acquired by: SELECT ... LOCK IN SHARE MODE. Exclusive lock (X): only one transaction at a time. Acquired by: UPDATE, DELETE, INSERT, SELECT ... FOR UPDATE. A plain SELECT in MVCC takes NO lock — it reads the snapshot. Lock compatibility: S + S = compatible. S + X = incompatible (X must wait). X + X = incompatible.

EHR: A CareMate medical review transaction uses SELECT ... FOR UPDATE on a patient record before updating their treatment plan — ensuring no other session can concurrently modify it.
OS/DB: The booking service acquires an X lock on a seat row before confirming it — preventing double-booking when two passengers request the same seat simultaneously.
Gap Locks — preventing phantom reads
In REPEATABLE READ, when InnoDB scans a range (e.g. WHERE id BETWEEN 10 AND 20), it locks not just the existing rows but the gaps between them. This prevents another transaction from inserting a new row into that range during your transaction — avoiding phantom reads. Gap locks are only used in REPEATABLE READ; READ COMMITTED eliminates them (using only record locks), which reduces contention at the cost of allowing phantom reads.

EHR: A gap lock on patient IDs 100–200 prevents inserting a new patient in that range while an audit query is scanning it. Usually harmless in EHR workloads.
OS/DB: On the ticket_scans table with range queries on station_id, gap locks can cause INSERT contention during high-frequency scan events. Switching scan-heavy transactions to READ COMMITTED eliminates the gap locks.
Intention Locks — table-level signals
Before acquiring a row-level lock, InnoDB places an intention lock on the table: Intention Shared (IS) or Intention Exclusive (IX). These are lightweight table-level locks that signal "I intend to lock some rows in this table." They allow concurrent row-level locking while preventing whole-table operations (like LOCK TABLE) from conflicting. You'll see them in SHOW ENGINE INNODB STATUS as IS, IX, S, X next to table names.

EHR: Concurrent UPDATE on two different patient rows: Session A places IX on the patients table, locks row 42. Session B places IX on the same table, locks row 107. IX + IX are compatible — both proceed concurrently.
OS/DB: A LOCK TABLES bookings WRITE waits until all IX locks are released — this is why table-level locking is avoided in OLTP; it blocks every concurrent DML.
Deadlock Anatomy
A deadlock occurs when Transaction A holds lock X and waits for lock Y, while Transaction B holds lock Y and waits for lock X — a circular dependency. InnoDB's deadlock detector runs on every lock request, checks for cycles, and resolves by rolling back the transaction with fewer undo records. The rolled-back transaction gets error 1213: "Deadlock found when trying to get lock; try restarting transaction." The application must handle this by retrying.

EHR: Session A locks patient 42 → tries to lock visit 100. Session B locks visit 100 → tries to lock patient 42. Deadlock. InnoDB rolls back B (1 row modified vs A's 2). Fix: always lock patient first, then visit — consistent order.
OS/DB: Two booking transactions locking seats in different orders cause deadlocks. Fix: ORDER BY seat_id in the application's lock acquisition sequence.
Lock Wait Timeout — the softer failure
Before a deadlock is detected, a transaction waiting for a lock that another holds will wait up to innodb_lock_wait_timeout seconds (default 50 seconds). If the wait exceeds this, InnoDB gives up with error 1205: "Lock wait timeout exceeded; try restarting transaction." Only the waiting transaction is rolled back, not the holder. This is different from a deadlock — it's just one transaction waiting too long.

EHR: A CareMate batch job holds a lock on 1,000 patient rows while processing. Concurrent API requests time out waiting for those locks. Fix: batch in smaller chunks to release locks between batches.
OS/DB: On MaxScale, a long-running write on the primary can cause all routing traffic queued behind it to exceed lock_wait_timeout. Monitor and set connection-level timeouts in application code.

⚠️ Common Pitfalls▾
⚠ Pitfall — Not Handling Error 1213 in Application Code

InnoDB resolves deadlocks by rolling back one transaction — but it does NOT retry it. The application receives error 1213 and must retry the entire transaction. Most ORM frameworks do NOT retry automatically. This means deadlock-prone code silently fails unless the application explicitly catches 1213 and retries (typically 2–3 times with jitter). Verify this in every service that writes to MariaDB.

⚠ Pitfall — SELECT ... FOR UPDATE Across Multiple Tables in the Wrong Order

The most reliable way to create deadlocks: Transaction A locks table X then Y; Transaction B locks table Y then X. Always document and enforce a canonical lock order in your team's coding standards. For OS GmbH: passengers → routes → bookings (in entity ID order when locking multiple rows of the same table).

⚠ Pitfall — SHOW ENGINE INNODB STATUS Only Shows the LATEST Deadlock

SHOW ENGINE INNODB STATUS retains only the most recent deadlock — it's overwritten on the next one. For deadlock analysis at scale, enable innodb_print_all_deadlocks=ON in my.cnf. This writes every deadlock to the MariaDB error log, where you can grep and aggregate them to find patterns. Essential for debugging intermittent deadlocks that only occur during peak traffic.

🎤 Interview Questions — Michael Nietsch▾
What happens inside InnoDB when you run SELECT * FROM patients WHERE id = 42?

InnoDB checks the buffer pool for the page containing row id=42 (a page lookup by PK value in the clustered index B-tree). If the page is in the pool (cache hit), the row is returned directly from RAM with zero disk I/O. If not (cache miss), InnoDB reads the 16KB page from the .ibd data file into the buffer pool, then returns the row. Because id is the primary key, this is a single B-tree traversal to the leaf page containing the row — extremely fast (O log n). A secondary index lookup would require two traversals: secondary index leaf (yields the PK), then PK B-tree to the data row.

⚓ Interview Anchor

"Buffer pool first — if the page is cached, zero disk I/O. PK = single B-tree lookup, returns row from clustered index leaf. Secondary index = two lookups: index leaf → PK, then PK → row data."

Why is innodb_flush_log_at_trx_commit=1 mandatory for production? What does it actually do?

On every COMMIT, value=1 writes the redo log buffer to the redo log file AND calls fsync() — this forces the OS to flush the file to physical storage. This guarantees that once "Query OK" is returned to the client, the change is on durable storage and will survive any crash. Value=2 writes to the redo log file but does NOT fsync — a kernel crash or power failure could lose the last ~1 second of commits even though they were "committed." Value=0 doesn't even write to the file on each commit, only every second — worst safety guarantee. For OS GmbH operating KRITIS infrastructure, any data loss is unacceptable, and only value=1 provides the guarantee.

⚓ Interview Anchor

"Value=1 means every COMMIT calls fsync — data is physically on disk before we return OK. Value=2 skips the fsync — an OS crash loses the last second of commits. For KRITIS infrastructure, only 1 is acceptable."

Your monitoring shows the InnoDB history list length has risen from 50 to 8,000 over the last 4 hours. Walk me through your diagnosis.

The history list length represents the backlog of undo log entries that the purge thread hasn't been able to clean. It rises when an active transaction holds a read view from 4 hours ago — the purge thread cannot remove any undo records created after that snapshot. My steps: (1) Query INNODB_TRX ordered by trx_started — find the transaction that started ~4 hours ago. (2) Check trx_mysql_thread_id and trx_query — if trx_query is NULL, it's an idle open transaction. (3) KILL the thread — the purge thread immediately resumes and the history list drops rapidly. (4) Investigation: was this a developer debug session, a reporting query, or a connection pool bug? (5) Prevention: set wait_timeout=300 to kill idle connections, add connection-level transaction timeouts in the application.

⚓ Interview Anchor

"History list rising = a long-running transaction blocking the purge thread. Find it in INNODB_TRX ordered by trx_started. KILL the thread. Monitor the history list drop. Prevent it: wait_timeout and connection pooling with explicit transaction timeouts."

What's the difference between a deadlock and a lock wait timeout, and how does InnoDB handle each?

A deadlock is a circular lock dependency: Transaction A waits for B, B waits for A. Neither can proceed. InnoDB's deadlock detection algorithm runs synchronously on each lock request, detects cycles using a depth-first graph traversal, and immediately resolves the deadlock by rolling back the transaction with fewer undo records — returning error 1213 to the victim's application. A lock wait timeout is simpler: Transaction A is waiting for a lock held by Transaction B, but no circular dependency exists — B will eventually release the lock. If the wait exceeds innodb_lock_wait_timeout (default 50 seconds), InnoDB gives up and returns error 1205 to the waiting transaction only — the holder is not affected. Application handling differs: 1213 (deadlock) should always be retried. 1205 (timeout) may indicate a real issue (B is stuck) and requires more analysis before retry.

⚓ Interview Anchor

"Deadlock = circular dependency, detected immediately, one victim rolled back (1213). Lock timeout = one-way wait that exceeds innodb_lock_wait_timeout, only the waiter fails (1205). Both need application-level retry logic."

Explain how MVCC allows a long-running SELECT to return consistent data even while concurrent UPDATEs are happening on the same rows.

When a transaction starts in REPEATABLE READ, InnoDB creates a read view: it records the current transaction ID, the list of all active transaction IDs at that moment, and the maximum committed transaction ID. For every row it reads, InnoDB checks the row's DB_TRX_ID against the read view. If the row's transaction ID is in the "active" list (i.e., it was being modified when our snapshot was taken), that version is invisible — InnoDB follows the DB_ROLL_PTR to the previous version in the undo log, and repeats until it finds a version that was committed before our snapshot. This means the SELECT sees a frozen snapshot of the data as of when it started, regardless of how many concurrent UPDATEs occur — without taking any locks on the rows it reads.

⚓ Interview Anchor

"MVCC: each row has a transaction ID and a pointer to its previous version in the undo log. The read view filters out rows modified after our snapshot. We walk the undo log chain backward until we find a visible version — no row locks needed for reads."

What is a gap lock and when would you disable it?

A gap lock prevents insertion into a range between index values. For example, if you lock WHERE id BETWEEN 100 AND 200, InnoDB also locks the "gap" between 100 and 200 so no new row can be inserted there. This prevents phantom reads in REPEATABLE READ. Gap locks are only applied in REPEATABLE READ isolation level — READ COMMITTED removes them, using only record locks on existing rows. You would disable gap locking by switching to READ COMMITTED in scenarios where: (1) High-concurrency INSERT workloads are being slowed by gap lock contention. (2) Phantom reads are acceptable for your application. (3) You're running on a Galera cluster where extra gap locking overhead isn't needed (Galera handles its own certification). The trade-off is clear: READ COMMITTED means SELECT might see different committed rows in different statements of the same transaction — acceptable for OLAP/reporting, risky for OLTP write logic.

⚓ Interview Anchor

"Gap locks prevent phantom reads by locking index ranges, not just existing rows. Only active in REPEATABLE READ. Set READ COMMITTED to eliminate them — higher INSERT concurrency at the cost of allowing phantom reads."

Show model answer ↓
✓ Model Answer (60 seconds)

"When a transaction commits, InnoDB first writes the change to the undo log — that's what enables MVCC reads to see the old version and what lets us roll back if needed. The actual data change is made in the buffer pool — in RAM, on the 16KB page. The redo log buffer records the physical change. On COMMIT with flush_log=1, we fsync the redo log to disk — that's the durability guarantee. The data page is marked dirty in the buffer pool and flushed to disk later by the background thread. The binary log is also fsynced if sync_binlog=1 — that's for replication and PITR. If the server crashes before the dirty page is written but after the redo log fsync, InnoDB replays the redo log on restart and the data is safe. That's the write-ahead guarantee — log first, data second."
