Phase 05 · Lesson 01 · 3h
my.cnf Deep Dive

InnoDB buffer pool sizing, connection pool tuning, binary log configuration, and the critical variables Michael Nietsch expects you to know cold. Every setting here has a direct operational consequence.

Click any section to expand
💡 Why This Matters▾

The /etc/my.cnf (or /etc/mysql/mariadb.conf.d/50-server.cnf) file is the control panel for all of MariaDB's runtime behaviour. Misconfigured defaults — a 128MB buffer pool, default max_connections=151, or disabled binary logging — are the single most common cause of performance disasters and unrecoverable data loss on production systems. For OS GmbH's 24/7 railway infrastructure and the Morafek CareMate EHR platform, getting this file right is a prerequisite for everything else: replication, Galera, MaxScale, and PITR all depend on correct base configuration.

🎯 Interview Anchor

"The first thing I do on a new MariaDB server is audit four areas of my.cnf: InnoDB buffer pool (should be 70–80% of RAM), innodb_log_file_size (at least 512MB for write-heavy workloads), binary logging (must be ON with ROW format for replication and PITR), and max_connections versus innodb_thread_concurrency to avoid the thundering herd problem."

🧠 Core Concepts▾
innodb_buffer_pool_size
The single most important InnoDB parameter. It defines how much RAM InnoDB uses to cache data pages and index pages. The goal is to keep your entire working set (hot data) in memory to avoid disk reads. Set to 70–80% of total RAM. On a 128GB server, use 96G. With multiple pools set innodb_buffer_pool_instances to 8–16 (one per 6–8GB) to reduce mutex contention.

🩺 EHR: CareMate on a 32GB server → set 24G to cache patient + vitals hot rows in RAM and avoid disk hits on dashboard queries.
🚂 OS/DB: Fraport cluster nodes (128GB) → set 96G; 16 pool instances to support thousands of concurrent booking reads without I/O bottleneck.
innodb_log_file_size & innodb_log_files_in_group
The redo log (InnoDB's write-ahead log) determines how much write data can be buffered before being flushed. A small redo log (default 48MB) causes frequent checkpoints which stall I/O. For write-heavy workloads set to 512MB–2GB per file, with 2 files in the group. Changing this requires a clean shutdown + removal of old ib_logfile* files on older MariaDB; 10.8+ handles it online.

🩺 EHR: Bulk FHIR import batches that insert thousands of vitals rows per second benefit from a 512MB log file to prevent checkpoint stalls.
🚂 OS/DB: ticket_scans table receives millions of inserts per day; 1GB log files prevent I/O spikes during peak boarding windows.
max_connections & thread handling
max_connections caps how many simultaneous client sessions are allowed. Each connection consumes ~1MB RAM overhead. Setting this too high on a system without a connection pool (MaxScale, PgBouncer) causes RAM exhaustion. Pair with thread_handling=pool-of-threads (MariaDB's thread pool) to reuse OS threads and prevent the thundering-herd problem where all connections wake simultaneously.

🩺 EHR: Flask + SQLAlchemy app → max_connections=200, thread pool on, since most Flask connections are short-lived.
🚂 OS/DB: MaxScale handles the connection front-end; MariaDB nodes set max_connections=500 as a safety ceiling only.
Binary Logging (log_bin, binlog_format, expire_logs_days)
Binary logging records every data-changing statement (or row change in ROW mode) for replication and point-in-time recovery. Always use binlog_format=ROW — it is idempotent and required by Galera. expire_logs_days (or binlog_expire_logs_seconds in 10.6+) controls automatic purge. Set retention to cover your backup window plus 1–2 days buffer.

🩺 EHR: binlog_format=ROW is mandatory for DSGVO audit trails and Galera replication of CareMate data.
🚂 OS/DB: binlog retention of 7 days covers the weekly full backup cycle with headroom for PITR after any incident.
innodb_flush_log_at_trx_commit & sync_binlog
These two settings control durability (the D in ACID). Both set to 1 = full ACID compliance, one fsync per commit. Set to 0 or 2 = faster writes but potential data loss on crash. For any financial or medical system these must both be 1. For read replicas or analytics nodes, you may relax them.

🩺 EHR: Both = 1 on all CareMate nodes — patient data and medication records cannot be lost even on OS crash (DSGVO requirement).
🚂 OS/DB: Both = 1 on Galera primaries. Read-only analytics replica may use flush_log=2 to reduce I/O pressure.

⚠️ Common Pitfalls▾
⚠ Pitfall — Leaving defaults on production

The default MariaDB install has innodb_buffer_pool_size=128M, max_connections=151, and binary logging off. These defaults are designed for developer laptops, not servers. Every production MariaDB deployment must have a tuned my.cnf before go-live. "We kept the defaults" is not an acceptable answer in an OS GmbH incident debrief.

⚠ Pitfall — Setting innodb_buffer_pool_size > available RAM

If the buffer pool size plus OS needs plus per-connection memory overhead exceeds physical RAM, the kernel will use swap. Swapped database pages cause millisecond-range latency spikes that look like random slow queries. Rule: buffer_pool = RAM × 0.75, then subtract per-thread overhead (sort_buffer_size + join_buffer_size) × max_connections.

⚠ Pitfall — Changing innodb_log_file_size without clean shutdown

On MariaDB before 10.8, changing innodb_log_file_size requires: (1) clean shutdown with SET GLOBAL innodb_fast_shutdown=0; SHUTDOWN;, (2) delete ib_logfile0 and ib_logfile1, (3) start with new size. Skipping the clean shutdown risks InnoDB crash recovery finding a mismatch between the new file size and existing log records — the server will refuse to start.

🚨 3am Scenario▾
🚨 3AM SCENARIO — "MariaDB Refuses to Start After Config Change"

Situation: A junior admin changed innodb_log_file_size from 48M to 512M in my.cnf and restarted MariaDB with a forced kill. The server now refuses to start. Error log shows: InnoDB: Log file ./ib_logfile0 is of different size 50331648 bytes than specified in the .cnf file 536870912 bytes!

Your immediate diagnostic steps:

  1. Check the error log: tail -100 /var/log/mysql/error.log — confirm the size mismatch message
  2. Verify what size the existing log files actually are: ls -lh /var/lib/mysql/ib_logfile*
  3. Check if there is an incomplete checkpoint: look for "starting crash recovery" in error.log — if present, you cannot delete log files yet
  4. If no crash recovery in progress: revert my.cnf to old size (48M), start MariaDB, verify it comes up clean
  5. Once running: execute a clean shutdown: SET GLOBAL innodb_fast_shutdown=0; SHUTDOWN;
  6. Delete old log files: rm /var/lib/mysql/ib_logfile0 /var/lib/mysql/ib_logfile1
  7. Set new size in my.cnf, start MariaDB — InnoDB will create new log files at correct size

Resolution: Server back online in ~8 minutes. No data loss (redo log was intact at old size).

Prevention: Always do a clean shutdown before changing log file size. Add a pre-flight checklist to the runbook: "innodb_log_file_size change = clean shutdown required."

🎤 Interview Questions▾
How do you size innodb_buffer_pool_size on a new server, and what happens if it's set too low?

Start with 70–80% of physical RAM as a baseline. On a dedicated 32GB MariaDB server, set 24G. Then refine: check Innodb_buffer_pool_read_requests vs Innodb_buffer_pool_reads — if the disk read ratio is above 1% (hit ratio below 99%), increase the pool. If set too low, every query touching hot data that's not cached causes a physical disk read — on spinning disk this is 4–8ms per page, on SSD 0.1ms, but even SSD misses at scale become the bottleneck. At OS GmbH scale a low buffer pool means the fast NVMe storage becomes the limiting factor instead of compute.

🎯 Anchor

"innodb_buffer_pool_size = 75% of RAM, then validate with the hit ratio: (1 - reads/read_requests) × 100 must exceed 99%."

Why must binlog_format=ROW for Galera clusters? What risk does STATEMENT mode introduce?

Galera's certification-based replication certifies write sets — sets of row changes — not SQL statements. ROW format records the exact before/after state of each changed row, making the write set deterministic and idempotent. STATEMENT format sends the SQL text, which can be non-deterministic (functions like NOW(), RAND(), user-defined functions, triggers). Two nodes executing the same STATEMENT at different times could produce different results, breaking consistency. Galera will actually refuse to apply non-deterministic statements and abort the transaction. For OS GmbH, a split-brain caused by non-deterministic replication on the ticketing DB would be catastrophic.

🎯 Anchor

"ROW format makes replication deterministic. STATEMENT mode with functions like NOW() can produce different results on different nodes — Galera will reject these transactions."

What is the thundering herd problem with max_connections and how does the thread pool help?

When a slow event (GC pause, lock contention, slow disk) holds up connections, new connections pile up. If max_connections=1000 and all 1000 connections wake simultaneously when the bottleneck clears, the server is overwhelmed processing all requests at once — CPU spikes, more lock contention, cascading failures. The MariaDB thread pool (thread_handling=pool-of-threads) separates client connections from OS threads. You can have 1000 client connections but only 32 OS threads doing actual work. Requests queue within the pool and are dispatched one-at-a-time per thread, preventing the stampede. For OS GmbH, MaxScale also helps by bounding the connection count reaching each MariaDB node.

🎯 Anchor

"Thread pool decouples connections from OS threads — thousands of connections, but only thread_pool_size threads execute concurrently. MaxScale adds another layer by pooling connections before they reach MariaDB."

Phase 05 · Lesson 02 · 2h
Users, Roles & Security

Creating users with the ed25519 authentication plugin, granting least-privilege access using roles, auditing privileges, and hardening MariaDB against common attack vectors. DSGVO compliance requires strict access control on CareMate patient data.

Click any section to expand
💡 Why This Matters▾

DSGVO (GDPR) Article 32 requires technical measures ensuring that personal data can only be accessed by authorised personnel. On CareMate, patient records, diagnoses, and vitals are special-category data under Article 9. Any DBA at OS GmbH must be able to create least-privilege accounts, assign roles, enforce strong authentication, and immediately revoke access. The ed25519 plugin is MariaDB's modern, performance-efficient alternative to the older mysql_native_password — it provides faster authentication and stronger cryptographic guarantees without requiring TLS for the auth exchange.

🎯 Interview Anchor

"My security model is: one role per application function (read-only, read-write, admin), users granted only the role they need, ed25519 authentication plugin instead of mysql_native_password, and a quarterly privilege audit using information_schema.USER_PRIVILEGES."

🧠 Core Concepts▾
ed25519 Authentication Plugin
MariaDB's native ed25519 plugin uses Elliptic Curve Diffie-Hellman for challenge-response authentication. Faster than sha256_password, no TLS required for the auth handshake itself (though TLS for the connection is still recommended). Available since MariaDB 10.1.22. Install with: INSTALL SONAME 'auth_ed25519';

🩺 EHR: All CareMate application accounts use ed25519 — stronger than SHA-1-based native passwords for DSGVO-regulated patient data access.
🚂 OS/DB: MaxScale service accounts use ed25519; MaxScale 6+ natively supports it for backend authentication.
Roles (MariaDB 10.0+)
A role is a named collection of privileges that can be granted to multiple users. This implements RBAC (Role-Based Access Control). Roles are not automatically active — users must either run SET ROLE rolename; or set a default role with SET DEFAULT ROLE rolename FOR user;

🩺 EHR: Roles: caremate_app (SELECT/INSERT/UPDATE on carematephr.*), caremate_readonly (SELECT only), caremate_dba (all + GRANT OPTION).
🚂 OS/DB: Roles: railway_app, railway_report, railway_dba. MaxScale monitoring user has role: maxscale_monitor (REPLICATION CLIENT, SHOW DATABASES).
Least Privilege Principle
Application users should only have the minimum privileges required to do their job. Never grant SUPER or GRANT OPTION to application accounts. Never grant global (*.*) privileges when database-level (db.*) or table-level (db.table) suffice. The SUPER privilege in particular bypasses read_only mode and can silently write to replicas — a catastrophic mistake on Galera nodes.

🩺 EHR: The Flask CareMate app gets SELECT/INSERT/UPDATE/DELETE on carematephr.* only — no CREATE, no DROP, no SUPER.
🚂 OS/DB: The ticket booking microservice gets INSERT/SELECT on bookings, ticket_scans only. Report queries use a separate readonly account limited to SELECT.
mysql_secure_installation & Hardening
After every fresh MariaDB install: run mysql_secure_installation to set root password, remove anonymous users, remove test database, disallow remote root login. Additionally: set skip_networking if only local connections needed (not applicable when MaxScale connects remotely), enable local_infile=OFF to prevent LOAD DATA LOCAL INFILE attacks.

🩺 EHR: local_infile=OFF is mandatory for DSGVO; prevents rogue SELECT ... INTO OUTFILE data exfiltration.
🚂 OS/DB: Remote root login disabled; root only accessible via socket from localhost or maintenance jump host.

⚠️ Common Pitfalls▾
⚠ Pitfall — Granting SUPER to application users

SUPER bypasses read_only mode (writing to a Galera read-only node), bypasses query limits, and can kill any connection. Application accounts must never have SUPER. On a Galera cluster, an app accidentally writing to a non-primary node with SUPER causes split data that the cluster cannot reconcile.

⚠ Pitfall — Using 'user'@'%' wildcards too broadly

Creating users with @'%' (any host) means that user can connect from the internet if the port is exposed. Always restrict to the specific subnet ('10.10.0.%') or exact IP of the application server. Verify with: SELECT user, host FROM mysql.user WHERE host = '%'; — any result is a security finding.

⚠ Pitfall — Not running FLUSH PRIVILEGES after direct mysql.user edits

If you ever bypass GRANT statements and edit the mysql.user table directly (strongly discouraged), changes are not active until FLUSH PRIVILEGES;. But GRANT/REVOKE/CREATE USER statements flush automatically. Document clearly in runbooks: "use SQL DDL, never edit grant tables directly."

🎤 Interview Questions▾
What grants does the MaxScale monitoring user need, and why can't you just give it SUPER?

The MaxScale GaleraMonitor needs: REPLICATION CLIENT (to run SHOW MASTER STATUS), REPLICATION SLAVE (to check replication state), SELECT on mysql.* (to read user/host info for connection routing), and SHOW DATABASES. SUPER is not required for monitoring — it bypasses read_only which means a MaxScale bug or misconfiguration could accidentally write to a node that should be read-only. Least privilege means we grant only what's documented as required. For MariaDB 10.5+ you can replace some of these with finer-grained dynamic privileges.

🎯 Anchor

"MaxScale monitor needs REPLICATION CLIENT, REPLICATION SLAVE, SELECT on mysql.*, and SHOW DATABASES — no SUPER. SUPER bypasses read_only and violates least privilege."

A developer asks for SELECT on all tables to debug a production issue. How do you handle this?

I create a time-limited read-only account with SELECT on the specific database, restricted to their workstation IP, valid for 24 hours using the EXPIRE clause, and log the action in the incident ticket. I do not grant global SELECT on *.* — I scope it to the specific schema. After the incident, I immediately DROP USER or set the account to EXPIRE. On CareMate, any access to patient data must be logged in the DSGVO access log — I'd add a note to the audit trail. I would also consider whether a data export (with PII masked) would meet their debugging needs without exposing production.

🎯 Anchor

"Scoped to specific DB, IP-restricted, time-limited, and logged in the ticket. On DSGVO systems, access to patient data must be in the audit trail."

How does ed25519 differ from mysql_native_password, and when would you use each?

mysql_native_password uses SHA-1 with a server-generated challenge — SHA-1 is considered cryptographically weak (collision attacks exist). ed25519 uses modern elliptic curve cryptography with a 256-bit key — orders of magnitude stronger against brute force and cryptanalysis. ed25519 authentication is also faster (no hash iterations needed). The drawback: older clients and some connectors don't support ed25519 yet. Use ed25519 for all new accounts where client compatibility allows. Keep mysql_native_password only for legacy application connectors that can't be updated. At OS GmbH, MariaDB Connector/J 3.x and MaxScale 6+ support ed25519 natively.

🎯 Anchor

Backup with mariabackup

mariabackup (the MariaDB fork of Percona XtraBackup) performs hot physical backups without locking tables. Master full + incremental backup strategy, the prepare phase, and complete restore + PITR procedures.

Click any section to expand
💡 Why This Matters▾

mysqldump produces logical backups (SQL statements) which are too slow to restore on large databases. A 500GB dump takes hours. mariabackup produces physical backups (raw InnoDB files) that can be restored in minutes. It copies data while the server runs by streaming changes from the redo log during the copy phase. For OS GmbH's railway ticketing database (potentially hundreds of GBs) and the CareMate EHR platform, mariabackup is the only production-grade backup tool. It also serves as the SST method for bringing new Galera nodes online.

🎯 Interview Anchor

"Our backup strategy is: daily full with mariabackup at 02:00, hourly incrementals, 7-day retention. The prepare step applies the redo log to make the backup consistent — you cannot restore a raw mariabackup without running --prepare first. RTO target is 30 minutes from snapshot to data available."

🧠 Core Concepts▾
Hot Backup — Copy + Apply-Log Phases
mariabackup works in two logical phases: (1) Copy phase: copies .ibd files and streams redo log changes that happen during the copy. (2) After backup completes, run --prepare: this applies the captured redo log to make the files consistent (like crash recovery). Without --prepare, the backup is in an inconsistent state and cannot be used.

🩺 EHR: CareMate full backup: copy phase copies all carematephr .ibd files; prepare makes them consistent for restore.
🚂 OS/DB: Fraport backup runs on the replica node to avoid I/O overhead on the primary; prepare runs on the backup server.
Incremental Backups & LSN (Log Sequence Number)
Each full backup records its ending LSN (a monotonically increasing counter of redo log position). Incremental backups copy only pages modified since that LSN. Chain: full → inc1 → inc2 → inc3. To restore, prepare the full backup, then apply each incremental in order with --apply-log-only, and finally a full prepare on the last increment.

🩺 EHR: Full at Sunday 02:00 (LSN recorded), hourly incremental Monday–Saturday, RPO of ~1 hour for CareMate data.
🚂 OS/DB: Full backup Sun night, 15-minute incrementals during peak booking hours for near-zero RPO on ticketing data.
Point-in-Time Recovery (PITR) with mariabackup + binlog
Restore procedure: (1) restore mariabackup to get data as of the backup end time, (2) use mysqlbinlog to replay binary logs from the backup endpoint to the target time/position. The backup's xtrabackup_binlog_info file records the exact binlog filename and position at the moment the backup completed — this is your PITR starting point.

🩺 EHR: A doctor accidentally deletes all visits for a patient. PITR to 5 minutes before the DELETE — exact recovery with zero other data loss.
🚂 OS/DB: A bad deployment inserts corrupt booking data. PITR to the exact second before the deployment ran.
Backup User Privileges
The mariabackup user needs: RELOAD (FLUSH), LOCK TABLES (brief lock at end of MyISAM copy if present), REPLICATION CLIENT (SHOW MASTER STATUS for binlog position), PROCESS (SHOW ENGINE INNODB STATUS), and BACKUP_ADMIN (MariaDB 10.4.2+). Do not run backups as root.

🩺 EHR: User 'caremate_backup'@'localhost' with RELOAD, LOCK TABLES, REPLICATION CLIENT, PROCESS.
🚂 OS/DB: Same grants, running on the Galera replica node from a scheduled systemd timer.

⚠️ Common Pitfalls▾
⚠ Pitfall — Restoring without --prepare

A mariabackup taken with --backup is NOT consistent. The copy phase captures files at different points in time while the redo log was streaming changes. Without --prepare, restoring these files will result in InnoDB crash recovery on startup — or worse, a server that starts but serves inconsistent data. ALWAYS run --prepare before --copy-back. The prepare step is idempotent; run it as many times as needed until it exits 0.

⚠ Pitfall — Applying incrementals out of order

Incrementals must be applied in chronological order: full → inc1 → inc2 → inc3. If you skip inc1 and apply inc2 directly, the LSN chain is broken and the prepare will fail or produce corrupt data. Always name incremental directories with timestamps and sort alphabetically before iterating. Use ls -1t | tail -r (oldest first) or name dirs with sortable YYYYMMDD_HHMMSS format.

⚠ Pitfall — Not testing restores

A backup that has never been tested is a theory, not a recovery plan. At OS GmbH, restore tests must be scheduled quarterly (at minimum) on a staging server. Run the full backup → incremental → prepare → restore workflow on a non-production host, then run mysqlcheck and spot-check critical table counts. Document the RTO (time to restore). "We have backups" is not acceptable — "we restored from backup in 23 minutes last Thursday" is.

🎤 Interview Questions▾
Why can't you restore a mariabackup without running --prepare first?

During the --backup phase, mariabackup copies data files while the server is running. Different pages are captured at different points in time. InnoDB maintains consistency through the redo log — but the backup has a redo log snapshot, not a consistent state. The --prepare step performs crash recovery on the backup: it applies the captured redo log to bring all pages to the same consistent point in time, and rolls back any transactions that were in progress but not committed. Without this, the restored data directory is in the same state as a server that just crashed — InnoDB would run crash recovery on startup, which might succeed or might not, and either way the data is unreliable.

🎯 Anchor

"Without --prepare, the backup files are inconsistent across pages — InnoDB would need to crash-recover them at startup. --prepare applies the captured redo log offline, producing a clean consistent state."

How does mariabackup serve as an SST method for Galera? Walk me through the flow.

When a Galera node has been offline too long for IST (Incremental State Transfer) to cover the gap, it requests a full SST (State Snapshot Transfer). With wsrep_sst_method=mariabackup, the donor node runs a full mariabackup --backup streamed over a socket to the joiner. The joiner applies --prepare on the received backup, then starts the MariaDB process with the recovered data. The joiner then replays any additional write sets that arrived during the SST from the Galera GCache. This is superior to rsync SST because mariabackup doesn't require the donor to lock tables — the donor stays fully operational during SST. At OS GmbH with 99.999% availability requirements, this is critical: SST can take 20–30 minutes on a large DB, and the donor must continue serving traffic throughout.

🎯 Anchor

"mariabackup SST is hot on the donor — no table locks. The donor streams a physical backup to the joiner while continuing to serve write traffic. Joiner runs --prepare then rejoins the cluster."

What is the difference between RPO and RTO, and what values would you target for OS GmbH?

RPO (Recovery Point Objective) is how much data loss is acceptable — the maximum age of the data you can restore to. RTO (Recovery Time Objective) is how long the recovery process takes — how long the system is down. For OS GmbH railway ticketing (high criticality, financial transactions): RPO should be near-zero, achieved by Galera synchronous replication (no data loss on node failure) plus 15-minute incremental backups (max 15 minutes of binlog replay needed if all nodes fail). RTO should be under 30 minutes, achieved by pre-prepared standby with mariabackup already restored and hot-standby in MaxScale. For CareMate EHR: RPO of 1 hour (hourly incrementals), RTO of 60 minutes is acceptable since it's not real-time transactional.

🎯 Anchor

"RPO = how much data can we lose; RTO = how long to recover. For railway ticketing: RPO near-zero via Galera sync replication, RTO under 30 minutes via pre-staged standby host."

#!/bin/bash
# backup_verify.sh — restore to staging and spot-check row counts
set -euo pipefail

BACKUP_DIR="/backup/full_$(date +%Y-%m-%d)"
STAGING_DATADIR="/var/lib/mysql-staging"
STAGING_SOCKET="/var/run/mysqld/mysqld-staging.sock"
PROD_HOST="mariadb-primary"
THRESHOLD=0.01   # Allow <1% row-count drift

echo "=== Backup Verification $(date) ==="

# 1. Prepare
mariabackup --prepare --target-dir="$BACKUP_DIR"

# 2. Restore to staging
systemctl stop mariadb-staging || true
rm -rf "$STAGING_DATADIR"/*
mariabackup --copy-back --target-dir="$BACKUP_DIR" \
  --datadir="$STAGING_DATADIR"
chown -R mysql:mysql "$STAGING_DATADIR"
systemctl start mariadb-staging
sleep 10

# 3. Row-count spot-check
declare -A TABLE_DB=(
  [patients]=carematephr [visits]=carematephr [vitals]=carematephr
  [bookings]=osdb        [ticket_scans]=osdb
)
FAIL=0
for TABLE in "${!TABLE_DB[@]}"; do
  DB="${TABLE_DB[$TABLE]}"
  S=$(mysql -S "$STAGING_SOCKET" -se "SELECT COUNT(*) FROM ${DB}.${TABLE}")
  P=$(mysql -h "$PROD_HOST" -se "SELECT COUNT(*) FROM ${DB}.${TABLE}")
  DIFF=$(echo "scale=4; ($P-$S)/$P" | bc)
  echo "  $TABLE: staging=$S prod=$P drift=$DIFF"

    echo "  ❌ FAIL: $TABLE drift exceeds threshold"; FAIL=1
  fi
done

               || { echo "❌ Backup verification FAILED"; exit 1; }

Every DBA at OS GmbH must be able to find the exact binlog event that caused an incident (an accidental DELETE, a bad migration, a corrupt UPDATE) and execute PITR to the exact second before that event. mysqlbinlog is the tool that decodes binary log files into human-readable SQL — essential both for PITR and for forensic auditing of "what actually happened."

🧠 Core Concepts▾
Binary Log Format — ROW vs STATEMENT vs MIXED
STATEMENT: logs the SQL text. Compact but non-deterministic. MIXED: uses STATEMENT normally, switches to ROW for non-deterministic functions. ROW: logs before/after of each changed row. Verbose but deterministic and idempotent. Use ROW for all production Galera + PITR setups. MariaDB 10.2.4+ defaults to MIXED — change to ROW explicitly.

🩺 EHR: ROW format provides full audit trail of exactly which patient record was changed, required for DSGVO Art. 30 records of processing activities.
🚂 OS/DB: ROW required by Galera; also allows Debezium CDC (Change Data Capture) to stream booking changes to analytics pipeline.
mysqlbinlog — Decoding & Filtering
mysqlbinlog converts binary log files to human-readable SQL (or raw event data). Key flags: --start-position / --stop-position for exact byte positions, --start-datetime / --stop-datetime for time-based filtering, -vv for ROW event pseudo-SQL, --database=dbname for filtering to one DB.

🩺 EHR: After a bad migration deletes visits rows, use --stop-position to replay everything up to but not including the DELETE event.
🚂 OS/DB: Find the exact position of a corrupt booking INSERT by scanning binlogs with --start-datetime.
xtrabackup_binlog_info — The PITR Bridge
Every mariabackup writes a file xtrabackup_binlog_info containing the binlog filename and position at the moment the backup completed. This is your PITR starting point: restore the backup (data up to T0), then replay binlogs from this position up to the target time (T1). Without this file you don't know where to start mysqlbinlog.

🩺 EHR: Full backup at 02:00 gives binlog position. Replay 02:00–07:42 to recover to 5 minutes before an incident at 07:47.
🚂 OS/DB: binlog_info on the replica backup (taken with --slave-info) gives the correct replica binlog position for PITR.
SHOW BINARY LOGS & SHOW BINLOG EVENTS
SQL-level commands to inspect binary logs without shell access. SHOW BINARY LOGS lists all log files and sizes. SHOW BINLOG EVENTS IN 'filename' FROM pos LIMIT N shows events in a specific file from a position. Useful for quick forensic inspection when you need to find the event position of a specific operation without decoding the whole file.

🩺 EHR: Quick check: "find all events on carematephr after 08:00" without leaving the MySQL shell.
🚂 OS/DB: Identify the exact position of a TRUNCATE TABLE event before calling PITR stop position.

⚠️ Common Pitfalls▾
⚠ Pitfall — Using --stop-datetime without checking for clock skew

Server clocks can drift, especially in VMs. If the MariaDB server clock is 2 minutes ahead of the DBA's workstation, a --stop-datetime="09:15:31" might miss events that happened at "real" 09:15:31 but were logged at 09:17:31. Always use --stop-position (exact byte position) when precision matters. Find the position with SHOW BINLOG EVENTS first, then use that position for PITR.

⚠ Pitfall — RESET MASTER in production

RESET MASTER deletes ALL binary log files and resets the log index to 000001. If any replica hasn't consumed recent binlogs, it will be permanently broken. This command must never be run in production. Use PURGE BINARY LOGS TO 'filename' or PURGE BINARY LOGS BEFORE 'datetime' to safely remove old logs after confirming all replicas/Galera nodes have consumed them.

⚠ Pitfall — Binary log on a full disk

Binary logs grow continuously. If the disk partition holding binlogs fills up, MariaDB will halt writes and the server becomes read-only (or crashes in older versions). Always put binlogs on a separate partition with monitoring. Set expire_logs_days (or binlog_expire_logs_seconds) to automatically purge old logs. Alert when the binlog partition exceeds 70% used.

🚨 3AM SCENARIO — "Bad migration UPDATE corrupted 40,000 booking records"

Situation: 02:47. A deployment script ran UPDATE bookings SET status='CANCELLED' WHERE 1=1 instead of WHERE status='PENDING'. 40,000 bookings (including CONFIRMED, CHECKED_IN) are now CANCELLED. Trains are departing. Refund emails are firing from the application. You have 20 minutes before the situation becomes a customer relations catastrophe.

Your immediate diagnostic steps:

  1. Stop the application from sending more refund emails — coordinate with AppOps to scale down the notification service
  2. Find the event: SHOW BINLOG EVENTS IN 'railway-bin.000042' — look for the mass UPDATE event around 02:47, note its position (e.g., position 4198762)
  3. Decode it: mysqlbinlog -vv --start-position=4198762 --stop-position=4200000 railway-bin.000042 — confirm it shows UPDATE bookings affecting thousands of rows
  4. Option A (faster if binlog_row_image=FULL): extract just the before-images from the ROW event — these are the original values. Write a reverse-UPDATE script from the -vv output
  5. Option B: PITR restore to a side server (parallel), verify counts, then use it as the source for a corrective UPDATE on production
  6. Apply corrective UPDATE on production using the recovered values — do this in a transaction: BEGIN; UPDATE bookings SET status=... WHERE ...; SELECT COUNT(*); COMMIT;

Resolution: Use Option A (reverse-UPDATE) if the script can be generated in <10 minutes. Use Option B PITR if the corruption is complex. With binlog_row_image=FULL, the before-image is always in the binlog — this is your undo log.

Prevention: All migration scripts go through DRY RUN (transaction + ROLLBACK) in staging first. Automated deployment must NEVER run without a WHERE clause test (add CHECK CONSTRAINT or application layer validation). Enable the query_response_time plugin to alert on unexpectedly long mass-update queries.

Why is binlog_row_image=FULL important for incident recovery?

With binlog_row_image=FULL, every ROW event contains both the complete before-image (all column values before the change) and the after-image (all values after). This means for any UPDATE or DELETE, you can reconstruct the original data without a backup restore — just decode the binlog with mysqlbinlog -vv and read the ### WHERE section (before-image). With MINIMAL mode, only columns used in the WHERE clause and changed columns are logged — useful for reducing binlog size but means you can't reconstruct a full before-image. At OS GmbH, FULL is worth the extra disk space for the ability to do surgical row-level recovery from binlogs alone.

Slow Query Analysis

Enable and tune the slow query log, use pt-query-digest to find the worst offenders, and leverage Performance Schema for live query diagnosis. The essential toolkit for any on-call DBA facing unexplained latency spikes.

Click any section to expand
💡 Why This Matters▾

The most common performance complaint at OS GmbH will be "the application is slow" with no other information. The slow query log is the evidence base. pt-query-digest (Percona Toolkit) condenses thousands of slow log entries into a ranked report showing the top 10 query patterns by total execution time — turning a 100MB log file into a 2-page diagnosis. Performance Schema provides real-time visibility while the slowness is happening. Together these tools let you move from "the DB is slow" to "this specific query pattern is responsible for 73% of the latency" in under 10 minutes.

🎯 Interview Anchor

"My slow query workflow: check the slow log first with pt-query-digest for the top-5 patterns, then EXPLAIN each pattern, then check if the issue is a missing index, a bad join, or a full table scan. If it's live, I use Performance Schema events_statements_summary_by_digest to find the worst queries without needing a log file."

🧠 Core Concepts▾
Slow Query Log Configuration
Enabled via: slow_query_log=1, long_query_time=N (seconds, supports decimals like 0.5), log_queries_not_using_indexes=1 (catches full-table scans even if they're fast), min_examined_row_limit=1000 (ignore tiny scans). Can be enabled at runtime: SET GLOBAL slow_query_log=1; — no restart needed.

🩺 EHR: long_query_time=1.0 for CareMate (1 second is unacceptable for a patient dashboard); log_queries_not_using_indexes=1 catches FHIR export full scans.
🚂 OS/DB: long_query_time=0.5 — sub-second threshold for real-time booking ops. Any query over 500ms on the ticketing path is a SLA breach candidate.
pt-query-digest (Percona Toolkit)
Takes a slow query log file and produces a ranked digest report. Groups queries by their "fingerprint" (normalising literals to ?), shows: total time, average time, number of calls, worst case. The top-5 fingerprints by total time are almost always where the optimisation effort should go. Install: dnf install percona-toolkit (RHEL) or from Percona repo.

🩺 EHR: Weekly pt-query-digest run on CareMate slow log, piped to a report for the dev team — identifies patterns like "SELECT * FROM visits WHERE patient_id=? ORDER BY created_at" missing an index.
🚂 OS/DB: pt-query-digest run after every release deployment — catch regressions before the shift begins.
Performance Schema — events_statements_summary_by_digest
The Performance Schema table performance_schema.events_statements_summary_by_digest is the live equivalent of pt-query-digest — it tracks cumulative stats per query fingerprint since server start (or since last TRUNCATE). Columns: DIGEST_TEXT (normalised query), COUNT_STAR (executions), SUM_TIMER_WAIT (total time in picoseconds), SUM_ROWS_EXAMINED, SUM_ROWS_SENT. Essential for live incidents when the slow log hasn't captured enough yet.

🩺 EHR: Query this table during a CareMate slowdown to immediately identify which query pattern is hammering the DB — no log file needed.
🚂 OS/DB: During a booking rush (Weihnachten, major events) query this table to identify queries that suddenly increased in call frequency.
EXPLAIN & EXPLAIN FORMAT=JSON
EXPLAIN shows the query execution plan: which indexes are used, estimated row counts, join type (ALL = full scan, ref = index lookup, range = index range, eq_ref = unique index). EXPLAIN FORMAT=JSON gives more detail including cost estimates. EXPLAIN ANALYZE (MariaDB 10.9+) actually runs the query and shows actual vs. estimated row counts — the most useful form for diagnosis.

🩺 EHR: EXPLAIN a slow patient dashboard query — see if it uses the patient_id index on visits or does a full scan.
🚂 OS/DB: EXPLAIN the booking lookup query — type=ALL on a 100M row table means a missing index on passenger_id or route_id.

⚠️ Common Pitfalls▾
⚠ Pitfall — long_query_time=0 in production

Setting long_query_time=0 logs every single query — including the millions of fast booking reads. This creates a multi-GB slow log in minutes, fills the disk, and itself becomes a performance problem. Use 0.5–1.0 seconds as a production threshold. For short-term debugging of specific sessions, enable it at the session level only: SET SESSION long_query_time = 0;

⚠ Pitfall — Optimising the wrong query

pt-query-digest ranks by total time, not by average time. A query that runs 1 million times at 0.5ms each might appear above a query that runs 10 times at 10s each. For user-facing latency, focus on the highest avg_ms queries that the user actually waits for. Read the pt-query-digest report fully — it shows both total time rank and call count, helping you distinguish "high-frequency fast" from "low-frequency slow" patterns.

⚠ Pitfall — EXPLAIN on live production tables during incident

EXPLAIN on a complex query with subqueries may itself be slow if the optimizer needs to execute subqueries for cardinality estimation. On a table under heavy load, an EXPLAIN can acquire metadata locks. For large tables, use EXPLAIN on a read replica or with a query that has a known LIMIT. On Galera, always run diagnostic queries on a non-primary node to avoid adding load to the write path.

What does type=ALL in EXPLAIN output mean, and is it always bad?

type=ALL means a full table scan — InnoDB reads every row in the table to find the matches. It's bad on large tables (millions of rows) because it reads every page into the buffer pool, evicting hot pages and causing I/O. However, it's not always wrong: on a table with 100 rows, a full scan is faster than a B-tree index lookup because it's a single sequential I/O vs. multiple random I/Os. Also, if a query returns >30–40% of rows, a full scan may be cheaper than following the index. The key question is: how many rows does the query actually return vs. how many does EXPLAIN estimate it examines? A ratio of examined/returned > 100 is the red flag, not type=ALL alone.

🎯 Anchor

"type=ALL is only bad if the table is large. The real signal is rows_examined / rows_returned — a ratio over 100x means we're reading far too much data to find too little."

Reading SHOW ENGINE INNODB STATUS

SHOW ENGINE INNODB STATUS is the most information-dense diagnostic command in MariaDB. Learn to read each section under pressure: transactions, lock waits, deadlocks, buffer pool state, I/O, and semaphores. This is the first command Michael Nietsch will expect you to run in a 3am incident.

Click any section to expand
💡 Why This Matters▾

SHOW ENGINE INNODB STATUS is a point-in-time snapshot of InnoDB's internal state. It contains information not available anywhere else: the last detected deadlock with full SQL, current lock waits with blocking transaction IDs, buffer pool internals, redo log write performance, and semaphore waits indicating mutex contention. A senior DBA can diagnose 80% of performance incidents from this single command. The output is dense and intimidating — but you only need to know which sections to look at for each symptom type.

🎯 Interview Anchor

"When something is wrong with InnoDB, SHOW ENGINE INNODB STATUS is my first command. I scan: TRANSACTIONS section for long-running transactions and lock waits, LATEST DEADLOCK for deadlock forensics, BUFFER POOL for dirty page ratio and I/O pressure, and SEMAPHORES for mutex contention indicating thread concurrency problems."

🧠 Core Concepts — Section by Section▾
TRANSACTIONS Section
Lists active InnoDB transactions. Key signals: "ACTIVE X seconds" — any transaction over 30 seconds is suspicious; over 300 seconds is a crisis. "lock wait X lock struct(s)" means the transaction is blocked. "LOCK WAIT" state means it's currently waiting for a row lock. Look for "TRX HAS BEEN WAITING" with the table and index name to identify the contested resource.

🩺 EHR: A CareMate session left open by a Flask connection that never committed — appears as "ACTIVE 847 seconds", blocking other writers on the visits table.
🚂 OS/DB: A booking transaction waiting for a lock on a specific seat_number row — visible here before it appears in application logs.
LATEST DETECTED DEADLOCK Section
Shows the last deadlock detected by InnoDB: both transactions involved, the SQL they were running, which transaction held which locks, and which transaction was the victim (rolled back). This is the authoritative deadlock forensic record — more detailed than the application error log. Appears only when a deadlock has occurred since last server start.

🩺 EHR: Deadlock between two Flask workers both updating vitals and visits for the same patient — revealed by the LATEST DEADLOCK section showing exact SQL and lock order.
🚂 OS/DB: Deadlock between the booking service and the scan service both locking bookings rows — the section shows which indexes were contested.
BUFFER POOL AND MEMORY Section
Key metrics: "Buffer pool size" (configured), "Free buffers" (available pages — should be >0), "Database pages" (pages holding data), "Modified db pages" = dirty pages (high = I/O bottleneck), "Pending reads/writes" (backlog — high = I/O saturation), "Pages read/created/written per second" (I/O rates — watch for spikes).

🩺 EHR: "Modified db pages 91423" (90% of pool dirty) = disk can't flush fast enough — increase innodb_io_capacity or add faster storage.
🚂 OS/DB: "Pending writes: LRU 0, flush list 847" = 847 pages queued for flush = I/O bottleneck on the NVMe at peak boarding time.
LOG Section — Redo Log Health
Shows: Log sequence number (current LSN), Log flushed up to (what's persisted), Last checkpoint at (what's checkpointed to disk), and "X MB/sec" write speed. The gap between LSN and checkpoint is the "redo log checkpoint age" — if this approaches the total redo log size, InnoDB will stall writes (furious flushing). A large checkpoint gap indicates the redo log is too small or I/O can't keep up.

🩺 EHR: Checkpoint age of 480MB on a 512MB log file = checkpoint stall imminent — increase innodb_log_file_size immediately.

⚠️ Common Pitfalls▾
⚠ Pitfall — Ignoring History List Length

A long history list (> 100,000) means many old row versions are being kept in the undo log because a long-running transaction is holding an MVCC snapshot. Every query doing a consistent read must walk back through these undo records to find the correct row version — this makes seemingly-simple SELECTs slow. Find the long-running transaction with SHOW ENGINE INNODB STATUS TRANSACTIONS section and kill it. A history list > 1,000,000 is a critical incident on its own.

⚠ Pitfall — Misreading the checkpoint age

The "max checkpoint age" in the LOG section is the maximum allowed gap (limited by log file size), not the current gap. Current checkpoint age = Log sequence number - Last checkpoint at. If this gap approaches max checkpoint age, InnoDB enters furious flushing (writes stall). Don't confuse the two numbers — calculate the gap manually from the LSN values shown.

⚠ Pitfall — Only reading SHOW ENGINE INNODB STATUS once

INNODB STATUS is a point-in-time snapshot. A deadlock that happened 10 minutes ago is still shown — but a current lock wait might not be. Run it multiple times (every 5–10 seconds) during an incident to catch transient events. The snapshot script above (capturing every 30s) is the production-grade approach for incident analysis.

🎤 Interview Questions▾
What does a high "History list length" in INNODB STATUS tell you, and what causes it?

History list length is the count of undo log records that InnoDB's purge thread hasn't yet cleaned up. It grows when old row versions (from MVCC) accumulate faster than the purge thread can remove them. The primary cause is a long-running transaction that holds an MVCC snapshot — because InnoDB must keep all row versions that are newer than the oldest active transaction's snapshot. With a 3-hour-old open transaction, InnoDB must keep 3 hours of row version history. The effect: every consistent-read query must traverse back through potentially thousands of undo records to find the correct version, causing seemingly-fast queries to become slow. Resolution: find and kill the long-running transaction. Preventions: wait_timeout, innodb_lock_wait_timeout, and Prometheus alerting on history list length.

🎯 Anchor

"History list length spikes when a long-running transaction prevents purge from cleaning undo records. Find the ACTIVE N seconds transaction in INNODB STATUS and KILL it."

A deadlock just happened. How do you find out what caused it using SHOW ENGINE INNODB STATUS?

SHOW ENGINE INNODB STATUS contains a "LATEST DETECTED DEADLOCK" section that persists until the next deadlock or server restart. It shows: (1) both transactions involved — their IDs, how long they'd been active, what SQL they were running; (2) which locks each transaction held (table, index, lock mode); (3) which lock each was waiting for; (4) which transaction was the victim (rolled back). From this I can identify: what query pattern caused the deadlock (usually a lock-ordering issue — two transactions acquiring the same rows in different order), which table and index was contested, and which application code path triggered it. The fix is usually to ensure all transactions acquire locks in the same consistent order.

🎯 Anchor

"LATEST DEADLOCK section shows both transactions, what SQL they ran, which locks they held, and which lock each was waiting for — complete forensics without needing to reproduce the deadlock."

Explain the difference between wait_timeout, innodb_lock_wait_timeout, and net_read_timeout. When would each trigger?

wait_timeout: how long the server waits for activity on a non-interactive connection before closing it. Prevents leaked Flask/JDBC connections that hold resources. Triggers on idle connections (in Sleep state). innodb_lock_wait_timeout: how long a transaction waits to acquire a row lock before being rolled back (with a "Lock wait timeout exceeded" error). Prevents long lock queues during contention. Triggers when one transaction is waiting for another to release a row lock. net_read_timeout: how long the server waits for a client to send the next packet when it expects one (during a query). Prevents half-open TCP connections hanging server threads. Triggers during network instability between client and server. At OS GmbH: wait_timeout=120 (MaxScale reconnects fast), innodb_lock_wait_timeout=50 (fail fast on lock contention), net_read_timeout=30.

🎯 Anchor

"wait_timeout = idle connection cleanup. innodb_lock_wait_timeout = lock queue abort. net_read_timeout = incomplete packet abort. Three different layers: session, InnoDB, network."

A developer wants to run ALTER TABLE visits ADD COLUMN consulting_fee DECIMAL(10,2) on production (100M rows). How do you handle this?

Direct ALTER TABLE on a 100M row table acquires a metadata lock and rebuilds the entire table — potentially taking hours and blocking all reads and writes. The correct approach: (1) Test the DDL in staging first, measure duration on a copy of production data. (2) If MariaDB 10.8+ with Instant DDL is available, check if ADD COLUMN supports ALGORITHM=INSTANT — adding a column with a default can be done in milliseconds with no table rebuild. (3) If INSTANT is not available, use pt-online-schema-change or gh-ost — these create a shadow table, copy rows in batches, apply changes via triggers on the original, then swap tables. This runs transparently while traffic continues. (4) Schedule during a maintenance window as a fallback. On Galera, DDL propagates cluster-wide — the schema change must be tested on all nodes and the TOI (Total Order Isolation) mode means all nodes process the DDL together.

🎯 Anchor

"First check ALGORITHM=INSTANT (MariaDB 10.8+). If not available, pt-online-schema-change or gh-ost for zero-downtime DDL. Never run ALTER TABLE directly on a multi-million row production table."

What would you look at to determine if a MariaDB server is I/O bound vs. CPU bound?

CPU bound indicators: top shows MariaDB process at 90%+ CPU; SHOW PROCESSLIST shows many queries in state "executing" or "sorting result"; Performance Schema shows high CPU time in events_waits_summary. For CPU-heavy queries, EXPLAIN reveals bad query plans, missing indexes, large filesorts. I/O bound indicators: iostat -x shows device utilization near 100%, high await times; SHOW ENGINE INNODB STATUS shows "Pending reads/writes" > 0, high dirty page count, low buffer pool hit ratio (reads/read_requests > 1%); MariaDB processes in state "waiting for page to be read". I/O bound is most common at OS GmbH due to ticketing's high write throughput — fix by increasing innodb_io_capacity, adding buffer pool, or upgrading to faster NVMe. CPU bound is less common and usually indicates missing indexes or a suboptimal query plan.

🎯 Anchor

"I/O bound: iostat shows disk saturation + high InnoDB dirty pages + pending reads. CPU bound: high CPU in top + many 'executing' queries in PROCESSLIST + poor query plans in EXPLAIN."

How do you handle a MariaDB server that has run out of disk space on the datadir partition?

This is a critical incident. Immediate steps: 1. Don't restart MariaDB — it may not be able to start if disk is full. 2. Check what's using the space: du -sh /var/lib/mysql/* sorted by size. Common causes: binary logs not purging, large temporary tables, undo log growth, or a rogue large table. 3. If binary logs: PURGE BINARY LOGS BEFORE 'date' — but verify replicas have consumed them first. 4. If tmp files: check tmp_table_size and max_heap_table_size — a rogue sort may have created a huge temp file. KILL the query. 5. If undo log: find and kill long-running transactions (see INNODB STATUS). 6. Emergency: move binary logs to a different partition: SET GLOBAL log_bin = '/newpath/binlog'; — but this is non-trivial. 7. Long-term: separate binlog and datadir onto different partitions. Alert at 70%, page at 85%.

🎯 Anchor

"Don't restart a full-disk MariaDB. Identify what's growing (binlogs, tmp files, undo log), remove the largest safe-to-remove files first, then address root cause. Alert at 70% disk usage."

Zero-Downtime Schema Changes

ALTER TABLE on a large InnoDB table can block writes for 20–40 minutes and stall an entire Galera cluster. Master pt-online-schema-change, gh-ost, and Galera DDL modes to evolve your schema on live 24/7 systems without downtime.

Click any section to expand
💡 Why This Matters▾

A bare ALTER TABLE vitals MODIFY COLUMN bp_value DECIMAL(6,2) on a 50-million-row table acquires a metadata lock (MDL) that blocks all writes for the duration of the rebuild — potentially 30+ minutes on spinning disk. In Galera, that stall propagates cluster-wide and triggers flow control, pausing the entire cluster. Knowing how to execute schema changes safely is a core OS GmbH DBA competency.

🎯 Interview Anchor — What Nietsch Expects

"For tables under 1M rows I'll use bare ALTER with ALGORITHM=INPLACE during a low-traffic window. For large tables I reach for pt-osc or gh-ost. On Galera I always switch to RSU mode first — SET SESSION wsrep_OSU_method='RSU' — so the DDL runs locally without stalling the cluster. My default for large Galera tables is gh-ost with RSU."

🧠 Core Concepts▾
Why Bare ALTER TABLE Blocks Writes
Even with ALGORITHM=INPLACE, LOCK=NONE, a column-type change requires a full table rebuild — every row is read, transformed, and written to a new structure. InnoDB holds a brief exclusive metadata lock at the start and end of this rebuild. Any DML that arrives during the rebuild queues behind the MDL. In Galera, the node applying the DDL enters flow control and pauses cluster replication for the rebuild duration, stalling all other nodes.

🩺 EHR: The vitals table (55M rows, written every 15 min per patient) cannot tolerate a 30-minute write stall during day shifts.
🚂 OS/DB: bookings (80M rows) has continuous INSERT activity — a 60-second MDL block drops thousands of booking attempts visible to end-users.
pt-online-schema-change (pt-osc)
How it works: pt-osc creates a shadow table with the new schema, installs three triggers (INSERT, UPDATE, DELETE) on the original table to mirror live DML to the shadow, copies rows in configurable chunks, then swaps the tables with a fast RENAME requiring only a brief MDL — not for the full rebuild.

Installation: 
apt install percona-toolkit   # Debian/Ubuntu
yum install percona-toolkit   # RHEL/CentOS
Worked example — change bp_value FLOAT → DECIMAL(6,2) on CareMate vitals: 
pt-online-schema-change \
  --host=mariadb-primary \
  --user=dba_user --ask-pass \
  --database=carematephr \
  --table=vitals \
  --alter="MODIFY COLUMN bp_value DECIMAL(6,2)" \
  --chunk-size=5000 \
  --max-load="Threads_running=50" \
  --critical-load="Threads_running=100" \
  --no-drop-old-table \
  --print --execute
--chunk-size: rows per copy iteration — lower = less I/O pressure.
--no-drop-old-table: keeps original as _vitals_old for rollback.

Caveats: triggers add ~10–15% write overhead. Avoid on tables with existing triggers (conflict). Foreign key constraints require extra care.
gh-ost — When to Prefer It Over pt-osc
How it differs: gh-ost acts as a replica — connecting to the primary's binlog stream and applying row changes directly to the ghost table. No triggers are installed on the production table, eliminating trigger overhead and all trigger-conflict issues.

Prefer gh-ost when:
  • Table already has triggers (pt-osc cannot co-exist)
  • Very high write rate where trigger overhead is measurable
  • Galera environment where you want RSU (no trigger replication issues)
  • You need pause/resume without losing state
  • You want fine-grained cut-over control via postpone flag file
gh-ost \
  --host=mariadb-primary \
  --user=dba_user --password=... \
  --database=carematephr \
  --table=vitals \
  --alter="MODIFY COLUMN bp_value DECIMAL(6,2)" \
  --chunk-size=5000 \
  --max-load=Threads_running=50 \
  --allow-on-master \
  --execute
DDL on Galera — TOI vs RSU (wsrep_OSU_method)
TOI (Total Order Isolation) — default: The DDL is broadcast to all nodes simultaneously. All nodes pause new transactions and apply the DDL at the same global sequence position. Safe for small tables; catastrophic for large ones — the entire cluster freezes for the rebuild duration.

RSU (Rolling Schema Upgrade): The DDL runs only on the local node. That node de-syncs from the cluster, applies the DDL independently, then re-syncs. Other nodes continue serving traffic. Application must tolerate a brief schema version mismatch window. 
-- Switch to RSU for this session only
SET SESSION wsrep_OSU_method = 'RSU';
ALTER TABLE vitals MODIFY COLUMN bp_value DECIMAL(6,2);
SET SESSION wsrep_OSU_method = 'TOI';  -- restore default
🩺 EHR: For CareMate's 3-node Galera cluster, RSU + gh-ost on one node at a time is the safe pattern for large-table DDL.
🚂 OS/DB: Fraport cluster uses RSU for all production DDL on tables over 5M rows.
🗺 Decision Guide▾
✅ DDL Decision Tree

Small table (<1M rows), standalone MariaDB → Bare ALTER TABLE … ALGORITHM=INPLACE, LOCK=NONE during a low-traffic window. Verify the algorithm is actually INPLACE with EXPLAIN ALTER first.

Large table (>1M rows), standalone MariaDB → pt-osc (no existing triggers, lower write rate) or gh-ost (existing triggers or very high write rate). Always use --no-drop-old-table / postpone flag for rollback.

Galera cluster + large table → gh-ost with RSU. Run gh-ost against the primary with SET SESSION wsrep_OSU_method='RSU' to isolate the rebuild to one node. MaxScale continues routing reads to other Synced nodes.

Emergency — must cut over at a specific moment → gh-ost with --postpone-cut-over-flag-file; the final table swap is under your manual control.

⚠ Pitfalls▾
⚠ Pitfall — Bare ALTER on a Large Table in Production

The most common DBA mistake Nietsch encounters: bare ALTER TABLE during business hours on a multi-million-row table "because it only changes one column." The MDL blocks every subsequent DML. On OS GmbH's system, booking failures become user-visible within 30 seconds. Always check row count before ALTER: SELECT TABLE_ROWS FROM information_schema.TABLES WHERE TABLE_NAME='vitals'.

⚠ Pitfall — pt-osc and Foreign Keys

pt-osc drops and re-creates foreign keys on the shadow table during the swap, which can cause brief constraint-check gaps. For tables with FK relationships, either test thoroughly on staging or prefer gh-ost.

⚠ Pitfall — TOI on Large Tables in Galera

Using the default TOI mode for a large-table ALTER is a cluster outage. Every node pauses for the full rebuild duration. Flow control kicks in within seconds of the MDL being held. Never run bare ALTER on a large Galera table without switching the session to RSU first.

🎤 Interview Q&A▾
How do you ALTER a live production table without downtime?

My decision depends on table size and environment. Under 1M rows: bare ALTER with ALGORITHM=INPLACE, LOCK=NONE in a low-traffic window. Over 1M rows I use pt-osc or gh-ost — pt-osc uses triggers to mirror live writes to a shadow table during the copy phase; gh-ost uses the binlog stream instead, no triggers, lower overhead, safer on Galera. On a Galera cluster I always switch to RSU mode first: SET SESSION wsrep_OSU_method='RSU' — this isolates the DDL to one node without stalling the cluster. I prefer gh-ost on Galera because RSU + triggers can cause certification conflicts; gh-ost with binlog-based apply is cleaner. I always keep the old table (--no-drop-old-table) as a rollback option.

"Small table → bare ALTER; large table → pt-osc or gh-ost; Galera + large table → gh-ost with RSU. Never bare ALTER a multi-million-row table in production."
