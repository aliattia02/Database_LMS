Install & Connect

Get MySQL running and connected in DBeaver. Takes about 10 minutes. You only do this once — then you build the Morafek CareMate database for all lessons.

Terminal

Lesson 01 · ~30 min
Your First Table

CREATE TABLE defines the structure. INSERT puts data in. You'll build two PHR tables — patients and vitals — that you'll use for all 8 lessons.

Click any section to expand
Key concepts▾
Table = a spreadsheet with enforced rules
A table has columns (like fields in a MongoDB document) and rows (like documents). The difference: every row must follow the exact same column structure and data types. No optional or extra fields.
PRIMARY KEY — the unique ID for every row
Every table needs a column that uniquely identifies each row. AUTO_INCREMENT means MySQL assigns the ID automatically: 1, 2, 3… You never insert this value manually. This is the same as MongoDB's _id, but numeric.
NOT NULL vs NULL
NOT NULL means the field is required — MySQL rejects rows without it. Omitting this allows NULL (missing). In Morafek, patient full_name is NOT NULL (required), but notes on a vital reading can be NULL (optional).
Create the patients table▾
🍃 From MongoDB

In MongoDB you'd do db.createCollection('patients') (or just start inserting). In SQL, you must define the exact structure upfront — column names, data types, and constraints — before inserting a single row.

SQL🩺 EHR
USE caremate_db; CREATE TABLE patients ( id INT AUTO_INCREMENT PRIMARY KEY, full_name VARCHAR(120) NOT NULL, date_of_birth DATE NOT NULL, blood_type VARCHAR(5) DEFAULT NULL, city VARCHAR(80) NOT NULL, email VARCHAR(160) UNIQUE NOT NULL, created_at DATETIME DEFAULT NOW() ); -- Equivalent e-commerce table for reference: -- CREATE TABLE customers (id INT AUTO_INCREMENT PRIMARY KEY, -- name VARCHAR(120) NOT NULL, email VARCHAR(160) UNIQUE NOT NULL, -- city VARCHAR(80), created_at DATETIME DEFAULT NOW());
Create the vitals table▾
SQL🩺 EHR
CREATE TABLE vitals ( id INT AUTO_INCREMENT PRIMARY KEY, patient_id INT NOT NULL, systolic SMALLINT NOT NULL, -- blood pressure: upper value (mmHg) diastolic SMALLINT NOT NULL, -- blood pressure: lower value (mmHg) pulse SMALLINT DEFAULT NULL, -- heart rate (bpm), optional weight_kg DECIMAL(5,2) DEFAULT NULL, -- body weight, optional notes TEXT DEFAULT NULL, recorded_at DATETIME DEFAULT NOW() ); -- Equivalent e-commerce table for reference: -- CREATE TABLE products (id INT AUTO_INCREMENT PRIMARY KEY, -- name VARCHAR(160) NOT NULL, category VARCHAR(80), -- price DECIMAL(10,2) NOT NULL, stock INT DEFAULT 0);
Data types used

SMALLINT — whole numbers up to 32,767. Perfect for BP values (60–300 range). Saves space vs INT. DECIMAL(5,2) — exact decimal: 5 digits total, 2 after the point — e.g. 72.50 kg. Use for anything requiring exact precision (never FLOAT for medical values). TEXT — unlimited string, used for free-text notes.

🚂 OS/DB Example — bookings schema at OS scale▾
🚂 OS/DB — Frankfurt · Fraport KRITIS

At operational services, the bookings table processes millions of rows per day across Fraport, VW, and S-Bahn Berlin. The same CREATE TABLE fundamentals apply — but data-type choices have SLA consequences. A DECIMAL(10,2) for price is non-negotiable; a FLOAT would accumulate rounding error in financial settlement queries.

SQL🚂 OS/DB
CREATE TABLE passengers ( id INT AUTO_INCREMENT PRIMARY KEY, name VARCHAR(120) NOT NULL, bahn_card_number VARCHAR(20) UNIQUE DEFAULT NULL, -- nullable: not all passengers have BahnCard loyalty_tier ENUM('standard','silver','gold','senator') DEFAULT 'standard', email VARCHAR(160) UNIQUE NOT NULL ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4; CREATE TABLE bookings ( id INT AUTO_INCREMENT PRIMARY KEY, passenger_id INT NOT NULL, route_id INT NOT NULL, seat_number VARCHAR(5) DEFAULT NULL, class ENUM('1','2') NOT NULL DEFAULT '2', price DECIMAL(10,2) NOT NULL, -- DECIMAL, never FLOAT for money status ENUM('confirmed','cancelled','refunded') NOT NULL DEFAULT 'confirmed', booked_at DATETIME DEFAULT NOW() ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4; -- DESCRIBE shows column definitions — useful when joining unfamiliar tables DESCRIBE bookings;
💉 DiaTwin — append-only from the start

DiaTwin's glucose_readings table omits UPDATE permissions entirely at the DB user level. The table structure is designed so that new readings are always INSERTs — never UPDATEs to existing rows. This is DiGA compliance built into the schema design, not bolted on later.

🎤 Interview Questions▾
Why use DECIMAL instead of FLOAT for a price column in a bookings table?

FLOAT is a binary floating-point type — it cannot represent all decimal fractions exactly, so 29.90 might be stored as 29.89999… Summing thousands of rows introduces rounding drift that appears in financial settlement reports. DECIMAL(10,2) stores exact decimal values with no rounding error, which is required for any monetary or legally auditable data.

⚓ Anchor

At Fraport, a FLOAT in the price column would cause booking settlement discrepancies — DECIMAL is non-negotiable for financial columns.

When would you choose ENUM vs VARCHAR for a status column?

ENUM is ideal when the set of valid values is small, fixed, and business-critical — like booking status ('confirmed','cancelled','refunded'). It enforces constraint at the storage layer, uses only 1–2 bytes, and is self-documenting. VARCHAR makes sense when values are user-defined, frequently added, or need to be managed without a schema migration. The tradeoff: adding a new ENUM value requires ALTER TABLE, which can be a concern on large tables.

⚓ Anchor

ENUM trades flexibility for enforced integrity and storage efficiency — correct for fixed business states, wrong for user-defined categories.

A colleague adds a column with ALTER TABLE on a 500M-row production table during peak hours. What happens?

In older MariaDB/MySQL versions, ALTER TABLE acquires a full table lock and rebuilds the table — blocking all reads and writes for the duration, which can be minutes or hours on a 500M-row table. MariaDB 10.x supports instant ADD COLUMN for InnoDB (Algorithm=INSTANT) for columns added at the end — this completes in milliseconds without a rebuild. Always check whether the operation supports ALGORITHM=INSTANT or INPLACE before running on large tables. At OS, this is a change-management process, not an ad-hoc operation.

⚓ Anchor

ALTER TABLE on large tables requires checking Algorithm=INSTANT first — blocking a 500M-row table during peak hours is a production incident.

Lesson 02 · ~30 min
Reading Data: SELECT

SELECT is the most-used SQL command — you'll write it hundreds of times a day. Learn to read all columns, specific columns, rename them, and compute derived values like pulse pressure.

Click any section to expand
SELECT anatomy▾
The basic structure

SELECT [what columns] FROM [which table]

That's it. Everything else (WHERE, ORDER BY, etc.) is optional and added after. You always need SELECT and FROM.

🍃 From MongoDB

db.patients.find() → SELECT * FROM patients
db.patients.find({}, {full_name:1, email:1}) → SELECT full_name, email FROM patients

Run these queries now▾
SQL🩺 EHR
-- Get ALL columns from all patient rows (* means everything) SELECT * FROM patients; -- Get only specific columns (most efficient — avoid * in production) SELECT full_name, email FROM patients; SELECT full_name, city, blood_type FROM patients; -- Rename a column in the result using AS (alias) SELECT full_name AS patient_name, email AS contact FROM patients; -- Compute pulse pressure (systolic - diastolic) in SELECT -- Pulse pressure > 60 can indicate arterial stiffness SELECT patient_id, systolic, diastolic, systolic - diastolic AS pulse_pressure FROM vitals; -- DISTINCT — show unique cities only SELECT DISTINCT city FROM patients; -- DISTINCT blood types used by patients SELECT DISTINCT blood_type FROM patients; -- Concatenate name + city into one field SELECT CONCAT(full_name, ' (', city, ')') AS patient_info FROM patients; -- Count totals SELECT COUNT(*) AS total_patients FROM patients; SELECT COUNT(*) AS total_readings FROM vitals;
Column expressions & functions▾
SQL🩺 EHR
-- String functions on patient names SELECT full_name, UPPER(full_name) AS name_upper, LOWER(email) AS email_lower, LENGTH(full_name) AS name_length, SUBSTRING(full_name,1,5) AS first_5_chars FROM patients; -- Number functions on vitals SELECT patient_id, systolic, diastolic, ROUND((systolic + diastolic * 2) / 3.0, 1) AS mean_arterial_pressure, -- MAP ≈ (systolic + 2×diastolic) / 3 — a clinical metric systolic - diastolic AS pulse_pressure FROM vitals; -- Date functions on patient records SELECT full_name, date_of_birth, YEAR(NOW()) - YEAR(date_of_birth) AS approx_age, DATEDIFF(NOW(), created_at) AS days_since_registered FROM patients;

🚂 OS/DB Example — querying passengers and bookings▾
🚂 OS/DB — VW supplier portal query pattern

At OS scale, SELECT column projection matters: a VW supplier portal query that returns SELECT * across a 200M-row bookings table will saturate I/O. Explicit column lists let the optimizer use covering indexes, keeping read throughput predictable under load.

SQL🚂 OS/DB
-- Never SELECT * in production queries on large tables -- Explicit columns = smaller network payload + index coverage possible SELECT p.name, p.loyalty_tier, b.seat_number, b.class, b.price, b.status FROM passengers p JOIN bookings b ON b.passenger_id = p.id WHERE b.status = 'confirmed' LIMIT 100; -- LIMIT is mandatory in application queries — no unbounded scans -- Computed column: display human-readable class label SELECT id, CONCAT('Class ', class) AS class_label, price, price * 1.19 AS price_with_vat FROM bookings WHERE status = 'confirmed' LIMIT 20;
🎤 Interview Questions▾
What is the difference between WHERE and HAVING?

WHERE filters individual rows before grouping — it runs against the raw table data. HAVING filters groups after aggregation — it runs against the result of GROUP BY. You cannot use aggregate functions like COUNT() or AVG() in WHERE; you must use HAVING for those. A common mistake is using HAVING when WHERE would work, which forces aggregation on unnecessary rows.

⚓ Anchor

WHERE filters rows before aggregation; HAVING filters groups after — misusing them produces wrong results or full-table aggregates when a simple WHERE would suffice.

Why is SELECT * problematic in a production application query?

SELECT * reads every column, including large TEXT/BLOB fields and columns the application doesn't use. This increases I/O, network transfer, and memory. It also breaks covering-index optimizations: InnoDB can satisfy a query entirely from the index if the SELECT list matches the index columns; SELECT * always requires a full row lookup. Additionally, SELECT * makes code fragile — if the schema changes (new column added), application result sets change silently.

⚓ Anchor

SELECT * prevents covering indexes, bloats result sets, and creates silent schema coupling — always list explicit columns in application queries.

A Flask endpoint returns a patient's latest vital reading. How would you write that SELECT?

Use a correlated subquery or ORDER BY + LIMIT pattern. The most readable and index-friendly approach: SELECT * FROM vitals WHERE patient_id = ? ORDER BY recorded_at DESC LIMIT 1. With an index on (patient_id, recorded_at DESC), this is a single-row index scan. Avoid a subquery like WHERE recorded_at = (SELECT MAX(recorded_at)...) — it reads the same data twice and is harder to optimize.

⚓ Anchor

ORDER BY timestamp DESC LIMIT 1 on an indexed (patient_id, recorded_at) composite is the canonical "latest record" pattern — one index scan, one row returned.

Filter & Sort

WHERE filters which rows you get back. ORDER BY sorts them. LIMIT caps how many. These three let you answer "find me all patients with high blood pressure" or "show the 5 most recent vitals".

WHERE — basic comparisons▾
🍃 From MongoDB

db.patients.find({city: "Berlin"}) → SELECT * FROM patients WHERE city = 'Berlin'

SQL🩺 EHR
-- Find patients in Berlin SELECT * FROM patients WHERE city = 'Berlin'; -- Find patients NOT in Berlin SELECT * FROM patients WHERE city != 'Berlin'; -- Find vitals with high systolic (Stage 2 hypertension: ≥140 mmHg) SELECT * FROM vitals WHERE systolic >= 140; -- Find vitals in normal range (< 120 systolic AND < 80 diastolic) SELECT * FROM vitals WHERE systolic < 120; -- Find the hypertensive crisis reading (systolic > 180 — urgent!) SELECT * FROM vitals WHERE systolic > 180; -- E-commerce equivalent for context: -- SELECT * FROM products WHERE price > 50; -- SELECT * FROM products WHERE stock = 0; -- out of stock
AND / OR / NOT — combine conditions▾
SQL🩺 EHR
-- AND: both conditions must be true -- High Stage 2 hypertension: systolic ≥140 AND diastolic ≥90 SELECT * FROM vitals WHERE systolic >= 140 AND diastolic >= 90; -- OR: either condition is true -- All elevated readings (Stage 1 OR Stage 2) SELECT * FROM vitals WHERE systolic >= 130 OR diastolic >= 80; -- Patients from Berlin OR Hamburg (two major German cities) SELECT * FROM patients WHERE city = 'Berlin' OR city = 'Hamburg'; -- NOT: invert the condition -- Patients who do NOT have blood type O+ SELECT * FROM patients WHERE blood_type != 'O+'; -- Use parentheses to control AND/OR precedence -- Crisis readings: systolic > 180 OR diastolic > 120 SELECT * FROM vitals WHERE (systolic > 180) OR (diastolic > 120);
LIKE, IN, BETWEEN, IS NULL▾
SQL🩺 EHR
-- LIKE: partial string match (% = any characters) -- Find patients whose name starts with 'S' SELECT * FROM patients WHERE full_name LIKE 'S%'; -- Find patients whose name contains 'er' SELECT * FROM patients WHERE full_name LIKE '%er%'; -- IN: match any value in a list -- Find patients in any of these three cities SELECT * FROM patients WHERE city IN ('Berlin', 'Hamburg', 'Munich'); -- Find patients with blood type A+ or O+ (universal donors) SELECT * FROM patients WHERE blood_type IN ('A+', 'O+', 'O-'); -- BETWEEN: inclusive range -- Elevated BP: systolic between 120 and 139 SELECT * FROM vitals WHERE systolic BETWEEN 120 AND 139; -- IS NULL: find vitals where notes were not recorded SELECT * FROM vitals WHERE notes IS NULL; -- IS NOT NULL: only vitals that have notes SELECT * FROM vitals WHERE notes IS NOT NULL;
ORDER BY & LIMIT▾
SQL🩺 EHR
-- ORDER BY: sort results -- Most recent vital readings first SELECT * FROM vitals ORDER BY recorded_at DESC; -- Highest systolic first (worst readings at the top) SELECT patient_id, systolic, diastolic FROM vitals ORDER BY systolic DESC; -- Sort patients alphabetically by name SELECT full_name, city FROM patients ORDER BY full_name ASC; -- ASC is the default, can be omitted -- LIMIT: only return top N rows -- Top 3 most dangerous BP readings SELECT patient_id, systolic, diastolic FROM vitals ORDER BY systolic DESC LIMIT 3; -- OFFSET: skip N rows (useful for pagination in Morafek's API) -- Page 2: skip first 5 patients, show next 5 SELECT * FROM patients ORDER BY full_name LIMIT 5 OFFSET 5; -- E-commerce equivalent for context: -- SELECT * FROM products WHERE price > 50 ORDER BY price ASC LIMIT 5;

🚂 OS/DB Example — WHERE patterns at Fraport scale▾
🚂 OS/DB — S-Bahn Berlin gate scan filtering

Fraport's ticket validation system runs thousands of WHERE queries per second during peak boarding windows. Index selectivity determines whether a filter uses a fast index scan or a catastrophic full-table scan. A WHERE on status='confirmed' across 500M rows requires the status column to be indexed — otherwise every gate swipe causes a full-table read.

SQL🚂 OS/DB
-- Find all confirmed bookings for a specific route (departure scan scenario) -- Fraport: every passenger at gate G37 in the next 30 minutes SELECT b.id, p.name, b.seat_number, b.class FROM bookings b JOIN routes r ON r.id = b.route_id JOIN passengers p ON p.id = b.passenger_id WHERE b.status = 'confirmed' AND r.origin_station = 'Frankfurt Airport' AND r.scheduled_departure BETWEEN NOW() AND DATE_ADD(NOW(), INTERVAL 30 MINUTE) ORDER BY r.scheduled_departure ASC, b.seat_number ASC; -- Find the 5 highest-price confirmed bookings today (revenue check) -- At VW: highest-cost supplier access sessions for billing report SELECT id, passenger_id, price, booked_at FROM bookings WHERE status = 'confirmed' AND booked_at >= CURDATE() ORDER BY price DESC LIMIT 5;
🎤 Interview Questions▾
A query uses WHERE LOWER(email) = 'user@example.com'. What's the problem?

Wrapping a column in a function like LOWER() prevents the optimizer from using an index on that column. The index stores the original values — the function-wrapped expression is not in the index, so MySQL must apply the function to every row (full table scan). The fix: store emails as lowercase on INSERT, or use a functional index if MariaDB version supports it. For case-insensitive comparisons, a case-insensitive collation (utf8mb4_unicode_ci) on the column handles this transparently without function wrapping.

⚓ Anchor

Functions on indexed columns disable index usage — store data in the form you query it, or use collation for case-insensitive matching.

When does OR in a WHERE clause cause a performance problem?

When OR connects conditions on different columns, MySQL may not be able to use a single index to satisfy both. For example, WHERE city = 'Berlin' OR blood_type = 'O+' requires scanning both a city index and a blood_type index separately and merging results (index merge) — which is often slower than a full scan. IN() is preferable when OR conditions are on the same column (MySQL handles IN() as an index range scan). For multi-column OR, consider UNION ALL of two indexed queries.

⚓ Anchor

OR across different columns often defeats single-index optimization — rewrite as UNION ALL or restructure the query to use IN() on the same column.

You need to find all patients born in the 1980s. How do you write the WHERE and why?

Use WHERE date_of_birth BETWEEN '1980-01-01' AND '1989-12-31' or equivalently WHERE date_of_birth >= '1980-01-01' AND date_of_birth < '1990-01-01'. Avoid WHERE YEAR(date_of_birth) BETWEEN 1980 AND 1989 — the YEAR() function call disables the index. The range scan version allows the optimizer to use an index on date_of_birth for an efficient range scan.

⚓ Anchor

Date range filtering with BETWEEN on the raw column preserves index usage; wrapping in YEAR() or DATE() kills it.

🎤 Interview Questions▾
A developer runs UPDATE patients SET city = 'Berlin' — no WHERE clause. What happens and how do you recover?

Every row in the table is updated — all patients now show city='Berlin'. Recovery requires point-in-time recovery (PITR) from binlog. The exact procedure: (1) identify the timestamp of the bad UPDATE from the binlog with mysqlbinlog, (2) restore from the last full backup, (3) replay binlog events up to the position just before the bad statement. This is why binary logging must be enabled (log_bin=ON) and why developers should not have UPDATE privileges without a WHERE clause restriction in production. In MariaDB you can also set sql_safe_updates=1, which rejects UPDATE/DELETE without a WHERE on an indexed column.

⚓ Anchor

Recovery from a bare UPDATE requires PITR via binlog replay — and sql_safe_updates=1 prevents it from happening in the first place.

Why do production systems use soft deletes (status flag) instead of DELETE?

Hard DELETE is irreversible without backup recovery. Soft deletes preserve the full audit trail, satisfy GDPR right-to-erasure with a separate anonymization step, maintain FK referential integrity (no orphaned related records), and support undo/restore workflows. For KRITIS systems like Fraport, the audit trail of who did what and when is a regulatory requirement — deleted rows can't be audited. The tradeoff is that soft-deleted rows accumulate and require filtering in queries, which is managed with partial indexes or views.

⚓ Anchor

Soft deletes preserve audit trails, FK integrity, and regulatory compliance — KRITIS environments require hard-delete alternatives by default.

How would you safely run a DELETE that removes 10 million rows without locking the table for hours?

Batch the deletion in loops with a LIMIT clause: DELETE FROM table WHERE condition LIMIT 10000 repeated in a script until ROW_COUNT() returns 0. Each batch is a separate transaction, releasing row locks after each commit. Sleep 100ms between batches to give the replication stream and other queries breathing room. This avoids a single multi-million-row transaction that would hold locks for the entire duration and generate a huge undo log. On Galera clusters, large single-batch deletes also trigger flow control on slow nodes.

⚓ Anchor

Batch DELETE with LIMIT in a loop — one 10M-row DELETE transaction locks everything; 1000 batches of 10K rows keep the system responsive.

How GROUP BY works
GROUP BY splits all rows into buckets by the column you specify, then runs the aggregate function on each bucket. "GROUP BY patient_id" gives you one result row per patient. "GROUP BY blood_type" gives one row per blood type.

WHERE filters rows before grouping. HAVING filters groups after grouping. Rule: if you're filtering based on an aggregate result (COUNT, AVG, MAX…) you must use HAVING, not WHERE.

🎤 Interview Questions▾
What does COUNT(*) count vs COUNT(column)?

COUNT(*) counts every row including rows with NULL values in all columns. COUNT(column) counts only rows where that specific column is NOT NULL. This distinction matters when counting optional fields: COUNT(notes) on a vitals table counts only readings that have a note — patients with NULL notes are excluded. For total row counts, always use COUNT(*). Using COUNT(nullable_column) to count rows is a subtle bug that produces incorrect results when NULLs exist.

⚓ Anchor

COUNT(*) = all rows including NULLs; COUNT(col) = only non-NULL values in that column — a bug waiting to happen if you mix them up.

A GROUP BY query runs in 45 seconds on a 100M-row table. What do you check first?

Run EXPLAIN — look for "Using filesort" and "Using temporary" in the Extra column. These indicate MySQL is creating an in-memory or on-disk temporary table to perform the grouping, because no suitable index exists. The fix: add a composite index on (GROUP BY columns + WHERE filter columns). For example, GROUP BY route_id WHERE status='confirmed' benefits from an index on (status, route_id). Also check if the query can be rewritten to narrow the row count with a stronger WHERE before grouping.

⚓ Anchor

"Using temporary; Using filesort" in EXPLAIN means no index serves the GROUP BY — the composite index fix is the first thing to reach for.

Can you use a column alias defined in SELECT inside the same query's WHERE clause?

No — in standard SQL, aliases defined in SELECT are not available to WHERE because WHERE is evaluated before SELECT in the logical query processing order. The processing order is: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. Aliases are only available from ORDER BY onward (and HAVING in MariaDB/MySQL as an extension). To filter on an expression, repeat it in WHERE or use a subquery/CTE.

⚓ Anchor

SQL logical processing order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY — SELECT aliases aren't defined when WHERE runs.

🎤 Interview Questions▾
What is the difference between ON DELETE RESTRICT and ON DELETE CASCADE?

RESTRICT (and NO ACTION, which behaves identically in InnoDB) prevents deletion of the parent row if child rows exist — the delete fails with an error. CASCADE automatically deletes all child rows when the parent is deleted. RESTRICT is the safer default: it forces the application to handle deletion order explicitly, which prevents accidental data loss. CASCADE is appropriate only when child records have no independent meaning — e.g. deleting an order deletes its line items. In KRITIS systems, RESTRICT is preferred because cascading deletes can silently remove audit-relevant data.

⚓ Anchor

RESTRICT is the safe default — it surfaces the dependency so you handle deletion order deliberately; CASCADE can silently wipe child records you need for audits.

You need to load 50 million rows from a CSV into a table with FK constraints. How do you speed this up?

Disable FK checks before the bulk load with SET foreign_key_checks = 0, load the data (LOAD DATA INFILE is the fastest path), then re-enable with SET foreign_key_checks = 1. After re-enabling, run a validation query to check for orphans (LEFT JOIN / IS NULL pattern). This approach is safe if you control the data — dangerous if the source data may contain FK violations, as they'll be silently inserted and only detected on re-enable (which doesn't retroactively validate existing rows in all MariaDB versions).

⚓ Anchor

SET foreign_key_checks=0 for bulk loads, then validate with a LEFT JOIN IS NULL check after re-enabling — never skip the validation step.

What normalisation form requires that no non-key column depends on a partial key?

Second Normal Form (2NF). A table violates 2NF when it has a composite primary key and a non-key column that depends on only part of that composite key. For example, a bookings table with PK (passenger_id, route_id) that also stores passenger_name would violate 2NF — passenger_name depends only on passenger_id, not the full composite key. The fix: move passenger_name to the passengers table, referenced by FK. This is exactly the normalization decision behind CareMate's schema: patient names live in patients, visit data lives in visits.

⚓ Anchor

2NF: every non-key column must depend on the whole primary key — partial dependency = move the column to the parent table it actually belongs to.

A missing index on bookings.route_id means a nested-loop join that scans bookings for every route row — at 500M bookings this takes minutes instead of milliseconds.

SQL🚂 OS/DB
-- 3-table join: passenger names + booking details + route info -- Used by Fraport gate display boards and mobile check-in apps SELECT p.name AS passenger_name, p.loyalty_tier, b.seat_number, b.class, r.origin_station, r.destination_station, r.scheduled_departure FROM bookings b -- drive from bookings (filtered first) INNER JOIN passengers p ON p.id = b.passenger_id INNER JOIN routes r ON r.id = b.route_id WHERE r.origin_station = 'Frankfurt Airport' AND r.scheduled_departure BETWEEN '2024-06-10 06:00' AND '2024-06-10 08:00' AND b.status = 'confirmed' ORDER BY r.scheduled_departure, b.seat_number; -- LEFT JOIN to find passengers with NO bookings in the last 90 days -- Marketing: lapsed customers at VW corporate travel portal SELECT p.name, p.email, p.loyalty_tier FROM passengers p LEFT JOIN bookings b ON b.passenger_id = p.id AND b.booked_at >= DATE_SUB(CURDATE(), INTERVAL 90 DAY) WHERE b.id IS NULL; -- NULL on the right side = no matching booking
🎤 Interview Questions▾
Explain INNER JOIN vs LEFT JOIN with a concrete example from a PHR system.

INNER JOIN returns only rows where a match exists in both tables. LEFT JOIN returns all rows from the left table, with NULLs for right-table columns where no match exists. In CareMate: SELECT * FROM patients INNER JOIN visits ON visits.patient_id = patients.id returns only patients who have at least one visit — new patients with no visits disappear from results. The LEFT JOIN version returns all patients, with NULL visit columns for those without visits. The wrong choice here produces subtle bugs: a patient count using INNER JOIN would miss patients who haven't had their first appointment yet.

⚓ Anchor

INNER JOIN silently excludes non-matching rows — use LEFT JOIN when you need to preserve all records from the driving table regardless of whether a match exists.

A JOIN query on two large tables runs without an index on the JOIN column. What happens at the execution level?

MySQL uses a nested-loop join: for each row in the outer table, it scans the entire inner table looking for matches. Without an index on the join column, the inner scan is a full table scan — so the total work is outer_rows × inner_rows (Cartesian product of scans). With an index, the inner lookup is an O(log n) index seek per outer row. At OS scale with 500M bookings, a missing index on bookings.route_id turns a 1ms query into a multi-minute full scan. EXPLAIN shows "type: ALL" on the inner table when the index is missing.

⚓ Anchor

JOIN without an index on the join column = nested-loop full scan; EXPLAIN type:ALL on the inner table is the diagnostic signal.

When would you use a self-join?

A self-join joins a table to itself, useful for hierarchical or comparative data. Examples: finding employees and their managers in an org-chart table where manager_id references the same employees table; finding patients who have both a normal reading and a hypertensive reading (by joining vitals to itself on patient_id with different WHERE conditions on each alias). Self-joins require table aliases to distinguish the two roles: FROM vitals v1 JOIN vitals v2 ON v1.patient_id = v2.patient_id WHERE v1.systolic > 180 AND v2.systolic < 120.

⚓ Anchor

Self-join = same table twice with different aliases — classic use cases are hierarchical lookups (manager → employee) and within-table comparisons.

Indexes

Without indexes, MySQL reads every single row to find data. With the right index, it jumps directly. This is why fetching "all vitals for patient 1" from a million-row vitals table can go from 10 seconds to 10 milliseconds.

Click any section to expand
What an index is▾
Index = sorted lookup table
An index on a column is like the index in a medical textbook: instead of reading every page to find "Hypertension", you jump to H in the index and get the exact page. MySQL builds and maintains this sorted structure automatically — you just tell it which column(s) to index.
When to add an index
Index columns that appear in WHERE clauses, JOIN conditions, or ORDER BY. In Morafek: vitals.patient_id (every vitals query filters by patient), visits.patient_id, visits.doctor_id. Don't index every column — each index slows down INSERT/UPDATE slightly.
The problem — a full table scan▾
SQL🩺 EHR
-- Step 1: Ask MySQL to explain how it would run this query EXPLAIN SELECT * FROM vitals WHERE patient_id = 1; -- Look at: type=ALL (full scan), rows=9 (reads all 9 rows for just 1 patient!) -- On a real PHR with 1 million vital readings, this would be disastrous.
⚠️ type=ALL means full table scan

In EXPLAIN output, type=ALL means MySQL is reading every single row. If your vitals table has 10 million rows (a realistic production PHR), this query would read all 10 million rows every time a patient opens their app. That's why indexes are critical.

CREATE INDEX — fix it▾
SQL🩺 EHR
-- Step 2: Create an index on vitals.patient_id -- (every vitals lookup in Morafek filters by patient_id) CREATE INDEX idx_vitals_patient ON vitals(patient_id); -- Step 3: Run EXPLAIN again EXPLAIN SELECT * FROM vitals WHERE patient_id = 1; -- type=ref, key=idx_vitals_patient ← index is now used! -- Also index visits for common Morafek queries CREATE INDEX idx_visits_patient ON visits(patient_id); CREATE INDEX idx_visits_doctor ON visits(doctor_id); -- Index recorded_at for time-range queries on vitals -- ("show me all readings in the last 30 days") CREATE INDEX idx_vitals_date ON vitals(recorded_at); -- E-commerce equivalent for context: -- CREATE INDEX idx_oi_product ON order_items(product_id); -- CREATE INDEX idx_orders_customer ON orders(customer_id);
List and drop indexes▾
SQL🩺 EHR
-- See all indexes on a table SHOW INDEX FROM vitals; SHOW INDEX FROM visits; -- Remove an index (if it turns out it's not needed or slowing writes) DROP INDEX idx_vitals_date ON vitals;
EXPLAIN output — how to read it▾
Key EXPLAIN columns
  • type: ALL = full scan (bad for large tables), ref/range/eq_ref = index used (good)
  • key: which index was used (NULL = no index used)
  • rows: estimated number of rows MySQL will examine — lower is better
  • Extra: "Using where" = filtering applied; "Using filesort" = may benefit from an index on ORDER BY column

🚂 OS/DB — Fraport KRITIS index design

Index design at OS scale is not optional performance tuning — it's the difference between a gate manifest query returning in 2ms and blocking the boarding process for 20 seconds. Over-indexing is also a real cost: each index adds write overhead on INSERT/UPDATE, which matters on Galera clusters where write-sets must be certified across all nodes.

SQL🚂 OS/DB
-- Composite index for the gate manifest query pattern: -- WHERE status='confirmed' AND route_id=? ORDER BY seat_number ALTER TABLE bookings ADD INDEX idx_booking_route_status (route_id, status, seat_number); -- Covers: WHERE route_id=? AND status=? ORDER BY seat_number -- All three columns in the index = covering index, no row lookup needed -- Index for passenger lookup by email (login / single-record fetch) ALTER TABLE passengers ADD UNIQUE INDEX idx_passenger_email (email); -- Index for revenue aggregation: status + booked_at date range ALTER TABLE bookings ADD INDEX idx_booking_status_date (status, booked_at); -- VERIFY with EXPLAIN — look for key: idx_... and rows ≈ 1 -- Bad: type:ALL, key:NULL → full table scan -- Good: type:ref or range, key:idx_..., rows:small number EXPLAIN SELECT id, passenger_id, seat_number FROM bookings WHERE route_id = 301 AND status = 'confirmed' ORDER BY seat_number; -- Find unused indexes (wasted write overhead — drop these) SELECT object_name, index_name, count_star FROM performance_schema.table_io_waits_summary_by_index_usage WHERE object_schema = 'os_db' AND index_name != 'PRIMARY' AND count_star = 0 ORDER BY object_name;
🎤 Interview Questions▾
What is a covering index and why does it matter for read performance?

A covering index includes all columns that a query needs — both the filter columns (WHERE) and the output columns (SELECT list). When a covering index exists, InnoDB can satisfy the query entirely from the index B-tree without reading the actual table rows (the clustered index). This eliminates a second I/O operation (the "row lookup") per matching row. On high-read queries at OS scale — like the gate manifest query that runs thousands of times per minute — the covering index can reduce I/O by 50–90%. EXPLAIN shows "Using index" in the Extra column when a covering index is in use.

⚓ Anchor

Covering index = query satisfied entirely from the index, no row lookup — "Using index" in EXPLAIN confirms it; it's the single highest-impact index optimization.

Why does column order matter in a composite index?

InnoDB composite indexes are ordered left-to-right — the index is sorted first by the leftmost column, then by the next, and so on. A query can use the index only if it filters on a left-prefix of the index columns. An index on (status, route_id) can satisfy WHERE status='confirmed' or WHERE status='confirmed' AND route_id=5, but NOT a query that only filters on route_id (no left-prefix match). Put the most selective filter first (fewest distinct values that narrow the scan the most) — in practice, equality conditions before range conditions.

⚓ Anchor

Composite index left-prefix rule: columns must be used from left to right — skipping the leftmost column skips the index entirely.

A table has 15 indexes. What problems does this cause and how do you identify which to remove?

Too many indexes slow down writes: every INSERT/UPDATE/DELETE must update all affected indexes. On a Galera cluster, this also enlarges the write-set that must be certified across nodes, increasing certification latency. To identify unused indexes: query performance_schema.table_io_waits_summary_by_index_usage where count_star=0 — these indexes have never been used since the last server restart and are candidates for removal. Also look for duplicate indexes (same leading columns) with pt-duplicate-key-checker from Percona Toolkit. Remove unused indexes in a maintenance window, after verifying in slow query log that no queries relied on them.

⚓ Anchor

Unused indexes identified via performance_schema and removed with pt-duplicate-key-checker — on Galera, excess indexes hurt write-set certification speed.

Column modifiers, table-level rules, and built-in value generators. These are the tools that enforce data quality at the database layer — so your app never has to.
⚙️ Column Attributes — modify how a column stores or generates values
AUTO_INCREMENT Numeric
Automatically assigns the next integer value on each INSERT. Always paired with PRIMARY KEY or UNIQUE. Starts at 1 by default.

Removes negative range and doubles the positive range. Applied to INT, TINYINT, BIGINT, etc. Use for any column that can never be negative (IDs, counts, ages).

Sets the fallback value used when a column is omitted from an INSERT statement. Can be a literal value or a function like CURRENT_TIMESTAMP.
SQL

NOT NULL Any type
Forbids NULL values. Every INSERT or UPDATE must supply a value (or fall back on DEFAULT). Without this, columns accept NULL implicitly.
SQL

Explicitly allows NULL values (the implicit default for most columns). Useful for optional fields where absence of data is a valid state.
SQL

ON UPDATE CURRENT_TIMESTAMP Timestamp
Automatically updates the column's value whenever any column in the row is modified. The standard pattern for updated_at audit columns.

Creates a virtual or stored column whose value is derived from an expression. You never INSERT into it — the DB computes it automatically.

🔒 Table Constraints — enforce rules across rows and tables
PRIMARY KEY
Uniquely identifies each row. Implicitly NOT NULL and UNIQUE. Every table should have exactly one. Creates a clustered index in InnoDB.
SQL
PRIMARY KEY (id) -- or composite: PRIMARY KEY (flight_id, passenger_id)
UNIQUE
Ensures no two rows have the same value in this column (or combination of columns). Unlike PRIMARY KEY, a table can have multiple UNIQUE constraints, and NULLs are allowed (multiple NULLs are permitted).
SQL
UNIQUE (email) UNIQUE (flight_number, departure_date) -- composite unique
FOREIGN KEY
Links a column in this table to the PRIMARY KEY of another table. Enforces referential integrity — you cannot insert a row that references a non-existent parent row.
SQL
FOREIGN KEY (passenger_id) REFERENCES passengers(id) ON DELETE CASCADE -- delete booking when passenger deleted ON UPDATE CASCADE -- update FK when PK changes
CHECK
Validates that column values meet a condition before accepting the row. Supported in MariaDB 10.2+ and MySQL 8.0.16+. Older versions parse but silently ignore CHECK.
SQL
CHECK (ticket_price > 0) CHECK (departure_at < arrival_at) CHECK (seat_count BETWEEN 1 AND 853)
