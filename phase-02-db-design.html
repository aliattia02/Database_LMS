Phase 02 · Lesson 01 · 4h
Normalization — 1NF → 3NF

Normalization is the systematic process of structuring a database to reduce redundancy and improve data integrity. Understanding the normal forms tells an interviewer you can reason about schema correctness — not just write CREATE TABLE.

Click any section to expand
Why This Matters▾

Normalization is the foundation of every good relational schema. In an OS GmbH interview, Michael Nietsch will care less about your exact definition of 3NF and more about whether you can look at a schema and immediately spot the redundancy that will cause update anomalies in production. For Morafek CareMate, proper normalization is also a DSGVO requirement — patient data duplicated across tables means more surfaces for inconsistency and audit failure.

🎯 Interview Anchor

"I normalize first, always to at least 3NF, then I selectively denormalize only where I have a measured read-performance problem and document the trade-off explicitly."

Core Concepts▾
1NF — Atomic values, no repeating groups
Every column holds exactly one indivisible value. No comma-separated lists in a cell. No phone1/phone2/phone3 columns. Each row must be uniquely identifiable by a primary key.

EHR: An early CareMate schema stored allergies as "penicillin,aspirin,latex" in one VARCHAR column. Querying for all penicillin patients required a LIKE '%penicillin%' scan. Fix: a separate patient_allergies(patient_id, allergy) table.
OS/DB: A booking system that stored multiple seat numbers as "12A,13A,14A" in a single column could not enforce seat uniqueness or answer "which passengers are in row 12" without string parsing.
2NF — No partial dependencies (applies to composite PKs)
Every non-key column must depend on the entire primary key, not just part of it. This only applies when the primary key is composite (multi-column). If a column depends on only one part of the composite PK, it belongs in a separate table.

EHR: A visits table with PK (visit_id, patient_id) that also stored patient_name violated 2NF — patient_name depends on patient_id alone. Move it to the patients table.
OS/DB: A bookings_details table with PK (booking_id, route_id) that stored route_description violated 2NF — that description depends only on route_id.
3NF — No transitive dependencies
Non-key columns must depend only on the primary key — not on other non-key columns. If column B determines column C, and B is not the PK, then C should be in a separate table keyed by B.

EHR: Storing icd10_code and icd10_description in the visits table violates 3NF — the description is determined by the code, not the visit. A separate icd10_codes(code, description, category) lookup table solves this.
OS/DB: Storing station_id and station_city in the routes table violates 3NF — city is determined by station_id, not by route_id.
BCNF — Boyce-Codd Normal Form (stricter 3NF)
For every functional dependency A → B, A must be a superkey. BCNF catches edge cases that 3NF misses when there are multiple overlapping candidate keys. In practice, 3NF is sufficient for most operational databases — BCNF matters most for complex multi-key tables.

EHR: Rarely relevant in CareMate's operational tables. Arises in cases like scheduling where multiple candidate keys overlap (doctor + time + room).
OS/DB: Train assignment tables where (train_number, date) and (driver_id, date) are both candidate keys can create BCNF violations.
When to deliberately denormalize
Denormalization is a conscious trade-off: you duplicate data to improve read performance, at the cost of write complexity and risk of inconsistency. Always normalize first — measure the problem — then denormalize with documentation.

EHR: A CareMate analytics dashboard that aggregates vitals by city over 5 years might denormalize city onto the vitals table rather than joining patients every time.
OS/DB: Deutsche Bahn's reporting layer uses a star schema (denormalized) for monthly KPI queries, while the operational OLTP system remains in 3NF.
EHR Example — Normalizing CareMate▾
🩺 Morafek CareMate Context

An early version of CareMate stored visit data in a single flat table. The normalization process below fixes all three normal form violations, producing the production schema used throughout this course.

SQL 🩺 EHR
-- ❌ BEFORE: Un-normalized "fat" visits table (violates all three NFs) CREATE TABLE visits_bad ( visit_id INT PRIMARY KEY, patient_id INT, patient_name VARCHAR(200), -- violates 2NF: depends on patient_id, not visit patient_allergies VARCHAR(500), -- violates 1NF: comma-separated list doctor_id INT, doctor_name VARCHAR(200), -- violates 2NF: depends on doctor_id doctor_specialty VARCHAR(100), -- violates 2NF: depends on doctor_id icd10_code VARCHAR(10), icd10_description VARCHAR(300), -- violates 3NF: depends on icd10_code reason TEXT, created_at DATETIME ); -- ✅ AFTER: Normalized to 3NF — the actual CareMate schema -- Patients table: stores patient facts once CREATE TABLE patients ( id BIGINT AUTO_INCREMENT PRIMARY KEY, full_name VARCHAR(200) NOT NULL, date_of_birth DATE NOT NULL, blood_type ENUM('A+','A-','B+','B-','AB+','AB-','O+','O-'), city VARCHAR(100), email VARCHAR(255) UNIQUE, created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ); -- Fix 1NF violation: allergies in their own table (one row per allergy) CREATE TABLE patient_allergies ( patient_id BIGINT NOT NULL, allergy VARCHAR(100) NOT NULL, severity ENUM('mild','moderate','severe') NOT NULL DEFAULT 'moderate', recorded_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, PRIMARY KEY (patient_id, allergy), -- composite PK prevents duplicates FOREIGN KEY (patient_id) REFERENCES patients(id) ON DELETE CASCADE ); -- Fix 3NF: ICD-10 codes in their own lookup table CREATE TABLE icd10_codes ( code VARCHAR(10) PRIMARY KEY, description VARCHAR(300) NOT NULL, category VARCHAR(100) ); -- Clean visits table: references parent tables, no redundant data CREATE TABLE visits ( id BIGINT AUTO_INCREMENT PRIMARY KEY, patient_id BIGINT NOT NULL, doctor_id BIGINT NOT NULL, icd10_code VARCHAR(10), reason TEXT, status ENUM('scheduled','completed','cancelled') NOT NULL DEFAULT 'scheduled', notes TEXT, created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, FOREIGN KEY (patient_id) REFERENCES patients(id) ON DELETE RESTRICT, FOREIGN KEY (doctor_id) REFERENCES doctors(id) ON DELETE RESTRICT, FOREIGN KEY (icd10_code) REFERENCES icd10_codes(code) ON DELETE SET NULL );
OS/DB Example — Normalizing Booking Data▾
🚂 OS / Deutsche Bahn Context

Operational services manages millions of daily bookings. A poorly normalized bookings table would mean updating a route's departure time in thousands of rows instead of one.

SQL 🚂 OS/DB
-- ❌ BEFORE: flat booking record (update anomaly nightmare) -- Updating a route departure time requires touching every booking row CREATE TABLE bookings_bad ( booking_id INT PRIMARY KEY, passenger_name VARCHAR(200), -- 2NF violation passenger_bahn_card VARCHAR(20), -- 2NF violation origin_station VARCHAR(100), -- 2NF violation destination_station VARCHAR(100), -- 2NF violation scheduled_departure DATETIME, -- 2NF violation origin_city VARCHAR(100), -- 3NF: city determined by station seat_number VARCHAR(5), price DECIMAL(8,2) ); -- ✅ AFTER: 3NF — normalized OS production schema CREATE TABLE stations ( id INT AUTO_INCREMENT PRIMARY KEY, name VARCHAR(100) NOT NULL, city VARCHAR(100) NOT NULL, -- city lives here, not in routes eva_code VARCHAR(10) UNIQUE -- Deutsche Bahn station code ); CREATE TABLE passengers ( id BIGINT AUTO_INCREMENT PRIMARY KEY, name VARCHAR(200) NOT NULL, bahn_card_number VARCHAR(20) UNIQUE, loyalty_tier ENUM('none','silver','gold','senator') DEFAULT 'none', email VARCHAR(255) UNIQUE NOT NULL ); CREATE TABLE routes ( id INT AUTO_INCREMENT PRIMARY KEY, origin_station_id INT NOT NULL, destination_station_id INT NOT NULL, scheduled_departure DATETIME NOT NULL, scheduled_arrival DATETIME NOT NULL, FOREIGN KEY (origin_station_id) REFERENCES stations(id), FOREIGN KEY (destination_station_id) REFERENCES stations(id) ); CREATE TABLE bookings ( id BIGINT AUTO_INCREMENT PRIMARY KEY, passenger_id BIGINT NOT NULL, route_id INT NOT NULL, seat_number VARCHAR(5), class ENUM('1st','2nd') NOT NULL DEFAULT '2nd', price DECIMAL(8,2) NOT NULL, status ENUM('confirmed','cancelled','used') NOT NULL DEFAULT 'confirmed', booked_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, FOREIGN KEY (passenger_id) REFERENCES passengers(id) ON DELETE RESTRICT, FOREIGN KEY (route_id) REFERENCES routes(id) ON DELETE RESTRICT ); -- Now updating Frankfurt Hbf departure time touches ONE row in routes -- Not millions of rows in bookings — this is the power of normalization
Common Pitfalls▾
⚠️ Storing lists in a single column

The most common 1NF violation: tags VARCHAR(500) storing "urgent,cardiology,follow-up". This breaks indexing, prevents proper FK relationships, and makes any query on individual tags a full-table scan with LIKE. Always create a junction table.

⚠️ Premature denormalization "for performance"

Adding redundant columns before measuring an actual performance problem is technical debt, not optimization. Joins in a properly indexed 3NF schema are extremely fast. Denormalize only after you have EXPLAIN output proving a join is the bottleneck — and document the decision in a schema comment.

⚠️ Forgetting that normalization applies to data, not just structure

You can have a perfectly normalized schema that still stores denormalized data. Example: storing the full address string "Bahnhofstr. 12, 60329 Frankfurt" in a single VARCHAR when you will later need to filter by city or postal code. Break it into street, postal_code, city columns from the start.

🎤 Interview Questions▾
What is the difference between 2NF and 3NF? Give a concrete example.

2NF eliminates partial dependencies — where a non-key column depends on only part of a composite primary key. This only applies when the PK is composite. 3NF eliminates transitive dependencies — where a non-key column depends on another non-key column rather than directly on the primary key. Example: in a visits table, storing doctor_name alongside doctor_id violates 2NF (doctor_name depends on doctor_id, not visit_id). Storing icd10_description alongside icd10_code violates 3NF (description depends on code, which is a non-PK column).

🎯 Anchor

"2NF is about composite keys — partial dependencies. 3NF is about non-key columns — transitive dependencies. In practice I normalize to 3NF as a default."

When would you intentionally denormalize a schema?

Denormalization is justified when: (1) a specific read query is measured to be slow due to expensive joins, (2) the data is more read-heavy than write-heavy, and (3) the write complexity of maintaining duplicated data is manageable. Common use cases are analytics/reporting tables, materialized aggregations, and data warehouse star schemas. I always normalize first, measure the problem, then denormalize the specific part that's slow — and leave a comment in the schema explaining why.

🎯 Anchor

"Normalize first, measure second, denormalize last — and always document the reason in the schema itself."

What are update anomalies and how does normalization prevent them?

An update anomaly occurs when the same fact is stored in multiple places and an update to one copy creates inconsistency. For example, storing doctor specialty in the visits table means updating a doctor's specialty requires touching every visit row — and a missed row creates inconsistent data. Normalization ensures each fact is stored in exactly one place, so updates are atomic and consistent. Delete anomalies (losing facts when deleting a row) and insert anomalies (inability to insert without unrelated data) are similarly resolved by proper normalization.

🎯 Anchor

"Normalization eliminates update, insert, and delete anomalies by ensuring each fact exists in exactly one place."

Phase 02 · Lesson 02 · 3h
Keys, Constraints & Relationships

Constraints are the database enforcing your business rules automatically — so your application code doesn't have to. Understanding primary keys, foreign keys, and referential integrity is critical for any DBA role.

Click any section to expand
Why This Matters▾

At OS GmbH, you are responsible for databases where a deleted parent row could cascade and corrupt booking or operational data. Understanding ON DELETE behaviour is not theoretical — it is what prevents 3am disasters. For CareMate, proper foreign key constraints are also a DSGVO safeguard: referential integrity ensures you cannot have patient vitals pointing to a non-existent patient.

🎯 Interview Anchor

"I use surrogate BIGINT primary keys almost universally — natural keys change over time, contain PII, and slow down joins. Foreign key constraints are always ON by default in my schemas unless there's a documented performance reason to disable them."

Core Concepts▾
Natural key vs Surrogate key
A natural key is a real-world identifier — email address, passport number, VAT ID. A surrogate key is a system-generated integer (AUTO_INCREMENT / BIGINT). Surrogate keys are almost always better: natural keys change (email addresses), contain PII, can be long strings that slow B-tree traversal, and cause complications if the business requirement changes.

EHR: Patient email is tempting as a natural key but patients change email addresses. BIGINT AUTO_INCREMENT id is the correct PK.
OS/DB: Train numbers look stable but Deutsche Bahn renumbers trains seasonally. Internal BIGINT id is the correct PK for trains table.
ON DELETE / ON UPDATE referential actions
RESTRICT (default): refuse the parent delete/update if children exist. Safest option — the application must handle deletion explicitly.
CASCADE: automatically delete/update children when parent changes. Convenient but dangerous if misconfigured.
SET NULL: set the FK column to NULL when parent is deleted. Requires the FK column to be nullable.
NO ACTION: same as RESTRICT in MariaDB, checked at end of statement rather than immediately.

EHR: vitals.patient_id FK ON DELETE RESTRICT — never allow deleting a patient who has vitals records without explicit cleanup.
OS/DB: ticket_scans.booking_id FK ON DELETE CASCADE — if a booking is cancelled and purged, the scan records are meaningless and should go too.
UNIQUE constraints vs UNIQUE indexes
A UNIQUE constraint and a UNIQUE index are functionally equivalent in MariaDB/MySQL — defining one creates the other. They prevent duplicate values across one or more columns. UNIQUE on a nullable column allows multiple NULL values (NULL ≠ NULL in SQL).

EHR: patients.email UNIQUE prevents two patient records sharing an email — critical for CareMate login and communication.
OS/DB: A composite UNIQUE on (route_id, seat_number) in bookings prevents double-booking the same seat on the same train.
Many-to-many: junction tables
A direct many-to-many relationship cannot be expressed with a FK — you need an intermediate junction (associative/bridge) table. Its PK is typically a composite of both FK columns. The junction table can carry its own attributes (the relationship data).

EHR: patient_doctors(patient_id, doctor_id, assigned_at, is_primary) — a patient has multiple treating doctors, a doctor treats multiple patients.
OS/DB: train_staff(train_id, staff_id, role, shift_date) — multiple staff per train, each staff member may work multiple trains.
CHECK constraints — database-enforced business rules
A CHECK constraint validates a condition at INSERT/UPDATE time, at the database level — independent of application code. MariaDB has supported enforced CHECK constraints since 10.2.1.

EHR: CHECK (systolic > diastolic) on the vitals table prevents medically impossible blood pressure readings from being inserted.
OS/DB: CHECK (scheduled_arrival > scheduled_departure) on routes prevents logically impossible timetable entries.

OS/DB Example — Booking Integrity▾
🚂 Deutsche Bahn Context

With millions of daily transactions, a missing FK constraint means inconsistent data reaching the KRITIS railway control system. Constraints are non-negotiable in operational services infrastructure.

SQL 🚂 OS/DB
-- Bookings with all constraints preventing operational data corruption CREATE TABLE bookings ( id BIGINT AUTO_INCREMENT PRIMARY KEY, passenger_id BIGINT NOT NULL, route_id INT NOT NULL, seat_number VARCHAR(5) NOT NULL, class ENUM('1st','2nd') NOT NULL DEFAULT '2nd', price DECIMAL(8,2) NOT NULL, status ENUM('confirmed','cancelled','used') NOT NULL DEFAULT 'confirmed', booked_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, -- Prevent double-booking: same route + same seat must be unique CONSTRAINT uk_booking_seat UNIQUE (route_id, seat_number), -- Price must be non-negative CONSTRAINT chk_price CHECK (price >= 0), -- Referential integrity: RESTRICT prevents orphaned bookings CONSTRAINT fk_booking_passenger FOREIGN KEY (passenger_id) REFERENCES passengers(id) ON DELETE RESTRICT ON UPDATE CASCADE, CONSTRAINT fk_booking_route FOREIGN KEY (route_id) REFERENCES routes(id) ON DELETE RESTRICT ON UPDATE CASCADE ); -- Query: find all routes with at least one booking, show occupancy SELECT r.id, s_orig.name AS origin, s_dest.name AS destination, r.scheduled_departure, COUNT(b.id) AS bookings_count, t.capacity, ROUND(COUNT(b.id) * 100.0 / t.capacity, 1) AS occupancy_pct FROM routes r JOIN stations s_orig ON r.origin_station_id = s_orig.id JOIN stations s_dest ON r.destination_station_id = s_dest.id JOIN trains t ON r.train_id = t.id LEFT JOIN bookings b ON b.route_id = r.id AND b.status = 'confirmed' GROUP BY r.id, s_orig.name, s_dest.name, r.scheduled_departure, t.capacity ORDER BY r.scheduled_departure;
Common Pitfalls▾
⚠️ SET foreign_key_checks = 0 left in production scripts

Disabling FK checks for a data import and forgetting to re-enable them is a production hazard. Always use a transaction and verify foreign_key_checks=ON is restored. In MariaDB, this is a session variable — re-enabling affects only the current session. The safer pattern is to use LOAD DATA with the constraints pre-validated.

⚠️ Using CASCADE DELETE without fully understanding the cascade chain

ON DELETE CASCADE is powerful but dangerous in deep relationship trees. Deleting a parent can silently remove thousands of child rows across multiple tables. Always map the full cascade chain before enabling it. For anything touching medical or financial data at CareMate or OS, prefer RESTRICT and write explicit cleanup logic.

⚠️ Integer overflow on surrogate keys — using INT instead of BIGINT

An INT (4 bytes) maxes out at ~2.1 billion. Deutsche Bahn processes millions of ticket scans per day — INT on ticket_scans.id overflows in months. Always use BIGINT (8 bytes, ~9.2 × 10¹⁸) for any table that will see significant volume. The storage difference is 4 bytes per row — negligible.

🎤 Interview Questions▾
What is the difference between ON DELETE RESTRICT and ON DELETE CASCADE? When would you use each?

RESTRICT prevents the parent row deletion if any child rows reference it — the database returns an error and the application must handle the cleanup explicitly. This is the safest default for critical data (patient records, financial transactions). CASCADE automatically deletes all child rows when the parent is deleted — useful for junction tables or data that is meaningless without the parent (e.g., ticket scan records without the booking). In a healthcare or railway system I default to RESTRICT on all business entities and only use CASCADE on technical/log tables where orphaned rows have no business meaning.

🎯 Anchor

"RESTRICT for business data, CASCADE only for dependent technical records — and I always map the full cascade chain before enabling it."

Why should you generally prefer surrogate keys over natural keys as primary keys?

Natural keys (email, VAT ID, train number) seem convenient but have three major problems in practice: (1) they change — a patient changes their email address, breaking all child row references; (2) they often contain PII (email, passport) that you want to be able to anonymize independently of the row's relationships; (3) string PKs are slower in B-tree traversal than integer PKs, and foreign key joins become expensive at scale. BIGINT AUTO_INCREMENT surrogate keys are fast, stable, PII-free, and unambiguous. The natural key can still have a UNIQUE constraint for business deduplication without being the PK.

🎯 Anchor

"Natural keys change and contain PII — I use surrogate BIGINT PKs and put the natural key on a UNIQUE constraint separately."

How do you model a many-to-many relationship in a relational database?

With a junction (associative/bridge) table that holds the foreign keys to both parent tables. The junction table's primary key is typically the composite of both FKs, which also acts as a natural deduplication constraint. The junction table can carry its own attributes — timestamps, roles, status — that describe the relationship itself rather than either entity. For example, patient_doctors(patient_id, doctor_id, assigned_at, is_primary) — the relationship has its own metadata. Always index both FK columns for bidirectional join performance.

🎯 Anchor

"Junction table with composite PK, FKs to both parent tables, and additional attributes for any data that belongs to the relationship itself."

Phase 02 · Lesson 03 · 3h
Index Design Strategy

Indexes are the single most impactful performance tool in a DBA's toolkit. The right index turns a 5-second query into 4ms. The wrong index silently slows down every write. This lesson teaches you how to think about indexes strategically.

Click any section to expand
Why This Matters▾

At OS GmbH, query performance directly affects train schedule reliability. A slow query in the booking system at peak time (Friday 17:00 departures) causes cascading delays. Michael Nietsch will almost certainly ask you to read an EXPLAIN output — this is the most common practical DBA task, and it requires understanding how indexes work at the B-tree level.

🎯 Interview Anchor

"When I see a slow query, my first action is always EXPLAIN ANALYZE — not guessing. I look at the type column: ref or range is good, ALL means full table scan. Then I identify the access pattern and create the minimal covering index that serves that pattern."

Core Concepts▾
How B-tree indexes work
A B-tree index is a sorted tree structure. The leaf nodes contain: the indexed value(s) + a pointer back to the row (the clustered index key / PK value). A lookup traverses the tree in O(log n) — for a 1M-row table, that's ~20 comparisons instead of 1,000,000. The engine uses the index when the WHERE clause can use the sorted order.

EHR: An index on visits.patient_id turns "all visits for patient #42" from a full-table scan of 2M rows to a 20-step tree traversal.
OS/DB: An index on ticket_scans.scanned_at allows range queries ("all scans in the last 5 minutes") to be answered by reading only the relevant leaf pages.
Composite index column order — the leftmost prefix rule
A composite index on (A, B, C) can satisfy queries filtering on: A, (A, B), or (A, B, C). It cannot efficiently satisfy a query filtering only on B or only on C. The rule: put the most-queried column first, then the second-most, etc. Equality conditions before range conditions in the index.

EHR: Index (patient_id, recorded_at) on vitals serves both "all vitals for patient X" and "all vitals for patient X in the last 30 days". An index on just (recorded_at) cannot serve the first query efficiently.
OS/DB: Index (route_id, status, booked_at) on bookings serves "all confirmed bookings for route 42 in May" efficiently. Query filtering only on status would not use this index.
Covering indexes — index-only scans
A covering index includes all columns needed by a query — SELECT columns + WHERE columns. The engine can answer the query entirely from the index without touching the main table (row data). This is the fastest possible query path. EXPLAIN shows "Using index" in the Extra column.

EHR: CREATE INDEX idx_vitals_cover ON vitals(patient_id, recorded_at, systolic, diastolic) — the dashboard query "last 3 BP readings for patient X" never reads the table.
OS/DB: Index (route_id, status, seat_number) on bookings — "is seat 12A confirmed on route 99?" is answered from the index alone.
Index selectivity — when indexes help vs hurt
Selectivity = unique values / total rows. High selectivity (patient_id: thousands of unique values) → index is very useful. Low selectivity (status: 3 values for millions of rows) → the optimizer may skip the index and do a full scan (cheaper to scan than to random-access the index for 30% of rows).

EHR: An index on visits.status where 80% of rows have status='completed' has poor selectivity. A composite index (status, created_at) with a date range becomes useful again.
OS/DB: An index on bookings.class ('1st'/'2nd') is near-useless alone. An index on bookings.booking_reference (unique per booking) has perfect selectivity.
When NOT to index — write overhead
Every index must be maintained on INSERT, UPDATE, and DELETE. A table with 10 indexes pays the price of updating 10 B-trees per write. Over-indexing slows bulk loads, high-write OLTP tables, and replication lag. Remove indexes that are not used.

EHR: The CareMate vitals table receives ~50,000 inserts per day. Every unnecessary index on it adds write latency and replication load on the Galera cluster.
OS/DB: The ticket_scans table receives millions of inserts per day. Only index columns that are actually queried — scanned_at for time-range lookups, booking_id for booking verification.
EHR Example — Index Analysis with EXPLAIN▾
SQL 🩺 EHR

Common Pitfalls▾
⚠️ Indexing every column "just in case"

A common mistake is adding an index on every column of a table. Each index adds overhead to every INSERT, UPDATE, and DELETE. On a high-write table like ticket_scans (millions of inserts/day), this can increase write latency by 200-400%. Index only columns that appear in WHERE, JOIN ON, or ORDER BY clauses of actual production queries.

⚠️ Wrong column order in composite indexes

Index on (status, patient_id) does NOT efficiently serve WHERE patient_id = 42. The leftmost prefix rule means the first column must match the query's equality filter. Always put the column with the highest cardinality (most unique values) first unless the query pattern dictates otherwise. Test with EXPLAIN before deploying.

⚠️ Functions in WHERE clauses bypass indexes

WHERE YEAR(created_at) = 2024 cannot use an index on created_at because the function is applied to every row. Rewrite as WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31 23:59:59' — the range can use the B-tree index. This is one of the most common query optimization mistakes.

🎤 Interview Questions▾
How do you diagnose a slow query? Walk me through your process.

First, I run EXPLAIN (or EXPLAIN ANALYZE for MariaDB 10.9+) on the query. I look at the 'type' column: const/eq_ref/ref is good, range is acceptable, ALL means full table scan. Then I check 'rows' — how many rows the optimizer estimates it will examine. Then 'Extra' — 'Using filesort' and 'Using temporary' are warnings. If the type is ALL on a large table, I look at the WHERE clause to identify which columns need an index. I also check if functions applied to columns are preventing index use. Finally I verify the index is actually being used after creation with EXPLAIN, not just assumed.

🎯 Anchor

"EXPLAIN first, always. I look at type=ALL as the red flag, then rows, then Extra. I never add an index without verifying with EXPLAIN that it's actually used."

What is a covering index and when would you use one?

A covering index contains all the columns referenced by a query — both the WHERE filter columns and the SELECT result columns. When the optimizer uses a covering index, it never reads the main table: the data is served entirely from the index pages, which is faster because index pages are smaller and more cache-friendly. I use covering indexes for high-frequency queries that have stable column requirements — like the CareMate patient dashboard query that always requests the same 4 columns, or the booking scanner query at OS that runs 500 times per second. EXPLAIN shows "Using index" in Extra when a covering index is used.

🎯 Anchor

"A covering index includes SELECT columns + WHERE columns so the engine never reads the table — it's the fastest possible access path. EXPLAIN shows 'Using index' to confirm."

Why might the query optimizer ignore a perfectly valid index?

Several reasons: (1) Low selectivity — if 80% of rows match the WHERE condition, a full table scan reads pages sequentially which is faster than random-access index lookups for that proportion of rows; (2) Table statistics are stale — ANALYZE TABLE updates the statistics the optimizer uses to estimate costs; (3) A function is applied to the indexed column in WHERE, preventing index use; (4) The index columns don't match the leftmost prefix of the query's filter; (5) The table is very small and the optimizer correctly decides a full scan is cheaper. Running ANALYZE TABLE and checking EXPLAIN EXTENDED often reveals which case applies.

🎯 Anchor

"Low selectivity, stale statistics (fix with ANALYZE TABLE), functions on indexed columns, or wrong column order in the index. EXPLAIN EXTENDED shows which path the optimizer chose and why."

Schema Design Patterns

Beyond normalization, experienced designers use recurring patterns to solve common problems. Soft deletes, audit trails, and status state machines appear in virtually every production system — including CareMate and OS railway operations.

Click any section to expand
Why This Matters▾

Schema patterns are how you communicate design intent to other engineers. At OS GmbH, schemas are maintained by teams across shifts — the deleted_at pattern, explicit status enumerations, and audit columns make the schema self-documenting. For CareMate, DSGVO requires auditability by law: knowing who created or modified a medical record, and when, is not optional.

🎯 Interview Anchor

"Every table I create gets created_at and updated_at at minimum. For DSGVO-relevant data like patient records, I add created_by and updated_by. And I use soft deletes (deleted_at) instead of hard DELETE for any data with audit or compliance requirements."

Core Concepts▾
Soft deletes — the deleted_at pattern
Instead of issuing DELETE, set deleted_at = NOW(). Application logic always queries WHERE deleted_at IS NULL. Benefits: undo/restore capability, audit trail, forensic analysis. Cost: every query must filter deleted rows — easy to forget, causing "ghost data" bugs.

EHR: Patient records are never hard-deleted in CareMate — DSGVO right-to-erasure is handled through anonymization, not row deletion. The record structure remains for audit trail integrity.
OS/DB: Cancelled bookings set status='cancelled' (not deleted) so the booking history and revenue analytics remain intact.
Audit columns — created_at, updated_at, created_by
Standard audit columns on every table: created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP, and for compliance-sensitive tables: created_by BIGINT FK→users, updated_by BIGINT FK→users.

EHR: Every medical record table (patients, vitals, visits) carries created_by and updated_by referencing the staff member who made the change — mandatory for HL7 FHIR audit log export and DSGVO Article 30 records.
OS/DB: Route and schedule modifications carry a staff_id audit trail for liability documentation in case of operational incidents.
ENUM for state machines — explicit valid states
ENUM columns enforce a closed set of valid values at the database level. Combined with a clear naming convention, they document the state machine of your entities. Alternative: a separate status lookup table (more flexible for dynamic states, but joins are needed for constraints).

EHR: visits.status ENUM('scheduled','in_progress','completed','cancelled','no_show') — only these five states are valid. The ENUM prevents typos and application bugs from inserting invalid states.
OS/DB: bookings.status ENUM('pending','confirmed','checked_in','used','cancelled','refunded') — the booking lifecycle in one place.
Polymorphic associations (with caution)
A polymorphic association allows one table to relate to multiple other tables via a (record_type, record_id) pair instead of separate FK columns. Common in ORMs. Problems: no foreign key constraint possible (the DB can't enforce referential integrity), queries are complex, and the pattern often indicates a normalization opportunity.

EHR: An audit_log table with (entity_type VARCHAR, entity_id BIGINT) is a polymorphic pattern — acceptable for append-only logs where FK enforcement is not needed.
OS/DB: Avoid polymorphic associations in transactional tables — use separate notification_bookings and notification_routes tables instead.
Versioning / temporal tables
MariaDB 10.3+ supports system-versioned tables (temporal tables). The engine automatically maintains a history of all row changes. Queries can include FOR SYSTEM_TIME AS OF '2024-01-01' to see historical data without any application-level versioning.

EHR: Enabling system versioning on the patients table provides a DSGVO-compliant audit trail with zero application code changes — the DB tracks every patient record modification automatically.
OS/DB: System versioning on the routes table captures every schedule change for post-incident analysis and regulatory reporting.

Common Pitfalls▾
⚠️ Forgetting to filter deleted_at IS NULL

Soft-deleted rows are invisible to business logic only if every query includes WHERE deleted_at IS NULL. Forgetting this filter — especially in JOIN conditions — causes deleted data to re-appear in reports. Mitigate with database views that pre-apply the filter, or with an ORM-level default scope. Always test soft-delete queries explicitly.

⚠️ Using VARCHAR for status columns instead of ENUM

Storing status as VARCHAR(50) allows any string — including typos, mixed case, and invalid states. An ENUM enforces the valid state set at the database level, not just the application. If states need to change over time, prefer a lookup table with a FK; but for a stable state machine, ENUM is cleaner and more storage-efficient.

🎤 Interview Questions▾
How do you implement a DSGVO-compliant audit trail in MariaDB?

Several complementary approaches: (1) Application-level audit columns — created_at, created_by, updated_at, updated_by on every relevant table, populated by the application layer; (2) MariaDB system versioning (temporal tables) with WITH SYSTEM VERSIONING — the engine automatically maintains a complete history of every row change with timestamps, queryable via FOR SYSTEM_TIME; (3) An audit_log table recording change type, entity type/id, old values, new values, and acting user — populated by application triggers or framework. For CareMate, I'd combine all three: application audit columns for everyday queries, system versioning for point-in-time recovery and compliance investigation, and an audit_log for user-facing change history.

🎯 Anchor

"Application audit columns for everyday queries, MariaDB system versioning for automatic historical tracking, and audit_log for compliance investigation — I use all three for DSGVO-relevant medical data."

What are the trade-offs of soft deletes vs hard deletes?

Soft deletes preserve data for audit, undo, and analytics — essential for medical records and financial data. But they add complexity: every query must filter deleted rows (easy to forget), unique constraints must account for deleted rows (a re-registered patient email would conflict with the soft-deleted record — solve with a unique index on (email, deleted_at)), and the table grows indefinitely without archival. Hard deletes are simpler and keep tables smaller but lose data permanently. For OS GmbH's booking data and CareMate patient records, soft deletes are the correct default. For high-volume ephemeral data like session logs, hard deletion or partitioned archival is better.

🎯 Anchor

"Soft deletes for any data with audit, compliance, or analytics requirements. Hard deletes or archival partitioning for high-volume ephemeral data. The unique constraint interaction with deleted_at is a common implementation pitfall."

Design Fundamentals▾
You're given a 20-column flat table with no normalization. How do you approach redesigning it?

I start by identifying the primary entity (what is the main subject of a row?). Then I look for repeating values — columns where the same value appears in many rows, suggesting the data belongs in a lookup table. I check for columns that don't depend on the PK directly (transitive dependencies). I identify any multi-valued columns (comma-separated lists). Then I draw the entity relationships and design the normalized schema. Finally, I write a migration script that is reversible — with the new schema and a rollback plan. For a production table at OS GmbH, I'd also benchmark the migration and plan for a maintenance window.

🎯 Anchor

"Identify the main entity, spot repeating values and multi-valued columns, draw the ERD, normalize to 3NF, write a reversible migration with rollback."

What is the difference between a clustered and non-clustered index in InnoDB?

In InnoDB, the primary key IS the clustered index — the actual table data is stored in PK order in a B-tree. There is exactly one clustered index per table. Every secondary (non-clustered) index stores the indexed column values plus the PK value as the row pointer — not a physical address. A secondary index lookup therefore requires two B-tree traversals: one to find the PK from the secondary index, and one to find the row data from the clustered index. This is called a "double lookup" or "bookmark lookup". Covering indexes avoid this by including all needed columns in the secondary index.

🎯 Anchor

"In InnoDB, PK is the clustered index — data is physically ordered by PK. Secondary indexes store PK as pointer, requiring two lookups. Covering indexes eliminate the second lookup."

A developer adds 8 indexes to a table to speed up their queries. What concerns would you raise?

Three main concerns: (1) Write overhead — every index adds maintenance cost on INSERT, UPDATE, and DELETE. For high-write tables (ticket_scans at OS), this can significantly increase latency and replication lag in Galera cluster; (2) Memory usage — all active index pages should fit in the InnoDB buffer pool. 8 indexes on a large table may exceed available RAM, causing disk I/O on queries; (3) Optimizer confusion — with many indexes, the query optimizer's cost estimates become more complex and it can occasionally choose a suboptimal index. I'd review the actual query patterns, use the performance_schema to identify which indexes are actually used after 30 days, and drop unused ones.

🎯 Anchor

"Write overhead, buffer pool pressure, and optimizer confusion. I'd monitor with performance_schema.table_io_waits_summary_by_index_usage for 30 days and drop any indexes with count_read = 0."

How would you model the CareMate EHR schema to support HL7 FHIR export requirements?

FHIR (Fast Healthcare Interoperability Resources) maps very naturally to a well-normalized relational schema. The key requirements: (1) every resource needs a unique identifier — our BIGINT PKs satisfy this; (2) resources must be versionable — MariaDB system versioning or explicit version columns with versionId; (3) audit trail with author references — our created_by/updated_by FK columns satisfy this; (4) standard coding systems — ICD-10 codes in a separate lookup table with the FHIR CodeSystem URI stored alongside; (5) no hard deletes — soft deletes with deleted_at; (6) meta.lastUpdated maps directly to updated_at. The existing 3NF schema with system versioning and audit columns is already FHIR-aligned — the export layer maps the relational data to FHIR JSON bundles.

🎯 Anchor

"A well-normalized schema with BIGINT PKs, system versioning, audit columns, and standard code lookup tables maps directly to FHIR resource requirements — the export layer handles the JSON transformation."

When would you use a partial index and does MariaDB support them?

A partial index (also called filtered index) covers only a subset of rows matching a WHERE condition. Classic use cases: an index on bookings(status) WHERE status='confirmed' — confirmed bookings are queried constantly but cancelled/used bookings rarely need to be found by status. This makes the index much smaller and faster. MariaDB 10.5+ supports partial indexes in a limited form via invisible indexes and expression indexes, but true partial indexes (WHERE clause filtered) are a PostgreSQL strength. In MariaDB, I approximate partial indexes by creating a generated/virtual column that is NULL for unwanted rows and indexing that column — NULL values are typically excluded from index lookups.

🎯 Anchor

"Partial indexes filter index entries by a WHERE condition — great for indexing a minority of rows. MariaDB doesn't support them natively; I approximate with a nullable virtual column and index that."
