Lesson 01 · ~3 hours
Architecture & Certification-Based Replication

The wsrep protocol and write-set certification model — why Galera is fundamentally different from traditional async replication. This is the foundation every other lesson builds on.

Click any section to expand
⏪ Traditional async replication in 90 seconds▾
🚧 Prerequisite — Complete Phase 05b (Traditional Replication) First

This lesson assumes you can already describe the async replication pipeline from memory: binary log events, the IO thread, the relay log, the SQL thread, GTID tracking, and the conditions under which data loss occurs. These are covered in depth in Phase 05b (10 hours).

If you have not completed Phase 05b, stop here and do that first. The Galera concepts in this lesson — write-set certification, commit latency, SST/IST — are defined in contrast to the async model. Without that baseline the comparisons have no anchor.

Why this matters▾

Galera is not replication in the MySQL sense. It is a synchronous multi-master replication plugin that replaces the binlog-based async model entirely. Nietsch will probe whether you understand the certification model — why Galera can have multiple writable nodes simultaneously without split-brain, and what the cost of that synchrony is. Fraport's KRITIS gate systems cannot tolerate replication lag — this is exactly why Galera is used.

⚓ Interview anchor

Galera uses optimistic locking via write-set certification — transactions commit locally first, then the write-set is broadcast and certified against other nodes' concurrent transactions; there are no distributed locks during execution.

Core concepts▾
Write-Set — the unit of replication
A write-set is a binary package containing the transaction's SQL changes (row images) plus metadata about which rows were read (the "keys"). It is created at COMMIT time and broadcast to all cluster members simultaneously via the wsrep API.

🩺 CareMate: An INSERT into vitals generates a write-set containing the new row plus the PK key of the patient row it references. 🚂 OS/DB: A Fraport booking INSERT generates a write-set certified against concurrent seat_number updates on the same route.
Certification — conflict detection without locks
Every node independently certifies each incoming write-set against its own record of recent concurrent write-sets. If two write-sets modified the same row concurrently, certification fails and the later transaction is rolled back. This is optimistic concurrency: no global locks, conflicts detected at commit.

🚂 OS/DB: Two agents book the last available seat simultaneously — one write-set certifies, one is rejected with deadlock-equivalent error.
Synchronous multi-master — what "synchronous" actually means
Galera is synchronous at the write-set delivery level, not the apply level. COMMIT returns only after the write-set is certified and acknowledged by all nodes — but application to the storage engine may lag slightly (local apply queue). This is called virtually synchronous.

🩺 CareMate: If node1 accepts a vitals INSERT, all other nodes have certified it before the COMMIT returns to Flask — no reading stale data on failover.
wsrep API — the Galera plugin interface
The wsrep (Write-Set REPlication) API is the interface between MariaDB and the Galera library (galera.so). MariaDB calls wsrep hooks at key points: pre_commit, post_commit, apply. The library handles all group communication (GCS), certification, and state transfer. wsrep_provider in my.cnf points to the galera.so library path.
Group Communication System (GCS) — the underlying protocol
Galera uses its own GCS layer (based on Totem protocol) for reliable ordered multicast. GCS ensures all nodes receive write-sets in the same total order — this ordering is the foundation of deterministic certification. The GCS also handles membership changes (node join/leave) and quorum votes.

Deep dive — write-set lifecycle▾
Transaction begins on node1 → DML executes (reads keys + writes rows locally, no locks on other nodes) → COMMIT issued by application → wsrep pre-commit hook fires: 1. Write-set assembled: {keys read} + {row changes} 2. Write-set assigned seqno (sequence number) by GCS 3. Write-set broadcast to ALL nodes simultaneously → Each node CERTIFIES the write-set: Compare keys in write-set against keys of all concurrent write-sets since the last certified seqno this transaction depended on PASS → write-set queued for apply FAIL → node1 rolls back its transaction, app receives deadlock error → node1 applies its own write-set (commits to InnoDB) → node2, node3 apply write-set in background (apply queue) → COMMIT returns to application on node1
Key implication

The certification step is where Galera catches conflicts that would cause inconsistency. If two nodes write to the same row concurrently, only one will certify — the other rolls back. This is deterministic: all nodes reach the same certification decision independently because they all have the same write-set history.

Common pitfalls▾
Pitfall: Writing to multiple nodes simultaneously without coordination

Galera allows writes to any node, but certification conflicts increase when the same rows are modified on different nodes. Result: frequent deadlock-equivalent errors returned to the application. Detect: monitor wsrep_local_cert_failures (should be near zero). Prevent: use MaxScale ReadWriteSplit to route all writes to a single "primary" node by default; allow multi-master writes only for explicitly sharded workloads.

Pitfall: Large transactions — MyISAM tables or non-transactional DDL

MyISAM tables are not supported in Galera — they don't have row-level locking or transactions, so write-sets cannot be assembled reliably. Result: MyISAM writes on one node are not replicated. Detect: CHECK TABLE engine=MyISAM. Prevent: enforce wsrep_replicate_myisam=OFF (default) and migrate all tables to InnoDB before cluster setup.

Pitfall: wsrep_on=OFF disables replication silently

Setting wsrep_on=OFF on a session or globally tells Galera to skip write-set certification for that session's transactions. The transaction commits locally but is NOT replicated. Result: silent data divergence between nodes — the cluster will not detect this until an SST is triggered. Prevent: never set wsrep_on=OFF in production; this is only for disaster recovery procedures when you know exactly what you're doing.

🎤 Interview Questions▾
What is a write-set in Galera and how does certification work?

A write-set is a binary package created at COMMIT time containing the row-level changes of the transaction plus the primary keys of rows that were read (the "key set"). It is broadcast to all cluster members via the GCS. Each node independently certifies the write-set by checking whether any of its keys conflict with keys from concurrent write-sets that have been applied since this transaction's snapshot. If no conflict: the write-set is queued for apply and the originating node's COMMIT returns. If conflict: the later transaction is rolled back and the application receives a deadlock-equivalent error. Certification is deterministic — every node makes the same decision from the same write-set history.

⚓ Anchor

Write-sets contain row changes + key set; certification is per-node, deterministic conflict detection at commit time — no global locks during execution.

How is Galera replication different from MariaDB async replication?

Traditional async replication: the primary commits locally, writes to binlog, replica reads binlog and applies events — there is a lag between primary commit and replica apply, and the replica may serve stale reads. Galera: the write-set must be certified by all nodes before COMMIT returns to the application. There is no lag at the certification level (though apply queues may have brief delays). Galera is multi-master — any node can accept writes simultaneously. Async replication is single-master (or semi-sync with one standby). The tradeoff: Galera adds network latency to every COMMIT (the certification round-trip); async replication commits locally without network involvement.

⚓ Anchor

Galera: synchronous certification before COMMIT, multi-master, no replication lag — but adds network RTT to every write. Async: zero write overhead, but stale reads and data loss on failover.

Can Galera have two "primary" nodes simultaneously? What prevents split-brain?

Yes — Galera has no designated primary; all Synced nodes accept writes simultaneously. Split-brain prevention comes from the quorum mechanism: a group of nodes only forms a Primary Component (and continues accepting writes) if it has a strict majority of the configured cluster nodes. With 3 nodes, quorum requires 2. If a network partition splits the cluster into a 1-node partition and a 2-node partition, only the 2-node side continues as Primary Component — the 1-node side transitions to a non-primary state and rejects writes. This is enforced by the GCS layer automatically.

⚓ Anchor

Split-brain is prevented by quorum: only a partition with strict majority (≥N/2+1) continues as Primary Component and accepts writes.

On a 3-node Galera cluster, node1 and node2 simultaneously receive an UPDATE to bookings WHERE id=500 (setting different seat numbers). Walk through the certification process and explain which transaction wins and how.

Show answer ↓
Answer

Both transactions execute locally without locks. At COMMIT time, both generate a write-set with key=bookings:500. The GCS assigns seqnos — say node1's write-set gets seqno 101, node2's gets seqno 102. Both write-sets are broadcast to all nodes. Certification: node1 certifies seqno 101 first (no conflict yet), applies it. Seqno 102 arrives — certification checks: does seqno 102's key (bookings:500) conflict with any write-set between 102's snapshot and its seqno? Yes — seqno 101 also touched bookings:500. Seqno 102 fails certification. node2 rolls back its transaction and returns a deadlock error (ER_LOCK_DEADLOCK) to the application. node1's write wins because it got the lower seqno. The decision is identical on all three nodes.

🎤 Interview Questions▾
A Galera node shows wsrep_local_state_comment = Joining. MaxScale has already removed it from the pool. The other two nodes are Synced. Is the cluster at risk? What do you do?

The cluster is not at immediate risk — the two Synced nodes form a quorum (2 of 3) and continue processing writes normally. The Joining node is receiving a state transfer (SST or IST) and will rejoin automatically when complete. Action: confirm the donor node (state=2, Donor/Desynced) is not experiencing excessive load — SST via mariabackup is CPU/IO intensive. Monitor wsrep_local_recv_queue on the joining node; it will drop to 0 when IST completes. Do not restart the joining node — that would abort the transfer and restart from scratch. Alert if the join takes more than 30 minutes.

⚓ Anchor

Joining = state transfer in progress, not a failure. 2 Synced nodes = quorum intact. Action is monitor and wait, not restart.

wsrep_flow_control_paused = 0.35 on node2 at 09:00. What does this mean and what do you do?

wsrep_flow_control_paused is the fraction of the last monitoring interval (default 1 second) during which the cluster was paused waiting for node2 to catch up. 0.35 means the entire cluster was effectively 35% slower than it would be without node2's slowness. This is a significant degradation — at Fraport's peak check-in window, 35% slower means gate assignment queries are backing up. Investigation: check node2's disk I/O (iostat), CPU, and the slow query log. The cause is usually a slow disk on the apply thread, a large transaction being applied, or a hardware issue. Short-term fix: drain node2 from MaxScale to stop it receiving new write-sets while you diagnose.

⚓ Anchor

flow_control_paused=0.35 means the cluster lost 35% of throughput waiting for one node — drain it from the pool and diagnose its disk/CPU before the 09:00 peak hour hits.

How do you check replication lag between Galera nodes without using Seconds_Behind_Master?

Seconds_Behind_Master is an async replication concept — it has no meaning in Galera. To check lag, compare wsrep_last_committed across all nodes: SELECT @@hostname, VARIABLE_VALUE FROM information_schema.GLOBAL_STATUS WHERE VARIABLE_NAME='wsrep_last_committed'. If all nodes show the same seqno (or within 1-2), they are in sync. A node with a significantly lower seqno is behind — check its wsrep_local_recv_queue to see how many write-sets are queued waiting to apply. For a more automated check, the Prometheus wsrep exporter exposes wsrep_last_committed as a gauge per node.

⚓ Anchor

Galera lag = seqno delta between nodes via wsrep_last_committed — Seconds_Behind_Master is an async-only metric that means nothing in Galera.

Lesson 03 · ~3 hours
SST vs IST — State Transfers

State Snapshot Transfer (SST) and Incremental State Transfer (IST) are how nodes rejoin a cluster. Understanding when each triggers and how to control them is essential DBA knowledge — an SST at the wrong moment can take a donor node offline during peak hours.

Click any section to expand
Why this matters▾
⚓ Interview anchor

IST is preferred — it transfers only the missing write-sets from gcache. SST transfers the entire data directory — it's a full backup/restore operation that makes the donor a Donor/Desynced node (state 2) for the duration. Always size gcache large enough to cover your expected node-down window to favour IST over SST.

Core concepts — SST vs IST decision▾
IST (Incremental State Transfer) — the fast path
IST sends only the write-sets the joining node missed, directly from gcache. It is triggered when: (a) the joiner's grastate.dat has a valid seqno, AND (b) the donor's gcache still contains all write-sets since that seqno. IST is fast (seconds to minutes) and does NOT desynch the donor. 🩺 CareMate: A node restarted after a 10-minute maintenance window — gcache contains the missing 600 write-sets, IST completes in 30 seconds.
SST (State Snapshot Transfer) — the full resync
SST is triggered when IST is not possible: the joiner's grastate.dat is invalid (seqno=-1), or the donor's gcache doesn't go back far enough. SST transfers the entire data directory — effectively a live backup/restore. Duration: proportional to database size. During SST, the donor enters state=2 (Donor/Desynced) and may have degraded write performance. 🚂 OS/DB: At Fraport with a 500GB database, SST via mariabackup takes 45-90 minutes.
gcache — the write-set ring buffer
gcache (Galera cache) is an on-disk ring buffer that stores recent write-sets. Its size (wsrep_provider_options="gcache.size=2G") determines how far back IST can go. If gcache runs out of space, old write-sets are evicted — if the joiner needs those evicted write-sets, SST is triggered instead. Size gcache to cover your longest expected node-down window × peak write rate. A 2GB gcache covering 4 hours at 128MB/h peak gives you IST for anything under 4 hours of downtime.
mariabackup SST method — the correct choice
wsrep_sst_method=mariabackup is the recommended SST method. It uses MariaDB Backup (a fork of Percona XtraBackup) to take a hot backup without locking the donor. The alternative, mysqldump, is far slower and locks tables. The donor runs mariabackup --backup during SST; the joiner receives the stream and restores it. Both donor and joiner must have mariabackup installed.

Common pitfalls▾
Pitfall: gcache too small → constant SSTs

If gcache evicts write-sets faster than nodes can be restarted, every node restart triggers a full SST instead of a fast IST. Detect: Check mariadb error log — "IST sender limit reached" or "State transfer required" with a full SST. Fix: Increase gcache.size in wsrep_provider_options. Calculate: peak_write_rate_bytes/hour × expected_maintenance_window_hours × 1.5 safety margin.

Pitfall: SST during peak hours takes the donor offline

When SST is triggered during Fraport's 06:00-09:00 check-in window, the donor enters Donor/Desynced state — writes may be throttled or paused depending on wsrep_desync configuration. Prevent: Schedule node maintenance in the 23:00-05:00 low-traffic window. Set wsrep_sst_donor to a specific node to prevent the primary writer from being chosen as donor: SET GLOBAL wsrep_sst_donor='fra-db3'.

Pitfall: Wrong SST user permissions

Fix: GRANT the correct privileges and re-attempt join.

🎤 Interview Questions▾
When does Galera choose IST over SST? What can prevent IST from being used?

IST is used when two conditions are met: (1) the joining node has a valid seqno in grastate.dat (not -1), meaning it knows where it last left off; and (2) the donor's gcache contains all write-sets from that seqno to the current seqno. If either condition fails — seqno is -1 (dirty crash), or gcache was too small and evicted the needed write-sets — SST is triggered instead. A node that crashes without a clean shutdown will have seqno=-1 written to grastate.dat by Galera on restart. The implication: always stop MariaDB cleanly (mysqladmin shutdown or systemctl stop mariadb) to preserve the seqno for IST eligibility.

⚓ Anchor

IST requires valid seqno in grastate.dat AND gcache coverage. Clean shutdown preserves seqno; crash forces SST. Size gcache for your longest maintenance window.

Why is wsrep_sst_method=mysqldump dangerous in production?

mysqldump SST acquires a global read lock (FLUSH TABLES WITH READ LOCK) on the donor for the entire duration of the dump — which can be hours on large databases. This blocks all writes to the donor node during SST, effectively taking it out of write service. With mariabackup, the hot backup runs without global locks — only a brief lock at the end for consistent position. mysqldump is acceptable only for tiny databases (<1GB) in test environments. Any Galera cluster used for KRITIS workloads must use mariabackup.

⚓ Anchor

mysqldump SST holds FTWRL on the donor for the entire transfer duration — on a 500GB Fraport database that's hours of write-blocked donor. mariabackup hot-copies without locking.

How do you calculate the correct gcache.size for a production cluster?

Formula: peak_write_throughput_bytes_per_second × maximum_expected_downtime_seconds × safety_factor (1.5–2x). Example: OS cluster doing 50MB/s writes at peak, and the SLA allows 4-hour maintenance windows: 50MB/s × 14400s × 2 = 1,440,000MB ≈ 1.4TB. That's impractical, so you either reduce the maintenance window or accept that long outages will SST. In practice, most OS clusters use 2-4GB gcache (covering 1-3 hours at typical write rates) and schedule all maintenance for low-traffic windows (<4h). Monitor gcache utilisation with wsrep_provider_options status output.

⚓ Anchor

gcache.size = peak_write_rate × max_downtime × 2. Under-sized gcache = every restart triggers SST. Monitor gcache ring-buffer utilisation to validate sizing.

Quorum, Split-Brain & Recovery

This is the hardest lesson — and the most critical for the interview. grastate.dat, safe_to_bootstrap, and the quorum recovery procedure are 3am knowledge that separates a DBA from a developer with database access.

Click any section to expand
Why this matters▾

When all three nodes are down simultaneously — power failure, datacenter outage — the cluster cannot restart automatically. Someone must manually determine which node has the most recent data, set safe_to_bootstrap=1 on that node, and bootstrap from it. Doing this wrong means bootstrapping from a stale node and losing committed transactions. Doing it right recovers the cluster in minutes.

⚓ Interview anchor

Never bootstrap from a node with safe_to_bootstrap=0. Find the node with the highest seqno in grastate.dat — that node last committed data. Set safe_to_bootstrap=1 ONLY on that node, bootstrap it, then start the others normally.

Core concepts▾
Quorum — why 3 nodes, not 2
Quorum requires a strict majority: ⌊N/2⌋+1. With 3 nodes, quorum=2. A 2-node cluster has quorum=2 — if one node goes down, the remaining node has exactly 1 of 2 = no majority → the cluster goes non-primary and rejects writes entirely. Three nodes allow one failure while maintaining quorum. For cross-datacenter setups, 3 nodes in 3 DCs provides both HA and geographic redundancy.

🚂 OS/DB: Fraport runs node1+node2 in Frankfurt DC1, node3 in Frankfurt DC2 as an arbiter-like setup. Any single DC failure leaves 2 nodes with quorum.
grastate.dat — the cluster position file
Written by Galera on clean shutdown. Contains the cluster UUID, the node's last committed seqno, and safe_to_bootstrap flag. The seqno identifies exactly where in the write-set history this node last was. On dirty crash, seqno=-1 is written (or the file may be stale). On cluster-wide power failure, all nodes will have their last clean seqno in this file — the highest seqno wins.
safe_to_bootstrap — the safety interlock
Galera sets safe_to_bootstrap=1 only on the last node to leave the cluster (the node that processed the most recent write-sets). This prevents accidentally bootstrapping from a stale node. If you manually set safe_to_bootstrap=1 on the wrong node and bootstrap, you create a new cluster with older data — all subsequent joins will overwrite their data with this stale snapshot.
pc.weight — arbiter and split-brain prevention
wsrep_provider_options="pc.weight=2" assigns a node a weight of 2 votes in quorum calculations. Useful in a 2+1 datacenter setup where DC1 has 2 nodes (each weight=1, total=2) and DC2 has 1 node (weight=2). If the WAN link drops, DC2 has 2 votes — it can maintain quorum alone and continue writes. The DC1 side (2 nodes, 2 votes = 50%) goes non-primary and stops writes.

Pitfall: Bootstrapping from the wrong node

Flow Control — Diagnosing a Slow Node

Flow control is Galera's backpressure mechanism. When one node falls behind applying write-sets, it sends FC_PAUSE messages that slow down the entire cluster. Understanding the cause and fix is essential for any OS production DBA role.

Click any section to expand
Why this matters▾
⚓ Interview anchor

Flow control affects the entire cluster — one slow node degrades write throughput for all. wsrep_flow_control_paused near 1.0 means the cluster is nearly halted. Diagnosis: find why the slow node can't keep up (disk, CPU, large transaction, misconfigured wsrep_slave_threads). Immediate mitigation: drain the slow node from MaxScale to stop new write-sets arriving while you fix the root cause.

Core concepts▾
Flow control mechanism
When a node's apply queue (wsrep_local_recv_queue) exceeds gcs.fc_limit (default: 16 write-sets), it sends FC_PAUSE messages to the GCS. All nodes pause sending new write-sets until the queue drains below gcs.fc_factor (default: 1). This creates a cluster-wide pause — every committing transaction waits. 💉 DiaTwin: A burst of 10,000 glucose sensor readings arriving simultaneously can fill the recv_queue on a slow cloud node, triggering FC and delaying confirmations to all devices.
wsrep_flow_control_paused — the key metric
The fraction of the last second the cluster spent paused due to flow control from this node. Values: 0.0 = no FC (healthy), 0.01-0.09 = mild (monitor), 0.10-0.50 = significant (diagnose), >0.50 = critical (immediate action), 1.0 = cluster completely halted. This metric is per-node — the node generating the highest value is the bottleneck.
wsrep_slave_threads — parallelising apply
Galera can apply write-sets in parallel using multiple apply threads. wsrep_slave_threads=8 (recommended: nCPU/2) allows 8 write-sets to apply simultaneously on the joiner/replica node. The default of 1 is a common bottleneck on write-heavy clusters. Increasing this reduces the recv_queue and therefore reduces flow control events. Requires MariaDB 10.4+ for optimal parallel apply.

What three wsrep metrics would you page on-call for immediately, and what are the thresholds?

1. wsrep_cluster_size < 3 — any node absence reduces redundancy and quorum margin. Alert for=1m to avoid false positives from transient restarts. 2. wsrep_cluster_status ≠ Primary (value=0 in Prometheus) — quorum lost, writes rejected by this node. Alert for=0m (immediate). 3. wsrep_flow_control_paused > 0.10 for >60s — sustained FC pausing is causing visible cluster degradation, not just a brief spike. These three cover the critical failure modes: membership loss, quorum loss, and throughput degradation.

⚓ Anchor

Page on: cluster_size < 3 (node loss), cluster_status=Non-primary (quorum lost), flow_control_paused > 0.10 for 60s (throughput degraded >10%).

How do you expose wsrep metrics to Prometheus at OS?

Use the Prometheus mysqld_exporter with the --collect.global_status flag enabled (it's on by default). Configure it as a scrape target for each Galera node: host:9104. The exporter reads SHOW GLOBAL STATUS and exposes all wsrep_ variables with the prefix mysql_global_status_. For wsrep_cluster_status (a string value Primary/Non-primary), the exporter converts it to 1/0 numerically. The exporter also needs the wsrep_* variables enabled in the MariaDB user granted to the exporter (PROCESS and REPLICATION CLIENT privileges minimum).

⚓ Anchor

mysqld_exporter scrapes SHOW GLOBAL STATUS → exposes mysql_global_status_wsrep_* metrics. wsrep_cluster_status is converted to 1=Primary/0=Non-primary numerically.

🔴 Scenario A — all 3 nodes down, power failure recovery▾
🔴 Scenario A — Complete cluster down after Fraport UPS failure

Trigger: All three Fraport DB nodes unreachable. Power restored. It's 03:17, gate systems open at 05:30.

  1. SSH to all three nodes. Confirm MariaDB is stopped. Do not start anything yet.
  2. Read grastate.dat seqno on all three: for n in db1 db2 db3; do ssh $n "grep seqno /var/lib/mysql/grastate.dat"; done
  3. Identify the bootstrap candidate: highest seqno = most recent data = bootstrap here.
  4. Check safe_to_bootstrap: If the highest-seqno node has safe_to_bootstrap=1 → proceed. If all show 0 (simultaneous crash) → manually set it on the highest-seqno node: sed -i 's/safe_to_bootstrap: 0/safe_to_bootstrap: 1/' /var/lib/mysql/grastate.dat
  5. Bootstrap: galera_new_cluster on the chosen node. Wait for Primary status: SHOW STATUS LIKE 'wsrep_cluster_status'
  6. Start the other two: systemctl start mariadb one at a time. They will IST (if gcache covers the downtime) or SST.
  7. Confirm recovery: wsrep_cluster_size=3 on all nodes. Notify MaxScale and application teams.
  8. Expected time: 5-15 minutes from power-up to full cluster. SST adds 45-90 min if gcache insufficient.
🔴 Scenario B — wsrep_flow_control_paused = 0.87▾
🔴 Scenario B — Cluster nearly halted, 13 minutes to VW shift change

Trigger: 08:47. Prometheus fires GaleraFlowControlHigh on fra-db3 (FC=0.87). Cluster throughput dropped 87%. VW shift change login storm starts at 09:00.

  1. Drain fra-db3 immediately: maxctrl set server fra-db3 drain. This stops fra-db3 receiving new write-sets — removes it from the FC feedback loop. Cluster throughput should recover within seconds.
  2. Confirm recovery on fra-db1/db2: SHOW STATUS LIKE 'wsrep_flow_control_paused' should drop toward 0.
  3. Diagnose fra-db3 — disk first: iostat -x 1 5 — %util near 100% confirms disk saturation as the cause.
  4. Check wsrep_slave_threads on fra-db3: SHOW VARIABLES LIKE 'wsrep_slave_threads'. If low (1-2), increase: SET GLOBAL wsrep_slave_threads=12
  5. Check for large transaction: SELECT trx_id, trx_rows_modified FROM information_schema.INNODB_TRX\G
  6. Watch recv_queue drain: watch -n2 "mysql -e \"SHOW STATUS LIKE 'wsrep_local_recv_queue'\""
  7. Re-add when queue = 0 sustained: maxctrl clear server fra-db3 drain
  8. Post-incident: Add wsrep_slave_threads=12 to my.cnf. Investigate VW login storm — stagger logins or add connection pooling.
🔴 Scenario C — network split, 2 nodes Frankfurt / 1 node Berlin▾
🔴 Scenario C — WAN link drops: Frankfurt ↔ Berlin partition

Trigger: 03:45. WAN link between Fraport DC (fra-db1, fra-db2) and Berlin DC (fra-db3) drops. All three nodes were running and Synced.

What each side sees:

  • Frankfurt (fra-db1 + fra-db2, 2 nodes): Has 2 of 3 votes = quorum. wsrep_cluster_status remains Primary. wsrep_cluster_size drops to 2 (fra-db3 lost). Continues accepting reads and writes normally. fra-db3 transitions to wsrep_local_state=0 (disconnected) on the Frankfurt view.
  • Berlin (fra-db3, 1 node): Has 1 of 3 votes = no quorum. wsrep_cluster_status flips to Non-primary. wsrep_ready flips to OFF. fra-db3 rejects all write queries and notifies MaxScale via wsrep_ready=OFF.
  • MaxScale (located in Frankfurt): GaleraMonitor polls fra-db3 — sees wsrep_ready=OFF. Removes fra-db3 from the write pool within 1 poll interval. Continues routing all writes to fra-db1 and fra-db2.

DBA actions during split:

  1. Confirm the split is network, not node failure: SSH to fra-db3 via management network (separate from DB network) succeeds → MariaDB is running, network is partitioned.
  2. Do NOT try to bootstrap fra-db3 — it would create a separate cluster with old data.
  3. GaleraNodeDown alert fires (cluster_size=2). Notify app teams that fra-db3 is temporarily unavailable but cluster is operational with 2 nodes.
  4. Escalate to network team to restore the WAN link.

When WAN link restores:

  1. fra-db3 detects the Primary Component on the Frankfurt side and initiates IST to catch up on the write-sets it missed during the partition.
  2. IST completes, fra-db3 returns to state=Synced. MaxScale re-adds it to the pool.
  3. Cluster returns to wsrep_cluster_size=3. No data loss — Frankfurt side was authoritative throughout.
🎤 Synthesis Interview Questions▾
You're setting up a new Galera cluster for a KRITIS client. Walk through the key my.cnf decisions you'd make and why.

Key decisions: (1) wsrep_sst_method=mariabackup — never mysqldump for production. (2) gcache.size: peak_write_rate × maintenance_window × 2. For a 100GB/day write workload with 4h maintenance windows: 100GB/24h × 4h × 2 = 33GB gcache. (3) wsrep_slave_threads=nCPU/2 — parallelise apply from the start. (4) innodb_autoinc_lock_mode=2 — required for Galera multi-master AUTO_INCREMENT. (5) binlog_format=ROW — required. (6) wsrep_cluster_address listing all 3 nodes — use internal hostnames not IPs for easier node replacement. (7) Dedicated SST user with minimal required privileges. (8) wsrep_node_name matching server hostname for easy MaxScale correlation in logs.

⚓ Anchor

mariabackup SST, right-sized gcache, nCPU/2 slave threads, innodb_autoinc_lock_mode=2 — these four decisions determine whether a Galera cluster runs smoothly or constantly fights SSTs and flow control.

A developer says "Galera is eventually consistent." How do you respond?

That's incorrect — Galera is virtually synchronous. Writes are not acknowledged to the client until the write-set has been certified by all nodes, meaning at COMMIT time all nodes have agreed on the transaction. There is no read-your-writes lag from the writing client's perspective. What is eventually consistent is the apply queue on non-originating nodes — a node that has certified a write-set may not have applied it to InnoDB yet, so a read on another node could theoretically return stale data in the brief apply window. This is called "causal reads" and can be addressed with wsrep_sync_wait if needed. But Galera is definitively not eventually consistent in the NoSQL/Dynamo sense.

⚓ Anchor

Galera is virtually synchronous — COMMIT waits for certification on all nodes. Apply queue lag is milliseconds, not seconds. "Eventually consistent" is wrong terminology for Galera.

What are the limitations of Galera that an experienced DBA should know before deploying it?

Key limitations: (1) Write scalability doesn't scale linearly — every write is replicated to all nodes, so more nodes = more certification overhead, not more write capacity. (2) Large transactions are dangerous — they generate huge write-sets that cause long certification pauses across the cluster. Transactions must be kept small. (3) DDL (ALTER TABLE) is replicated with Total Order Isolation (TOI) — a schema change blocks all writes on all nodes during execution. Use online schema change tools (pt-online-schema-change, gh-ost) for large table alterations. (4) No MyISAM support. (5) LOAD DATA INFILE requires wsrep_load_data_splitting=ON to avoid one giant transaction. (6) Explicit table locks (LOCK TABLES) are not replicated. (7) AUTO_INCREMENT requires innodb_autoinc_lock_mode=2 — values may have gaps.

⚓ Anchor

Galera limitations: write throughput doesn't scale with nodes, large transactions dangerous, DDL uses TOI (blocking), no MyISAM, AUTO_INCREMENT gaps. Know these before deploying.

How does Galera handle DDL — ALTER TABLE — and what is the risk?

By default, Galera uses Total Order Isolation (TOI) for DDL: the ALTER TABLE is replicated to all nodes and executed in the same total order as all other write-sets. During TOI, all concurrent write-sets on all nodes are blocked until the DDL completes. On a small table this is milliseconds. On a 500M-row bookings table this is minutes to hours — a cluster-wide outage. The safer alternative is wsrep_OSU_method=RSU (Rolling Schema Upgrade): the DDL runs locally on one node at a time without blocking the cluster. RSU requires the old and new schema to be compatible for the duration of the upgrade. For KRITIS tables, always use pt-online-schema-change or gh-ost in combination with RSU to achieve zero-downtime schema changes.

⚓ Anchor

TOI = DDL blocks entire cluster (safe but blocking). RSU = rolling per-node DDL (non-blocking but requires schema compatibility). Use pt-osc/gh-ost for large production tables.
