Architecture & Installation

⚡ Why This Matters▾

MaxScale is the critical middleware layer at OS GmbH — every application query passes through it before reaching the Galera cluster. Nietsch will expect you to draw the four-component architecture from memory and explain why each component is distinct. For Morafek CareMate EHR, MaxScale enables zero-downtime node maintenance by transparently re-routing queries when a node enters maintenance mode.

⚓ Interview Anchor

"MaxScale has four logical components: Servers (the MariaDB backends), a Monitor (polls health via SQL), a Router Service (contains routing logic + server references), and a Listener (the TCP port clients connect to). Traffic flows: Client → Listener → Router Service → Monitor-aware server pool → Backend. Removing any layer breaks the chain."

🧠 Core Concepts▾
Server — a backend MariaDB node
A [server] stanza defines one backend DB host: address, port, and protocol. MaxScale sends health-check queries to it via the monitor. Server objects are stateless — they carry no routing logic, only connection parameters.

EHR: Three server objects represent the three Galera nodes running the CareMate database — db1, db2, db3 in the Frankfurt datacenter.
OS/DB: OS GmbH runs five Galera nodes across two availability zones; each is a separate server object so MaxScale can individually drain or quarantine AZ-2 nodes during maintenance.
Monitor — health intelligence
The monitor (galeramon for Galera, mariadbmon for primary/replica) periodically queries every server and assigns state flags: Master, Slave, Synced, Down, Maintenance. The router reads these flags to decide routing.

EHR: galeramon detects when a CareMate node's wsrep_local_state drops to 2 (Donor/Desynced) and immediately removes it from the read pool.
OS/DB: Monitor interval is set to 1000ms (1s) on OS's cluster — a 2s detection window for node failure is acceptable for 99.999% SLA; longer would increase client error exposure.
Router Service — the routing brain
A [service] stanza references a router module (readwritesplit, readconnroute, schemarouter) and a list of servers. It holds connection credentials, session limits, retry policies, and filter chains. Multiple services can share the same server pool with different routing policies.

EHR: Two services — caremate-rw (readwritesplit for Flask app) and caremate-ro (readconnroute for reporting queries).
OS/DB: OS uses a separate ticketing-rw service for real-time bookings and a analytics-ro service pointing only to replica-tier nodes — protecting OLTP from batch reporting.
Listener — the client-facing TCP port
A [listener] stanza binds a TCP address:port and links to a service. Clients connect here using the standard MariaDB protocol. You can run multiple listeners on different ports linking to different services — a common pattern for separating OLTP, read-only, and admin traffic.

EHR: Listener on port 3306 for the Flask API (readwritesplit) and port 3307 for Metabase reporting (readconnroute, max_connections=20).
OS/DB: Port 3306 for booking engine, 3307 for Prometheus exporter queries, 3308 for DBA admin access with localhost bind only.
Filter — query inspection middleware
Filters are optional middleware inserted into the query pipeline within a service. Common filters: qlafilter (query log), regexfilter (rewrite queries), throttlefilter (rate limit), cachefilter (result cache). They are chained in order and can modify, log, or reject queries.

EHR: qlafilter logs all queries touching the patients table to a DSGVO-compliant audit file, satisfying Article 30 GDPR record-of-processing requirements.
OS/DB: throttlefilter limits the analytics service to 500 queries/sec, preventing a runaway reporting job from saturating the cluster during peak operations.

⚠️ Common Pitfalls▾
⚠️ Wrong User Privileges

Using a monitor user without REPLICATION CLIENT means MaxScale cannot run SHOW SLAVE STATUS and will mark all servers as Running without any Galera state — it will happily route to a Donor node performing SST. Always verify monitor user privileges immediately after install.

⚠️ Firewall Blocks Monitor Queries

MaxScale connects to each backend on port 3306. If the server firewall or iptables blocks the MaxScale host, the monitor marks all backends as Down and the proxy goes into a state where it refuses all new connections. Check with telnet db1 3306 from the MaxScale host before going live.

⚠️ Binding to 0.0.0.0 on the Listener

A listener bound to 0.0.0.0 exposes MaxScale to all network interfaces, including external ones. In OS GmbH's network topology, MaxScale should bind only to the internal app-tier VLAN IP. Use address=10.0.2.10 not address=0.0.0.0 in the listener stanza.

🎤 Interview Questions▾
What are the four components of MaxScale and how do they relate to each other?

MaxScale has Servers (backend MariaDB nodes with address/port), a Monitor (polls servers and assigns state flags like Synced/Down/Maintenance), a Router Service (applies routing logic using server state flags, holds credentials and filter chains), and a Listener (the TCP endpoint clients connect to, linked to one service). Traffic flows strictly: Listener receives TCP connection → Service router makes routing decision → selected backend Server. The monitor runs independently in a background thread and updates state flags that the router reads per-connection.

⚓ Anchor Phrase

"The monitor and router are decoupled — monitor updates state asynchronously every N milliseconds; the router reads those state flags on every new connection. This is why MaxScale reacts to node failures within one monitor interval."

Why does MaxScale require two separate database users, and what minimum privileges does each need?

The monitor user needs only REPLICATION CLIENT, REPLICATION SLAVE, SHOW DATABASES, and BINLOG MONITOR — it never touches application data, only runs SHOW STATUS queries. The service/router user needs DML privileges on application schemas. Keeping them separate means a leaked router credential cannot be used to query replication topology, and a failed monitor doesn't affect session credentials.

⚓ Anchor Phrase

"Least privilege and blast-radius isolation — monitor user touches zero application data, router user has zero replication visibility."

How would you verify a new MaxScale installation is routing correctly without touching the application?

First run maxctrl list servers to confirm all backends show Master, Synced, Running or Slave, Synced, Running. Then connect directly via mysql -h 127.0.0.1 -P 3306 and run SELECT @@hostname multiple times — writes should always return the master node name, SELECTs should round-robin across slaves. Check maxctrl list services for connection counts and journalctl -u maxscale for any routing errors.

⚓ Anchor Phrase

"maxctrl list servers is always my first command — if states are wrong, no amount of routing config matters."

⚡ Why This Matters▾

Nietsch will almost certainly hand you a broken or incomplete maxscale.cnf and ask you to identify the problem. Every parameter in this file has a reason; being able to explain why a parameter exists separates a competent admin from someone who just copy-pasted a config. For OS GmbH's 24/7 operations, a misconfigured master_failure_mode or wrong monitor_interval can mean the difference between a 2-second blip and a 30-second outage.

⚓ Interview Anchor

"In maxscale.cnf I focus on three areas: the monitor interval (how fast we detect failures), master_failure_mode (what to do with in-flight writes when master goes down), and slave_selection_criteria (how we choose which replica to use). Everything else is important, but those three determine your availability profile."

🧠 Core Concepts — Key Parameters▾
monitor_interval — detection latency
monitor_interval=1000ms means MaxScale polls each server every second. A node going down is detected within 1–2 polling cycles (1–2 seconds). Lower values mean faster detection but more DB connections consumed by health checks. For OS GmbH's 99.999% SLA, 500ms–1000ms is typical. Never set below 200ms in production — the health check queries themselves add load.

EHR: CareMate uses 2000ms — acceptable for an EHR where a 2s routing delay during failover won't cause patient harm.
OS/DB: 500ms on ticket-processing cluster — a train departure system cannot tolerate long routing errors during peak boarding.
master_failure_mode — in-flight write handling
Controls what happens to active write sessions when the master node disappears. fail_instantly — return error to client immediately (safe, predictable). error_on_write — allow reads to continue but fail the next write. fail_on_write — close session on next write attempt. For Galera, fail_instantly is safest because the Galera cluster will elect a new master quickly.

EHR: error_on_write — allows a doctor mid-session to finish reading patient history even if the master just failed, but blocks the next SAVE action.
OS/DB: fail_instantly — booking transactions are short-lived; it's better to retry cleanly than to allow partial session state.
slave_selection_criteria — replica load balancing
Determines which slave receives a read query. Options: LEAST_CURRENT_OPERATIONS (fewest active queries — best for heterogeneous hardware), LEAST_BEHIND_MASTER (lowest replication lag — best for read-after-write consistency), LEAST_ROUTER_CONNECTIONS (fewest MaxScale-managed connections), ADAPTIVE_ROUTING (response-time-aware).

EHR: LEAST_CURRENT_OPERATIONS — CareMate's two slave nodes have identical hardware; distribute by current query load.
OS/DB: LEAST_BEHIND_MASTER — for ticket scan lookups, reading stale data (e.g. a scan not yet replicated) could cause double-boarding. Prefer the freshest replica.
max_slave_connections — connection pool sizing
Limits how many backend connections MaxScale opens to each slave. Default is 255. Setting this too low causes connection queuing; too high exhausts max_connections on the backend. Rule of thumb: MaxScale backend connections per server = (expected concurrent sessions × 1.5) / number of slaves.

EHR: CareMate has max 50 concurrent Flask connections; with 2 slaves, max_slave_connections=40 is appropriate with headroom for spikes.
OS/DB: OS allows 500 concurrent app connections; with 4 slaves, max_slave_connections=200 per slave (800 total slave capacity vs 500 app demand).

⚠️ Common Pitfalls▾
⚠️ transaction_replay with non-idempotent writes

transaction_replay=true can replay a transaction against a new master after failover. If the transaction contained NOW(), UUID() or row counts, the replayed result may differ from the original attempt. Always set transaction_replay_max_size to a small value and ensure application logic handles duplicate-key errors gracefully.

⚠️ causal_reads adds latency for every read

causal_reads=true (strict mode) adds a GTID-based synchronization step before every slave read to guarantee read-your-writes consistency. This adds 1–5ms to every SELECT. Use causal_reads=fast which only tracks per-session GTIDs, or set it only on services where stale reads would cause business problems.

⚠️ Forgetting to reload config after changes

MaxScale does not automatically detect config file changes. After editing maxscale.cnf, run systemctl reload maxscale (or maxctrl reload config for hot-reload of supported parameters). A full restart drops all existing client connections — avoid during business hours without draining first.

🎤 Interview Questions▾
What does master_failure_mode=fail_instantly do and when would you choose it over error_on_write?

fail_instantly causes MaxScale to immediately return an error to the client the moment the master is detected as down — even for in-progress sessions. error_on_write allows existing sessions to continue reading until they attempt a write. Choose fail_instantly for high-frequency short transactions (ticketing, payment) where a clean fast retry is better than an uncertain partial session. Use error_on_write for user-facing sessions where interrupting a read-heavy workflow (e.g., a doctor reviewing patient history) creates worse UX than a delayed write error.

⚓ Anchor Phrase

"fail_instantly trades session continuity for predictability — critical for booking systems where you'd rather have a fast retry than a zombie session."

Explain causal_reads and when it's necessary in a Galera setup.

In Galera, all nodes are writable, but replication is synchronous only in terms of certification — a committed transaction on node db1 might not be visible on db2 for a few milliseconds. Without causal_reads, a session that does INSERT then immediately SELECT could read its own write from a slave that hasn't applied it yet. causal_reads=fast makes MaxScale track the session's latest GTID and wait for the chosen slave to reach that GTID before sending the SELECT. It's necessary in any flow where a user writes data and immediately reads it back — like a doctor saving a vital sign and then the UI refreshing to show the saved value.

⚓ Anchor Phrase

"causal_reads=fast solves read-your-writes in Galera without requiring all reads to hit the master — it's the feature that makes readwritesplit viable for user-facing apps."

How do you change a MaxScale parameter without dropping client connections?

Some parameters support dynamic reconfiguration via maxctrl alter service, maxctrl alter monitor, etc — these take effect immediately without restart. For example: maxctrl alter monitor galera-monitor monitor_interval=500ms. For parameters that require a restart, the safe procedure is: use maxctrl set server db1 maintenance to drain connections to MaxScale itself first, then use a load balancer or keepalived VIP to redirect traffic to a standby MaxScale instance while performing the restart.

⚓ Anchor Phrase

"maxctrl alter is the first tool — check if the parameter is dynamic. If not: drain via VIP failover, restart, validate, re-enable. Never cold-restart MaxScale with live traffic."

⚡ Why This Matters▾

The Galera Monitor is the component Nietsch is most likely to probe deeply — because it's the intelligence that keeps traffic away from a desynced or split-brained node. Understanding how galeramon uses wsrep_local_state, wsrep_cluster_size, and wsrep_local_index to make routing decisions is what separates a MaxScale user from a MaxScale administrator.

⚓ Interview Anchor

"galeramon selects the primary node using wsrep_local_index — the node with index 0 becomes Master. It only routes to nodes with wsrep_local_state=4 (Synced). If available_when_donor=false, a node performing SST (state=2) is immediately removed from the pool, preventing slow donor I/O from bleeding into application queries."

🧠 Core Concepts▾
wsrep_local_state — what galeramon reads
galeramon queries SHOW STATUS LIKE 'wsrep_local_state' on each server every monitor_interval. The returned integer determines routing eligibility: 1=Joining (excluded), 2=Donor/Desynced (excluded if available_when_donor=false), 3=Joined (excluded, still catching up), 4=Synced (eligible — routes traffic here). Only state 4 means the node is fully caught up and safe for reads and writes.

EHR: During a planned CareMate DB upgrade, galeramon detects db3 entering state 3 (Joined, applying IST) and keeps it excluded until state reaches 4 — patients never hit an outdated node.
OS/DB: A ticket scan arriving at a state-3 node could return stale seat availability — galeramon's strict state-4 requirement prevents this class of booking error.
wsrep_local_index — primary election
galeramon reads wsrep_local_index (the node's position in the membership list) and designates the Synced node with index 0 as Master (write target). This is a logical label — all Galera nodes accept writes, but MaxScale routes all writes through one to avoid certification conflicts. If node 0 leaves, the node with the next lowest index becomes Master.

EHR: On initial startup of CareMate cluster, db1 gets index 0 and becomes MaxScale Master. After db1 maintenance, db2 (index 1) is promoted automatically — no config change needed.
OS/DB: With 5-node cluster, if the current index-0 node is drained for AZ maintenance, galeramon re-elects based on index order deterministically — no tie-breaking needed.
available_when_donor — SST protection
When a Galera node performs SST (State Snapshot Transfer — full resync), it enters state 2 (Donor/Desynced) and may pause query processing for minutes or hours. Setting available_when_donor=false (recommended) removes the donor node from the routing pool immediately. If true, the donor stays in pool but queries may hang while it streams state to the joiner.

EHR: A new CareMate replica node joining via SST takes 20 minutes to sync 50GB. With available_when_donor=false, the donor node (db2) disappears from routing during this window — Flask only uses db1 and db3 for reads.
OS/DB: A 200GB booking database SST would freeze the donor for 30+ minutes. available_when_donor=false is non-negotiable on OS's cluster — donor exclusion is automatic and immediate.
disable_master_failback — controlling re-election
After the original master node (index 0) recovers from maintenance or failure, should MaxScale re-elect it as Master? disable_master_failback=true keeps the current elected master stable — avoids a second routing disruption. false (default) re-elects the lowest-index synced node, which may be the original master. For planned maintenance, true gives you manual control of failback timing.

EHR: disable_master_failback=true during CareMate DB patching — allows controlled failback after verifying the patched node is stable, rather than automatic re-election.
OS/DB: OS uses false in normal operation — automatic re-election to the primary node is preferred to avoid load imbalance from keeping a non-optimal primary long-term.

🎤 Interview Questions▾
What wsrep variables does galeramon query and what does it do with them?

galeramon queries wsrep_local_state (4=Synced is required for routing), wsrep_local_index (used to elect Master — lowest index among Synced nodes), wsrep_cluster_size (quorum check — if below majority, nodes are marked Non-Primary), wsrep_connected and wsrep_ready (fast health checks). Based on these, it assigns state flags used by the router for every routing decision.

⚓ Anchor Phrase

"wsrep_local_state=4 is the entry ticket to the routing pool. wsrep_local_index=0 is what makes one Synced node the Master. These two variables are what galeramon cares about most."

What happens in MaxScale when a Galera node goes into SST donor state?

When a node transitions to wsrep_local_state=2 (Donor/Desynced), galeramon detects this on the next polling cycle. If available_when_donor=false (recommended), MaxScale immediately removes that node from the routing pool — any existing sessions to it are closed, and new queries are routed only to the remaining Synced nodes. If the donor was the current Master (index 0), galeramon promotes the next lowest-index Synced node to Master.

⚓ Anchor Phrase

"available_when_donor=false is critical — an SST donor can pause for minutes. We never route production traffic to a donating node."

How does galeramon handle a split-brain scenario where two partitions each believe they are the primary?

In a genuine split-brain, both partitions will have nodes reporting wsrep_cluster_size less than the majority. galeramon checks the quorum: if a partition's wsrep_cluster_size is below ceil(total_nodes / 2 + 1), those nodes are in a Non-Primary Component and MaxScale marks them as non-eligible for routing. However, if one partition has majority quorum (e.g., 3 of 5 nodes), those nodes remain Synced and available. MaxScale will only route to the majority partition — protecting against writing to both sides.

⚓ Anchor Phrase

"galeramon doesn't resolve split-brain — Galera does. But it enforces that only quorum-bearing nodes receive traffic, preventing writes to isolated minority partitions."

Read/Write Split Router

The readwritesplit router is MaxScale's most powerful module. Understand routing rules, transaction boundaries, sticky sessions, and the edge cases that catch operators off guard.

Click any section to expand
⚡ Why This Matters▾

readwritesplit is what makes a Galera cluster useful as a scale-out read platform. Getting routing wrong — a SELECT inside a transaction going to a slave, or a temporary table created on one node then queried on another — causes application bugs that are hard to reproduce. Nietsch will test whether you understand transaction boundaries, sticky sessions, and the difference between autocommit SELECTs and transactional SELECTs.

⚓ Interview Anchor

"readwritesplit routes all writes and any statement inside an open transaction to the Master. Autocommit SELECTs go to slaves. The critical edge cases are: temporary tables (always sticky to the node that created them), stored procedures (always to master by default), and causal_reads ensuring read-your-writes after a write completes."

🧠 Core Concepts▾
Transaction routing — writes lock to master
Once a client sends BEGIN or START TRANSACTION, readwritesplit pins the entire transaction to the Master — including any SELECTs within it. This is the safest behavior: it prevents a SELECT from reading an older snapshot on a replica that may lag. After COMMIT or ROLLBACK, the session returns to normal split routing.

EHR: A CareMate Flask transaction that begins a consultation (BEGIN), writes a vitals row (INSERT), then reads it back (SELECT) — all three statements go to the master, ensuring the SELECT sees the uncommitted INSERT.
OS/DB: A ticket booking transaction (BEGIN → check seat → reserve seat → COMMIT) all goes to master — never a race condition with a seat being double-booked due to stale slave reads.
Sticky sessions — special connection states
Certain SQL constructs force readwritesplit to pin all subsequent statements to one backend node for the lifetime of the session: CREATE TEMPORARY TABLE (temp tables are connection-scoped), GET_LOCK() (advisory locks are connection-scoped), LAST_INSERT_ID() without causal_reads. Once sticky, the session stays on that node until disconnected.

EHR: A legacy CareMate stored procedure uses CREATE TEMPORARY TABLE tmp_vitals — that connection becomes sticky to the master for its lifetime. Important to track with maxctrl list sessions.
OS/DB: An ETL job using temporary tables causes all 20 ETL connections to become master-sticky — effectively routing all ETL load to master during the batch window. Switch ETL to CTEs instead.
use_sql_variables_in — WHERE variables route
use_sql_variables_in=master ensures that any query using SQL user variables (@user_var) is routed to master. all allows them on slaves too. User variables are session-scoped — routing a SET @var to master then a SELECT using @var to a slave would read an empty variable. Default is master and should stay that way unless you fully control variable usage.

EHR: Some legacy CareMate queries use SET @patient_id=42; SELECT * FROM vitals WHERE patient_id=@patient_id; — use_sql_variables_in=master routes both to master, maintaining variable state.
OS/DB: Never change to all — variable-using queries in the booking engine must always hit master to guarantee seat inventory consistency.
retry_failed_reads — slave query retry
retry_failed_reads=true automatically retries a failed read on another slave (or the master) if the original slave drops mid-query. This is safe for SELECTs since they are idempotent. For writes, there is no automatic retry — use transaction_replay instead. retry_failed_reads reduces transient slave failures from reaching the application.

EHR: If db3 crashes mid-SELECT during a patient history page load, MaxScale retries the query on db2 transparently — the Flask API never sees the error.
OS/DB: With retry_failed_reads=true and 5 nodes, a single node crash is invisible to the application for read queries — critical for maintaining perceived availability during rolling updates.

⚠️ Common Pitfalls▾
⚠️ Stored Procedures with SELECT-Only Logic

MaxScale routes all CALL statements to the master by default, even stored procedures that only do SELECTs. If you have a heavy reporting stored procedure, it will always hammer the master. Workaround: rewrite as a plain multi-statement query or use a dedicated readconnroute service for the reporting connection that only reads from slaves.

⚠️ Connection Pooling Libraries vs MaxScale

Many ORM connection pools (SQLAlchemy, HikariCP) reuse connections and may issue a SET SESSION on connection checkout, causing that session to become sticky to master. Always test your ORM's connection initialization queries through MaxScale with log_info=true to identify hidden stickiness sources before going to production.

⚠️ max_slave_connections Set Too Low

If max_slave_connections is set to a lower value than the application's connection pool size divided by the number of slaves, MaxScale will queue new slave connections — adding latency to reads under load. Monitor with maxctrl show server dbN | grep "Max Connections" and ensure you have headroom.

🎤 Interview Questions▾
A developer reports that after inserting a record they sometimes can't find it immediately. What MaxScale setting addresses this?

This is the classic read-after-write (read-your-writes) problem. After an INSERT on the master, the next SELECT is routed to a slave that may not have replicated the new row yet, returning 0 results. The solution is causal_reads=fast — MaxScale records the GTID of the completed write and ensures the next slave read waits until that slave has caught up to that GTID. The fast mode only tracks the GTID for the duration of the session, adding minimal overhead compared to true (strict) mode.

⚓ Anchor Phrase

"causal_reads=fast — it records the write GTID and holds the next slave read until that slave has applied it. Fixes read-after-write without forcing everything to master."

How would you diagnose why the master server is receiving 80% of queries despite having two slaves?

Start with maxctrl list servers to see connection counts per server. If master has disproportionately more connections, check maxctrl list sessions --json filtered for sessions routed to master — look at the query type. Common causes: application is using transactions or temporary tables (creating sticky sessions), stored procedure calls, or use of SET @variables. Enable log_info=true temporarily to see routing decisions in the log and identify which query patterns cause master routing.

⚓ Anchor Phrase

"maxctrl list sessions plus log_info=true is the diagnostic combo — connection counts tell me there's an imbalance, log_info tells me which queries are causing it."

⚠️ Common Pitfalls▾
⚠️ maxctrl alter vs config file — persistence

maxctrl alter changes are applied to the running MaxScale instance but NOT written back to maxscale.cnf. After a MaxScale restart, the original config file values will be used. Always update maxscale.cnf after any runtime change you want to persist, or use maxctrl sync config (MaxScale 22+) to write runtime state back to config.

⚠️ Forgetting to drain before stopping MariaDB

Stopping MariaDB directly on a backend node without first setting it to maintenance in MaxScale causes MaxScale to route queries to a suddenly-unreachable server, resulting in connection errors for clients. Always: maxctrl set server dbN maintenance → wait for drain → then stop MariaDB. The maintenance state persists across MaxScale restarts.

⚠️ maxctrl requires MaxScale to be running

If MaxScale itself crashes, maxctrl cannot connect to its REST API and all commands fail. Have a fallback: know how to connect applications directly to the MariaDB cluster (bypassing MaxScale) via the backup connection string, and know systemctl restart maxscale to recover. MaxScale itself is not clustered by default — a second standby MaxScale instance is the HA answer.

🎤 Interview Questions▾
What are the limits of MaxScale's automatic failover capability?

MaxScale handles routing failover automatically — it stops sending traffic to failed nodes, re-elects a new Master from Synced nodes, and resumes routing. What it cannot do: it cannot restart a crashed MariaDB process, cannot bootstrap a Galera cluster from a fully down state, cannot resolve a network partition causing split-brain, cannot repair data corruption, and cannot take over if MaxScale itself crashes (unless a standby MaxScale is in place with a VIP). MaxScale is a proxy — its failure domain is routing decisions, not DB recovery.

⚓ Anchor Phrase

"MaxScale handles the proxy layer — traffic routing. Everything below that is Galera's job or the DBA's job. Never confuse MaxScale failover with Galera recovery."

MaxScale itself is a single point of failure. How do you make it highly available?

Run two MaxScale instances (active/passive) with a shared VIP managed by keepalived or Pacemaker. Applications connect to the VIP. The keepalived health check script runs maxctrl list servers — if MaxScale is unhealthy, the VIP moves to the standby MaxScale within 3–5 seconds. Both MaxScale instances have identical configs (managed by Ansible), they both connect to all Galera backends independently. On VIP failover, all client connections reconnect to the standby MaxScale — application must implement reconnect logic.

⚓ Anchor Phrase

"Two MaxScale instances, one keepalived VIP, health check via maxctrl. The VIP is the HA mechanism — MaxScale itself is stateless, so standby is ready immediately."

A Galera node that was the MaxScale Master comes back after a crash. What happens?

When the crashed node restarts, it attempts to rejoin the Galera cluster. Depending on how long it was down and the cluster's gcache size, it will rejoin via IST (fast — applies missing transactions from gcache) or SST (slow — full resync). During this process, galeramon tracks its wsrep_local_state: it stays in Down or Joining state until it reaches Synced (state=4). Once Synced, galeramon re-evaluates wsrep_local_index. If disable_master_failback=false, and this node has the lowest index, it gets re-elected as Master. If disable_master_failback=true, the currently elected Master keeps that role and the recovered node joins as a Slave.

⚓ Anchor Phrase

"The recovered node re-enters the routing pool only when wsrep_local_state=4. Master election depends on disable_master_failback — I prefer true during recovery windows to control timing manually."

🎤 Advanced Interview Questions▾
MaxScale's readwritesplit is routing too many queries to master. The application team says they didn't change anything. What do you investigate?

Start with maxctrl list servers — if the number of Synced slaves has dropped (node Down or in Maintenance), MaxScale has fewer places to route reads and master gets more. If servers look fine, check maxctrl show service caremate-rw-service for per-server query counts. Then enable log_info=true temporarily and search logs for "routed to master" entries — the surrounding context shows which query type is triggering master routing. Common suspects after "nothing changed": a new ORM version that added a SET statement on connection init, a new feature using stored procedures, an increase in transaction usage, or a new query using user variables.

⚓ Anchor Phrase

"Server count first, then query pattern analysis with log_info=true. 'Nothing changed in the app' is never fully true — something changed, even if it's the ORM version."

Compare MaxScale readwritesplit vs ProxySQL for a Galera cluster. When would you choose one over the other?

MaxScale readwritesplit is Galera-aware via galeramon — it understands wsrep state natively and handles Galera-specific scenarios (donor exclusion, SST awareness, wsrep_local_index-based election) without custom rules. ProxySQL has no native Galera awareness — you need to write custom health check scripts or use Percona's ProxySQL Galera scripts to maintain the hostgroup membership. MaxScale has better out-of-the-box Galera support and is the natural choice when running the MariaDB Enterprise stack. ProxySQL offers more granular query routing rules and a richer admin interface, making it popular in Percona XtraDB Cluster environments. At OS GmbH with MariaDB Enterprise + Galera, MaxScale is the correct choice.

⚓ Anchor Phrase

"MaxScale is Galera-native — galeramon understands wsrep state out of the box. ProxySQL needs custom scripts to achieve the same result. On a MariaDB Enterprise stack, MaxScale is the right tool."

A DSGVO audit requires you to log all SELECT queries touching the patients table, with the username and timestamp. How do you implement this in MaxScale without touching the application?

Use a qlafilter (Query Log All) in the MaxScale service. Configure it with match=patients to capture only queries mentioning the patients table, log_type=unified for a single time-ordered log, and specify a filebase path on a filesystem with appropriate access controls and rotation. The log includes: timestamp, client IP, database user, query text. Add this filter to the service stanza's filters= list. Then configure logrotate to rotate and archive the audit logs to cold storage, retaining for the GDPR-mandated period (typically 3 years in Germany for operational logs).

⚓ Anchor Phrase

"qlafilter with match=patients, log_type=unified, locked-down filebase path. Zero application changes — the audit is transparent to the Flask app and compliant with Article 30 GDPR."

What happens to MaxScale's routing if the entire Galera cluster loses quorum? How does MaxScale behave and what do you do?

When Galera loses quorum (fewer than majority nodes remain in the Primary Component), the surviving nodes enter Non-Primary Component state — their wsrep_ready becomes OFF. galeramon detects this and marks those nodes as non-Synced. If all nodes lose quorum (e.g., a full cluster partition), MaxScale marks all servers as Down and refuses all new connections — preventing split-brain writes. Existing connections get errors. The DBA must: 1) Identify why quorum was lost (network partition? simultaneous node failures?), 2) If a majority partition exists, ensure those nodes re-elect primary component (may need SET GLOBAL wsrep_provider_options='pc.bootstrap=YES' on one node), 3) Rejoin remaining nodes. MaxScale resumes routing automatically once nodes show Synced again.

⚓ Anchor Phrase

"MaxScale goes dark when quorum is lost — it's protecting you from writing to a split cluster. My job is to restore quorum; MaxScale resumes automatically the moment a node shows wsrep_local_state=4."

Describe the difference between MaxScale's connection pooling and persistent connections. How does connection_keepalive affect backend connections?

MaxScale maintains a pool of backend connections to each server. When a client connects to MaxScale, MaxScale either reuses an idle backend connection or opens a new one. MaxScale does not do true connection multiplexing in readwritesplit mode — one client session maps to dedicated backend connections (one per server it's connected to). connection_keepalive controls how long MaxScale keeps an idle backend connection open before closing it. Setting it to 300s means even if a Flask connection pool has been idle for 5 minutes, the backend connection stays warm — avoiding reconnect latency on the next query. Without it, MariaDB's wait_timeout may close idle connections and MaxScale has to reconnect, adding 1–3ms per first query after idle.

⚓ Anchor Phrase

"connection_keepalive prevents the cold-start reconnect penalty — set it slightly less than MariaDB's wait_timeout so MaxScale closes connections before MariaDB does."

🚨 3am Scenario 1 — causal_reads Causing Query Timeouts▾
🚨 3AM — BOOKING QUERIES TIMING OUT AFTER DB NODE REPLACEMENT

Situation: 02:30. After replacing a Galera node earlier today (new hardware, rejoined via SST), booking queries are timing out at 60% rate. Errors: "Lost connection to MySQL server during query". The new node (db5) shows Slave, Synced in MaxScale. causal_reads=fast is enabled.

What's happening: The new node db5 just finished SST and is now Synced, but its GTID position may be slightly behind the other nodes — it's catching up on transactions it missed during SST. causal_reads=fast is routing reads to db5 but then waiting for it to apply GTIDs from recent writes, causing the wait to exceed the query timeout.

Immediate steps:

  1. Check db5's replication lag: mysql -h db5 -e "SHOW STATUS LIKE 'wsrep_local_recv_queue%';" — if wsrep_local_recv_queue_avg > 0, db5 has a write backlog.
  2. Temporarily set db5 to maintenance: maxctrl set server db5 maintenance — stops routing to it while it catches up.
  3. Monitor db5's recv queue until it drops to 0: watch -n 1 "mysql -h db5 -e \"SHOW STATUS LIKE 'wsrep_local_recv_queue';\""
  4. Once queue = 0, clear maintenance and verify query timeouts resolve: maxctrl clear server db5 maintenance

Prevention: After any SST, keep the new node in maintenance mode for 5–10 minutes after reaching state=4 to allow it to fully absorb the write queue before routing traffic to it. Add a health check script that validates wsrep_local_recv_queue_avg < 1 before clearing maintenance.

🚨 3am Scenario 2 — MaxScale Memory Leak / High CPU▾
🚨 3AM — MAXSCALE PROCESS AT 95% CPU, QUERY LATENCY SPIKING

Situation: 04:15. MaxScale is running but query latency has climbed from 2ms to 800ms over the last hour. MaxScale process shows 95% CPU in top. The Galera cluster itself is healthy.

Immediate steps:

  1. Check active session count: maxctrl list sessions | wc -l — compare to normal baseline. A session leak (sessions accumulating without closing) causes memory and CPU growth.
  2. Identify longest-running sessions: maxctrl list sessions --json | python3 -c "import json,sys; [print(s['id'], s['attributes'].get('connected_since','')) for s in json.load(sys.stdin)['data']]"
  3. Kill zombie sessions: maxctrl kill session <id> for sessions open for hours with no activity.
  4. If CPU still high after session cleanup, check MaxScale version for known CPU bugs: maxscale --version and check MariaDB JIRA.
  5. Emergency VIP failover to standby MaxScale: update keepalived priority to force VIP to maxscale-2, then restart maxscale-1 clean.

Root cause: A connection pooling library was leaking connections — each Flask worker opened a MaxScale connection but never closed it on crash. Over 4 hours, 4000 zombie sessions accumulated. Fix: Set net_write_timeout and net_read_timeout in maxscale.cnf to auto-close idle sessions, and fix the Flask connection pool leak.

🚨 3am Scenario 3 — Write Routing to Wrong Node▾
🚨 3AM — DUPLICATE BOOKING INSERTS DETECTED

Situation: 01:55. Duplicate booking records are appearing in the ticketing system. Each booking_id appears twice with slightly different timestamps. Data integrity alert fired. No application code changes today.

What to suspect: If MaxScale's transaction_replay is replaying transactions that DID commit on the original master before it failed, duplicate INSERTs can occur — the original committed but MaxScale didn't receive the acknowledgement before the TCP connection dropped.

Explain the complete query path for a Flask INSERT to the CareMate database, from TCP packet to committed row.

Flask opens a TCP connection to MaxScale VIP port 3306. MaxScale Listener accepts it and passes to caremate-rw-service.
